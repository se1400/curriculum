Why Objects?

The Problem — Strings Aren't Enough

Analogy: Imagine a filing cabinet. So far, you've been storing sticky notes with single words — "Tokyo", "Lisbon". But a real product catalog needs a full folder per item: name, price, category, image. Objects are those folders — they group related data together under one name.

You've built card grids from arrays of strings:

const cities = ['Tokyo', 'Lisbon', 'Chicago'];

But real web apps need richer data. A product needs a name, price, category, and stock status. You can't fit all of that into a single string.

The old way (don't do this): Separate arrays for each property — names[], prices[], categories[] — that must stay in sync by index. One mistake and everything is misaligned.

The solution: JavaScript objects — group all related data into one item.

JavaScript Objects

Creating an Object

An object is a collection of key-value pairs wrapped in curly braces.

JavaScript

const product = {
  name:     'Wireless Headphones',
  price:    79.99,
  category: 'Electronics',
  inStock:  true
};
PartWhat it isExample
Key (property name)The label — like a column headingname, price
ValueThe data — any type: string, number, boolean'Wireless Headphones', 79.99
Colon :Separates key from valuename: 'Headphones'
Comma ,Separates each key-value pairAfter each property
Think of it like a contact card: A contact has a name, phone number, email, and address — all grouped under one person. An object groups related data under one variable.
JavaScript Objects

Dot Notation — Reading & Writing

Analogy: Dot notation is like opening a folder and pulling out a specific document. product.name means "open the product folder, get the name document."

Reading properties

console.log(product.name);      // 'Wireless Headphones'
console.log(product.price);     // 79.99
console.log(product.inStock);   // true

Writing / updating properties

product.price  = 69.99;   // Update an existing property
product.rating = 4.5;     // Add a new property

JavaScript Objects

Bracket Notation — Dynamic Access

An alternative syntax that uses a string (or variable) inside square brackets:

const key = 'name';
console.log(product[key]);   // 'Wireless Headphones' — same as product.name

Dot Notation

product.name

Use this 99% of the time. Shorter, more readable.

Bracket Notation

product['name'] or product[key]

Use when the property name is stored in a variable or computed at runtime.

Arrays of Objects

Arrays of Objects — The Core Data Structure

Analogy: If an object is a contact card, then an array of objects is your entire contacts list. Each entry is one full card with all the details. This is the data shape behind every product listing, team page, and content gallery on the web.

JavaScript

const products = [
  { id: 1, name: 'Headphones',  price: 79.99, category: 'Electronics' },
  { id: 2, name: 'Sneakers',    price: 49.99, category: 'Footwear'    },
  { id: 3, name: 'Coffee Mug',  price: 12.99, category: 'Kitchen'     },
  { id: 4, name: 'Yoga Mat',    price: 34.99, category: 'Fitness'     }
];
  • Each object is one record — one product, one recipe, one team member
  • The array holds all the records
  • Every object has a unique id — you'll need it when users click cards
Arrays of Objects

Accessing Nested Data

Combine array index access with dot notation:

const first = products[0];        // The first product object
console.log(first.name);          // 'Headphones'

console.log(products[1].price);   // 49.99 — second product's price
Reading it aloud: products[1].price means "go to the array products, get item at index 1 (the second object), then read its price property." Two steps chained together.

Rendering Cards

Rendering Arrays of Objects as Cards

The renderCards() pattern from last lesson — now with object properties.

HTML

<div class="grid"></div>

CSS

.grid { display: grid; grid-template-columns: repeat(auto-fill, minmax(200px, 1fr)); gap: 1rem; }
.card { background: #16213e; padding: 1rem; border-radius: 12px; }
.badge { font-size: .65rem; background: rgba(108,92,231,.15); padding: .15rem .5rem; border-radius: 100px; }

JavaScript

function renderProducts(items) {
  container.innerHTML = items.map(product => `
    <div class="card" data-id="${product.id}">
      <span class="badge">${product.category}</span>
      <h3>${product.name}</h3>
      <p class="price">$${product.price.toFixed(2)}</p>
    </div>
  `).join('');
}
  • product.price.toFixed(2) — formats to exactly 2 decimal places
  • data-id="${product.id}" — stores the record's ID in the HTML for later lookup

Sorting Arrays

Sorting Arrays of Objects

Analogy: Sorting is like rearranging a deck of playing cards. The .sort() method needs you to tell it the rule: "put the lower price first" or "alphabetical by name." You provide this rule as a comparator function.

JavaScript

// Sort by price, low to high
const byPriceAsc = [...products].sort((a, b) => a.price - b.price);

// Sort by price, high to low
const byPriceDesc = [...products].sort((a, b) => b.price - a.price);

// Sort by name, alphabetically
const byName = [...products].sort((a, b) => a.name.localeCompare(b.name));
How the comparator works: The function compares two items at a time. Return a negative number → a comes first. Return positive → b comes first. Return zero → keep current order. For numbers, a.price - b.price produces the right sign automatically.
Sorting Arrays

Always Sort a Copy

Warning: .sort() modifies the original array! Always sort a copy using the spread operator: [...products].sort(). If you sort the original, there's no way to restore the default order without reloading the page.

Bad — mutates original

products.sort((a, b) =>
  a.price - b.price);
// products is now permanently
// reordered — can't undo!

Good — sorts a copy

const sorted = [...products].sort(
  (a, b) => a.price - b.price
);
// products unchanged
// sorted is the new order

The data-id Pattern

Connecting Cards to Data

When a user clicks a card, how does your JS know which item they clicked?

Analogy: Imagine name tags at a conference. Each person wears a badge with a unique number. When you meet someone, you read their badge number and look them up in the attendee list. data-id is that badge — it's stamped in the HTML, and your JS reads it back to find the matching object.

HTML that gets generated

<div class="card" data-id="3">
  <h3>Coffee Mug</h3>
  <p>$12.99</p>
</div>

JavaScript — reading it back

const id = Number(card.dataset.id);       // Convert string → number
const product = products.find(p => p.id === id); // Look up the object
  • dataset.id always returns a string — wrap in Number() before comparing with ===
  • .find() returns the first matching object, or undefined if none match
The data-id Pattern

Event Delegation

One listener on the parent — catches clicks on all cards, even ones added later.

HTML

<div class="grid" id="grid">
  <!-- cards are generated by JS — not in the HTML at load time -->
</div>

JavaScript

container.addEventListener('click', function(event) {
  const card = event.target.closest('.card');
  if (!card) return;  // Click was not on a card

  const id      = Number(card.dataset.id);
  const product = products.find(p => p.id === id);

  if (product) {
    showDetail(product);
  }
});
Why event delegation? Cards generated by JS aren't in the HTML when the page loads — you can't attach listeners to elements that don't exist yet. A listener on the static parent container catches all clicks because events bubble up through the DOM.

Empty States

Handling Empty States

Why this matters: When a filter or search produces zero results, showing a blank space makes users think the page is broken. A good empty state tells them "nothing matched" — it's part of strong UX design.

HTML

<div class="grid"></div>

JavaScript

function renderProducts(items) {
  if (items.length === 0) {
    container.innerHTML = '<p class="empty-state">No items match your search.</p>';
    return;
  }

  container.innerHTML = items.map(product => `
    <div class="card">...</div>
  `).join('');
}

Rule: A render function that returns nothing when its input is empty is not finished. Always add a check and a user-friendly message.
Building Objects

Building Objects from Form Input

Data doesn't have to be hardcoded — you can build objects from what the user types.

Analogy: Think of a registration form. Each field the user fills in becomes a property on the new object. Clicking "submit" creates the object, pushes it into the array, and re-renders the grid — the new card appears alongside all the others.

HTML

<form id="add-form">
  <input type="text" id="name" placeholder="Product name">
  <input type="number" id="price" placeholder="29.99">
  <button type="submit">Add</button>
</form>

JavaScript

form.addEventListener('submit', function(event) {
  event.preventDefault();

  const newProduct = {
    id:       products.length + 1,
    name:     document.querySelector('#name').value,
    price:    Number(document.querySelector('#price').value),
    category: document.querySelector('#category').value
  };

  products.push(newProduct);
  renderProducts(products);
});

Best Practices

Best Practices

Give every object a unique id
Even if you don't need it immediately, you will need it the moment users interact with individual cards. Adding it from the start is cheap; retrofitting later is not.
Sort copies, not originals
[...products].sort() every time. The original array is your single source of truth — protect it. There's no undo for .sort() on the original.
Convert numeric IDs immediately
dataset.id returns a string. Wrap it in Number() before comparing with === to numeric IDs, otherwise "3" === 3 is false.
Always handle the empty state
A render function that returns nothing when input is empty is not finished. Add a check and a user-friendly message.
Use data that feels real
Placeholder names like "Item 1, Item 2" make it hard to spot layout bugs. Use realistic names, prices, and categories from the start.
