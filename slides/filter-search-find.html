Filter, Search, Find

Array iteration methods that turn a static gallery into an interactive, searchable, filterable interface.

  • forEach() — run a function for each item
  • map() — transform each item into something new
  • filter() — keep only items that pass a test
  • find() — get the first item that matches a condition
  • Build category filter buttons that update in real time
  • Build a live search input that filters as you type
  • Chain filter() and map() together
  • Event delegation for dynamically generated content
Why This Matters

From Static Grid → Real Interface

You already have a gallery that renders, sorts, and responds to clicks. These four methods add filtering and search — the two features that separate a real app from a demo.

The pattern behind the magic: Every e-commerce site, search page, and content library you use relies on the same four tools — forEach, map, filter, find. Once you understand them, you can read and build almost any data-driven interface on the web.

What the methods have in common

  • They are called iteration methods — they go through every item for you
  • Each one takes a function as its argument
  • That function runs once per item in the array
  • The result is code that reads almost like plain English
Old way (manual loops): for (let i = 0; i < arr.length; i++) { ... } — you manage the index, the length check, and the increment.

New way (iteration methods): arr.filter(p => p.price < 50) — you say what you want, not how to get it.
forEach()

forEach() — Do Something With Each Item

Runs a function once for each item. Returns nothing. Use it when you need a side effect, not a new array.

Analogy: Imagine a teacher handing out the same worksheet to each student in a row. The teacher doesn't collect anything back — they just do the thing for each student. That's forEach().

JavaScript

const products = ['Headphones', 'Shoes', 'Mug'];

products.forEach(product => console.log(product));
// Logs: 'Headphones'
// Logs: 'Shoes'
// Logs: 'Mug'

Reads as: "For each product in products, log the product."

A common real-world use — resetting button states

HTML

<div class="filters">
  <button class="filter-btn active">All</button>
  <button class="filter-btn">Electronics</button>
</div>

CSS

.filter-btn.active { background: #a78bfa; color: white; }

JavaScript

const filterBtns = document.querySelectorAll('.filter-btn');
// Strip .active off every button before marking the clicked one
filterBtns.forEach(btn => btn.classList.remove('active'));
When to reach for forEach(): You want to do something for each item (update the DOM, toggle a class, log a value). When you want to produce a new array, use map() instead.

map()

map() — Transform Each Item

Runs every item through a function and collects the results into a new array of the same length. The original array is unchanged.

Analogy: map() is an assembly line. Items enter one end, a machine (your function) transforms each one, and transformed items come out the other end. Same number of items, new shape.
[79.99, 49.99, 12.99]
→
p => '$' + p.toFixed(2)
→
['$79.99', '$49.99', '$12.99']

JavaScript

const prices = [79.99, 49.99, 12.99];

const formatted = prices.map(price => '$' + price.toFixed(2));
// ['$79.99', '$49.99', '$12.99']

console.log(prices);     // [79.99, 49.99, 12.99]  ← original unchanged

filter()

filter() — Keep Only What Passes

Runs a test function on each item. Items that return true are kept. Items that return false are removed. Result: a new, shorter array.

Analogy: Think of a bouncer at a club. Each person walks up, the bouncer checks the ID, and either lets them in or sends them away. filter() is your bouncer — it lets only the matching items into the new array.
6 products
→
p => p.price < 50
→
2 products (the cheap ones)

JavaScript

const products = [
  { name: 'Headphones', price: 79.99, category: 'Electronics' },
  { name: 'Shoes',      price: 49.99, category: 'Footwear'    },
  { name: 'Mug',        price: 12.99, category: 'Kitchen'     }
];

// Keep only Electronics
const electronics = products.filter(p => p.category === 'Electronics');

// Keep only items under $50
const affordable = products.filter(p => p.price < 50);

Reads as: "For each product p, keep it if p.category equals 'Electronics'."

find()

find() — Get the First Match

Works like filter(), but stops at the first match and returns the item itself — not an array.

Analogy: You're looking for your friend Sarah in a crowded room. filter() would point out every Sarah. find() just taps the shoulder of the first Sarah it sees and hands her to you. If there is no Sarah, it returns undefined.
filter()find()
ReturnsAn array (maybe empty)The item itself, or undefined
Stops atEnd of the array (checks every item)The first match
Use whenYou want all matchesYou want one specific item (usually by id)

Quick Reference

forEach · map · filter · find

Method What it does Returns Use it when…
forEach() Runs a function on each item Nothing (undefined) You want a side effect (DOM change, log)
map() Transforms each item New array, same length You want to convert data (usually to HTML)
filter() Keeps items that pass a test New array, shorter or empty You want a subset of the data
find() Stops at the first match One item, or undefined You want a specific item (usually by id)
Reading trick: All four take a function. The function runs once per item. p (or whatever name you pick) is the current item being inspected. The difference between them is what they do with the result.
p is not a keyword — it's just a name you chose. You could write products.filter(item => ...) or products.filter(thing => ...) or products.filter(🐱 => ...) (don't, but you could). Most devs pick a short letter that matches the array — p for products, u for users, e for events. It's just a placeholder for "the current item."
Arrow-function refresher: p => p.price < 50 is the same as function(p) { return p.price < 50; }. The arrow is shorter — that's it. No magic. The thing before => is the input, the thing after is the output.

Category Filter Buttons

Wiring Buttons to filter()

Buttons change a variable. The variable drives the filter. The filter rebuilds the grid.

Analogy: Think of a TV remote with channel buttons. Each press flips the TV to a different channel — but the TV itself never moves. Here, the buttons change which slice of products is "on-screen." The full product list never changes; you're just tuning in to a subset.

HTML

<div class="filters">
  <button class="filter-btn active" data-category="all">All</button>
  <button class="filter-btn" data-category="Electronics">Electronics</button>
  <button class="filter-btn" data-category="Kitchen">Kitchen</button>
</div>
<div class="grid"></div>

CSS

.filter-btn        { padding: .5rem 1rem; border: 1px solid #ccc; border-radius: 999px; }
.filter-btn.active { background: #6c5ce7; color: white; border-color: #6c5ce7; }

JavaScript

let activeCategory = 'all';

filterBtns.forEach(btn => {
  btn.addEventListener('click', function() {
    filterBtns.forEach(b => b.classList.remove('active'));
    btn.classList.add('active');

    activeCategory = btn.dataset.category;
    filterAndRender();
  });
});

function filterAndRender() {
  const results = activeCategory === 'all'
    ? products
    : products.filter(p => p.category === activeCategory);
  renderProducts(results);
}
The key idea: activeCategory lives outside the event handler. The handler updates it; filterAndRender() reads it. One source of truth, called from anywhere.

Live Search

Filter as You Type

The input event fires on every keystroke. That's exactly what live search needs.

Analogy: Imagine a librarian standing next to you. Every time you say another letter of the book title, she pulls matching books off the shelf and re-stacks them in front of you. She doesn't wait for you to finish — she reacts on every syllable. That's the input event.

HTML

<input type="search" id="search" placeholder="Search products…">
<div class="grid" id="grid"></div>

CSS

#search {
  width: 100%;
  padding: .6rem 1rem;
  border: 1px solid #ccc;
  border-radius: 8px;
  font-size: 1rem;
}
#search:focus { outline: none; border-color: #a78bfa; }

JavaScript

let searchQuery = '';

searchInput.addEventListener('input', function() {
  searchQuery = searchInput.value.toLowerCase();
  filterAndRender();
});

function filterAndRender() {
  const results = products.filter(p => {
    const matchesCategory = activeCategory === 'all' || p.category === activeCategory;
    const matchesSearch   = p.name.toLowerCase().includes(searchQuery);
    return matchesCategory && matchesSearch;
  });
  renderProducts(results);
}

The three moves that make search feel right

  • .toLowerCase() on both sides — so "shoes" matches "Running Shoes"
  • .includes() — matches anywhere in the string, not just at the start
  • Combined with category filter using && — both must pass

Chaining

Chaining filter() and map()

Because filter() returns an array, you can call .map() directly on its result. Each step feeds into the next.

Analogy: Think of a sandwich shop. Ingredients go on a conveyor belt. Station 1 (filter) removes anything you're allergic to. Station 2 (map) adds mayo to what's left. Station 3 (join) wraps the whole thing in paper. Each station does one job and passes the tray to the next. That's chaining.

JavaScript — the full pipeline

container.innerHTML = products
  .filter(p => p.category === activeCategory)
  .map(p => `<div class="card">${p.name}</div>`)
  .join('');

Reads as: "Take the products, keep the ones in the active category, turn each into an HTML string, glue them together, set as innerHTML."

Keep chains short: Two methods is a sweet spot. Three or more methods makes bugs hard to trace — when something is wrong, you can't tell which step broke. If your chain is getting long, store intermediate results in a variable.

When to break a chain apart

Hard to debug

data
  .filter(...)
  .map(...)
  .sort(...)
  .slice(0, 5)
  .join(...);

Easier to read

const matching = data.filter(...);
const sorted   = matching.sort(...);
const top5     = sorted.slice(0, 5);
top5.map(...).join(...);
Event Delegation

Why Event Delegation?

When cards are generated by JavaScript, they don't exist at page load. You can't attach listeners to elements that aren't there yet.

Analogy: Think of a mailroom. Instead of putting a mailbox at every desk in the building, you put one mailroom at the entrance. All mail flows through that single point. Event delegation is the same idea — one listener on a stable parent, handling events for any number of children.

The problem this solves

  • Cards are created and destroyed as filters change
  • Old cards' listeners are garbage-collected every re-render
  • Adding listeners to new cards every render is wasteful — and easy to forget
  • One listener on the parent survives every re-render

How it works — two DOM facts

  1. Events bubble up. A click on <h3> inside a card also fires on the card, and on the grid, and on the body.
  2. closest() walks up. From the clicked element, it climbs the DOM tree looking for the selector you give it. Returns the nearest match, or null.

Event Delegation

The Code Pattern

HTML — the container is stable; cards come and go

<div class="grid" id="grid">
  <!-- cards are rendered by JS — none at load time -->
</div>

CSS — pointer cursor cues that cards are clickable

.grid        { display: grid; gap: 1rem; }
.card        { cursor: pointer; transition: .2s; }
.card:hover  { transform: translateY(-2px); border-color: #a78bfa; }
.card.selected { box-shadow: 0 0 0 2px #a78bfa; }

JavaScript

// ONE listener, on the stable parent — not on each card
container.addEventListener('click', function(event) {
  const card = event.target.closest('.card');
  if (!card) return;  // clicked something that isn't inside a card

  const id      = Number(card.dataset.id);
  const product = products.find(p => p.id === id);
  if (product) openModal(product);
});
Why event.target.closest('.card')? The user might click on the card's <h3>, badge, price, or anything nested inside. event.target is the specific element clicked; closest('.card') climbs up to find the card that wraps it — no matter what child was clicked.
The magic: This listener keeps working after every renderProducts() call. Filter, search, sort, reset — doesn't matter. The grid container never gets replaced. Its listener never has to be reattached.

Best Practices

Best Practices

Keep filter logic and render logic separate
A filterAndRender() function that (1) computes the filtered array and (2) calls renderProducts() is easier to debug than mixing both inside one event handler.
Store filter state in variables outside handlers
let activeCategory = 'all' and let searchQuery = '' live at the top of your script. Every handler can read and update them. No passing arguments around.
Always .toLowerCase() search comparisons
A user typing "shoes" expects to match "Running Shoes". Lowercase both sides before comparing, or your search will feel broken.
Use event delegation for dynamic content
Any time cards are generated by JavaScript, attach the click listener to the stable parent, not the cards. One listener, works forever.
Always handle the empty state
When filter() returns an empty array, show "No results match your search" — not a blank space. Users assume blank means broken.
Cap chains at two methods
.filter().map() is great. .filter().sort().slice().map() is hard to debug. Break long chains into named intermediate variables.
