Background

The Evolution of Web Animation

1990s — Animated GIFs & <marquee>

<blink>, <marquee>, and looping GIF images were the only "animation." Everything else either snapped instantly or didn't move at all.

Late 1990s — DHTML & setInterval

Developers used JavaScript to nudge element.style.left every 16ms via setInterval. Brittle and browser-inconsistent — the UI froze whenever the tab was under load.

2000–2010 — Flash Dominates

Adobe Flash brought true frame-by-frame motion to the web. Then Apple refused Flash on iPhone (2007). HTML5 + CSS finished its replacement by 2015.

2006–2012 — jQuery.animate() & requestAnimationFrame

$.animate() made JS animation approachable — but it ran on the main thread. Under load: dropped frames and visible jank. requestAnimationFrame (2012) improved timing but didn't fix the thread problem.

2004–2012 — CSS Transitions Emerge ✅

Apple shipped -webkit-transition in Safari (~2004). W3C Working Draft published 2009. All major browsers shipped by 2012. Animation work moved to the GPU — completely off the main thread.

/* You may see this in OLD code — do not copy */
-webkit-transform: rotate(45deg);
   -moz-transform: rotate(45deg);
    -ms-transform: rotate(45deg);
        transform: rotate(45deg);  /* ← just this */
Vendor prefixes: Tutorials from 2009–2013 show -webkit-, -moz-, and -ms- variants. All modern browsers (since ~2015) only need the unprefixed version. Never copy the prefixed forms into new projects.

CSS Transitions

What is a CSS Transition?

A transition watches a CSS property for changes. When the property changes — hover, class toggle, focus — the browser animates smoothly from the old value to the new one instead of snapping instantly.

/* Without transition — instant */
button {
  background-color: #4f6df5;
}
button:hover {
  background-color: #e74c3c;  /* SNAP */
}

/* With transition — animated */
button {
  background-color: #4f6df5;
  transition: background-color 0.3s ease;
}
button:hover {
  background-color: #e74c3c;  /* SMOOTH */
}

The Four Transition Sub-Properties

button {
  background-color: #4f6df5;

  /* WHAT to watch */
  transition-property: background-color, transform;

  /* HOW LONG */
  transition-duration: 0.2s;

  /* THE PACE */
  transition-timing-function: ease-out;

  /* WAIT BEFORE STARTING */
  transition-delay: 0s;
}

button:hover {
  background-color: #3a56d4;
  transform: translateY(-2px);
}
PropertyWhat it controls
transition-propertyWhich CSS property to animate (all, opacity, transform, etc.)
transition-durationHow long the animation takes (0.3s = 300ms)
transition-timing-functionThe pace curve (ease, linear, cubic-bezier…)
transition-delayWait time before animation starts — useful for stagger effects

CSS Transitions

The Stagger Effect — transition-delay in Action

One of the most polished patterns in UI design: multiple elements sharing the same transition, but with incrementally increasing transition-delay values so they animate in sequence rather than all at once.

/* Each nav item gets a longer delay */
.nav-item:nth-child(1) { transition-delay: 0ms; }
.nav-item:nth-child(2) { transition-delay: 60ms; }
.nav-item:nth-child(3) { transition-delay: 120ms; }
.nav-item:nth-child(4) { transition-delay: 180ms; }
.nav-item:nth-child(5) { transition-delay: 240ms; }

/* All share the same transition */
.nav-item {
  opacity: 0;
  transform: translateX(-20px);
  transition: opacity 0.35s ease,
              transform 0.35s ease;
}

/* Triggered by a class toggle */
.nav.open .nav-item {
  opacity: 1;
  transform: translateX(0);
}
Generate delays with CSS custom properties or Sass loops in real projects. The 50–80ms sweet spot per item feels natural — shorter feels rushed, longer feels sluggish.

CSS Transitions

The Shorthand & The One Rule You Must Know

The Shorthand Syntax

/* Single property */
button {
  transition: background-color 0.2s ease-out;
}

/* Multiple — comma-separated */
button {
  transition: background-color 0.2s ease-out,
              transform 0.15s ease-out;
}

/* All (use with caution) */
button {
  transition: all 0.2s ease-out;
}
Order: property  duration  timing  delay
Only duration is required. The rest default to all, ease, 0s.
Avoid transition: all in production — it watches every CSS property, including ones you don't intend to animate. Always name them explicitly.

The One Rule: Default State, Not :hover

Put transition on the element's default state, never on :hover.
When the transition lives on the default state, the browser animates both entering and leaving the hover. Put it on :hover and the return snap instantly — because leaving hover removes the transition rule itself. 
/* ✅ Correct — animates IN and OUT */
button {
  transition: background-color 0.3s ease;
}
button:hover { background-color: #2ecc71; }

/* ❌ Wrong — animates IN, snaps back */
button:hover {
  background-color: #2ecc71;
  transition: background-color 0.3s ease;
}

Custom Curves: cubic-bezier(), steps(), linear()

/* All keywords are cubic-bezier shortcuts */
/* ease = */ cubic-bezier(0.25, 0.1, 0.25, 1.0)

transition: background-position 0.8s steps(8);

.spring {
  transition: transform 0.6s linear(
    0, 0.5, 1.05, 0.95, 1.02, 0.98, 1
  );
}

Transitions

What Can (and Cannot) Be Transitioned

✅ Can transition (numeric/color values)

  • opacity — 0 to 1 (GPU-accelerated)
  • transform — all functions (GPU-accelerated)
  • background-color, color, border-color
  • box-shadow, text-shadow
  • width, height, padding, margin
  • font-size, border-radius, max-height

❌ Cannot smoothly transition (discrete values)

  • display — none / block / flex (no interpolation)
  • font-family — no middle value between fonts
  • position — static / relative / absolute
  • visibility — partially supported
Avoid transitioning layout properties (width, height, margin, padding) — they trigger browser layout recalculation on every frame. Prefer transform and opacity.

Modern CSS

Animating display: none — Finally Solved

You toggle a dropdown or modal with display: none. You add a transition. Nothing animates — because display has no "halfway" value between none and block. Two features added in 2023 fix this.

/* Step 1: define transition on the hidden state
   — include display with allow-discrete */
.panel {
  display: none;
  opacity: 0;
  transform: translateY(12px);
  transition: opacity 0.3s ease,
              transform 0.3s ease,
              display 0.3s allow-discrete;
}

/* Step 2: open state — what it animates TO */
.panel.open {
  display: block;
  opacity: 1;
  transform: translateY(0);
}

/* Step 3: @starting-style — what it animates
   FROM when first appearing. Without this, the
   browser has no "before" and entry snaps. */
@starting-style {
  .panel.open {
    opacity: 0;
    transform: translateY(12px);
  }
}
How it works — plain language:
  • allow-discrete opts display into the transition. On enter: the element becomes visible immediately (display: block) and then the opacity/transform animate in. On exit: opacity/transform animate out first, and only then does display: none apply — so the element doesn't vanish mid-animation.
  • @starting-style is the "before" snapshot for elements entering from display: none. It gives the browser a starting point so it can interpolate.
The old workaround: developers combined visibility: hidden + pointer-events: none + opacity: 0, or used JS libraries. This is the native CSS solution.
Browser support: Chrome 117+, Firefox 129+, Safari 17.5+ (all current as of 2025).

Performance

Why transform & opacity Are Faster

Not all CSS properties are equal when animated. Two properties get a special fast path through the browser — everything else takes the slow road.

The fast lane: transform and opacity
The browser treats elements with transform or opacity transitions like a separate pre-painted image. During the animation, it only moves or fades that image — it doesn't touch the rest of the page at all. The GPU handles it independently.
The slow road: everything else
Animating left, width, height, margin, or padding forces the browser to recalculate the size and position of every element on the page on every frame. At 60fps, that's 60 full-page recalculations per second — which causes dropped frames and jank, especially on mobile.
Practical rule: To move something, use transform: translateX() — not left. To fade something, use opacity — not visibility. The visual result is usually identical. The browser work is not.

Performance

How the Browser Draws a Page

When a CSS property changes during a transition, the browser may need to redo some or all of its drawing work. There are three stages — and the earlier in the chain a property change starts, the more expensive it is.

① Layout

"Where does everything go?" The browser calculates the size and position of every element. A change to width, height, padding, margin, top, left, or font-size starts here.

🔴 Most expensive — forces Paint and Composite too.

→

② Paint

"What does it look like?" Fill in colors, borders, shadows, text. A change to background-color, color, box-shadow, or border-color starts here — it already knows where things are.

🟡 Medium cost — skips Layout, but must redraw pixels.

→

③ Composite ✅

"Move the layers." The browser keeps elements on pre-drawn layers and simply repositions them on screen. transform and opacity only trigger this step — the page is never re-calculated or redrawn.

🟢 Cheapest — handled by the GPU, independent of the page.

Animating left starts at Layout — the browser recalculates every element's position on every single frame. At 60fps on a complex page, this causes visible dropped frames.
Animating transform: translateX() jumps straight to Composite — the browser moves a pre-drawn layer. The page itself is never touched. Same visual result; far less work.
The takeaway: For smooth transitions at 60fps, keep your animated properties in the Composite column — transform and opacity. Avoid animating layout properties like width, height, left, or top.
Performance & Accessibility

Two CSS Properties Every Animated Page Needs

1. will-change — warn the browser ahead of time

Even though transform is GPU-accelerated, the browser still has to set up the GPU layer when the transition first starts. On slow or busy hardware, that setup takes a moment and causes a stutter on the very first hover.

will-change is like calling ahead to reserve a table: it tells the browser "prepare the GPU layer now — before any animation happens."

/* Browser prepares the GPU layer immediately,
   so the very first hover is instant */
.card {
  will-change: transform;
}
.card:hover {
  transform: scale(1.03);
  transition: transform 0.2s ease-out;
}
Use sparingly. Each will-change declaration reserves GPU memory. Only add it to elements you know will animate. Never apply it to every element — that wastes memory and can actually hurt performance.
Pro tip: Add will-change via JavaScript on mouseenter, and remove it on transitionend — so it only reserves memory exactly when needed.

2. prefers-reduced-motion — include every user

Some people have conditions where screen motion causes real physical discomfort — nausea, dizziness, or headaches. Every major OS (macOS, iOS, Windows, Android) has a "Reduce Motion" accessibility setting. When it's on, this media query matches:

/* Default: animated */
.button {
  transition: transform 0.2s ease-out,
              box-shadow 0.2s ease-out;
}
.button:hover { transform: translateY(-3px); }

/* User turned on "Reduce Motion" in their OS */
@media (prefers-reduced-motion: reduce) {
  .button {
    transition: none; /* make it instant instead */
  }
}
"Reduce motion" doesn't mean "no motion."
✅ Fine: color changes, simple fades, state feedback
❌ Disable: parallax, large fly-in animations, looping spinners, anything that moves large elements across the screen
Test it: Chrome DevTools → More tools → Rendering → Emulate CSS media → prefers-reduced-motion: reduce
2D Transforms

2D Transforms: translate() & rotate()

transform visually repositions/reshapes an element without affecting document flow — other elements don't shift to accommodate it.

translate() — Move

/* Move 50px right, 20px down */
transform: translate(50px, 20px);

/* Single axis shortcuts */
transform: translateX(50px);
transform: translateY(-20px);  /* UP */

/* % = relative to element's own size */
transform: translateX(100%);  /* = element width */

rotate() — Spin

transform: rotate(45deg);     /* clockwise */
transform: rotate(-90deg);    /* counter-clockwise */
transform: rotate(0.25turn);  /* quarter turn */
transform: rotate(1turn);     /* full spin */
A positive Y value moves DOWN. Positive X moves RIGHT. Degrees: positive = clockwise.

2D Transforms

scale(), skew() & Combining Transforms

scale() — Resize from center

transform: scale(1.2);        /* 20% larger */
transform: scale(1.5, 0.8);   /* wider & shorter */
transform: scaleX(2);         /* double width only */
transform: scale(0);          /* shrink to nothing */

skew() — Shear/Slant

transform: skewX(15deg);      /* slant left-to-right */
transform: skewY(5deg);       /* slant top-to-bottom */
transform: skew(15deg, 5deg); /* both axes */

Combining — order matters!

/* Applied right-to-left */
/* translate first, then rotate */
transform: translateX(100px) rotate(45deg);

/* rotate first, then translate along rotated axis */
transform: rotate(45deg) translateX(100px);

/* Practical hover effect */
.card:hover {
  transform: translateY(-8px) scale(1.02) rotate(-1deg);
}
Transforms are applied right to left. rotate(45deg) translateX(100px) translates along the rotated axis — very different from the reverse order.

  translate: 0 0;
  scale: 1;
  rotate: 0deg;

  transition: translate 0.2s ease-out,

}

3D Transforms: Perspective & Rotation

perspective — set on the PARENT

/* Small value = dramatic depth effect */
.scene { perspective: 200px; }

/* Typical UI range: 600–1000px */
.scene { perspective: 800px; }

/* Large value = subtle perspective */
.scene { perspective: 1200px; }

.card {
  transform: rotateY(30deg); /* now has depth */
}
perspective goes on the parent element (the "scene"), not on the element being transformed. Think of it as the viewer's distance from the screen — smaller = more dramatic.

3D Transforms

transform-style: preserve-3d & backface-visibility

/* The scene provides perspective */
.scene {
  perspective: 800px;
  width: 300px;
  height: 200px;
}

/* The card flips — MUST have preserve-3d
   so children exist in 3D space */
.card {
  width: 100%;
  height: 100%;
  position: relative;
  transform-style: preserve-3d;
  transition: transform 0.6s ease-in-out;
}

.scene:hover .card {
  transform: rotateY(180deg);
}

/* Both faces fill the card absolutely */
.card__face {
  position: absolute;
  width: 100%;
  height: 100%;
  backface-visibility: hidden; /* hide when facing away */
}

/* Back face starts pre-rotated */
.card__face--back {
  transform: rotateY(180deg);
}

How it works:

  • transform-style: preserve-3d on the card container — keeps children in 3D space. Without it, children are "flattened" to the parent's 2D plane.
  • backface-visibility: hidden on both faces — makes each face invisible when its back is toward the viewer.
  • Back face starts at rotateY(180deg) — so it faces away from the viewer by default and only becomes visible after a flip.
Common bug: overflow: hidden, clip-path, filter, or mix-blend-mode on an ancestor element will break preserve-3d and flatten the 3D context.

What's Next

Transitions Need a Trigger — Animations Don't

CSS transitions are reactive: they only run when a property changes (hover, class toggle, focus). For motion that plays automatically — on load, in a loop, or with multiple steps — you need CSS Animations with @keyframes.

Transition — needs a trigger

/* This does NOTHING until you hover */
.button {
  background: #4f6df5;
  transition: background 0.3s ease;
}
.button:hover {
  background: #e74c3c;  /* ← the trigger */
}
Limitation: You cannot chain multiple steps, loop automatically, or play on page load without JavaScript toggling a class.

@keyframes Animation — self-contained

/* Define the motion sequence */
@keyframes pulse {
  0%   { transform: scale(1); }
  50%  { transform: scale(1.1); }
  100% { transform: scale(1); }
}

/* Attach it — plays immediately, loops forever */
.badge {
  animation: pulse 1.2s ease-in-out infinite;
}
Key differences: Animations have multiple steps (keyframes), run on their own, can loop, can alternate direction, and can pause. Use animation-fill-mode to control the element's state before/after the animation plays.

These elements are running @keyframes animations right now — no JavaScript, no hover needed:

pulse (scale)
spin (rotate)
bounce (translateY)
progress (width)
Rule of thumb: Use transitions for state changes driven by user interaction (hover, click, focus). Use @keyframes animations for anything autonomous — loading states, attention pulses, page-load reveals, looping decorative motion.
Summary

Summary

CSS Transitions 4 sub-properties: property, duration, timing-function, delay. Put the transition on the default state, not :hover.
Timing Functions ease-out for most hover effects. cubic-bezier() for custom curves. linear() for spring physics (2023+).
GPU Performance Only transform and opacity are compositor-only. All other properties trigger layout/paint. Use sparingly — avoid transition: all.
2D Transforms translate, rotate, scale, skew never affect document flow. Order matters when combining. Modern: use individual translate/rotate/scale properties.
3D Transforms perspective on the parent. transform-style: preserve-3d on the container. backface-visibility: hidden on both faces. Back face pre-rotated 180°.
Accessibility Always add @media (prefers-reduced-motion: reduce). Functional transitions (state change) = OK. Decorative parallax/fly-ins = disable.
display: none Use @starting-style + transition-behavior: allow-discrete for enter/exit animations. Chrome 117+, Firefox 129+, Safari 17.5+.
Common Patterns Button lift (translateY + box-shadow), image zoom (scale inside overflow:hidden), fade+slide in (opacity + translateY).
Q & A

Common Questions

What's the difference between left: 100px and transform: translateX(100px)? They look the same.
They look identical, but the browser work is very different. Animating left requires layout recalculation on every frame. transform: translateX() is handled by the GPU compositor — it just moves a pre-composited layer. On a complex page, this is the difference between smooth 60fps motion and visible jank.
Should I always use transition: all to keep CSS simple?
Avoid it in production. transition: all monitors every property — including ones you didn't intend to animate. If a CSS variable changes during a theme switch, it will animate unexpectedly. Always name your properties: transition: transform 0.2s ease-out, opacity 0.2s ease-out.
The card flip shows the front face through the back. What's wrong?
Almost always: missing transform-style: preserve-3d on the card container, or missing backface-visibility: hidden on the faces. Also check that no ancestor has overflow: hidden, filter, clip-path, or mix-blend-mode — these break the 3D context.
How do I animate an element only in one direction — in on hover, but snap back instantly?
Put the transition on the :hover rule instead of the default state. When hover ends, the :hover styles (including the transition) are removed, so the return snaps instantly. Use sparingly — two-way transitions usually feel more polished.
I want an animation to play on page load, not just on hover. Can transitions do that?
Transitions need a property change (before/after). Classic approach: JS sets initial state in CSS, then setTimeout adds a class to trigger the transition. Modern approach: use @starting-style — define starting values, and the browser auto-transitions from them when the element first renders. Supported in all current browsers since 2024.
