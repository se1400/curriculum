Part 1 — Modern CSS

⬡ CSS Features

CSS Nesting

Previously only possible with Sass/SCSS. Now natively supported in all modern browsers — no build step required. The & selector references the parent rule.

Without Nesting

/* Selectors scattered everywhere */
.card { background: white; }
.card:hover { box-shadow: ...; }
.card .card-title { font-size: 1.25rem; }
.card .card-title:first-child { margin-top: 0; }
.card .card-body { padding: 1rem; }

With Nesting

/* All card styles in one block */
.card {
  background: white;

  &:hover { box-shadow: ...; }

  .card-title {
    font-size: 1.25rem;
    &:first-child { margin-top: 0; }
  }

  .card-body { padding: 1rem; }
}
Browser support: All modern browsers. Check caniuse.com/css-nesting. For legacy support, use PostCSS to compile nesting.

CSS Nesting — Demo

Nesting in Action

Nested Media Queries

Keep responsive overrides right next to the base styles they modify:

.hero {
  font-size: 1.5rem;
  padding: 1rem;

  @media (min-width: 768px) {
    font-size: 2.5rem;
    padding: 2rem;
  }

  @media (min-width: 1200px) {
    font-size: 3.5rem;
    padding: 4rem;
  }
}
Why this matters: Responsive overrides stay next to the base styles they modify. No more scrolling to find the media query.

What the Browser Compiles To

Nesting is syntax sugar. The browser expands it into flat selectors — the same CSS you'd have written before:

You write (nested)

.card {
  background: white;
  border-radius: 8px;

  &:hover {
    box-shadow: 0 4px 12px
      rgba(0,0,0,.2);
    transform: translateY(-2px);
  }

  .card-title {
    font-size: 1.25rem;
    &:first-child { margin-top: 0; }
  }

  @media (min-width: 768px) {
    padding: 2rem;
  }
}

Browser sees (compiled)

.card {
  background: white;
  border-radius: 8px;
}
.card:hover {
  box-shadow: 0 4px 12px
    rgba(0,0,0,.2);
  transform: translateY(-2px);
}
.card .card-title {
  font-size: 1.25rem;
}
.card .card-title:first-child {
  margin-top: 0;
}
@media (min-width: 768px) {
  .card { padding: 2rem; }
}
Same result, less repetition. You write .card once. The browser handles expanding every nested rule into its full selector path.

Modern CSS — :has()

⬡ CSS Features

:has() — The Parent Selector

For decades, there was no way to style a parent based on its children. :has() changes that — supported in all modern browsers.

Common Patterns

/* Card with image: remove top padding */
.card:has(img) { padding-top: 0; }

/* Label highlights when input is focused */
.field:has(input:focus) label {
  color: #4f6df5;
  font-weight: 600;
}

/* Form shows error border */
form:has(:invalid) {
  border: 2px solid #ef4444;
}

/* Hide empty sidebar */
.sidebar:not(:has(li)) {
  display: none;
}

/* List items with sub-lists */
li:has(ul) { list-style-type: "▸ "; }

The HTML Structure

Here is the HTML. The CSS selects the parent by checking what's inside it:

<!-- The PARENT we want to style -->
<div class="field">
  <label>Email Address</label>
  <input type="email">
</div>
/* Before :has(): needed JavaScript to
   add a class to .field on focus */

/* With :has(): pure CSS — no JS */
.field:has(input:focus) {
  border-color: #4f6df5; /* parent changes */
}
.field:has(input:focus) label {
  color: #4f6df5;        /* sibling changes */
  font-weight: 700;
}

Modern CSS — :is() and :where()

⬡ CSS Features

:is() and :where()

Both match any element that matches at least one selector in their list. They are identical except for one thing: specificity.

The Problem They Solve: Targeting heading elements in multiple containers used to mean writing every combination separately: header h1, header h2, main h1, main h2 { } — six selectors for two containers and two tags. :is(header, main) :is(h1, h2) does the same in one line. :where() adds zero specificity, making it perfect for resets that shouldn't fight your own styles.
:is(selector-list)

Takes the highest specificity of any selector in the list.

/* Replaces 6 selectors */
:is(header, main) :is(h1, h2, h3) {
  color: #0f172a;
}
Specificity: inherits from list
:where(selector-list)

Always zero specificity — any selector overrides it.

/* Base styles — easy to override */
:where(p, ul, ol, dl) {
  margin-block: 1em;
  /* 0 specificity */
}
Specificity: always 0
Use caseChooseWhy
Reset / base styles:where()Zero specificity — any rule can override
Component / utility styles:is()Normal specificity behavior expected
Reduce repetitive selectorsEitherBoth collapse long selector lists

Modern CSS — @layer

⬡ CSS Features

CSS Cascade Layers with @layer

Specificity wars happen when you write increasingly specific selectors just to beat library styles. @layer solves this by letting you declare which styles win — regardless of specificity.

Without @layer: You import Bootstrap or Tailwind, then fight its styles with !important or increasingly specific selectors like .my-app .container .card.special button. That's the specificity war. With @layer: you put third-party CSS in a layer(vendor) layer and all of your styles beat it — regardless of specificity — because you control the layer order.

Declaring Layers

/* Layer order: later = higher priority */
@layer base, components, utilities;

@layer base {
  * { box-sizing: border-box; }
  body { font-family: system-ui; }
}

@layer components {
  .card { background: white;
          border-radius: 8px; }
}

@layer utilities {
  /* Always wins */
  .hidden { display: none; }
}

/* Import a library INTO a layer */
@import url("bootstrap.css") layer(vendor);

Interactive Layer Demo

A .btn gets styles from three layers. Which one wins?

@layer base .btn { color: #94a3b8; border-radius: 0; }
@layer components .btn { color: #93c5fd; border-radius: 6px; }
@layer utilities .btn { color: #86efac; border-radius: 999px; }
.btn — styled by: utilities
Key rule: Unlayered styles (no @layer) always beat layered ones.

Modern CSS — Dark Mode

⬡ CSS Features

Dark Mode with prefers-color-scheme

Why it matters: 82% of smartphone users prefer dark mode (Android Authority, 2023). Users with migraines, light sensitivity, or low-vision conditions depend on it. Dark mode is a WCAG-adjacent accessibility feature — ignoring it means ignoring a large segment of your users.

The CSS Custom Properties Approach

/* Light mode tokens */
:root {
  --color-bg:      #ffffff;
  --color-surface: #f8f9fa;
  --color-text:    #0f172a;
  --color-accent:  #4f6df5;
}

/* Override tokens for dark mode */
@media (prefers-color-scheme: dark) {
  :root {
    --color-bg:      #0f172a;
    --color-surface: #1e293b;
    --color-text:    #f1f5f9;
    --color-accent:  #7c9bff;
  }
}

/* Components use tokens — auto-switch */
body { background: var(--color-bg);
       color: var(--color-text); }
.card { background: var(--color-surface); }

Modern CSS — Color Functions

⬡ CSS Features

light-dark() & color-mix()

Why these exist: light-dark() reduces dark mode from "define tokens in two places" to a single function call per property. color-mix() eliminates hard-coding hover states — instead of calculating a lighter version of your brand color manually, you describe the intent: "80% brand + 20% white."

light-dark() — One Line Theming

:root {
  /* Required opt-in */
  color-scheme: light dark;
}

body {
  /* light-dark(light-value, dark-value) */
  background: light-dark(#ffffff, #0f172a);
  color:       light-dark(#0f172a, #f1f5f9);
}

.card {
  background: light-dark(#f8f9fa, #1e293b);
}
Supported in all modern browsers since 2024. Requires color-scheme: light dark on the root element.

color-mix()

:root { --brand: #4f6df5; }

.btn-hover {
  /* 20% lighter: mix with white */
  background: color-mix(
    in srgb, var(--brand), white 20%
  );
}

.btn-ghost {
  /* Very light tint */
  background: color-mix(
    in srgb, var(--brand), transparent 85%
  );
}

Modern CSS — @supports

⬡ CSS Features

Feature Queries with @supports

Tests whether a browser supports a CSS property before applying a block. Enables progressive enhancement — working styles for all browsers, enhanced styles for modern ones.

Real-world scenario: You want to use display: grid with subgrid, but Safari just got support in 2023. Without @supports, you either drop the feature for everyone or accept broken layouts for older Safari users. With @supports, old browsers get a solid flexbox layout; modern browsers get the enhanced grid layout. Both work. No one sees broken CSS.
/* Basic layout — works everywhere */
.gallery { display: flex; flex-wrap: wrap; }
.gallery-item { flex: 1 1 200px; }

/* Enhance with Grid if supported */
@supports (display: grid) {
  .gallery {
    display: grid;
    grid-template-columns:
      repeat(auto-fill, minmax(200px, 1fr));
  }
  .gallery-item { flex: unset; }
}

/* Container queries if supported */
@supports (container-type: inline-size) {
  .card-wrapper {
    container-type: inline-size;
  }
}

/* Glass effect if backdrop-filter works */
@supports (backdrop-filter: blur(8px)) {
  .glass {
    backdrop-filter: blur(8px);
    background: rgba(255,255,255,0.3);
  }
}

When to Use @supports

  • Before using newer layout features (subgrid, container queries)
  • Before applying visual effects that degrade gracefully (backdrop-filter)
  • When a newer CSS property replaces an older approach
Progressive Enhancement Pattern:
1. Write baseline CSS that works everywhere
2. Add @supports block with the enhanced version
3. Browsers that don't understand the feature use the baseline
You can also negate:
@supports not (display: grid) { }
Or combine:
@supports (display: grid) and (gap: 1rem) { }
Check support: caniuse.com — paste any CSS property to see browser support tables instantly.
Test in the console: CSS.supports("display", "grid") returns true or false. In Chrome DevTools Styles panel, rules inside an unmatched @supports block appear greyed out — a quick visual indicator that the feature wasn't supported.
How to check: In Chrome DevTools → Sources, search for @supports to see which blocks matched. Unmatched blocks are greyed out in the Styles panel. You can also test CSS.supports("display", "grid") in the console — returns true or false.

Part 2 — Web Accessibility

♿ Web Accessibility

Why Accessibility Matters

Accessibility is designing and building products that people with disabilities can use. Approximately 1 billion people worldwide have some form of disability.

⚖ Legal

The ADA (US), Section 508, and the EU Web Accessibility Directive legally require accessible digital products. Lawsuits over inaccessible sites have cost companies millions of dollars.

⬡ Ethical

Building a product that excludes a significant portion of the population is wrong. Accessibility is not a feature — it is the baseline expectation of a professional developer.

📈 Business

Accessible sites rank better in search engines, perform better on mobile, and reach a larger audience. The accessible UX is often a better experience for everyone.

Disabilities include: blindness & low vision · deafness & hard of hearing · motor impairments · cognitive differences · situational limitations (bright sunlight, broken mouse, holding a baby)

Accessibility — WCAG

♿ Web Accessibility

WCAG — Web Content Accessibility Guidelines

The international standard for web accessibility, published by the W3C. Organized around four principles: POUR. Current version: WCAG 2.2 (2023). www.w3.org/TR/WCAG22/

P — PerceivableUsers must be able to perceive the information. Example: images need alt text; video needs captions.
O — OperableUsers must be able to operate the interface. Example: all functionality available by keyboard, no seizure-inducing content.
U — UnderstandableContent and UI must be understandable. Example: clear language, predictable behavior, helpful error messages.
R — RobustContent must work with assistive technologies today and as they evolve. Example: valid semantic HTML, correct ARIA usage.

Conformance Levels

Level AMinimum requirements. Failing these makes content completely inaccessible to some users.
Level AAThe legally required standard in most jurisdictions and the target for professional web projects.
Level AAAThe highest level. Aspirational — not all criteria can be met for all content types.

Accessibility — The Accessibility Tree

♿ Web Accessibility

The Accessibility Tree

Your browser builds two trees from your HTML. The DOM is for rendering and JavaScript. The Accessibility Tree is a parallel, stripped-down version that screen readers and assistive technology read from — not the pixels.

Same Page — Two Representations

❌ What you wrote
<div class="nav">
<div onclick> Menu </div>
<div onclick> Shop </div>
Screen reader sees: "Menu Shop" — no roles, no hints
✓ What the tree sees
navigation landmark
link · "Menu"
link · "Shop"
Announces: "navigation, Menu link, Shop link"
Every element gets mapped: the tree records each element's role (what it is), name (what it's called), state (checked? expanded? disabled?), and properties. Semantic HTML fills these in automatically. Non-semantic HTML leaves them blank. 
<!-- Correct: semantic HTML → rich tree -->
<nav aria-label="Main navigation">
  <a href="/">Menu</a>
  <a href="/shop">Shop</a>
</nav>

<!-- Wrong: divs → empty tree entries -->
<div class="nav">
  <div onclick="...">Menu</div>
</div>

How to Inspect the Tree in Chrome

  1. Open DevTools (F12)
  2. Select any element in the Elements panel
  3. Look for the Accessibility tab in the right panel
  4. You'll see: Role · Name · Description · States

Chrome DevTools → Elements → Accessibility tab

The Golden Rule

Use this…Not this…Why
<button><div onclick>Keyboard + role free
<nav><div class="nav">Landmark role free
<label>placeholder textName never disappears
<h1>–<h6><div class="title">Document outline free
Semantic HTML is the highest-leverage accessibility technique. It gives you roles, names, keyboard behavior, and focus management — all for free, before you write a single line of ARIA.

Accessibility — ARIA

♿ Web Accessibility

ARIA — Accessible Rich Internet Applications

ARIA extends the accessibility tree with additional roles, states, and properties — for when semantic HTML alone isn't enough (custom dropdowns, tab panels, live regions).

The First Rule of ARIA: Do not use ARIA if a native HTML element can do the same thing.
A <button> is always better than <div role="button">. Native elements have built-in keyboard behavior and semantics. ARIA only adds to the accessibility tree — it does not add keyboard behavior or change visual appearance.

What ARIA Adds

ARIA adds…ARIA does NOT add…
Roles to the treeKeyboard behavior
Names & descriptionsVisual changes
States (expanded, invalid)Focus management
Live region announcementsInteraction events

When ARIA IS Needed

  • Custom dropdown/combobox (not a native <select>)
  • Custom tab interface
  • Custom date picker
  • Live notifications (chat messages, stock tickers)
  • Dynamic content that needs to be announced
ARIA Authoring Practices Guide: www.w3.org/WAI/ARIA/apg/ — keyboard patterns for every widget type.

Accessibility — ARIA Roles & Labels

♿ Web Accessibility

ARIA Roles & Labels

Why labels matter: A screen reader announces elements by their accessible name. An icon button with no label gets announced as "button" — no context. A close button with aria-label="Close dialog" gets announced as "Close dialog, button." The label is what tells the user what the control does.

Landmark Roles

ARIA roleSemantic equivalent
role="banner"<header> (direct child of body)
role="navigation"<nav>
role="main"<main>
role="contentinfo"<footer>
role="complementary"<aside>
role="dialog"Modal window
role="alert"Error / critical message (live)
role="status"Success / polite live message

Accessible Labels

<!-- aria-label: no visible text -->
<button aria-label="Close dialog">✕</button>

<!-- aria-labelledby: point to another element -->
<section aria-labelledby="pricing-h2">
  <h2 id="pricing-h2">Pricing Plans</h2>
</section>

<!-- aria-describedby: linked description -->
<input type="password"
  aria-describedby="pw-hint">
<p id="pw-hint">8+ chars, include a number</p>

<!-- Distinguish multiple navs on a page -->
<nav aria-label="Main navigation">...</nav>
<nav aria-label="Breadcrumb">...</nav>
Common mistake: Using aria-label on elements that already have visible text creates two competing names. Only use aria-label when there is no visible text (icon buttons, search inputs without visible labels). Use aria-labelledby to point to visible text that already exists.

Accessibility — ARIA States

♿ Web Accessibility

ARIA States & Properties

Why states matter: When you build a dropdown that opens and closes visually, sighted users see it happen. A screen reader user hears nothing — the visual change was never communicated. aria-expanded, aria-checked, and aria-invalid are the vocabulary for telling assistive technology: "this thing changed, here's how." Without them, your custom UI is invisible to a screen reader.

<!-- aria-expanded: toggle state -->
<button aria-expanded="false"
        aria-controls="menu">Menu</button>
<ul id="menu" hidden>...</ul>

<!-- aria-current: active page -->
<nav>
  <a href="/">Home</a>
  <a href="/about"
     aria-current="page">About</a>
</nav>

<!-- aria-hidden: remove from tree -->
<span aria-hidden="true">★★★★☆</span>
<span class="sr-only">4 out of 5 stars</span>

<!-- aria-invalid: validation errors -->
<input type="email"
  aria-invalid="true"
  aria-describedby="email-err">
<p id="email-err" role="alert">
  Enter a valid email address.
</p>

Accessibility — Focus Management

♿ Web Accessibility

Focus Management & Keyboard Navigation

Who depends on this: Keyboard navigation is essential for users with motor disabilities who cannot use a mouse, for power users who navigate faster with a keyboard, and for anyone using a keyboard-heavy device. The hard rule: if an interactive element can't be reached by pressing Tab, a large portion of your users simply cannot use it.

Every interactive element must be reachable and operable by keyboard alone.

Tab Key Behavior

  • Tab — move focus forward
  • Shift+Tab — move focus backward
  • Enter — activate focused link or button
  • Space — activate button or checkbox
  • Esc — close modal / dismiss popup
  • Arrow keys — navigate within components (menus, tabs)

tabindex Values

tabindex="-1"Focusable via JS only — removed from tab order
tabindex="1+"⚠ Never use — overrides document order

Accessibility — Focus Styles

♿ Web Accessibility

Focus Styles & :focus-visible

The dilemma: Default browser focus outlines look jarring to mouse users — they appear when you click a button. So developers write *:focus { outline: none } to remove them. This destroys keyboard navigation: users literally cannot see where they are on the page. :focus-visible solves this by showing the ring only during keyboard navigation — invisible to mouse users, essential for keyboard users.

Visible focus indicators are required by WCAG 2.4.7 (Level AA). Never write *:focus { outline: none } without providing an alternative.

The Modern Pattern

/* Remove browser outline for mouse users */
:focus:not(:focus-visible) {
  outline: none;
}

/* Clear focus ring for keyboard users */
:focus-visible {
  outline: 3px solid #4f6df5;
  outline-offset: 2px;
  border-radius: 2px;
}

/* High-contrast mode support */
@media (forced-colors: active) {
  :focus-visible {
    outline: 3px solid ButtonText;
  }
}
:focus-visible fires only during keyboard navigation, not on mouse clicks — giving you control over when the ring appears without hurting mouse usability.

Accessibility — inert Attribute

♿ Web Accessibility

The inert Attribute

The problem it solves: When a modal opens, sighted mouse users see a dimmed background and can't click behind it. Keyboard users can still Tab to every link and button hidden behind the modal — called "focus leak." Before inert, you had to manually set tabindex="-1" on every focusable descendant, add aria-hidden="true", and apply pointer-events: none. Miss one element and focus leaks through. inert does all of that with one attribute.

Makes an element and all its descendants unfocusable, unclickable, and invisible to screen readers — with a single attribute. Supported in all modern browsers.

Before inert (the old way)

<!-- Had to do ALL of this manually -->
<main id="bg-content"
  aria-hidden="true">
  <a href="/" tabindex="-1">...</a>
  <button tabindex="-1">...</button>
  <input tabindex="-1">
  <!-- style="pointer-events:none" too -->
</main>

With inert (the new way)

<main id="page-content" inert>
  <!-- Everything inside is unreachable
       while the modal is open -->
  <a href="/">Unreachable</a>
  <button>Unreachable</button>
</main>

<dialog id="modal">
  <!-- Focus stays here -->
</dialog>

JS Pattern for Modals

const modal   = document.getElementById('modal');
const content = document.getElementById('page-content');
const openBtn = document.getElementById('open-btn');

// Open: inert the background
openBtn.addEventListener('click', () => {
  modal.showModal();
  content.inert = true;
});

// Close: remove inert, return focus
modal.addEventListener('close', () => {
  content.inert = false;
  openBtn.focus(); // Return focus to trigger
});
inert removes the need to set tabindex="-1" on every focusable child, toggle aria-hidden, and set pointer-events: none. One attribute replaces all of that.
Browser support: All modern browsers. caniuse.com/inert

Accessibility — Native <dialog>

♿ Web Accessibility

The Native <dialog> Element

Before <dialog>: Building a correct accessible modal meant: trap Tab/Shift+Tab inside it, listen for Esc, move focus to the first element on open, return focus to the trigger on close, add role="dialog", aria-modal="true", aria-labelledby pointing to the title, and lock body scroll. Miss any one step and the modal fails for keyboard and screen reader users. <dialog> gets you most of this automatically.

Building an accessible modal from scratch requires focus trapping, Escape key handling, aria-modal, focus return on close, and scroll lock. <dialog> handles most of this automatically.

<button id="openBtn">Open Dialog</button>

<dialog id="myDialog">
  <h2>Confirm Delete</h2>
  <p>This cannot be undone.</p>
  <div class="actions">
    <button id="cancelBtn">Cancel</button>
    <button id="deleteBtn">Delete</button>
  </div>
</dialog>

<script>
const dialog = document.getElementById('myDialog');

// showModal() = true modal:
// - adds ::backdrop
// - traps keyboard focus
// - Escape closes it
document.getElementById('openBtn')
  .addEventListener('click', () =>
    dialog.showModal());

document.getElementById('cancelBtn')
  .addEventListener('click', () =>
    dialog.close()); // Returns focus to opener
</script>

What showModal() Provides Free

  • ✓ ::backdrop pseudo-element (semi-transparent overlay)
  • ✓ Keyboard focus trapped inside dialog
  • ✓ Esc closes the dialog
  • ✓ Focus returns to the trigger on close
  • ✓ Correct role="dialog" in accessibility tree

Accessibility — Live Regions

♿ Web Accessibility

Live Regions with aria-live

The problem: Screen readers follow focus. When content changes without focus moving — a form error appears, a notification pops up, a cart count updates — a screen reader user hears nothing. They have no idea the page changed. aria-live solves this: it tells the browser "watch this region; announce any text changes to the user, even without a focus event."

When content changes dynamically (error after form submit, chat message, notification), screen readers don't automatically notice. aria-live tells the browser to announce changes.

<!-- Polite: announces when user isn't busy -->
<div aria-live="polite" aria-atomic="true">
  <p id="status"></p>
</div>

<!-- Assertive: interrupts immediately -->
<div role="alert" aria-live="assertive">
  <p id="error"></p>
</div>

<script>
// Updating content inside a live region
// triggers screen reader announcement
document.getElementById('status').textContent
  = 'Your changes have been saved.';
</script>

Accessibility — Skip Navigation

♿ Web Accessibility

Skip Navigation Links

The problem: Imagine being blind and loading any page on a large site. The navigation has 25 links. Every single page load means pressing Tab 25 times before you can read any content. A skip link is the very first focusable element on the page — pressing Enter jumps the user straight to the main content, skipping the nav entirely. WCAG 2.4.1 (Level A) requires it. It's invisible to mouse users but critical for keyboard users.

Keyboard users tab through every link at the top of a page before reaching content. A 30-link nav means 30 Tab presses. A skip link jumps directly to main content.

Implementation

<!-- First element in <body> -->
<a class="skip-link" href="#main">
  Skip to main content
</a>

<header>...navigation...</header>

<main id="main" tabindex="-1">
  <!-- Page content -->
</main>
/* Hidden by default; visible on focus */
.skip-link {
  position: absolute;
  top: -100%;
  left: 0;
  background: #4f6df5;
  color: white;
  padding: 0.75rem 1.5rem;
  font-weight: 600;
  z-index: 9999;
  transition: top 0.1s;
}
.skip-link:focus { top: 0; }

Accessibility — Common Mistakes

♿ Web Accessibility

Common Accessibility Mistakes

These errors account for the majority of accessibility failures found in web audits. Each has a fast fix:

Missing or empty alt text Without alt, a screen reader reads the filename: "img-hero-banner-2024.jpg." Decorative images: alt="" (skipped). Meaningful images: describe the content concisely — not "image of a…"
Insufficient color contrast Light gray on white looks clean but is illegible for low-vision users. WCAG 1.4.3 (AA) requires 4.5:1 for body text, 3:1 for large text. Fix: WebAIM Contrast Checker.
Removing focus outlines outline: none makes keyboard users blind — they cannot see where focus is. Fix: never remove focus styles without an equally visible replacement. Use :focus-visible.
Interactive <div>s and <span>s <div onclick> cannot receive keyboard focus and announces no role. Screen reader users and keyboard users can never activate it. Fix: <button> for actions, <a href> for links — always.
Form inputs without labels Placeholder text disappears on typing and is not reliably announced. Screen readers need a permanent <label for="..."> or aria-label. Fix: every input gets a label — no exceptions.
Inaccessible modals A modal that leaks focus to the background, ignores Esc, or loses focus on close strands keyboard users. Fix: use native <dialog> with showModal() and inert on background content.
Auto-playing media Autoplay competes with screen reader audio and can trigger seizures for photosensitive users. WCAG 1.4.2 (A) says stop/pause must be available within 3 seconds. Fix: autoplay off by default; provide visible controls.
Non-descriptive link text Screen readers can list all links on a page. A page with 15 "Read more" links is unusable. Fix: link text must describe the destination — "Download the 2024 Annual Report (PDF, 2.3 MB)" not "Download here."

Accessibility — Testing Tools

♿ Web Accessibility

Accessibility Testing Tools

Automated tools catch a large percentage of common errors instantly — but no tool catches everything. They are the starting point, not the finish line.

Why automated tools aren't enough: Tools can verify presence of alt text, check color contrast ratios, and validate ARIA syntax. They cannot judge whether the alt text is meaningful, whether the page's reading order makes sense, or whether a screen reader experience feels confusing. Studies show automated tools catch roughly 30–40% of real-world issues. A manual screen reader test is the only way to hear what your users actually hear.

Why test with a real screen reader: A screen reader announces page structure, element roles, labels, and state changes exactly as a blind user hears them — in sequence, no visual context. Q & A — Part 1

Q & A

Q. Does CSS Nesting work the same as Sass/SCSS nesting?
Almost. Native CSS Nesting requires the & selector or a nesting selector to identify descendent rules. Some older Sass patterns work in native CSS now too. The main difference is that native CSS computes selectors slightly differently in edge cases involving specificity and combinators — but for everyday usage, the syntax and behavior are effectively the same.
Q. What's the difference between :is() and :where() in practice?
They work identically except for specificity. :is(h1, h2) has the specificity of h1. :where(h1, h2) has zero specificity. Use :where() in base/reset stylesheets where you want any specific selector to easily override the rule. Use :is() in component or utility styles where normal specificity behavior is expected.
Q. If ARIA only adds to the accessibility tree and doesn't add keyboard behavior, what handles keyboard interaction for custom widgets?
JavaScript. When you build a custom widget — a date picker, combobox, tab interface — you are responsible for implementing all keyboard interactions. The WAI-ARIA Authoring Practices Guide at www.w3.org/WAI/ARIA/apg/ documents the expected keyboard behavior for every ARIA widget pattern. This is one of the strongest reasons to prefer native HTML elements: they come with keyboard behavior built in.

Q & A — Part 2

Q & A (continued)

Q. Is using a CSS variable for theming the same as using light-dark()?
They achieve the same result but differently. The CSS custom properties approach — defining tokens on :root and overriding in a media query — is more explicit and controllable. You can also toggle dark mode via a class (useful for user-controlled theme switching). light-dark() is more concise and automatic but relies solely on the OS preference and is harder to override with a manual toggle. For most production sites, the custom properties approach is recommended.
Q. How do I implement a "dark mode toggle" button that overrides the OS setting?
Store the user's preference in localStorage, then add data-theme="dark" to the <html> element. In CSS, define color tokens for both @media (prefers-color-scheme: dark) and [data-theme="dark"] — the manual class should come after so it wins. Critically: restore the preference from localStorage in a <script> tag in the <head> — before stylesheets load — to avoid a flash of the wrong theme on page load.
