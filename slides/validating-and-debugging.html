Debugging HTML with the Inspector

Common HTML Errors

  • Unclosed tags: <i> without </i>
  • Typos in tags: <ll> instead of <li>
  • Mismatched tags
  • Missing required attributes

How to Debug

  1. Right-click → Inspect Element
  2. Expand nodes in Elements panel
  3. Compare DOM to your source
  4. Look for unexpected elements
  5. Fix in source file and refresh

HTML Debug Example

Buggy Code

<h1><i>Favorite<i> Movies</h1>
<ul>
  <ll>Lord of the Rings</li>
  <li>Harry Potter</li>
</ul>

Problems: Missing / in closing </i> and typo <ll>

Fixed Code

<h1><i>Favorite</i> Movies</h1>
<ul>
  <li>Lord of the Rings</li>
  <li>Harry Potter</li>
</ul>

The inspector shows the DOM as the browser interprets it—differences reveal errors!

The Console Panel

What It Shows

  • Errors - Red, with stack trace
  • Warnings - Yellow background
  • Info - Blue/gray messages
  • Logs - Your console.log() output

Error Messages Include

  • Error type (SyntaxError, ReferenceError, etc.)
  • Description of what went wrong
  • File name and line number (clickable!)
  • Stack trace showing call history

Console Methods for Debugging

Try different console methods and open DevTools (F12) to see the output.

Common JavaScript Errors

SyntaxError

Invalid syntax - missing brackets, typos

console.log("Hi"  // Missing )
ReferenceError

Variable not defined

console.log(undefinedVar);
TypeError

Wrong type or calling non-function

null.toString();

Remember: JavaScript is case-sensitive! countIt() ≠ countit()

The Sources Panel (Debugger)

Key Features

  • View all loaded JavaScript files
  • Set breakpoints to pause execution
  • Step through code line by line
  • Watch variable values change
  • View the call stack

Debugging Controls

  • Resume/Pause: Continue or pause execution
  • Step Over: Execute current line, skip into functions
  • Step Into: Enter the function being called
  • Step Out: Finish current function, return to caller

Setting Breakpoints

In DevTools

  1. Open Sources panel
  2. Find your JavaScript file
  3. Click the line number to set breakpoint
  4. Blue marker appears
  5. Click again to remove

In Your Code

function calculate() {
  debugger; // Pauses here!
  const result = a + b;
  return result;
}

The debugger keyword pauses execution when DevTools is open.

Remove debugger statements before deploying to production!

Watching Variables

Scope Panel

When paused at a breakpoint:

  • Local: Variables in current function
  • Closure: Variables from outer scopes
  • Global: Window/global object properties

Watch Expressions

Add custom expressions to monitor:

  • Click + in Watch panel
  • Type any expression: count > 5
  • See value update as you step

Hover over any variable in the source code while paused to see its current value!

Debugging Example: Counter Not Working

Buggy Code

function incCount() {
  var count = 0;  // Problem!
  count += 1;
  return count;
}
// Always returns 1!

Issue: count resets to 0 every time the function is called.

Fixed Code

var count = 0;  // Moved outside

function incCount() {
  count += 1;
  return count;
}
// Now increments properly!

Fix: Declare count outside the function so it persists.

Understanding Source Maps

The Problem

Production code is often:

  • Minified: Whitespace removed, short variable names
  • Transpiled: Modern JS converted for older browsers
  • Bundled: Multiple files combined into one

This makes debugging nearly impossible!

The Solution

Source maps provide a mapping between:

  • Transformed code → Original source
  • Allows debugging original code
  • Shows original variable names
  • Generated automatically by build tools

Debugging on Mobile Devices

Desktop testing isn't enough—real devices reveal unique issues:

Android (Chrome)

  1. Enable Developer Options on device
  2. Enable USB Debugging
  3. Connect via USB
  4. Open chrome://inspect
  5. Click "Inspect" on your page

iOS (Safari)

  1. Settings → Safari → Advanced → Web Inspector
  2. Connect to Mac via USB
  3. Open Safari on Mac
  4. Develop menu → Your device
  5. Select page to inspect

Device Mode in DevTools

How to Enable

  • Click device icon in DevTools toolbar
  • Or press Ctrl+Shift+M (Cmd+Shift+M on Mac)

Features

  • Simulate different screen sizes
  • Test touch events
  • Throttle network speed
  • Emulate device pixel ratio

Important: Device mode is useful for quick testing, but always verify on real devices before launching!

Also Consider

  • BrowserStack: Test on real cloud devices
  • Sauce Labs: Automated cross-browser testing

Debugging Best Practices

  • Validate first - Run through W3C validators
  • Check the Console - Look for red errors
  • Use breakpoints - Step through problem code
  • Test multiple times - Don't stop at one successful test
  • Isolate the problem - Comment out code to find the issue
  • Read error messages - They often tell you exactly what's wrong
  • Check Network panel - For missing resources
  • Test in multiple browsers - Don't assume consistency
  • Test on real devices - Desktop simulation isn't perfect
  • Keep console clean - Remove debug code before production

Q&A

Q: Do I need the <!doctype> tag?

A: Yes! Without it, browsers guess which standard to follow, which can cause display issues.

Q: When should I use debugger vs manual breakpoints?

A: Manual breakpoints for quick sessions. Use debugger in code for complex paths or page load issues. Always remove before production.

Q: Should I worry about all Console warnings?

A: Focus on errors (red) first. Warnings (yellow) should be addressed but may not break your page. Aim for a clean console in production.

Q: How do I debug mobile-only issues?

A: Use remote debugging to connect your device to desktop DevTools via USB.
