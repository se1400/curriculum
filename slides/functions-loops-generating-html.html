Why Functions?

The Problem — Repeated Code

Analogy: Imagine you have a recipe for chocolate chip cookies. Every time someone asks for cookies, you don't write the recipe from scratch on a new piece of paper — you just say "follow the cookie recipe." Functions work the same way. Write the instructions once, give them a name, and call that name whenever you need them.

Here's a common situation — two buttons that need to do the same thing:

HTML

<button id="dark-btn">Toggle Dark Mode</button>
<button id="pref-btn">Preferences</button>
<span id="theme-label">Dark</span>

JavaScript — the problem

// Without a function — the same logic written TWICE
document.querySelector('#dark-btn').addEventListener('click', function() {
  document.body.classList.toggle('dark-mode');
  document.querySelector('#theme-label').textContent =
    document.body.classList.contains('dark-mode') ? 'Light' : 'Dark';
});

document.querySelector('#pref-btn').addEventListener('click', function() {
  document.body.classList.toggle('dark-mode');                // ← same code again
  document.querySelector('#theme-label').textContent =
    document.body.classList.contains('dark-mode') ? 'Light' : 'Dark';
});
What's wrong here? If you find a bug, you fix it in one place and forget the other. If the logic changes, you update two places and probably miss one. Copy-pasted code is a maintenance trap.
Why Functions?

The Solution — Write It Once

Same HTML — better JavaScript

<button id="dark-btn">Toggle Dark Mode</button>
<button id="pref-btn">Preferences</button>
<span id="theme-label">Dark</span>

JavaScript — the solution

// With a function — write the logic ONCE, call it ANYWHERE
function toggleDarkMode() {
  document.body.classList.toggle('dark-mode');
  document.querySelector('#theme-label').textContent =
    document.body.classList.contains('dark-mode') ? 'Light' : 'Dark';
}

// Both buttons call the same function — one source of truth
document.querySelector('#dark-btn').addEventListener('click', toggleDarkMode);
document.querySelector('#pref-btn').addEventListener('click', toggleDarkMode);

Shorter

No duplicated logic. Less code to write and read.

One place to fix

Bug in the toggle? Fix it once in the function. Every caller gets the fix.

Readable

toggleDarkMode tells you what happens. The handler reads like English.

You've already been calling functions! document.querySelector(), classList.toggle(), console.log(), Math.random() — these are all functions someone else wrote. Now you write your own.

Function Declarations

Parameters — Inputs to a Function

Analogy: Parameters are like blanks on a form. The function is the form template. When you call it, you fill in the blanks. A function can have zero blanks, one, or many — it depends on what information the function needs to do its job.

HTML — the elements these functions target

<h1>My Site</h1>
<p id="msg"></p>

CSS — the custom properties setTheme() changes

:root {
  --hue: 220;
  --saturation: 70%;
}
body { background: hsl(var(--hue), var(--saturation), 95%); }

JavaScript

// No parameters — always does the same thing
function showWelcome() {
  document.querySelector('h1').textContent = 'Welcome!';
}

// One parameter — behavior changes based on input
function greet(name) {
  document.querySelector('#msg').textContent = 'Hello, ' + name + '!';
}

// Two parameters — even more flexible
function setTheme(hue, saturation) {
  document.documentElement.style.setProperty('--hue', hue);
  document.documentElement.style.setProperty('--saturation', saturation);
}
Parameters vs arguments: Parameters are the variable names in the function definition (name). Arguments are the actual values you pass when calling it ('Ada'). Most people use the terms interchangeably — that's fine.
Function Declarations

Return Values — Getting Data Back

Analogy: Some functions are like a vending machine — you put something in (money), and it gives something back (a snack). The return keyword is what sends the result back to whoever called the function. Without return, the function does its work but doesn't hand anything back.

HTML

<span class="price"></span>
<h1>My Page</h1>

With return — produces a value

function formatPrice(price) {
  return '$' + price.toFixed(2);
}

const label = formatPrice(9.99);
// label = '$9.99'
document.querySelector('.price')
  .textContent = label;

The function computes a value and sends it back with return. The caller stores the result in a variable.

Without return — just performs an action

function showWelcome() {
  document.querySelector('h1')
    .textContent = 'Welcome!';
}

showWelcome();
// Returns undefined — that's OK
// It just does its job (updates the DOM)

Default Parameters

A parameter can have a fallback value that's used when no argument is passed:

function greet(name = 'friend') {
  return 'Hello, ' + name + '!';
}

greet('Maria');  // 'Hello, Maria!'
greet();         // 'Hello, friend!'  — default used

When to use defaults: Use them for optional settings — like a greeting that should work even if no name is provided. They prevent errors from missing arguments and make your functions more forgiving.

Arrow Functions

Arrow Functions — Reading the Syntax

An arrow function is a tiny recipe: input on the left, output on the right, connected by =>.

Read the => as "produces":
(price) => '$' + price.toFixed(2)
"Give me a price → it produces a dollar-formatted string."

Evolution — same function, four stages

// 1. Regular function declaration
function formatPrice(price) {
  return '$' + price.toFixed(2);
}

// 2. Same function, stored in a variable (no arrow yet)
const formatPrice = function(price) {
  return '$' + price.toFixed(2);
};

// 3. Replace 'function' with => — this IS an arrow function
const formatPrice = (price) => {
  return '$' + price.toFixed(2);
};

// 4. One-line body? Drop the { } and 'return' — they're implied
const formatPrice = (price) => '$' + price.toFixed(2);
Why the concise form works: The arrow => already says "the input produces this output." When there's only one expression on the right, JavaScript treats it as the return value automatically. No braces, no return keyword needed.
Arrow Functions

Reading Arrow Functions in English

Every one-liner follows the same pattern: input => output. Try reading each one aloud:

Arrow functionRead it as…
(n) => n * 2 "Take n → produces n times 2"
(n) => n % 2 === 0 "Take n → produces true if n is even"
(name) => `<div>${name}</div>` "Take a name → produces a div with that name inside"
(text) => text.toUpperCase() + '!' "Take text → produces it in CAPS with a ! on the end"
The (n) on the left is just a name you pick — call it n, num, x, whatever. It's the placeholder for whatever value gets passed in. The right side is the answer you get back.

Arrays

Quick Primer — Arrays

Before we loop through arrays, let's make sure you know what one is.

Analogy: An array is an ordered list — like a grocery list, a Spotify playlist, or people waiting in line for coffee. Each item has a position (its index) starting at 0 — not 1.

Creating an array

const cities = ['Tokyo', 'Lisbon', 'Chicago', 'Nairobi'];
//                [0]       [1]        [2]         [3]       ← the index of each item

Reading items by index

cities[0];        // 'Tokyo'    — the first item (index starts at 0!)
cities[2];        // 'Chicago'  — the third item
cities.length;   // 4          — how many items are in the array
The big idea: An array holds many values in one variable. Loops let you "walk through" those values one at a time so you can do something with each one — like build a card for every city.
Loops

The for Loop

A for loop repeats a block of code a specific number of times.

Analogy: A for loop is like doing laps around a track. Before you start, you set your lap counter to 0. Each time around, you check "have I done enough laps?" If not, you run another lap and add 1 to the counter. When you've done enough, you stop. 
for (let i = 0; i < 5; i++) {
  console.log(i);  // 0, 1, 2, 3, 4
}
PartWhat it doesRuns when?
let i = 0Initialization — create a counter starting at zeroOnce, before the loop starts
i < 5Condition — keep running while this is trueBefore each lap
i++Update — add 1 to the counterAfter each lap

Loops

Looping Through an Array

Use the array's .length as the condition to loop through every item:

HTML

<ul id="color-list"></ul>

JavaScript

const colors = ['red', 'blue', 'green'];

for (let i = 0; i < colors.length; i++) {
  console.log(colors[i]);
  // i=0 → 'red', i=1 → 'blue', i=2 → 'green'
}
How colors[i] works — step by step:
The array colors has 3 items at positions 0, 1, 2.
When i = 0: colors[0] is "red"
When i = 1: colors[1] is "blue"
When i = 2: colors[2] is "green"
When i = 3: i < colors.length is false → loop stops

Loops

The for...of Loop

The friendliest way to visit every item in an array — no counter, no indexes.

Analogy: Imagine customers waiting in line at a coffee shop. You serve one person, they leave, the next person steps up. You keep going until no one is left. for...of works exactly like that — JavaScript hands you one item at a time until the array is empty.

Read the syntax word-by-word

for (const color of colors) {
  console.log(color);
}
PieceWhat it means
for"I'm about to loop"
const color"Create a variable called color — it will hold one item each time"
of colors"…pulling from the array called colors"
{ ... }"Run this block once for each item"

Reads as: "For each color of the colors array, log the color."

Walk through ['red', 'blue', 'green']

Iteration 1:  color = 'red'    → console.log('red')
Iteration 2:  color = 'blue'   → console.log('blue')
Iteration 3:  color = 'green'  → console.log('green')
Array empty — loop stops.

Index-based for

for (let i = 0; i < colors.length; i++) {
  console.log(colors[i]);
}

You manage the counter + look up the item yourself.

for...of

for (const color of colors) {
  console.log(color);
}

JS hands you the item directly. No counter needed.

Generating HTML

Generating HTML with Loops

This is the pattern that makes card grids come alive.

Analogy: Imagine a printing press. You have a list of names and a card template. The press stamps out one card per name — same design, different content. That's exactly what this loop does: one template, many cards, generated from data.

HTML — An empty container

<div class="grid"></div>

JavaScript — Fill it with cards

const cities = ['Tokyo', 'Lisbon', 'Chicago', 'Nairobi', 'Sydney'];
let html = '';

for (const city of cities) {
  html += `<div class="card">${city}</div>`;
}

document.querySelector('.grid').innerHTML = html;
How it works: Start with an empty string. Each iteration adds one card's HTML to the string using a template literal. After the loop, one innerHTML assignment injects the entire result. The CSS card styles you already know apply automatically.

The renderCards() Pattern

Wrapping the Loop in a Function

The renderCards() pattern combines functions + loops into one reusable tool.

Why wrap it? If you render cards once on page load and again after a filter or shuffle, you'd need the same loop in two places. Wrapping it in a function means you write it once and call it whenever the data changes.

HTML — an empty container + a button to re-render

<button id="shuffle-btn">Shuffle</button>
<div class="grid"></div>

CSS — the grid layout that styles whatever cards JS creates

.grid { display: grid; grid-template-columns: repeat(auto-fill, minmax(180px, 1fr)); gap: 1rem; }
.card { background: #fff; padding: 1rem; border-radius: 8px; box-shadow: 0 2px 6px rgba(0,0,0,.1); }

JavaScript

const container = document.querySelector('.grid');

function renderCards(items) {
  let html = '';
  for (const item of items) {
    html += `
      <div class="card">
        <h3>${item}</h3>
      </div>
    `;
  }
  container.innerHTML = html;
}

renderCards(cities);  // Initial render

Call it once on load. Call it again after a filter. Call it after a shuffle. The grid rebuilds every time.

The renderCards() Pattern

map() — Transform Every Item

map() takes an array, runs every item through a function you provide, and gives you back a new array of the transformed results.

Analogy: map() is an assembly line. Items enter one end, a machine (your arrow function) transforms each one, and transformed items come out the other end — same number of items, new shape.

The three things map() always does

  • Walks through every item in the array, one by one
  • Passes each item into the function you provide
  • Collects whatever the function returns into a new array (the original is untouched)

Simplest example — numbers in, doubled numbers out

const numbers = [1, 2, 3, 4];

const doubled = numbers.map(n => n * 2);
//                          └──────┬──────┘
//                        "for each n, produce n * 2"

console.log(doubled);  // [2, 4, 6, 8]
console.log(numbers);  // [1, 2, 3, 4]  ← original untouched

Walk through it

Iteration 1:  n = 1  →  1 * 2  →  2
Iteration 2:  n = 2  →  2 * 2  →  4
Iteration 3:  n = 3  →  3 * 2  →  6
Iteration 4:  n = 4  →  4 * 2  →  8
Result:       [2, 4, 6, 8]

Same idea — strings in, HTML strings out

const cities = ['Tokyo', 'Lisbon', 'Chicago'];

const cards = cities.map(city => `<div class="card">${city}</div>`);
// cards is now:
// [
//   '<div class="card">Tokyo</div>',
//   '<div class="card">Lisbon</div>',
//   '<div class="card">Chicago</div>'
// ]

The renderCards() Pattern

join('') — Combine an Array Into One String

.join() turns an array of strings into a single string, putting the separator you choose between each item.

Analogy: Think of .join() like glue. You choose what goes between the items. join(', ') uses a comma-and-space. join('') uses nothing — the items touch directly. For HTML we always want nothing in between, so we pass '' (an empty string).

What .join() does

const parts = ['Tokyo', 'Lisbon', 'Chicago'];

parts.join(', ');   // 'Tokyo, Lisbon, Chicago'   ← comma + space between
parts.join(' - ');  // 'Tokyo - Lisbon - Chicago' ← dash between
parts.join('');     // 'TokyoLisbonChicago'       ← nothing between

Why we need it for HTML

After map() you have an array of HTML strings. You need one string to set on innerHTML. If you skip join('') and hand the array directly to innerHTML, JavaScript converts the array to a string the default way — with commas between every item:

Without join('')

container.innerHTML = [
  '<div>Tokyo</div>',
  '<div>Lisbon</div>'
];
// renders as:
// <div>Tokyo</div>,<div>Lisbon</div>
// ↑ commas appear visibly on the page!

With join('')

container.innerHTML = [
  '<div>Tokyo</div>',
  '<div>Lisbon</div>'
].join('');
// renders as:
// <div>Tokyo</div><div>Lisbon</div>
// ↑ clean — no stray commas

Putting map() and join('') together

function renderCards(items) {
  container.innerHTML = items
    .map(item => `<div class="card">${item}</div>`)  // array of HTML strings
    .join('');                                        // one big string
}

Reads as: "Take the items, turn each one into a card-HTML string, then glue them all together into one string, and set that as the grid's HTML."

Putting It Together

Functions + Loops + Template Literals

The three tools combine to create a clean separation between data and display.

HTML

<button id="shuffle-btn">Shuffle</button>
<div class="grid"></div>    <!-- JS fills this container -->

CSS

.grid { display: grid; grid-template-columns: repeat(auto-fill, minmax(180px, 1fr)); gap: 1rem; }
.card { background: #fff; padding: 1rem; border-radius: 8px; }

JavaScript — three pieces working together

1. Data

Lives in an array. Easy to add, remove, or reorder items.

const projects = [
  'Portfolio',
  'Weather App',
  'Quiz Game'
];

2. Function

Knows how to turn items into HTML. Reusable.

function renderCards(list) {
  container.innerHTML =
    list.map(name => `
    <div class="card">
      <h3>${name}</h3>
    </div>`).join('');
}

3. Trigger

Call it on load. Call it after an event. The grid rebuilds.

renderCards(projects);

shuffleBtn.addEventListener(
  'click', () => {
    renderCards(
      [...projects].reverse()
    );
});
Key insight: Changing what gets displayed means changing the array and calling the render function again. The CSS card layouts you've designed stay completely unchanged.

Best Practices

Best Practices

Name functions after what they do
renderCards(), formatPrice(), showError() — a reader should understand the function's purpose from its name alone. Verb + noun is a good pattern.
Keep functions focused
A function that renders cards should render cards, not also validate form input. If a function does two things, split it into two functions. Each function = one job.
Default to for...of for arrays
It's shorter and more readable than index-based for loops. Only use the index-based version when you specifically need the position number.
Always use join('') with map() for HTML
Without it, JavaScript inserts commas between array items. Those commas will appear in your rendered HTML as visible text.
Be careful with innerHTML and user input
When data comes from user input, use textContent on individual elements instead. When data is your own array (under your control), innerHTML is fine.
