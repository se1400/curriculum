The big shift
Before this lesson, every JavaScript change happened once — when the page loaded. The variables ran, the DOM updated, and then JavaScript went silent.

Events are how JavaScript listens to the user. Every click, keystroke, slider drag, and form submission — the browser tracks all of it. Your job is to tell JavaScript which events to listen for and what to do when they happen.
The visual payoff is immediate: range sliders that shift your entire color scheme as you drag them, password fields with live strength bars, forms that validate as you type. All of it runs on two ideas: listen for an event → decide what to do.
addEventListener

addEventListener — The Right Way

❌ Old way — logic lives in HTML:

<!-- onclick mixes JS into HTML -->
<button onclick="doSomething()">Click me</button>

// Limitation: only ONE handler per element.
// Second onclick= overwrites the first.

✅ Modern way — keep JS in your JS:

const btn = document.querySelector('#submit-btn');

btn.addEventListener('click', function() {
  console.log('Button clicked!');
});
//            ↑ event type   ↑ handler function
Read it out loud: "Find the element with id submit-btn. When a click event occurs on it, run this function." The function is called a handler. It runs every time the event fires — not just once.
addEventListener stacks. You can attach multiple listeners to the same element — each one runs. The old onclick= attribute can only hold one handler.

Events

The 4 Events You'll Use Most — Part 1

click

Fires when the user clicks anything — buttons, cards, labels, links. Most common event you'll use.

Your HTML — the button on the page:

<button class="toggle-btn">Toggle Dark Mode</button>

Your JS — listens for the click:

const btn = document.querySelector('.toggle-btn');
// ↑ find the button using the same class name

btn.addEventListener('click', function() {
  // ↑ "when someone clicks this button, run..."
  document.body.classList.toggle('dark-mode');
  // ↑ flip the dark-mode class on <body>
});

This box switches between two CSS states — click the button!

input

Fires on every single keystroke or value change — while the user is still typing or dragging. Use this for live previews and character counters.

Your HTML — a slider and a display:

<input type="range" id="hue-slider" min="0" max="360">
<div id="hue-display">Hue: 0</div>

Your JS — updates as you drag:

const slider = document.querySelector('#hue-slider');
const display = document.querySelector('#hue-display');

slider.addEventListener('input', function() {
  // runs on EVERY pixel the slider moves
  display.textContent = `Hue: ${slider.value}`;
});

Events

The 4 Events You'll Use Most — Part 2

change

Fires once, after the user has finished making a choice — not while they're scrolling through options. Best for dropdowns and checkboxes where only the final answer matters.

Your HTML — a dropdown menu:

<select id="font-select">
  <option value="system-ui">Default</option>
  <option value="Georgia">Georgia</option>
</select>

Your JS — responds once they pick:

const select = document.querySelector('#font-select');
select.addEventListener('change', function() {
  // only runs after a final selection
  document.body.style.fontFamily = select.value;
});

Font Switcher (change event)

Select a font — fires once when you make your choice

submit

Fires when a form is submitted — either by clicking the submit button or pressing Enter in a text field. You attach the listener to the <form> element, not the button.

Your HTML — a real form:

<form id="signup">
  <input type="text" name="name" placeholder="Your name">
  <button type="submit">Sign Up</button>
</form>

Your JS — intercept before the browser acts:

const form = document.querySelector('#signup');
form.addEventListener('submit', function(event) {
  event.preventDefault(); // ← ALWAYS first
  // Now safely read values
  const name = form.querySelector('input').value;
  console.log('Submitted!', name);
});

Form Submit (no page reload)

Type your name, click Sign Up — the page does NOT reload

Why does event.preventDefault() exist?

The analogy: think of a form submit like pressing the gas pedal in a car. The browser's default behavior is to drive straight to the server — it reloads the entire page and sends the form data to a URL. event.preventDefault() is you pressing the brake before it takes off. Now you're in control.

Without it, the moment a user clicks "Submit":

  • The page reloads completely
  • Every JavaScript variable you had is erased
  • Every DOM change you made is gone
  • The user is staring at a blank or refreshed page

With preventDefault(), you handle it instead:

form.addEventListener('submit', function(event) {
  event.preventDefault(); // brake!

  // Now YOU decide what happens:
  const name = document.querySelector('#name').value;

  if (!name) {
    showError('Name is required'); // validate
  } else {
    showSuccessMessage(name); // or do something
  }
});

Rule: event.preventDefault() is the very first line inside every submit handler. Always.

Event Object

The Event Object — Your Report Card from the Browser

The analogy: When an event fires, the browser writes up a report card about exactly what happened — "which element was involved, what type of event it was, what value the user entered." It hands this report to your handler automatically. That report is the event object.

To read the report, add a parameter to your handler function. You can name it anything, but everyone uses event or just e.

Your HTML — a button and an input:

<button id="my-btn">Click me</button>
<input id="search" type="text" placeholder="Search...">

Your JS — receive and read the report:

const btn = document.querySelector('#my-btn');
btn.addEventListener('click', function(event) {
//                              ↑ the report card
  console.log(event.type);   // → 'click'
  console.log(event.target); // → <button id="my-btn">
});

const input = document.querySelector('#search');
input.addEventListener('input', function(event) {
  const typed = event.target.value; // what user typed
  console.log('User is searching for:', typed);
});

event.type

What kind of event happened. 'click', 'input', 'submit', etc.

event.target

The exact HTML element the user touched. You can read its classList, id, value, anything.

event.target.value

The current value inside a form field — whatever the user typed or selected right now.

event.preventDefault()

A method on the report card. Tells the browser "stop what you were about to do automatically." Used on form submits.

Event Inspector — See the report card live

Why does this matter? Without the event object, you'd have to querySelector for the element again inside your handler. With event.target, the browser already hands you the element — no extra lookup needed. For inputs, event.target.value gives you what the user typed at that exact moment.

Form Values

Reading Form Values — Text, Number, Color

Text, Email, Password — HTML:

<input type="text" id="name" placeholder="Your name">
const nameInput = document.querySelector('#name');
const value = nameInput.value; // → "Sofia" (a string)

Number & Range — HTML:

<input type="range" id="volume" min="0" max="100">
const slider = document.querySelector('#volume');

// ❌ Don't do this for math — it's a string!
slider.value         // → "75"  ← string
"75" + 5             // → "755" ← WRONG

// ✅ Convert it first:
Number(slider.value) // → 75   ← number
75 + 5               // → 80  ← correct!
.value is ALWAYS a string — even for type="number" or type="range". Without Number(), adding values gives you concatenated strings instead of math. Most common beginner bug.

Color Picker — HTML:

<input type="color" id="accent">
// Then in JS:
const picker = document.querySelector('#accent');
picker.value // → '#6c5ce7'

Decisions

Making Decisions: if / else if / else

Events tell JavaScript when something happened. Conditional statements tell it what to do about it.

const score = Number(input.value);

if (score >= 90) {
  label.textContent = 'Excellent';
  label.className = 'badge badge--green';
} else if (score >= 70) {
  label.textContent = 'Good';
} else if (score >= 50) {
  label.textContent = 'Pass';
} else {
  label.textContent = 'Not yet';
  // catch-all — runs if nothing above matched
}
Order matters. JavaScript evaluates top to bottom and stops at the first condition that is true. Put the most specific (highest value) conditions first, the most general last.
Why use if/else? Without decisions, your code can only do one thing. With if/else, the same function behaves differently depending on what the user typed, what value a slider has, or whether a box is checked. This is the foundation of every validation, grade calculator, quiz, and live preview you'll build.

Decisions

Logical Operators — &&, ||, !

Combine multiple conditions into a single expression:

// AND — both must be true
if (password.length >= 8 && password.includes('@')) { }

// OR — at least one must be true
if (role === 'admin' || role === 'editor') { }

// NOT — flips true to false and vice versa
if (!isLoggedIn) { redirectToLogin(); }

&& (AND)

Both sides must be true. Like a double lock — both keys needed.

|| (OR)

At least one must be true. Like a backup key — either one works.

! (NOT)

Flips the result. !true → false. !false → true. Reads as "if not".

Truthy & Falsy

Truthy and Falsy — JavaScript's Yes/No System

The analogy — a bouncer at the door:
Imagine a bouncer who only asks one question: "Is this value empty, zero, or nothing?" If yes → falsy (turned away). If no → truthy (let in).

JavaScript runs this same test every time you use a value inside an if statement. You don't have to write === true or === "". JavaScript already knows.

The 6 values JavaScript treats as "no" (falsy):

false 0 "" ← empty string null undefined NaN

Everything else is truthy — any non-empty string, any non-zero number, any object or array.

Why this matters — checking if a field is empty:

// Your HTML has a text input:
// <input id="name" type="text">

const name = document.querySelector('#name').value;

// Long version — checks for empty string explicitly:
if (name === '') {
  showError('Name is required');
}

// Short version — empty string is falsy, so !name is true:
if (!name) {
  showError('Name is required');
}
// Both do the same thing. Pros use the short version.
A common gotcha: the string "false" is truthy — because it's a non-empty string. "0" is also truthy. Only the actual number 0 and the actual boolean false are falsy. If in doubt, test it in the demo →

Real-world uses in your code

// Is the input empty?
if (!nameField.value) { showError(); }

// Did the user pick anything from a dropdown?
if (!select.value) { showError(); }

// Did the user enter a non-zero number?
if (score) { showResult(score); }

// Is the checkbox NOT checked?
if (!checkbox.checked) { blockSubmit(); }

Decisions

The Ternary Operator — A One-Line if/else

The analogy — a coin flip:
Think of it like asking a yes/no question. "Is it raining?" → Yes: grab an umbrella. No: leave it. The ternary asks one question and picks one of two answers. That's it.

It's just a shorter way to write a simple if/else when you're choosing between two values.

The pattern:

// Structure:
// question ? answer if yes : answer if no

age >= 18 ? 'Adult' : 'Minor'

The same thing written as if/else:

if (age >= 18) {
  label = 'Adult';
} else {
  label = 'Minor';
}

// Ternary version — same result, one line:
label = age >= 18 ? 'Adult' : 'Minor';

Three simple real-world uses:

// 1. Button text changes when you click it
// <button id="btn">Start</button>
btn.textContent = isRunning ? 'Stop' : 'Start';

// 2. Show a greeting based on the hour
// <p id="greeting"></p>
greeting.textContent = hour < 12 ? 'Good morning!' : 'Good afternoon!';

// 3. Add a pass or fail CSS class to a badge
// <span id="badge"></span>
badge.className = score >= 60 ? 'pass' : 'fail';
When NOT to use it: if you need to do more than one thing in each branch, use a regular if/else. The ternary is only for choosing one value, not running a block of code.

Live CSS

Live CSS Updates with Events

Range sliders connected to CSS custom properties via addEventListener('input') give users a live control panel for your design. Every element using var(--hue) updates instantly as the slider moves — no button click required.

const hueSlider = document.querySelector('#hue');
const satSlider = document.querySelector('#saturation');

hueSlider.addEventListener('input', function() {
  document.documentElement.style
    .setProperty('--hue', hueSlider.value);
});

satSlider.addEventListener('input', function() {
  document.documentElement.style
    .setProperty('--saturation', satSlider.value + '%');
});
// Color picker → accent color:
const picker = document.querySelector('#accent');
picker.addEventListener('input', function() {
  document.documentElement.style
    .setProperty('--accent', picker.value);
});
The real-world pattern: dark mode toggles, animation speed controls, font size sliders, spacing adjusters — any design parameter expressed in a CSS custom property can be given to the user as a live control.

Best Practices

Best Practices

Always use event.preventDefault()
on form submissions

Without it, the page reloads and all your JavaScript state is lost — every variable, every DOM change, everything. One line prevents this permanently.

form.addEventListener('submit', function(e) {
  e.preventDefault(); // Always first!
  // then read values...
});

Use input for live updates,
change for final selections

input fires continuously as the user types or drags — use this for live previews and character counters. change fires once when they finish — better for dropdowns and checkboxes where intermediate states don't matter.

EventWhen it firesBest for
inputEvery keystroke/dragLive previews
changeAfter user is doneDropdowns, checkboxes

Keep handlers short — extract logic

If your handler function grows long, extract the logic into a named function and call it from the handler. This makes the code readable, testable, and reusable.

// ✅ Named function = reusable
function validateForm() { /* logic */ }

form.addEventListener('submit', function(e) {
  e.preventDefault();
  validateForm(); // short & clear
});
Always use === not ==. Always wrap range/number input values in Number(). Always call event.preventDefault() before reading form values. These three habits prevent the most common beginner bugs.
