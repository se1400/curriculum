The big shift
Before this lesson, every JavaScript change happened once — when the page loaded. The variables ran, the DOM updated, and then JavaScript went silent.

Events are how JavaScript listens to the user. Every click, keystroke, slider drag, and form submission — the browser tracks all of it. Your job is to tell JavaScript which events to listen for and what to do when they happen.
The visual payoff is immediate: range sliders that shift your entire color scheme as you drag them, password fields with live strength bars, forms that validate as you type. All of it runs on two ideas: listen for an event → decide what to do.
addEventListener

addEventListener — The Right Way

❌ Old way — logic lives in HTML:

<!-- onclick mixes JS into HTML -->
<button onclick="doSomething()">Click me</button>

// Limitation: only ONE handler per element.
// Second onclick= overwrites the first.

✅ Modern way — keep JS in your JS:

const btn = document.querySelector('#submit-btn');

btn.addEventListener('click', function() {
  console.log('Button clicked!');
});
//            ↑ event type   ↑ handler function
Read it out loud: "Find the element with id submit-btn. When a click event occurs on it, run this function." The function is called a handler. It runs every time the event fires — not just once.
addEventListener stacks. You can attach multiple listeners to the same element — each one runs. The old onclick= attribute can only hold one handler.

The 4 Events You'll Use Most — Part 2

change

Fires once after the user finishes interacting — leaves a field or makes a final selection. Best for dropdowns and checkboxes.

const select = document.querySelector('#font-select');
select.addEventListener('change', function() {
  document.body.style.fontFamily = select.value;
});

submit

Fires when a form is submitted — button click or Enter. Always pair with event.preventDefault().

const form = document.querySelector('#signup');
form.addEventListener('submit', function(event) {
  event.preventDefault(); // Stop page reload!
  // Now read values and do something with them
});

Fill in your name and submit — notice the page doesn't reload
Without event.preventDefault(): a form submission reloads the page — the default browser behavior since 1995. Every variable, DOM change, and data you had in memory is gone. One line prevents this.
Event Object

The Event Object

When an event fires, the browser automatically passes an event object to your handler. Add a parameter to your function — conventionally named event or e:

btn.addEventListener('click', function(event) {
  console.log(event.type);    // 'click'
  console.log(event.target);  // the <button> element
});

event.type

The name of the event: 'click', 'input', 'submit'. Useful when one handler handles multiple event types.

event.target

The exact element the user interacted with. No need to querySelector for it again.

event.target.classList
  .toggle('active');

event.target.value

For any form input, gives you the current value at the moment the event fired.

const query =
  event.target.value;
event.preventDefault() is also a method on the event object — that's why it has to be called on event.

Form Values

Reading Form Values — Text, Number, Color

Text, Email, Password, Search

const nameInput = document.querySelector('#name');
const value = nameInput.value; // Returns a string

Number and Range (always convert!)

const slider = document.querySelector('#volume');

// ❌ Don't do this for math:
slider.value         // → "75" (a string!)

// ✅ Wrap in Number() first:
Number(slider.value) // → 75 (a number)
.value is ALWAYS a string — even for type="number" or type="range". Without Number(), "5" + 3 gives you "53", not 8. This is one of the most common beginner bugs in JavaScript.

Color Picker

const picker = document.querySelector('#accent');
const hex = picker.value; // → '#6c5ce7'

Decisions

Making Decisions: if / else if / else

Events tell JavaScript when something happened. Conditional statements tell it what to do about it.

const score = Number(input.value);

if (score >= 90) {
  label.textContent = 'Excellent';
  label.className = 'badge badge--green';
} else if (score >= 70) {
  label.textContent = 'Good';
} else if (score >= 50) {
  label.textContent = 'Pass';
} else {
  label.textContent = 'Not yet';
  // catch-all — runs if nothing above matched
}
Order matters. JavaScript evaluates top to bottom and stops at the first condition that is true. Put the most specific (highest value) conditions first, the most general last.
Why use if/else? Without decisions, your code can only do one thing. With if/else, the same function behaves differently depending on what the user typed, what value a slider has, or whether a box is checked. This is the foundation of every validation, grade calculator, quiz, and live preview you'll build.

Comparison Operators

These operators compare two values and return true or false:

OperatorMeaningExample → Result

Always use ===, never ==.
Double equals (==) performs type coercion:

Decisions

Logical Operators — &&, ||, !

Combine multiple conditions into a single expression:

// AND — both must be true
if (password.length >= 8 && password.includes('@')) { }

// OR — at least one must be true
if (role === 'admin' || role === 'editor') { }

// NOT — flips true to false and vice versa
if (!isLoggedIn) { redirectToLogin(); }

&& (AND)

Both sides must be true. Like a double lock — both keys needed.

|| (OR)

At least one must be true. Like a backup key — either one works.

! (NOT)

Flips the result. !true → false. !false → true. Reads as "if not".

Truthy & Falsy

Truthy and Falsy

In JavaScript, every value is either truthy or falsy in a condition. You don't always need an explicit comparison.

These 6 values are falsy — everything else is truthy:

false 0 "" null undefined NaN

Why it matters — shorter validation code:

// These two are identical:
if (nameInput.value === '') { showError(); }
if (!nameInput.value)       { showError(); }
// Empty string is falsy → !falsy = true

// Check if a number is non-zero:
if (score)  { /* runs if score is not 0 */ }
if (!score) { /* runs if score IS 0     */ }
Quick guide: empty = falsy, zero = falsy, anything else = truthy. Any non-empty string ("false", "0") is truthy because it's not empty. Any non-zero number (negative, decimal) is truthy.

Decisions

The Ternary Operator

The ternary operator is a compact one-line if/else. Use it when you're choosing between two values — not running two blocks of code.

// Syntax:
condition ? valueIfTrue : valueIfFalse

// Set button text based on state:
btn.textContent = isDark
  ? 'Switch to Light'
  : 'Switch to Dark';

// Choose a CSS class:
badge.className = score >= 60
  ? 'badge badge--pass'
  : 'badge badge--fail';

// Set an attribute:
icon.setAttribute('aria-label',
  isOpen ? 'Close menu' : 'Open menu');
When to use it: picking one of two values. If you need to run multiple lines of code in each branch, use a regular if/else. Nested ternaries become unreadable fast.
Read it like a question: "Is the condition true? Then use the first value. Otherwise? Use the second."

Live CSS

Live CSS Updates with Events

Range sliders connected to CSS custom properties via addEventListener('input') give users a live control panel for your design. Every element using var(--hue) updates instantly as the slider moves — no button click required.

const hueSlider = document.querySelector('#hue');
const satSlider = document.querySelector('#saturation');

hueSlider.addEventListener('input', function() {
  document.documentElement.style
    .setProperty('--hue', hueSlider.value);
});

satSlider.addEventListener('input', function() {
  document.documentElement.style
    .setProperty('--saturation', satSlider.value + '%');
});
// Color picker → accent color:
const picker = document.querySelector('#accent');
picker.addEventListener('input', function() {
  document.documentElement.style
    .setProperty('--accent', picker.value);
});
The real-world pattern: dark mode toggles, animation speed controls, font size sliders, spacing adjusters — any design parameter expressed in a CSS custom property can be given to the user as a live control.

Best Practices

Best Practices

Always use event.preventDefault()
on form submissions

Without it, the page reloads and all your JavaScript state is lost — every variable, every DOM change, everything. One line prevents this permanently.

form.addEventListener('submit', function(e) {
  e.preventDefault(); // Always first!
  // then read values...
});

Use input for live updates,
change for final selections

input fires continuously as the user types or drags — use this for live previews and character counters. change fires once when they finish — better for dropdowns and checkboxes where intermediate states don't matter.

EventWhen it firesBest for
inputEvery keystroke/dragLive previews
changeAfter user is doneDropdowns, checkboxes

Keep handlers short — extract logic

If your handler function grows long, extract the logic into a named function and call it from the handler. This makes the code readable, testable, and reusable.

// ✅ Named function = reusable
function validateForm() { /* logic */ }

form.addEventListener('submit', function(e) {
  e.preventDefault();
  validateForm(); // short & clear
});
Always use === not ==. Always wrap range/number input values in Number(). Always call event.preventDefault() before reading form values. These three habits prevent the most common beginner bugs.
