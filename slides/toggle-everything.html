Core Concept

The Central Principle

CSS does all the visual work. JavaScript just flips a switch.

Think of it like a light switch on a wall. The wiring (CSS) decides what happens when the switch is flipped — lights turn on, fan spins, whatever. JavaScript is just the finger that flips the switch. It doesn't know or care about wiring. It just toggles the switch between ON and OFF.

1. HTML — The structure

<button id="toggle-btn">Highlight</button>
<div class="box">Click the button to highlight me!</div>

2. CSS — Two visual states

/* State 1: Default look */
.box {
  background: #f5f5f5;
  color: #333;
  padding: 1rem;
  transition: all 0.3s;
}

/* State 2: Highlighted look */
.box.highlight {
  background: #a78bfa;
  color: #fff;
}

CSS defines what the box looks like normally, and what it looks like when it has the .highlight class. The transition makes the change smooth.

3. JS — Flip the switch

const btn = document.querySelector('#toggle-btn');
const box = document.querySelector('.box');

btn.addEventListener('click', function() {
  box.classList.toggle('highlight');
});

That's the whole thing. JavaScript selects the element, listens for a click, and toggles the class. CSS handles the color change and the animation.

Why separate them? CSS handles how things look. JavaScript handles when things change. Keeping them separate makes your code cleaner and easier to debug.

Core Concept

One Pattern Powers Everything

This exact same pattern — CSS defines two states, JS toggles the class — powers almost every interactive element you've ever seen:

🌙 Dark Mode

Toggle .dark-mode on body — CSS custom properties change every color at once

📂 Accordion

Toggle .open on a panel — max-height transitions reveal content smoothly

☰ Hamburger Menu

Toggle .nav-open on nav — slides in from the side on mobile

🗂 Tabs

Toggle .active on tab + panel — one visible at a time

💬 Modal Dialog

Toggle .open on overlay — fades in over the page

🔔 Toast Notification

Toggle .visible on toast — slides up, auto-dismisses with setTimeout

Learn this pattern once → build everything.

classList Methods

These are the four methods you'll use to manage CSS classes from JavaScript:

MethodWhat it doesExample
.toggle('name') Adds the class if missing, removes it if present el.classList.toggle('open')
.add('name') Adds the class (no effect if already there) el.classList.add('active')
.remove('name') Removes the class (no effect if not there) el.classList.remove('open')
.contains('name') Returns true or false el.classList.contains('dark-mode')

Dark Mode Toggle

Dark Mode — The CSS Side

Analogy: Think of smart bulbs in a house. Every bulb is set to "use the current scene." Right now the scene is "Daytime" — bright white lights. When you tap "Night Mode" on one switch, every smart bulb in every room changes at once. You didn't rewire anything — you just changed what "the current scene" means. That's exactly how CSS custom properties work.

HTML — A button to toggle

<button id="theme-toggle">☾ Dark Mode</button>

CSS — Two color palettes

/* Light mode (default) */
:root {
  --bg:   #ffffff;
  --text: #111111;
  --card: #f5f5f5;
}

/* Dark mode (toggled by JS) */
body.dark-mode {
  --bg:   #1a1a2e;
  --text: #eeeeee;
  --card: #16213e;
}

:root sets the default palette. body.dark-mode overrides it. Because body is more specific when the class is present, the dark values win.

CSS — Elements use the palette

body {
  background: var(--bg);
  color: var(--text);
  transition: background 0.3s,
              color 0.3s;
}

.card {
  background: var(--card);
}
Key insight: Every element uses var(--bg), var(--text), etc. When JS adds .dark-mode to <body>, those variables change — and every element using them updates automatically.
Dark Mode Toggle

Dark Mode — The JavaScript

Here's the complete JavaScript. It selects the button, listens for a click, toggles the class on <body>, and updates the button label:

// 1. Select the button from the HTML
const btn = document.querySelector('#theme-toggle');

// 2. Listen for clicks
btn.addEventListener('click', function() {

  // 3. Toggle the class on <body> — this swaps the CSS palette
  document.body.classList.toggle('dark-mode');

  // 4. Check which state we're in now
  const isDark = document.body.classList.contains('dark-mode');

  // 5. Update the button label to match
  btn.textContent = isDark ? '☀ Light Mode' : '☾ Dark Mode';
});

What's the ? and : syntax?

That's the ternary operator — a one-line shortcut for choosing between two values:

condition ? valueIfTrue : valueIfFalse

Read it like a question: "Is it dark? If yes, use '☀ Light Mode'. If no, use '☾ Dark Mode'."

It does the same thing as an if/else, just in one line instead of four.

How it all connects: The HTML provides the <button>. The CSS defines two color palettes (:root and body.dark-mode). JavaScript listens for the click, toggles the class, and every element using those CSS variables updates automatically. Three languages, one feature.

Accordion

Accordion — Expand & Collapse

An accordion is a list of sections where clicking a heading reveals or hides its content.

Analogy: Think of an elevator. Only one floor's doors can be open at a time. When the doors open on floor 3, the doors on every other floor are closed. You press a button (click a header), the current floor closes, and the new one opens — with a smooth sliding animation, not a sudden jump.

Why use an accordion?

  • Reduce visual clutter — show content only when needed
  • FAQ pages, settings panels, product features
  • Used on nearly every website
  • The max-height trick makes it animate smoothly

The max-height trick

You can't transition height: auto. But you can transition max-height from 0 to a value larger than the content. The browser animates the change smoothly.

Why not display: none? Because display can't be transitioned. The element just pops in/out with no animation. max-height: 0 + overflow: hidden gives you a smooth reveal.
Accordion

Accordion — The Code

HTML — The structure

<div class="accordion-item">
  <button class="accordion-header">What is classList?</button>
  <div class="accordion-body">
    <p>It's a property that lets you add, remove, and toggle CSS classes.</p>
  </div>
</div>
<!-- Repeat for each section -->

CSS — Two states

/* Default: body is hidden */
.accordion-body {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.35s ease;
}

/* When parent has .open, reveal */
.accordion-item.open .accordion-body {
  max-height: 400px;
}

The .open class goes on the parent .accordion-item. The CSS descendant selector then targets the .accordion-body inside it.

JavaScript — Toggle + close others

const headers = document
  .querySelectorAll('.accordion-header');

headers.forEach(function(header) {
  header.addEventListener('click', function() {
    // parentElement = the element that wraps this one
    // The button is inside .accordion-item,
    // so header.parentElement gives us that div
    const item = header.parentElement;

    // Close all others first
    document.querySelectorAll('.accordion-item')
      .forEach(function(el) {
        if (el !== item)
          el.classList.remove('open');
      });

    // Toggle the clicked one
    item.classList.toggle('open');
  });
});
How it connects: HTML provides the header/body structure. CSS hides the body by default (max-height: 0) and reveals it when the parent has .open. JavaScript listens for clicks on each header, closes the other items, and toggles the clicked one. The CSS transition handles the smooth animation.

Hamburger Menu

Real Hamburger Menu

Replace the CSS-only checkbox hack with clean, readable JavaScript.

Analogy: A hamburger menu is a sliding door on a closet. The links are always there behind the door — you just can't see them. Clicking the ☰ button slides the door open. Clicking ✕ slides it closed. CSS controls the sliding animation. JavaScript just decides when to open and close.

The old way (CSS-only hack)

  • Hidden checkbox + :checked pseudo-class
  • Uses the sibling selector (~) to show/hide nav
  • Clever but hard to read and maintain
  • No way to update aria-expanded

The right way (JavaScript)

  • Toggle a class on the nav element
  • CSS handles the slide-in animation
  • JavaScript updates button label and aria-expanded
  • Cleaner, more readable, accessible
Same visual result. Better code. Better accessibility. The checkbox hack was a creative workaround. Now that you know JavaScript, you can do it properly.
Hamburger Menu

Hamburger Menu — The Code

HTML — The button and nav

<button id="menu-btn" aria-expanded="false">☰</button>

<nav class="nav-links">
  <a href="#home">Home</a>
  <a href="#about">About</a>
  <a href="#contact">Contact</a>
</nav>

CSS — slide from the right

/* Default: nav is off-screen */
.nav-links {
  position: fixed;
  top: 0;
  right: -100%;
  width: 75%;
  height: 100vh;
  background: var(--card);
  transition: right 0.3s ease;
  padding-top: 5rem;
}

/* When .nav-open is added, slide in */
.nav-links.nav-open {
  right: 0;
}

The nav starts at right: -100% (completely off-screen to the right). Adding .nav-open moves it to right: 0. The transition animates the slide.

JavaScript — toggle + accessibility

const menuBtn = document
  .querySelector('#menu-btn');
const navLinks = document
  .querySelector('.nav-links');

menuBtn.addEventListener('click',
  function() {
    // Toggle the nav visibility
    navLinks.classList.toggle('nav-open');

    // Check the new state
    const isOpen = navLinks
      .classList.contains('nav-open');

    // Update accessibility
    menuBtn.setAttribute(
      'aria-expanded', isOpen);

    // Swap icon: ✕ when open, ☰ when closed
    menuBtn.textContent =
      isOpen ? '✕' : '☰';
});

Tabs

Tab System

Shows one content panel at a time. Clicking a tab activates it and hides the rest.

Analogy: Think of tabs like a stack of papers in a folder with labeled dividers. Only one section is "on top" (visible) at a time. When you click a divider tab, that section comes to the front, and all others go behind it.

HTML — Tabs + panels connected by data-tab

<!-- Tab buttons — data-tab stores the ID of the panel each controls -->
<button class="tab active" data-tab="overview">Overview</button>
<button class="tab" data-tab="features">Features</button>
<button class="tab" data-tab="pricing">Pricing</button>

<!-- Content panels — IDs match the data-tab values above -->
<div id="overview" class="panel active">Overview content...</div>
<div id="features" class="panel">Features content...</div>
<div id="pricing" class="panel">Pricing content...</div>

CSS — Only .active panels show

/* Panels hidden by default */
.panel { display: none; }
.panel.active { display: block; }

/* Active tab gets highlight */
.tab.active {
  background: var(--accent);
  color: #fff;
}

JavaScript — Remove all, add to clicked

const tabs = document.querySelectorAll('.tab');
const panels = document.querySelectorAll('.panel');

tabs.forEach(function(tab) {
  tab.addEventListener('click', function() {
    // Deactivate all
    tabs.forEach(function(t) {
      t.classList.remove('active');
    });
    panels.forEach(function(p) {
      p.classList.remove('active');
    });

    // Activate clicked tab
    tab.classList.add('active');

    // Find matching panel via data-tab
    const targetId = tab.dataset.tab;
    document.querySelector('#' + targetId)
      .classList.add('active');
  });
});
How '#' + targetId works — step by step:
Say the user clicks the "Features" tab. That button has data-tab="features".
1. tab.dataset.tab reads the value → "features"
2. '#' + targetId glues a # in front → "#features"
3. querySelector('#features') finds the panel with id="features"
4. .classList.add('active') shows that panel
The data-tab value and the panel id must match — that's how the tab knows which panel to show.

Modal Dialog

Modal Dialog

An overlay that appears on top of everything, requiring the user's attention before continuing.

Analogy: Think of a pop-up notification on your phone that dims the screen behind it. You can see your app is still there, but you can't tap anything until you deal with the notification — either respond to it or dismiss it. A modal works the same way on a website.

When to use a modal

  • Confirmation dialogs ("Are you sure?")
  • Login/signup forms
  • Image lightboxes
  • Cookie consent banners

Three ways to close it

  • Click the ✕ button
  • Click the dark backdrop
  • Press the Escape key

Why opacity instead of display: none?

display: none removes the element from layout instantly — there is nothing to transition. The element just pops in/out.
opacity: 0 + pointer-events: none keeps the element present but invisible and non-interactive. Now you can transition opacity for a smooth fade in/out.
Modal Dialog

Modal — The Code

HTML — The overlay wraps the dialog box

<button id="open-btn">Open Modal</button>

<div class="modal-overlay">           <!-- dark backdrop -->
  <div class="modal-box">              <!-- white dialog -->
    <h2>Are you sure?</h2>
    <p>This action cannot be undone.</p>
    <button id="close-btn">Close</button>
  </div>
</div>

CSS — hidden by default

/* Overlay covers the whole screen */
.modal-overlay {
  position: fixed;
  inset: 0;
  background: rgba(0,0,0,0.6);
  display: flex;
  align-items: center;
  justify-content: center;
  opacity: 0;
  pointer-events: none;
  transition: opacity 0.2s;
}

/* When .open is added, fade in */
.modal-overlay.open {
  opacity: 1;
  pointer-events: auto;
}

opacity: 0 makes it invisible. pointer-events: none makes it non-clickable. Both are restored when .open is added.

JavaScript — open, close, backdrop

const overlay = document
  .querySelector('.modal-overlay');

function openModal() {
  overlay.classList.add('open');
}
function closeModal() {
  overlay.classList.remove('open');
}

// Open button
document.querySelector('#open-btn')
  .addEventListener('click', openModal);

// Close button
document.querySelector('#close-btn')
  .addEventListener('click', closeModal);

// Close on backdrop click
overlay.addEventListener('click',
  function(event) {
    if (event.target === overlay)
      closeModal();
  });
Why event.target === overlay? The dialog box sits inside the overlay. So when you click the dialog, that click also counts as clicking the overlay (because the dialog is part of it). Without this check, clicking the dialog would accidentally close the modal. The if statement says: "only close if the user clicked the dark background itself — not anything inside it."

Timing

setTimeout — Delayed Actions

Schedule a function to run once after a delay (in milliseconds).

Analogy: setTimeout is like setting a kitchen timer. You set it for 3 minutes, walk away, and when it rings, it does the thing — once. It doesn't keep ringing every 3 minutes. One delay, one action.

HTML — A toast element

<button id="save-btn">Save</button>

<div id="toast" class="toast">
  Saved successfully!
</div>

CSS — Two states for the toast

.toast {
  opacity: 0;
  transform: translateY(10px);
  transition: all 0.3s;
}
.toast.visible {
  opacity: 1;
  transform: translateY(0);
}

JavaScript — Show, then auto-dismiss

const toast = document.querySelector('#toast');
const saveBtn = document.querySelector('#save-btn');

saveBtn.addEventListener('click', function() {
  // Show the toast immediately
  toast.classList.add('visible');

  // Hide it after 3 seconds
  setTimeout(function() {
    toast.classList.remove('visible');
  }, 3000);
});

Cancel with clearTimeout

const id = setTimeout(myFunc, 5000);
// User dismisses manually — cancel timer
clearTimeout(id);

Timing

setInterval — Repeated Actions

Schedule a function to run repeatedly at a fixed interval.

Analogy: If setTimeout is a kitchen timer (rings once), setInterval is an alarm clock that goes off every morning until you smash it. It keeps running on repeat. And just like a real alarm — if you forget to turn it off, it will keep going forever.

HTML — A clock display

<div id="clock">--:--:--</div>
<button id="start">Start</button>
<button id="stop">Stop</button>

Always cancel when done

clearInterval(clockId);
Memory leak warning: A running interval that is never cancelled keeps executing even after the user navigates away. Every setInterval must have a matching clearInterval.

JavaScript — Live clock

const clock = document.querySelector('#clock');
let clockId = null;

function updateClock() {
  const now = new Date();  // current date & time
  const h = String(now.getHours())
    .padStart(2, '0');   // 9 → "09"
  const m = String(now.getMinutes())
    .padStart(2, '0');   // 5 → "05"
  const s = String(now.getSeconds())
    .padStart(2, '0');   // 3 → "03"
  clock.textContent = h + ':' + m + ':' + s;
}
// padStart(2, '0') means: "make sure this string
// is at least 2 characters. If it's shorter, add
// '0' to the front." So 9 becomes "09", but 12
// stays "12" because it's already 2 digits.

// Start: run updateClock every 1 second
document.querySelector('#start')
  .addEventListener('click', function() {
    updateClock();
    clockId = setInterval(updateClock, 1000);
  });

// Stop: cancel the interval
document.querySelector('#stop')
  .addEventListener('click', function() {
    clearInterval(clockId);
  });

Accessibility

aria-expanded — Accessible Toggles

One line of JavaScript makes your interactive components usable by everyone.

Analogy: Imagine you're in a dark room and a friend is describing a website to you. They say "there's a button that says FAQ." You click it — but did anything happen? Without aria-expanded, your friend doesn't know. With it, they can say "FAQ section — expanded" or "FAQ section — collapsed." That's what screen readers do for visually impaired users.

HTML — The starting point

<button
  aria-expanded="false"
  aria-controls="faq-panel">
  FAQ
</button>

<div id="faq-panel" class="panel">
  <p>Frequently asked questions...</p>
</div>

aria-expanded="false" tells screen readers the panel is currently collapsed. aria-controls tells them which element this button controls.

Where to use it

  • Accordion headers
  • Hamburger menu buttons
  • Dropdown toggles
  • Any button that shows/hides content

JavaScript — Update after toggling

const btn = document.querySelector('button');
const panel = document.querySelector('.panel');

btn.addEventListener('click', function() {
  // Toggle the visual state
  panel.classList.toggle('open');

  // Update accessibility to match
  const isOpen = panel
    .classList.contains('open');
  btn.setAttribute(
    'aria-expanded', isOpen
  );
});

After toggling the class, check the new state with classList.contains() and update aria-expanded to match. The screen reader now announces the correct state.

Tiny effort, huge impact. Not everyone uses a mouse. Users with visual impairments rely on screen readers to navigate. One attribute makes your components work for them.

Best Practices

Best Practices

Let CSS do the animating
Define both states in CSS with a transition. JavaScript only adds or removes the class. Never write animation timings in JavaScript.
Toggle aria-expanded on interactive components
Buttons that show/hide content need aria-expanded="true" when open and "false" when closed. One line makes your components accessible.
Always clear intervals you no longer need
Every setInterval should have a corresponding clearInterval. Forgetting this is a common source of subtle bugs and memory leaks.
Use opacity + pointer-events for animated show/hide
display: none makes transitions impossible. Use opacity: 0; pointer-events: none; as the hidden state.
Keep modal close logic in one function
Multiple triggers close the same modal (✕ button, backdrop click, Escape key). One closeModal() function called by all three is cleaner than duplicating logic.
