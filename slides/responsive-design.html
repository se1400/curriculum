The Problem

Fixed-Width Layouts Break on Mobile

In the early 2000s, most sites were built at 800px or 960px wide — because that's what desktop monitors were.

  • When smartphones arrived, fixed sites overflowed the screen
  • Users had to pinch-zoom and pan horizontally
  • The "solution" was two separate sites: www. and m.
  • Two codebases, two content sets, fragile user-agent detection
Ethan Marcotte coined "responsive web design" in 2010.
Three foundations: fluid grids, flexible images, CSS media queries.

Fixed-width site vs responsive site on a mobile viewport

Mobile devices account for over 50% of all global web traffic (Statcounter). Designing only for desktop is a failure.
Philosophy

Mobile-First Design

Design and code for the smallest screen first, then progressively add layout complexity for larger screens.

📋 Content Priority

Limited screen space forces you to identify what matters most. The most important content comes first — on all screens.

⚡ Performance

Mobile users are often on slower networks. Start with a minimal, fast baseline. Add enhancements for larger, faster screens.

🎨 CSS Cascade Efficiency

Adding styles for larger screens with min-width is simpler than stripping them back with max-width. Less to override.

✓ MOBILE-FIRST (recommended) 
.nav { flex-direction: column; } /* mobile */

@media (min-width: 640px) {
  .nav { flex-direction: row; }  /* wider */
}
✗ DESKTOP-FIRST (avoid) 
.nav { flex-direction: row; }   /* desktop */

@media (max-width: 639px) {
  .nav { flex-direction: column; } /* narrow */
}
Mobile-first does not mean ignoring desktop — it means building the foundation for mobile, then enhancing.
The Foundation

The Viewport Meta Tag

Before writing a single line of responsive CSS, this must be in your <head>:

<meta name="viewport" content="width=device-width, initial-scale=1">

Without it:

  • Mobile browsers apply a virtual viewport of ~980px
  • The browser then zooms out to fit 980px onto a 375px screen
  • Your media queries appear to fire too late or not at all
  • Text is tiny; users must pinch-zoom to read anything

With it:

  • width=device-width — viewport width = actual screen width
  • initial-scale=1 — start at 1:1 zoom, no scaling
  • Media queries fire at the right viewport widths
  • Text is readable at its natural size
Never use maximum-scale=1, user-scalable=no. Preventing zoom is an accessibility barrier. WCAG 1.4.4 requires text can be resized to 200%. Users with low vision depend on zoom.
CSS Media Queries

Basic Media Query Syntax

A media query is a conditional block of CSS: if the condition is true, the styles inside apply.

/* Applies to ALL screen sizes (default) */
.container {
  padding: 1rem;
}

/* Only when viewport is ≥ 640px */
@media (min-width: 640px) {
  .container {
    padding: 2rem;
  }
}

/* Only when viewport is ≥ 1024px */
@media (min-width: 1024px) {
  .container {
    padding: 3rem;
    max-width: 1200px;
    margin-inline: auto;
  }
}

Media types

  • @media screen — screens (default)
  • @media print — print preview
  • @media all — both (same as no type)

Common feature queries

  • (min-width: 640px) — viewport width
  • (prefers-color-scheme: dark)
  • (prefers-reduced-motion: reduce)
  • (orientation: landscape)
  • (hover: hover) — pointer device
The @media keyword is required. Parentheses wrap the condition. Curly braces contain the scoped CSS.
CSS Media Queries

min-width vs max-width

✓ min-width (mobile-first) "Apply these styles when the screen is at least this wide." Styles stack upward.
↓ max-width (desktop-first) "Apply these styles when the screen is at most this wide." Styles override downward. 
/* Mobile-first: single column → 2 col → 3 col */
.card-grid {
  display: grid;
  grid-template-columns: 1fr;          /* mobile default */
  gap: 1rem;
}
@media (min-width: 640px) {
  .card-grid { grid-template-columns: repeat(2, 1fr); }
}
@media (min-width: 1024px) {
  .card-grid { grid-template-columns: repeat(3, 1fr); }
}
min-width (mobile-first)max-width (desktop-first)
Start pointMinimal mobile stylesFull desktop styles
DirectionAdd complexity upwardStrip styles downward
Override countFewer overrides neededMore overrides needed
Modern standard✓ YesAvoid in new projects
CSS Media Queries

Combining Conditions

/* Between 640px and 1023px only */
@media (min-width: 640px)
    and (max-width: 1023px) {
  .sidebar { display: none; }
}

/* Screen OR print */
@media screen, print {
  .page { max-width: 1200px; }
}

/* NOT a narrow screen */
@media not (max-width: 639px) {
  .desktop-only { display: block; }
}

/* Level 4 range syntax (modern) */
@media (640px <= width < 1024px) {
  .sidebar { display: none; }
}

Keywords

  • and — both conditions must be true
  • , (comma) — acts like or
  • not — negates the condition
  • only — hides from older browsers (legacy)

Level 4 Range Syntax

The modern specification allows direct comparison operators:

(640px <= width < 1024px)
(height > 500px)
(aspect-ratio >= 16/9)
Range syntax is supported in all modern browsers since 2023. Older projects still use and with two conditions.

CSS Media Queries

Choosing Breakpoints

Don't choose breakpoints based on device sizes ("375px for iPhone, 768px for iPad"). Device sizes change every product cycle.

Instead: let your content decide. Slowly resize your browser and add a breakpoint when the layout looks awkward.

Common starting framework:

  • Default — < 640px — single col, stacked nav
  • Medium — ≥ 640px — 2-col, horizontal nav
  • Large — ≥ 1024px — 3+ col, sidebar
  • XL — ≥ 1280px — wide, generous whitespace

Fluid Typography

Responsive Typography with clamp()

clamp(minimum, preferred, maximum) returns the preferred value clamped between min and max. Using vw as the preferred value makes fonts scale fluidly with the viewport.

/* Scales from 1.5rem to 3rem */
h1 {
  font-size: clamp(1.5rem, 4vw + 1rem, 3rem);
}

h2 {
  font-size: clamp(1.25rem, 3vw + 0.5rem, 2rem);
}

p {
  font-size: clamp(1rem, 1.5vw + 0.5rem, 1.25rem);
  line-height: clamp(1.5, 1.5 + 0.5vw, 1.8);
}
4vw + 1rem means "4% of viewport width plus 16px offset." The + rem offset prevents text from becoming too small at narrow widths.

Fluid Spacing

Fluid Spacing with clamp()

clamp() works on any property that accepts a length — not just font sizes. Use it for padding, gap, and margin to make the entire page breathe proportionally.

/* Padding scales 1rem → 3rem */
.section {
  padding-block:  clamp(1rem, 5vw, 3rem);
  padding-inline: clamp(1rem, 8vw, 6rem);
}

/* Consistent fluid gap for grids and stacks */
:root {
  --space-fluid: clamp(1rem, 3vw, 2rem);
}

.card-grid { gap: var(--space-fluid); }

/* Fluid max-width with centering */
.container {
  width: min(100% - 2rem, 1200px);
  margin-inline: auto;
}

The Formula

clamp(minimum, preferred, maximum)

  • minimum — floor: smallest allowed value
  • preferred — fluid middle: usually Xvw + Yrem
  • maximum — ceiling: largest allowed value

min() and max()

/* Fluid container: never wider than 1200px,
   never narrower than its content */
width: min(100% - 2rem, 1200px);

/* Font never smaller than 1rem */
font-size: max(1rem, 2vw);

/* These nest: same as clamp() */
font-size: max(1rem, min(4vw, 3rem));

Responsive Images

Responsive Images: The Baseline

Global reset (every project)

img {
  max-width: 100%;
  height: auto;
}

Prevents images from overflowing their container. Maintains aspect ratio when width is constrained.

object-fit

When you need an image to fill a fixed-size container (card thumbnail, avatar, hero), object-fit controls how it's sized and positioned:

.card-thumb {
  width: 100%;
  height: 200px;
  object-fit: cover; /* fills, crops if needed */
  object-position: top;
}
/* cover | contain | fill | none | scale-down */

Responsive Images

srcset and sizes

The srcset attribute lists multiple versions of the same image at different resolutions. The browser picks the most appropriate file based on screen density and display width — saving bandwidth on mobile.

<img
  src="hero-800.jpg"
  srcset="hero-400.jpg  400w,
          hero-800.jpg  800w,
          hero-1600.jpg 1600w"
  sizes="(max-width: 640px)  100vw,
         (max-width: 1024px)  50vw,
         800px"
  alt="A panoramic mountain landscape"
  width="800"
  height="450"
  loading="lazy"
>

srcset descriptor

  • 400w — image is 400px wide
  • Browser chooses based on: available display width × device pixel ratio

sizes attribute

  • Tells the browser how wide the image will display at each breakpoint
  • Last value is the default (no media condition)
  • Browser uses this + device pixel ratio to pick the right file
Always include width and height attributes — they prevent layout shift (CLS) by reserving space before the image loads. Always add loading="lazy" for images below the fold.
Responsive Images

Art Direction with <picture>

The <picture> element lets you swap out a completely different image based on media conditions — not just a different resolution of the same image. Use it when a tightly cropped portrait works better on mobile than the wide landscape used on desktop.

<picture>
  <!-- WebP on supporting browsers -->
  <source
    type="image/webp"
    srcset="hero-wide.webp 1200w,
            hero-narrow.webp 600w"
    sizes="(max-width: 640px) 100vw, 1200px"
  >
  <!-- Portrait crop on mobile -->
  <source
    media="(max-width: 640px)"
    srcset="hero-portrait.jpg"
  >
  <!-- Fallback: wide landscape -->
  <img
    src="hero-wide.jpg"
    alt="A mountain panorama at sunrise"
    width="1200"
    height="630"
    loading="lazy"
  >
</picture>

How it works

  • Browser checks <source> elements in order — uses the first matching one
  • The <img> is the required fallback and carries alt, width, height
  • type="image/webp" — format switching (serve WebP to supporting browsers)
  • media="(max-width:640px)" — art direction switching

max-width:100%Basic overflow prevention
srcset + sizesSame image, multiple resolutions
<picture>Different image at different sizes / formats
Layout Patterns

Responsive Layout: Stack-to-Row

The most common pattern: items stack vertically on small screens and line up horizontally on wider screens.

/* Stack-to-row navigation */
.nav {
  display: flex;
  flex-direction: column;  /* stacked on mobile */
  gap: 0.5rem;
}

@media (min-width: 640px) {
  .nav {
    flex-direction: row;   /* horizontal on tablet+ */
    gap: 1.5rem;
  }
}

/* Holy grail layout */
.page {
  display: grid;
  grid-template-rows:
    auto 1fr auto;     /* header / main / footer */
}

@media (min-width: 1024px) {
  .page {
    grid-template-columns: 240px 1fr;
    /* sidebar + main */
  }
}

Pattern family

Stack → Row
Column on mobile, row on wider screens. Flexbox.
Sidebar
Full-width stacked on mobile, sidebar+main on desktop. Grid.
Card Grid
1 col → 2 col → 3 col. Grid with breakpoints or auto-fill.
Off-Canvas
Nav slides in from the side on mobile. Transform + toggle.
Flexbox for 1D layouts (navigation, toolbars, button groups). Grid for 2D layouts (page structure, card grids, form fields).
Layout Patterns

Intrinsic Grid — Zero Media Queries

CSS Grid's auto-fill with minmax() creates a truly responsive layout that adds and removes columns automatically — with no media queries at all.

/* The "RAM" pattern:
   Repeat, Auto, Minmax */
.card-grid {
  display: grid;
  grid-template-columns:
    repeat(auto-fill, minmax(280px, 1fr));
  gap: 1.5rem;
}

/*
  This grid:
  - Creates as many ≥280px cols as will fit
  - Stretches all cols to equal widths (1fr)
  - Adds/removes cols as container resizes
  - Zero media queries!
*/

auto-fill vs auto-fit

auto-fill Keeps empty column tracks. Items stay at minimum width. Consistent column widths.
auto-fit Collapses empty tracks. Filled items expand to take all space. Fewer items = wider cards.
When to use intrinsic grid vs media queries:

  • Intrinsic grid — content that naturally adapts (card grids, image galleries, product listings)
  • Media queries — explicit layout pivots (sidebar appears, navigation changes, hero switches orientation)
/* Common starting values */
minmax(240px, 1fr)  /* compact cards */
minmax(280px, 1fr)  /* standard cards */
minmax(320px, 1fr)  /* content-rich cards */

/* Limit max columns */
grid-template-columns:
  repeat(auto-fill,
    minmax(min(280px, 100%), 1fr));

Modern CSS

The aspect-ratio Property

Responsive video embeds have always required a CSS hack — a wrapper with a calculated padding-bottom percentage. The aspect-ratio property replaces it cleanly.

/* Old padding-bottom hack (no longer needed) */
.video-wrapper-old {
  position: relative;
  padding-bottom: 56.25%; /* 16÷9 */
  height: 0;
}
.video-wrapper-old iframe {
  position: absolute; inset: 0;
  width: 100%; height: 100%;
}

/* Modern: aspect-ratio */
.video-wrapper {
  width: 100%;
  aspect-ratio: 16 / 9;
}
.video-wrapper iframe {
  width: 100%; height: 100%;
}

/* Card thumbnail */
.card-image {
  width: 100%;
  aspect-ratio: 4 / 3;
  object-fit: cover;
}

/* Perfect squares */
.avatar { aspect-ratio: 1; }
.icon-box { aspect-ratio: 1; width: 52px; }

Common ratios

ValueUse case
16 / 9Video, widescreen hero
4 / 3Card thumbnails, older video
1 / 1Avatars, icon boxes, squares
3 / 2Photography (DSLR native)
2 / 3Portrait orientation cards
21 / 9Ultra-wide cinematic hero
aspect-ratio can be combined with min-height or max-height for flexibility.
aspect-ratio: auto respects the intrinsic ratio of replaced elements (images, video). 
/* Fallback for old browsers */
.video {
  aspect-ratio: 16/9; /* modern */
}
/* Older browsers see: auto height */
Modern CSS

Dynamic Viewport Units

The old 100vh has a problem on mobile: the address bar and tab bar take up part of the screen and appear/disappear as the user scrolls. A 100vh hero is often taller than the visible area.

svh / svw — Small
Based on the smallest possible viewport (browser bars fully visible). Nothing will be cut off behind browser chrome. Safe for overlay content.
lvh / lvw — Large
Based on the largest possible viewport (bars fully retracted). Use when you want to fill the full screen after browser chrome hides.
dvh / dvw — Dynamic
Updates in real time as browser chrome appears or disappears. Best for full-screen layouts that should always fill the visual area.
/* Old approach — may be hidden behind mobile address bar */
.hero { min-height: 100vh; }

/* Better: dvh for full-screen sections */
.hero { min-height: 100dvh; }

/* Safe: svh guarantees nothing is cut off */
.modal-overlay { height: 100svh; }

/* With fallback for older browsers */
.hero {
  min-height: 100vh;    /* fallback */
  min-height: 100dvh;   /* overrides if supported */
}
Supported in Chrome 108+, Safari 15.4+, Firefox 101+. Always include a vh fallback for broader compatibility.
Container Queries

Container Queries — Concept

Media queries respond to the viewport size. But components live inside containers of varying widths — a card in a narrow sidebar has less space than the same card in a wide main column.

Container queries solve this: declare a container on a parent element, then write queries that respond to that container's width — not the viewport.

/* Step 1: Declare containment context */
.card-wrapper {
  container-type: inline-size;
}

/* Optional: name the container */
.sidebar {
  container-type: inline-size;
  container-name: sidebar;
}

/* Step 2: Query the container width */
.card { display: flex; flex-direction: column; }

@container (min-width: 400px) {
  .card { flex-direction: row; }
}

/* Named container query */
@container sidebar (min-width: 300px) {
  .widget { font-size: 1.1rem; }
}

Container Queries

Container Queries — Browser Support

Container query syntax

/* container-type values */
container-type: inline-size; /* tracks width */
container-type: size;         /* tracks width + height */
container-type: normal;       /* off (default) */

/* Shorthand: name + type */
.card-wrapper {
  container: card / inline-size;
}

/* Container query conditions */
@container (min-width: 400px) { }
@container (max-width: 600px) { }
@container (400px <= width < 700px) { }

/* Container units (cqi, cqb, cqw, cqh) */
.card-title {
  font-size: clamp(1rem, 5cqi, 1.5rem);
  /* 5% of the container's inline size */
}
Container units (cqi, cqb, cqw, cqh) are like vw/vh but relative to the nearest container — enabling truly encapsulated fluid components.

Container queries — supported in all modern browsers (Chrome 105+, Firefox 110+, Safari 16+)

Check current support: caniuse.com/css-container-queries

Media vs Container queries

  • Media queries — page-level layout (columns, navigation mode)
  • Container queries — component-level adaptation (card layout, widget density)

Responsive Navigation

The Hamburger Menu Pattern

<header>
  <a href="/" class="logo">Brand</a>
  <button class="menu-toggle"
    aria-expanded="false"
    aria-controls="nav-menu"
    aria-label="Toggle navigation">
    <span></span><span></span><span></span>
  </button>
  <nav id="nav-menu" class="nav-menu" hidden>
    <a href="/">Home</a>
    <a href="/about">About</a>
    <a href="/contact">Contact</a>
  </nav>
</header>
/* Desktop: always show */
@media (min-width: 768px) {
  .menu-toggle { display: none; }
  .nav-menu[hidden] {
    display: flex !important;
    flex-direction: row;
  }
}

// JavaScript toggle
toggle.addEventListener('click', () => {
  const exp =
    toggle.getAttribute('aria-expanded') === 'true';
  toggle.setAttribute('aria-expanded', !exp);
  menu.hidden = exp;
  menu.classList.toggle('is-open', !exp);
});

Testing Responsive Designs

Chrome/Edge DevTools Device Mode

Open DevTools (F12), click the device toolbar icon (or Ctrl+Shift+M / Cmd+Shift+M):

  • Select preset devices (iPhone, Pixel, iPad, Galaxy)
  • Enter custom dimensions manually
  • Simulate different device pixel ratios (1×, 2×, 3×)

Chrome DevTools Device Mode — Ctrl+Shift+M

Testing workflow

1. Resize your browser Drag the window narrow and watch the layout adapt. Test at 320px, 375px, 768px, 1024px, 1440px.
2. DevTools emulation Use device presets to check common phone/tablet sizes. Enable "responsive" mode for free-drag.
3. Real devices Test on at least one real iOS and one real Android device. Touch, font rendering, scroll behavior, and browser chrome differ from emulation.
DevTools emulation is a useful starting point — not a substitute for physical device testing.

Summary

What You Learned

Mobile-First Design for small screens first. Add complexity upward with min-width media queries. Less CSS to override.
Viewport Meta Tag Non-negotiable. Without it, mobile browsers render at 980px virtual width — your media queries won't work.
@media Queries Conditional CSS based on viewport width, color scheme, motion preference, and more. Mobile-first = min-width.
clamp() Fluid values between a minimum and maximum — no breakpoint jumps. Works on font sizes, spacing, and any length.
Responsive Images srcset + sizes for resolution switching. <picture> for art direction. object-fit for contained thumbnails.
Intrinsic Grid repeat(auto-fill, minmax()) — grid adds/removes columns as space changes. Zero media queries.
dvh / svh / lvh New dynamic viewport units that account for mobile browser chrome appearing and disappearing.
Container Queries @container responds to a component's parent width — not the viewport. Portable, context-aware components.
Q & A

Common Questions

What is the difference between responsive and adaptive design?
Responsive design uses a fluid, flexible layout that adjusts continuously to any screen size through CSS. Adaptive design uses multiple fixed layouts, each designed for a specific breakpoint — the server or client detects the device and serves the appropriate layout. Responsive is the industry standard because one codebase handles the full spectrum of sizes.
Should I use px or rem for breakpoints?
Many accessibility experts recommend rem: if a user's browser base font size is larger than 16px, px-based queries fire at the same viewport width regardless — potentially breaking the layout for users with enlarged text settings. rem breakpoints scale with the user's font size preference. In practice, px is used in most popular frameworks (Tailwind, Bootstrap). The impact is small for most sites, but rem is the more accessible choice.
What is the difference between auto-fill and auto-fit in CSS Grid?
Both create as many columns as will fit in the available space. The difference appears when there are fewer items than possible columns. auto-fill maintains the column tracks even if some are empty, keeping item widths consistent. auto-fit collapses empty tracks and stretches the filled items to take all available space. For most card layouts, auto-fill produces more predictable column widths.
Q & A

More Common Questions

When should I use container queries instead of media queries?
Use container queries for components that need to adapt to their context, not the viewport. A card used in both a narrow sidebar and a wide main column should use container queries so it adapts to available container width. Use media queries for page-level layout changes — shifting to two columns, showing or hiding the mobile navigation, adjusting the hero layout.
Is it okay to use display: none to hide content on mobile?
From a CSS perspective, yes — display: none removes the element from the rendering tree and it consumes no space. However, the HTML is still downloaded, and screen readers may or may not ignore it. If you need to hide content from all users including screen readers, pair display: none with aria-hidden="true" or the hidden attribute. Never hide content that is the only way to access important information — always provide an alternative.
Do I need to add a fallback for dvh?
Yes, for broader compatibility. Add the vh value first, then dvh on the next line — browsers that support dvh will apply it; older browsers use the vh fallback. Example: min-height: 100vh; min-height: 100dvh; Supported in Chrome 108+, Safari 15.4+, Firefox 101+.
