The Form Element

The <form> Element

The <form> container defines two critical things:

  • action — URL where data is sent on submit. Omit to submit to the current URL.
  • method — how data travels: get or post
<form action="/submit" method="post">
  <!-- inputs here -->
</form>
Rule of thumb: Use get for searches (bookmarkable URLs). Use post for sensitive data, large payloads, or anything that creates/modifies server data.
GETPOST
Data in URLData in request body
Bookmarkable & shareableNot visible in URL
Limited data sizeNo practical size limit
Cached by browserNot cached
Use for: searches, filtersUse for: login, signup, checkout

GET appends data as a query string: ?q=hello&page=2

Labels & Inputs

Every Input Must Have a Label

Why it matters:

  • 🔊 Screen readers announce the label when the input receives focus
  • 👆 Clicking the label focuses the input — larger tap target on mobile
  • 🤖 Autofill, password managers & browser autocomplete use labels to understand fields

Method 1 — for / id

<label for="email">Email Address</label>
<input type="email" id="email" name="email">

Method 2 — Wrap the input

<label>
  Email Address
  <input type="email" name="email">
</label>

Both are valid. for/id gives more layout flexibility; wrapping is more concise.

⚠️ Don't use placeholder as a label.
Placeholder text disappears when the user starts typing — they lose context about what the field is for. Use placeholder for example values like jane@example.com, not as the field label.
The name attribute is required for submission.
name becomes the key in submitted data: email=jane@example.com. An input without name is never included in the form submission.
id vs name
id — links to <label>, used by CSS/JS.
name — the key sent with the form data. You usually need both.
Input Types

Text-Type Inputs

All use <input type="...">. Each type unlocks keyboard layouts, validation, and autofill.

textGeneric single-line text. Use when no specific type applies.
emailValidates email shape. Mobile keyboard shows @ prominently.
passwordCharacters masked. Password managers auto-recognize this type.
urlValidates URL format. Mobile keyboard shows / and . shortcuts.
telTriggers numeric phone keypad on mobile. Use pattern for format validation.
searchLike text, but browsers may add a clear ✕ button. Semantic search signal.
numberShows spinner arrows. Use min, max, step to constrain values.
<input type="text"     name="username" placeholder="Your username">
<input type="email"    name="email"    placeholder="jane@example.com">
<input type="password" name="password" placeholder="At least 8 characters">
<input type="url"      name="website"  placeholder="https://example.com">
<input type="tel"      name="phone"    placeholder="(555) 555-5555">
<input type="number"   name="qty"      min="1" max="99">

Input Types

Checkboxes & Radio Buttons

☑ Checkboxes — zero or more

Each checkbox is independent. Multiple can be selected.

<label>
  <input type="checkbox" name="toppings"
         value="cheese"> Cheese
</label>
<label>
  <input type="checkbox" name="toppings"
         value="peppers" checked> Peppers
</label>
  • Same name sends multiple values
  • Add checked to pre-select

⭕ Radio Buttons — exactly one

All radios in a group share the same name. Selecting one deselects the others.

<label>
  <input type="radio" name="size"
         value="small"> Small
</label>
<label>
  <input type="radio" name="size"
         value="medium" checked> Medium
</label>
<label>
  <input type="radio" name="size"
         value="large"> Large
</label>
Once a radio is selected, the user can only pick another option — they can't deselect entirely. Always wrap radio groups in a <fieldset>.
Input Types

Range & Color Inputs

🎚 Range — number slider

Creates a drag slider. The slider doesn't display the current value — pair it with <output> and JavaScript.

<label for="vol">
  Volume: <output id="vol-out">50</output>
</label>
<input type="range" id="vol"
       name="volume"
       min="0" max="100"
       value="50" step="1">
// Wire the slider to the output
const slider = document.getElementById('vol');
const out = document.getElementById('vol-out');
slider.addEventListener('input', () => {
  out.textContent = slider.value;
});

🎨 Color — native color picker

Opens the OS color picker. Value must be a 6-digit hex code.

<label for="theme">Favorite Color</label>
<input type="color"
       id="theme" name="theme-color"
       value="#4f6df5">
Named colors like "red" are ignored — always use #rrggbb hex format for the default value.
accent-color CSS property — style checkboxes, radios, ranges and progress bars with one line:
accent-color: #4f6df5;

Form Controls

The <select> Element

Use a drop-down list when you have more than 4–5 options.

<label for="country">Country</label>
<select id="country" name="country">
  <option value="">-- Select a country --</option>
  <option value="us">United States</option>
  <option value="ca">Canada</option>
  <option value="uk">United Kingdom</option>
</select>
  • First <option value=""> acts as a prompt
  • The selected option's value is submitted under the select's name

Grouping with <optgroup>

<select name="country">
  <optgroup label="North America">
    <option value="us">United States</option>
    <option value="ca">Canada</option>
    <option value="mx">Mexico</option>
  </optgroup>
  <optgroup label="Europe">
    <option value="uk">United Kingdom</option>
    <option value="de">Germany</option>
  </optgroup>
</select>
Select vs Radio buttons:
≤ 5 options → radio buttons (all visible at once)
6+ options → <select> (saves space)
Form Controls

<textarea> & <datalist>

📝 Textarea — multi-line text

<label for="msg">Your Message</label>
<textarea id="msg" name="message"
          rows="5"
          placeholder="Write here...">
</textarea>
  • Has a closing tag (unlike <input>)
  • Text between tags = default value
  • rows sets initial height
  • Control resize with CSS: resize: vertical (recommended), resize: none, or resize: both

🔍 Datalist — autocomplete suggestions

Suggests options as the user types, but still allows free-text entry. Hybrid of <input> and <select>.

<input type="text" name="browser"
       list="browser-list"
       placeholder="Start typing...">

<datalist id="browser-list">
  <option value="Chrome">
  <option value="Firefox">
  <option value="Safari">
  <option value="Edge">
</datalist>
  • The list attribute must match the datalist's id
  • User can ignore suggestions and type anything
Grouping Controls

Grouping with <fieldset> & <legend>

Wrap related inputs in <fieldset> and label the group with <legend>.

<fieldset>
  <legend>Contact Preferences</legend>

  <label>
    <input type="radio" name="contact"
           value="email"> Email
  </label>
  <label>
    <input type="radio" name="contact"
           value="phone"> Phone
  </label>
  <label>
    <input type="radio" name="contact"
           value="text"> Text Message
  </label>

</fieldset>

Legend appears inside the fieldset border by default

Screen reader experience:
With fieldset + legend, a screen reader announces:
"Contact Preferences, Email, radio button, 1 of 3"

Without fieldset, it only announces:
"Email, radio button" — no group context!

Use <fieldset> whenever you have a group of radio buttons, or logically related sections like billing vs. shipping address.

Special Inputs

Hidden Fields, Buttons & Output

🙈 Hidden Inputs

Store data submitted with the form but not visible or editable by the user.

<input type="hidden"
  name="user_id"
  value="1042">
<input type="hidden"
  name="source"
  value="homepage">
Not secure — anyone can view page source. Use for reference values only, never secrets.

🔘 Button vs Input Submit

Prefer <button> — it can contain HTML markup.

<!-- Old approach -->
<input type="submit"
  value="Send">

<!-- Modern (preferred) -->
<button type="submit">
  Send
</button>

<!-- With icon -->
<button type="submit">
  ✉ Send Message
</button>
Always include type="submit" explicitly — it's the default but being explicit makes intent clear.

📊 The <output> Element

Displays a calculated result, often paired with a range or number input.

<label for="price">
  Price:
  $<output id="price-out">
    10
  </output>
</label>
<input type="range"
  id="price"
  min="10" max="100"
  step="5" value="10">

Wire with JavaScript — update output.textContent on the input's input event.

Validation

HTML5 Validation Attributes

Browsers enforce these rules before submitting — no JavaScript required.

AttributeWhat it does
requiredField must not be empty
minlength / maxlengthCharacter count limits for text
min / maxValue range for number, date, time
patternRegex the value must match
stepValid increment (e.g. multiples of 5)
autocompleteHints for browser autofill
<!-- Required field -->
<input type="text" name="user" required>

<!-- Length limits -->
<input type="text" name="user"
       minlength="3" maxlength="20">

<!-- Custom regex pattern + helpful title -->
<input type="text" id="zip" name="zip"
       pattern="[0-9]{5}(-[0-9]{4})?"
       title="5-digit ZIP, e.g. 12345"
       placeholder="12345"
       required>
Autocomplete tokens:
"given-name" "family-name" "email" "tel"
"street-address" "postal-code" "new-password"

CSS Styling

Form Pseudo-Classes

Style inputs based on their current state — no JavaScript needed.

Pseudo-classWhen it matches
:validValue passes all constraints
:invalidValue fails any constraint
:requiredHas the required attribute
:optionalDoes NOT have required
:placeholder-shownField is empty (placeholder visible)
:user-invalidInvalid after user interaction
:focusHas keyboard focus
/* GREEN border when valid */
input:valid {
  border-color: #22c55e;
  background: #f0fdf4;
}

/* RED border — but ONLY after user typed */
input:user-invalid {
  border-color: #ef4444;
  background: #fef2f2;
}

/* Fallback for older browsers */
input:not(:placeholder-shown):invalid {
  border-color: #ef4444;
}

/* Required field accent */
input:required {
  border-left: 4px solid #4f6df5;
}
⚠️ Avoid bare :invalid — it matches empty required fields on page load, showing red errors before the user touches anything. Use :user-invalid instead.

Accessibility

Form Accessibility — ARIA Attributes

aria-describedby

Links help text to its input. Screen reader announces both label AND description when field is focused.

<label for="user">
  Username
</label>
<input id="user" type="text"
  aria-describedby="user-hint"
  minlength="3" maxlength="20">
<p id="user-hint">
  3–20 characters.
  Letters & numbers only.
</p>

Screen reader says: "Username — 3–20 characters, letters and numbers only"

aria-invalid

Set to "true" on a failing field after submission. Screen readers announce the invalid state when the field is focused.

<input type="email"
  id="email"
  aria-invalid="true"
  aria-describedby="email-err">

<p id="email-err"
   role="alert">
  Please enter a valid
  email address.
</p>

aria-invalid is typically toggled via JavaScript after the user attempts to submit.

role="alert"

Makes error messages announce immediately when they appear in the DOM — even if focus hasn't moved there.

<p id="err-msg"
   role="alert"
   hidden>
  Field is required.
</p>

When you remove hidden via JavaScript, the screen reader announces the message automatically.

Use novalidate on the <form> to disable browser popups and handle all errors with your own ARIA-friendly UI.
Accessibility

Tab Order & Keyboard Navigation

Every form must be fully usable with a keyboard alone.

  • Inputs, selects, textareas, and buttons are in the tab order by default
  • Tab order follows DOM order — if CSS repositions elements visually, keyboard users experience HTML order

tabindex values

ValueEffect
tabindex="0"Add custom element to natural tab order
tabindex="-1"Focusable via JS only; not in tab sequence
tabindex="3"❌ Avoid positive values — breaks natural order
Manual test: Set your mouse aside and Tab through the form. Every field should be reachable, every label clearly readable, and Enter should submit the form.

DevTools → Accessibility panel shows the focus order

Keep your visual layout and DOM order in sync. If they differ, keyboard users experience a disorienting non-linear tab flow.

  • axe DevTools ↗

Validation checklist

  • ✅ Every input has a <label>
  • ✅ Radio groups wrapped in <fieldset>
  • ✅ required on mandatory fields
  • ✅ pattern + title for custom formats
  • ✅ :user-invalid (not bare :invalid) for CSS
  • ✅ aria-describedby for help text
  • ✅ aria-invalid + role="alert" for errors
  • ✅ Tested by tabbing through without a mouse
Always validate server-side too! Browser validation (HTML5 or JavaScript) can be bypassed. Server validation is your last line of defense.
Summary

Key Takeaways

form element action = where data goes. method="get" for searches, method="post" for sensitive/state-changing data.
Labels are mandatory Every input needs a <label>. Use for/id or wrap the input. Never substitute placeholder.
Input types Use the right type — email, tel, date, number unlock mobile keyboards, native pickers, and autofill.
Grouping controls <fieldset> + <legend> groups related inputs and gives screen readers essential context.
HTML5 validation required, minlength, maxlength, min, max, pattern, step — no JavaScript needed.
CSS pseudo-classes Style based on state: :valid, :invalid, :required. Use :user-invalid to avoid errors on page load.
ARIA & accessibility aria-describedby for help text, aria-invalid for errors, role="alert" to announce dynamic messages.
Keyboard navigation Keep DOM order = visual order. Avoid positive tabindex. Test by tabbing through the form yourself.
Server validation too Browser validation can always be bypassed. Validate server-side as well for real security.
