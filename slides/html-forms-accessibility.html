The Form Element

The <form> Element

A <form> acts like a mailing envelope for user input — it tells the browser where to send the data and how to package it. Two attributes control this behavior:

  • action — the destination URL. When the user submits, the browser sends data here. Omit it to re-submit to the current page's URL (common for server-rendered forms).
  • method — the delivery mechanism:
    • get — appends all values directly in the URL as a query string
    • post — bundles data invisibly inside the HTTP request body
    • Default is get if you leave this attribute out
<form action="/search" method="get">
  <!-- inputs here -->
</form>
Rule of thumb: Use get for searches and filters — the resulting URL is bookmarkable and shareable. Use post for login forms, signups, checkouts, or anything that changes data on the server.
GETPOST
Data visible in URLData hidden in request body
Bookmarkable & shareableNot visible in URL bar
Limited data size (~2 KB)No practical size limit
Cached by browserNot cached
Use for: searches, filtersUse for: login, signup, checkout

GET appends data as a query string — each field becomes a name=value pair:
https://shopfinder.com/search?q=leather+boots&sort=price&color=brown

The URL above is the result of a form with three fields: q, sort, and color. Anyone can copy, paste, or bookmark it — the entire search is encoded in the URL.

Labels & Inputs

Every Input Must Have a Label

Why labels are not optional:

  • 🔊 Screen readers announce the label text when a user tabs into the input. Without one, the user hears only "text field" — no idea what to type.
  • 👆 Clicking or tapping anywhere on the label text activates the input, giving mobile users a much larger tap target than the tiny input box alone.
  • 🤖 Password managers, browser autofill, and autocomplete all use label text + input type + name together to identify fields and suggest the right saved values.
  • 🧪 Test automation tools reference labels to find fields — well-labeled forms are far easier to test.

Method 1 — for / id

<label for="email">Email Address</label>
<input type="email" id="email" name="email">

Method 2 — Wrap the input

<label>
  Email Address
  <input type="email" name="email">
</label>

Both are valid. for/id gives layout flexibility; wrapping is concise.

⚠️ Never use placeholder as a label.
Placeholder text disappears the moment typing begins — the user immediately loses context about what the field is for. Use placeholder only for example values like jane@example.com or e.g. New York, never as the field label itself.
The name attribute is required for submission.
name becomes the key in submitted data. A form field with value="Alice" but no name is silently excluded from the submission — the server never receives it.
id vs name — not the same thing
id — links to <label>, targeted by CSS and JavaScript. Must be unique on the page.
name — the key sent with form data to the server. Required for submission. Can repeat (e.g. checkboxes in a group).
No visible label? Use ARIA:
aria-label="Search products" — invisible label only screen readers hear.
aria-labelledby="heading-id" — points to existing page text as the label.

Input Types

Text-Type Inputs

All use <input type="...">. Choosing the right type unlocks the correct mobile keyboard, built-in validation, and browser autofill — for free.

textGeneral fallback for names, usernames, cities — any single line of free text.
emailValidates an "@" sign; shows email-friendly keyboard on mobile; enables autofill.
passwordMasks characters as dots; password managers detect and offer to save credentials.
urlValidates the value starts with a protocol (https://); does NOT verify the URL exists.
telOpens a numeric keypad on mobile; add a pattern attribute for format validation.
searchSignals "search" context to browsers; some add a ✕ clear button automatically.
numberSpinner arrows; use min, max, and step to constrain the valid range.
<input type="text"     name="username"
       placeholder="Your username">
<input type="email"    name="email"
       placeholder="jane@example.com">
<input type="password" name="password"
       placeholder="At least 8 characters">
<input type="url"      name="website"
       placeholder="https://example.com">
<input type="tel"      name="phone"
       placeholder="(555) 555-5555">
<input type="number"   name="qty"
       min="1" max="99">

Key universal attributes

AttributeWhat it does
placeholderHint text shown when field is empty — disappears on typing
valuePre-fills the field with a default value on page load
disabledGrays out the field — cannot interact with it, NOT submitted
readonlyShows the value but prevents editing — value IS submitted
autofocusAuto-focuses this field when the page loads
autocompleteHints the browser autofill system (e.g. "email", "given-name")

Date & Time Inputs

The browser renders a native date/time picker built into the OS — no JavaScript calendar widget needed. However, the picker's appearance (colors, layout) varies by browser and OS, and cannot be styled with CSS alone.

TypeValue formatBest for
dateYYYY-MM-DDBirthdays, travel dates
timeHH:MMMeeting times
datetime-localYYYY-MM-DDTHH:MMAppointments, events
monthYYYY-MMCredit card expiry
weekISO weekSprint planning
<!-- Set a default value, plus min and max -->
<input type="date" name="travel"
       value="2025-06-01"
       min="2025-01-01"
       max="2025-12-31">
Key insight: Regardless of how the picker displays (e.g. "June 1, 2025" in a US locale), JavaScript always reads input.value in the standardized ISO format shown above. This makes server-side date handling and comparisons reliable across every browser and locale.
For date-range pickers, custom styling, or calendar UIs, a JavaScript library (like Flatpickr or Pikaday) gives you full control.

Input Types

Checkboxes & Radio Buttons

☑ Checkboxes — zero or more

Each checkbox is completely independent. A user can select none, one, or all of them simultaneously. Use when the question is "which of these apply?" — toppings, interests, permissions.

<label>
  <input type="checkbox" name="toppings"
         value="cheese"> Cheese
</label>
<label>
  <input type="checkbox" name="toppings"
         value="peppers" checked> Peppers
</label>
  • Same name sends multiple values: toppings=cheese&toppings=peppers
  • Add checked attribute to pre-select on page load
  • Unchecked boxes send nothing — check on the server if a box was left empty

▶ Check items — see multiple values submitted:

toppings= (none checked)

⭕ Radio Buttons — exactly one

All radios sharing the same name form a single group. Selecting one automatically deselects whichever was previously selected — it's physically impossible for two in the same group to be active at once. Use when the question has exactly one right answer — size, payment method, preference.

<label>
  <input type="radio" name="size"
         value="small"> Small
</label>
<label>
  <input type="radio" name="size"
         value="medium" checked> Medium
</label>
<label>
  <input type="radio" name="size"
         value="large"> Large
</label>
Once a radio in a group is selected, users can only change to another option — they cannot deselect entirely. Always wrap radio groups in a <fieldset> for screen readers.

Input Types

Range & Color Inputs

🎚 Range — drag slider

Creates a familiar drag slider control. The slider itself does not display the current value — pair it with an <output> element and the oninput event to show it in real time. The min, max, and step attributes work the same as for type="number".

<label for="vol">
  Volume: <output id="vol-out">50</output>
</label>
<input type="range" id="vol"
       name="volume"
       min="0" max="100"
       value="50" step="1"
       oninput="document.getElementById('vol-out')
                .textContent = this.value">

The oninput event fires continuously as you drag. this.value is the current position as a string.

🎨 Color — native color picker

Opens the operating system's native color picker — the macOS color wheel, Windows color grid, etc. The returned value is always a 6-digit lowercase hex code regardless of how the user made the selection.

<label for="theme">Favorite Color</label>
<input type="color"
       id="theme" name="theme-color"
       value="#4f6df5">
Named colors like "red" are ignored in the value attribute — always use 6-digit hex format: #rrggbb.
accent-color CSS property
Style checkboxes, radio buttons, range sliders, and progress bars with one line of CSS — no custom components needed:
accent-color: #4f6df5;

Form Controls

The <select> Element

A <select> dropdown is ideal when you have 5 or more mutually exclusive options. Users click the control, scan the list, pick one, and the list closes. For fewer options, radio buttons (all visible at once) require fewer clicks.

<label for="country">Country</label>
<select id="country" name="country">
  <option value="">-- Select a country --</option>
  <option value="us">United States</option>
  <option value="ca">Canada</option>
  <option value="uk">United Kingdom</option>
</select>
  • The first <option value=""> acts as a prompt — an empty value fails required validation
  • The selected option's value attribute (not its visible text) is what gets submitted
  • Add multiple to allow multiple selections; add size="4" to show multiple rows

▶ Pick a country — see the submitted value:

country= (nothing selected)

Grouping with <optgroup>

Long lists become hard to scan. <optgroup> adds non-selectable headers that organize options into logical categories:

<select name="country">
  <optgroup label="North America">
    <option value="us">United States</option>
    <option value="ca">Canada</option>
    <option value="mx">Mexico</option>
  </optgroup>
  <optgroup label="Europe">
    <option value="uk">United Kingdom</option>
    <option value="de">Germany</option>
  </optgroup>
</select>
Select vs Radio buttons — quick rule:
≤ 4 options → radio buttons (all visible, fastest to scan)
5+ options → <select> (saves vertical space)
Multi-select: Add multiple to allow choosing several options (Ctrl/Cmd+click). But multi-select UIs are notoriously confusing on mobile — consider checkboxes instead for better usability.
Form Controls

<textarea> & <datalist>

📝 Textarea — multi-line text

Use <textarea> any time a user might write more than one sentence — comments, messages, bios, addresses.

<label for="msg">Your Message</label>
<textarea id="msg" name="message"
          rows="5"
          maxlength="500"
          placeholder="Write here...">
</textarea>
  • Has a closing tag — unlike <input>. Text between tags becomes the default value.
  • rows sets initial height; use CSS height for precise control.
  • Control resize with CSS: resize: vertical (recommended), resize: none, or resize: both.

🔍 Datalist — autocomplete suggestions

A <datalist> is a hybrid of <input> and <select>. It shows suggestions as the user types, but unlike a <select>, the user can ignore all suggestions and type anything they want. The value is never restricted to the list.

<label for="browser">Favorite Browser</label>
<input type="text" id="browser"
       name="browser"
       list="browser-list"
       placeholder="Start typing...">

<datalist id="browser-list">
  <option value="Chrome">
  <option value="Firefox">
  <option value="Safari">
  <option value="Edge">
</datalist>
  • The list attribute on the input must match the datalist's id
  • Suggestions appear as the user types and filter automatically
  • The user can still type values not in the list
browser= (type to see suggestions)
Grouping Controls

Grouping with <fieldset> & <legend>

Wrap related inputs in <fieldset> and label the entire group with <legend>. This is not just visual decoration — it gives screen readers the context they need to describe the group correctly.

<fieldset>
  <legend>Contact Preferences</legend>

  <label>
    <input type="radio" name="contact"
           value="email"> Email
  </label>
  <label>
    <input type="radio" name="contact"
           value="phone"> Phone
  </label>
  <label>
    <input type="radio" name="contact"
           value="text"> Text Message
  </label>

</fieldset>

Use <fieldset> for any logically related group: radio buttons, a section of shipping address fields, or an entire billing form section.

Legend appears inside the fieldset border by default

Screen reader comparison:
Without fieldset: "Email, radio button, 1 of 3" — no context.
With fieldset + legend: "Contact Preferences, Email, radio button, 1 of 3" — clear.
You can style away the default border with CSS (border: none), but keep the <fieldset> and <legend> in your HTML for accessibility. The semantic value is in the markup, not the visual border.

Special Inputs

Hidden Fields, Buttons & Output

🙈 Hidden Inputs

Store data that is submitted with the form but is never shown to the user — no input box, no label visible on screen.

<input type="hidden"
  name="user_id"
  value="1042">
<input type="hidden"
  name="source"
  value="homepage">

Common uses: passing a CSRF security token, tracking which page the user came from, storing a record ID for "edit" forms.

Hidden inputs are visible in page source — anyone can read or change them. Never use them for passwords, tokens that grant permissions, or anything security-sensitive.

🔘 Button Types

Prefer <button> over <input type="submit"> — it can contain HTML, icons, and images.

<!-- Old approach (text only) -->
<input type="submit"
  value="Send">

<!-- Modern (preferred) -->
<button type="submit">
  ✉ Send Message
</button>

<!-- Non-submit button -->
<button type="button"
  onclick="clearForm()">
  Clear
</button>
Always include type="submit" or type="button" explicitly. Inside a <form>, a button's default type is "submit" — an untyped button can accidentally trigger form submission.

📊 The <output> Element

A semantic container for a computed or calculated result. It's more meaningful than a plain <span> because browsers and accessibility tools understand it represents a "result" value, not just arbitrary text.

<label for="price">
  Price: $<output id="price-out">
    10
  </output>
</label>
<input type="range"
  id="price" name="price"
  min="10" max="100" step="5"
  value="10"
  oninput="document.
   getElementById('price-out')
   .textContent = this.value">

Validation

HTML5 Validation Attributes

Browsers enforce these rules automatically before submitting — no JavaScript required. If a field fails validation, the browser shows a native tooltip and the form never submits.

AttributeWhat it does
requiredField must not be empty on submit
minlengthText must be at least N characters
maxlengthPhysically prevents typing beyond N characters
min / maxValue range for number, date, time, range
patternValue must match a regular expression
stepValid increment (e.g. multiples of 5)
autocompleteHints browser autofill with token like "email"
disabledField cannot be interacted with; NOT submitted
readonlyValue is visible but not editable; IS submitted
Common autocomplete tokens:
"given-name" "family-name" "email" "tel"
"street-address" "postal-code" "new-password"
Full list on MDN ↗ 
<!-- Required field -->
<input type="text" name="user" required>

<!-- Length limits -->
<input type="text" name="user"
       minlength="3" maxlength="20">

<!-- Custom regex + descriptive title -->
<input type="text" id="zip" name="zip"
       pattern="[0-9]{5}(-[0-9]{4})?"
       title="5-digit ZIP, e.g. 12345"
       placeholder="12345"
       required>

<!-- Read-only pre-filled value -->
<input type="text" name="plan"
       value="Pro Plan" readonly>

CSS Styling

Form Pseudo-Classes

Style inputs based on their current state — no JavaScript needed. The browser applies and removes these pseudo-classes automatically as the user interacts with the form.

Pseudo-classWhen it applies
:focusField currently has keyboard focus
:validValue passes all constraints
:invalidValue fails any constraint — fires on page load! ⚠️
:user-invalidInvalid after user interaction — use this instead
:requiredHas the required attribute
:optionalDoes NOT have required
:placeholder-shownPlaceholder visible (field is empty)
:disabledHas the disabled attribute
:read-onlyHas the readonly attribute
/* GREEN border when valid */
input:valid:not(:placeholder-shown) {
  border-color: #22c55e;
}

/* RED border after user types something invalid */
input:user-invalid {
  border-color: #ef4444;
}

/* Fallback for older browsers */
input:not(:placeholder-shown):invalid {
  border-color: #ef4444;
}

/* Blue left bar on required fields */
input:required {
  border-left: 4px solid #4f6df5;
}
⚠️ Avoid bare :invalid — it immediately matches empty required fields on page load, showing red errors before the user has done anything. Always use :user-invalid instead.

Accessibility

Form Accessibility — ARIA Attributes

aria-describedby

Connects a visible help-text paragraph to an input. When the user tabs into the field, the screen reader reads: the label, the input type, then the description text. Use this for instructions, format hints, or character limits.

<label for="user">
  Username
</label>
<input id="user" type="text"
  aria-describedby="user-hint"
  minlength="3" maxlength="20">
<p id="user-hint">
  3–20 characters.
  Letters & numbers only.
</p>

Screen reader announces: "Username — 3–20 characters, letters and numbers only"

aria-invalid

Set aria-invalid="true" via JavaScript when you detect an error — typically after a failed submit or after the user leaves a field invalid. When true, screen readers announce "invalid entry" when the user re-focuses that field.

<input type="email"
  id="email"
  aria-invalid="true"
  aria-describedby="email-err">

<p id="email-err"
   role="alert">
  Please enter a valid
  email address.
</p>

Toggle back to aria-invalid="false" when the user fixes the error so screen reader users know it's resolved.

role="alert"

Elements with role="alert" are "live regions" — when their content changes (even without focus moving there), the screen reader interrupts and reads the new content aloud immediately. Perfect for error messages and success confirmations.

<p id="err-msg"
   role="alert"
   hidden>
  Field is required.
</p>

When JavaScript removes the hidden attribute, the screen reader announces the message automatically — no user navigation required.

Add novalidate to the <form> to disable browser native popups and manage all validation feedback with your own ARIA-friendly UI instead.
Accessibility

Tab Order & Keyboard Navigation

Every form must be fully usable with a keyboard alone — Tab (next), Shift+Tab (previous), Space/Enter (activate), arrow keys (radio/select).

  • Inputs, selects, textareas, and buttons are in the tab order by default
  • Tab order follows DOM order (the HTML source) — not the visual position on screen
  • CSS flexbox, grid, and absolute positioning can make visual and keyboard order diverge — test both!

tabindex values

ValueEffect
tabindex="0"Add a custom element (div, span) to the natural tab flow
tabindex="-1"Focusable via JavaScript only; excluded from Tab sequence
tabindex="2"❌ Avoid — positive values break the natural, predictable order
Manual test: Set your mouse aside, tab from the first field to the submit button. Every label should be readable, every field reachable, and Enter should submit.

DevTools → Accessibility panel reveals the focus order

When visual layout and DOM order differ, keyboard users experience a disorienting non-linear tab flow. Always keep the HTML source order matching the logical reading order of your form.

Accessibility checklist

  • ✅ Every input has a visible <label>
  • ✅ Icon-only inputs use aria-label
  • ✅ Radio groups wrapped in <fieldset> + <legend>
  • ✅ required on mandatory fields
  • ✅ pattern + title for custom formats
  • ✅ :user-invalid CSS (not bare :invalid)
  • ✅ aria-describedby for help text
  • ✅ aria-invalid + role="alert" for errors
  • ✅ Tested by tabbing through without a mouse
Always validate server-side too! Browser and JavaScript validation can both be bypassed by anyone with DevTools or Postman. Server-side validation is your final and most important line of defense.
Summary

Key Takeaways

form element action = where data goes. method="get" for searches (bookmarkable URL). method="post" for sensitive or state-changing data.
Labels are mandatory Every input needs a <label>. Use for/id, wrap the input, or use aria-label. Never substitute placeholder for a real label.
Input types Always use the right type — email, tel, date, number unlock mobile keyboards, native pickers, built-in validation, and smarter autofill.
Grouping controls <fieldset> + <legend> groups related inputs and gives screen readers the context to announce the group name with each option.
HTML5 validation required, minlength, maxlength, min, max, pattern, step — browser enforces these before submit, no JavaScript needed.
CSS pseudo-classes Style based on state: :valid, :focus, :required. Always use :user-invalid (not bare :invalid) to avoid red borders on page load.
ARIA & accessibility aria-describedby for help text, aria-invalid for errors, role="alert" to announce dynamic messages to screen reader users.
Keyboard navigation Keep DOM order = visual order. Avoid positive tabindex. Tab through your own forms to verify every field is reachable and every label is clear.
Server validation too Browser validation is user-friendly but can always be bypassed. Real security and data integrity require server-side validation as the final authority.
