This lesson is intentionally ambitious — it covers two large topics in a single day. The first half surveys the most impactful recent additions to the CSS language. The second half introduces web accessibility: what it is, why it matters legally and ethically, and how to build it in from the start rather than bolting it on as an afterthought.

These topics are not unrelated. Good CSS architecture and good accessibility share the same root value: clarity of intent expressed through clean code.

Part 1: Modern CSS Features

CSS Nesting

For years, nested selectors were only possible with CSS preprocessors like Sass or Less. In Sass you could write .card { &:hover { } } and the preprocessor would compile it to .card:hover { }. Native CSS Nesting — now supported in all modern browsers — brings this capability directly to the browser, no build step required.

Nesting keeps related styles co-located, making large stylesheets easier to scan and understand. The & selector represents the parent rule.

LISTING 17.1 — CSS Nesting

/* Without nesting — selectors scattered throughout the file */
.card { background: white; border-radius: 8px; }
.card:hover { box-shadow: 0 8px 24px rgba(0,0,0,0.12); }
.card .card-title { font-size: 1.25rem; }
.card .card-title:first-child { margin-top: 0; }
.card .card-body { padding: 1rem; color: #475569; }

/* With nesting — all card styles in one block */
.card {
  background: white;
  border-radius: 8px;

  &:hover {
    box-shadow: 0 8px 24px rgba(0,0,0,0.12);
  }

  .card-title {
    font-size: 1.25rem;

    &:first-child { margin-top: 0; }
  }

  .card-body {
    padding: 1rem;
    color: #475569;
  }
}

You can also nest media queries directly inside a selector, keeping the responsive overrides right next to the base styles they modify:

LISTING 17.2 — Nesting Media Queries

.hero {
  font-size: 1.5rem;
  padding: 1rem;

  @media (min-width: 768px) {
    font-size: 2.5rem;
    padding: 2rem;
  }

  @media (min-width: 1200px) {
    font-size: 3.5rem;
    padding: 4rem;
  }
}

CAUTION: CSS Nesting is not yet available in the most ancient browsers. Check caniuse.com/css-nesting for support. If you need to support older browsers, use a build tool like PostCSS to compile nesting syntax.

The :has() Selector — The Parent Selector

For decades, CSS had no way to style a parent based on what it contained. You could style a .card that had a certain class, but not a .card based on whether it contained an <img>. JavaScript was required. :has() changes that.

:has(selector) matches an element if any of its descendants match the argument. It is often called "the parent selector," though it is more accurately "the relational selector" — it can check for any descendant, sibling, or relative.

LISTING 17.3 — :has() in Practice

/* Style a figure that contains a figcaption differently */
figure:has(figcaption) {
  border-bottom: 2px solid #e2e8f0;
  padding-bottom: 0.75rem;
}

/* Add extra padding to cards that contain images */
.card:has(img) {
  padding-top: 0;  /* image provides visual top spacing */
}

/* Mark a form as invalid when it contains an invalid input */
form:has(:invalid) {
  border: 2px solid #ef4444;
}

/* Show a sidebar navigation only when the nav contains items */
.sidebar:not(:has(li)) {
  display: none;
}

/* Style the label of a focused input */
.field:has(input:focus) label {
  color: #4f6df5;
  font-weight: 600;
}

/* Add indicator to list items that contain a nested list */
li:has(ul) {
  list-style-type: "▸ ";
}

:has() is supported in all modern browsers. It has fully replaced a significant amount of JavaScript that was previously required to conditionally apply styles based on dynamic content.

:is() and :where()

Both selectors match any element that matches at least one selector in their argument list. They behave identically with one critical difference: specificity.

  • :is(selector-list) — takes the highest specificity of any selector in its list
  • :where(selector-list) — always has zero specificity regardless of its contents

LISTING 17.4 — Reducing Repetition with :is()

/* Without :is() — repetitive */
header h1,
header h2,
header h3,
main h1,
main h2,
main h3 {
  color: #0f172a;
}

/* With :is() — concise */
:is(header, main) :is(h1, h2, h3) {
  color: #0f172a;
}

/* :where() for zero-specificity base styles (easy to override) */
:where(p, ul, ol, dl) {
  margin-block: 1em;
  /* This rule has 0 specificity — any selector overrides it */
}

Use :where() for base/reset styles that you always want to be easily overridden. Use :is() when you want the matched selector's specificity to apply normally.

CSS Cascade Layers with @layer

Specificity wars — where developers write increasingly specific selectors just to override library or framework styles — are one of the most common sources of CSS frustration. @layer introduces a new layer above and below the existing specificity system, giving you explicit control over which styles win.

Think of layers as named buckets. Styles in later-declared layers always beat styles in earlier-declared layers, regardless of specificity. This is a fundamentally different — and more predictable — cascade resolution system.

LISTING 17.5 — Declaring and Using @layer

/* Declare layer order: later layers win */
@layer base, components, utilities;

@layer base {
  /* Low-priority resets */
  * { box-sizing: border-box; margin: 0; }
  body { font-family: system-ui, sans-serif; }
}

@layer components {
  .card {
    background: white;
    border-radius: 8px;
    padding: 1.5rem;
  }
}

@layer utilities {
  /* Highest priority — utility classes always win */
  .p-0 { padding: 0 !important; }
  .hidden { display: none; }
}

/* Importing a library into a layer (prevents it from overriding your code) */
@import url("third-party.css") layer(vendor);

Unlayered styles (rules not inside any @layer) always beat layered styles. So your existing CSS without layers will not break when you introduce @layer into a project.

Dark Mode with prefers-color-scheme and light-dark()

Operating systems now expose a user preference for light or dark color schemes. CSS provides the prefers-color-scheme media query to respond to this preference, and two newer functions — color-mix() and light-dark() — to make theming cleaner than ever.

The Media Query Approach

LISTING 17.6 — Dark Mode with CSS Custom Properties

/* Define color tokens on :root for light mode */
:root {
  --color-bg:      #ffffff;
  --color-surface: #f8f9fa;
  --color-text:    #0f172a;
  --color-muted:   #64748b;
  --color-accent:  #4f6df5;
  --color-border:  #e2e8f0;
}

/* Override tokens for dark mode */
@media (prefers-color-scheme: dark) {
  :root {
    --color-bg:      #0f172a;
    --color-surface: #1e293b;
    --color-text:    #f1f5f9;
    --color-muted:   #94a3b8;
    --color-accent:  #7c9bff;
    --color-border:  #334155;
  }
}

/* Components use tokens — they automatically switch */
body {
  background: var(--color-bg);
  color: var(--color-text);
}

.card {
  background: var(--color-surface);
  border: 1px solid var(--color-border);
}

The light-dark() Function

The light-dark() function (supported in all modern browsers as of 2024) lets you specify both light and dark values in a single line — no media query block needed. You must set color-scheme on the root element to opt in.

LISTING 17.7 — light-dark() Function

:root {
  color-scheme: light dark; /* required to activate light-dark() */
}

body {
  /* light-dark(light-value, dark-value) */
  background: light-dark(#ffffff, #0f172a);
  color:       light-dark(#0f172a, #f1f5f9);
}

.card {
  background: light-dark(#f8f9fa, #1e293b);
  border-color: light-dark(#e2e8f0, #334155);
}

The color-mix() Function

color-mix(in colorspace, color1 percentage, color2 percentage) blends two colors in a specified color space. It is particularly useful for generating tints and shades of a brand color without hardcoding every variation.

LISTING 17.8 — color-mix() for Color Palettes

:root {
  --brand: #4f6df5;
}

.btn-primary {
  background: var(--brand);
}

.btn-primary:hover {
  /* 20% lighter: mix brand with white */
  background: color-mix(in srgb, var(--brand), white 20%);
}

.btn-ghost {
  /* Very light tint background */
  background: color-mix(in srgb, var(--brand), transparent 85%);
  color: var(--brand);
}

Feature Queries with @supports

@supports tests whether the browser supports a given CSS property and value before applying a block of styles. It enables progressive enhancement: you provide a basic working style for all browsers, then layer on advanced styles only for browsers that understand them.

LISTING 17.9 — Progressive Enhancement with @supports

/* Basic layout — works everywhere */
.gallery { display: flex; flex-wrap: wrap; gap: 1rem; }
.gallery-item { flex: 1 1 200px; }

/* Enhance with Grid if supported */
@supports (display: grid) {
  .gallery {
    display: grid;
    grid-template-columns: repeat(auto-fill, minmax(200px, 1fr));
  }
  .gallery-item { flex: unset; }
}

/* Use container queries only if supported */
@supports (container-type: inline-size) {
  .card-wrapper { container-type: inline-size; }
  @container (min-width: 400px) {
    .card { flex-direction: row; }
  }
}

/* Test for a property-value pair */
@supports (backdrop-filter: blur(8px)) {
  .glass-card {
    backdrop-filter: blur(8px);
    background: rgba(255, 255, 255, 0.3);
  }
}

Part 2: Web Accessibility

Why Accessibility Matters

Accessibility is the practice of designing and building products that people with disabilities can use. On the web, this encompasses a wide range of conditions: blindness and low vision, deafness and hard of hearing, motor impairments that prevent mouse use, cognitive and learning differences, and situational limitations (holding a baby, being in bright sunlight, a broken mouse).

Approximately one billion people worldwide have some form of disability. Here is why accessibility is non-negotiable:

  • Legal. In the United States, the Americans with Disabilities Act (ADA) and Section 508 of the Rehabilitation Act require accessible digital products for many organizations. The EU Web Accessibility Directive imposes similar requirements in Europe. Lawsuits over inaccessible websites have cost companies millions of dollars.
  • Ethical. Building a product that excludes a significant portion of the population is wrong. Accessibility is not a feature — it is the baseline expectation of a professional.
  • Business. Accessible websites rank better in search engines, perform better on mobile, and reach a larger audience. The accessible user experience is often a better experience for everyone.

The Web Content Accessibility Guidelines (WCAG)

WCAG is the international standard for web accessibility, published by the W3C. It organizes requirements into four principles — content must be Perceivable, Operable, Understandable, and Robust (POUR). Within those principles are success criteria at three conformance levels:

  • Level A — Minimum requirements. Failing these makes content completely inaccessible to some users.
  • Level AA — The legally required standard in most jurisdictions and the target for most professional web projects.
  • Level AAA — The highest level. Not all success criteria at this level can be met by all content types; it is an aspirational target.

The current standard is WCAG 2.2, published in 2023. See the full specification at www.w3.org/TR/WCAG22/.

The Accessibility Tree

Every browser maintains two parallel representations of a web page: the DOM (Document Object Model) — the structure of HTML elements you write — and the accessibility tree, which is a simplified, semantic version of the DOM that assistive technologies like screen readers, braille displays, and voice control software use to understand and present content.

The accessibility tree represents elements not as <div>s and <span>s but as semantic roles: "button," "heading level 2," "list with 5 items," "required text input labeled Full Name." When you use semantic HTML correctly, the browser builds this tree automatically. When you use <div>s for everything, the tree is meaningless — assistive technology cannot interpret it.

[IMG - Chrome DevTools Accessibility panel showing the accessibility tree for a form]

NOTE: You can inspect the accessibility tree in Chrome DevTools: open the Elements panel, select an element, then click the "Accessibility" tab in the panel on the right. This shows the element's role, name, states, and properties as seen by assistive technology.

Semantic HTML is the single most impactful accessibility technique. Using <button> instead of <div onclick>, <nav> instead of <div class="nav">, and <h1> through <h6> in a meaningful hierarchy makes the accessibility tree rich and correct for free.

ARIA: Accessible Rich Internet Applications

ARIA (Accessible Rich Internet Applications) is a specification of HTML attributes that extend the accessibility tree with additional roles, states, and properties. ARIA fills the gaps where semantic HTML alone is not sufficient — for example, when building a custom dropdown, a tab panel, or a live chat interface.

NOTE: The first rule of ARIA: do not use ARIA if a native HTML element can do the same thing. A <button> is always preferable to <div role="button">. Native elements have built-in keyboard behavior, states, and accessibility semantics. ARIA only adds to the accessibility tree — it does not add keyboard behavior or change visual appearance.

ARIA Roles

The role attribute overrides or augments the implicit role of an element. The landmark roles are the most important for page structure:

  • role="banner" — the page header (equivalent to <header> as a direct child of <body>)
  • role="navigation" — equivalent to <nav>
  • role="main" — equivalent to <main>
  • role="contentinfo" — the page footer (equivalent to <footer>)
  • role="complementary" — equivalent to <aside>
  • role="dialog" — a modal or non-modal dialog window
  • role="alert" — important live information that is announced immediately (error messages, warnings)
  • role="status" — polite live information (success messages, loading states)

ARIA Labels

ARIA labels provide accessible names for elements when the visible text is insufficient or absent:

LISTING 17.10 — ARIA Labeling

<!-- aria-label: provides a label when there is no visible text -->
<button aria-label="Close dialog">✕</button>

<!-- aria-labelledby: references another element's text as the label -->
<section aria-labelledby="pricing-heading">
  <h2 id="pricing-heading">Pricing Plans</h2>
  ...
</section>

<!-- aria-describedby: references a longer description -->
<input
  type="password"
  aria-describedby="pw-hint"
  aria-required="true"
>
<p id="pw-hint">Must be at least 8 characters and include a number.</p>

<!-- Multiple nav elements on a page need distinguishing labels -->
<nav aria-label="Main navigation">...</nav>
<nav aria-label="Breadcrumb">...</nav>

ARIA States and Properties

LISTING 17.11 — ARIA States

<!-- Indicate whether a section is expanded or collapsed -->
<button aria-expanded="false" aria-controls="menu">Menu</button>
<ul id="menu" hidden>...</ul>

<!-- aria-current marks the active page in navigation -->
<nav>
  <a href="/">Home</a>
  <a href="/about" aria-current="page">About</a>
  <a href="/contact">Contact</a>
</nav>

<!-- aria-hidden removes content from the accessibility tree -->
<span aria-hidden="true">★★★★☆</span>
<span class="sr-only">4 out of 5 stars</span>

<!-- aria-invalid marks a form field as having an error -->
<input type="email" aria-invalid="true" aria-describedby="email-error">
<p id="email-error" role="alert">Please enter a valid email address.</p>

<!-- aria-live for dynamically updated regions -->
<div aria-live="polite">
  <!-- Content added here is announced by screen readers -->
  <p id="status-message"></p>
</div>

Focus Management and Keyboard Navigation

Every interactive element on a web page — buttons, links, form fields — must be reachable and operable using only the keyboard. This is essential for people who cannot use a mouse, including many users with motor impairments and power users who prefer keyboard navigation.

The Tab key moves focus forward through interactive elements; Shift+Tab moves backward. Enter activates a focused link or button. Space activates a focused button or checkbox.

The tabindex Attribute

The tabindex attribute controls whether an element is focusable and in what order:

  • tabindex="0" — includes a non-interactive element (like a <div>) in the natural tab order. Use this when you must make a custom interactive widget focusable.
  • tabindex="-1" — removes an element from the tab order but allows it to be focused programmatically with JavaScript (element.focus()). Used to manage focus in modals, menus, and carousels.
  • Positive values (tabindex="1", etc.) — never use these. They override the natural document order, creating a confusing and unpredictable tab sequence for keyboard users.

Focus Styles

Visible focus indicators are required by WCAG 2.4.7 (Level AA). Never write *:focus { outline: none; } without providing an alternative visible focus style. The :focus-visible pseudo-class targets keyboard focus specifically (not mouse clicks), letting you show focus rings only when navigating by keyboard:

LISTING 17.12 — Accessible Focus Styles

/* Remove browser outline only for mouse users */
:focus:not(:focus-visible) {
  outline: none;
}

/* Show a clear focus ring for keyboard users */
:focus-visible {
  outline: 3px solid #4f6df5;
  outline-offset: 2px;
  border-radius: 2px;
}

/* High-contrast mode: use a thicker, high-contrast outline */
@media (forced-colors: active) {
  :focus-visible {
    outline: 3px solid ButtonText;
  }
}

The inert Attribute

The inert attribute is a modern HTML attribute that makes an element and all its descendants unfocusable, unclickable, and invisible to screen readers — with a single attribute. Before inert, developers had to manually set aria-hidden="true", tabindex="-1" on every focusable child, and pointer-events: none. The inert attribute replaces all of that.

LISTING 17.13 — Using the inert Attribute

<!-- When a modal is open, make the rest of the page inert -->
<main id="page-content" inert>
  <!-- All focus and interaction is trapped in the modal -->
  <a href="/">This link is unreachable while inert</a>
</main>

<dialog id="my-modal">
  <h2>Are you sure?</h2>
  <button id="confirm">Confirm</button>
  <button id="cancel">Cancel</button>
</dialog>

<script>
const modal   = document.getElementById('my-modal');
const content = document.getElementById('page-content');

// Open: inert the background
document.getElementById('open-btn').addEventListener('click', () => {
  modal.showModal();
  content.inert = true;
});

// Close: remove inert, return focus
document.getElementById('cancel').addEventListener('click', () => {
  modal.close();
  content.inert = false;
  document.getElementById('open-btn').focus();
});
</script>

The Native <dialog> Element

Building accessible modal dialogs from scratch is notoriously difficult: you must trap focus inside the dialog, close on Escape, manage aria-modal, return focus to the trigger on close, and prevent scrolling of background content. The native HTML <dialog> element handles most of this automatically.

LISTING 17.14 — The <dialog> Element

<button id="openBtn">Open Dialog</button>

<dialog id="myDialog">
  <h2>Confirm Delete</h2>
  <p>This action cannot be undone. Delete this item?</p>
  <div class="dialog-actions">
    <button id="cancelBtn">Cancel</button>
    <button id="deleteBtn">Delete</button>
  </div>
</dialog>

<script>
const dialog    = document.getElementById('myDialog');
const openBtn   = document.getElementById('openBtn');
const cancelBtn = document.getElementById('cancelBtn');
const deleteBtn = document.getElementById('deleteBtn');

// showModal() creates a true modal — Escape closes it, focus is trapped
openBtn.addEventListener('click', () => dialog.showModal());

// close() closes the dialog and returns focus to the opener automatically
cancelBtn.addEventListener('click', () => dialog.close());

deleteBtn.addEventListener('click', () => {
  // Do the delete action
  dialog.close('deleted'); // Optional return value
});

// Handle close event
dialog.addEventListener('close', () => {
  console.log('Dialog closed, return value:', dialog.returnValue);
});
</script>

The showModal() method creates a true modal: it adds a ::backdrop pseudo-element behind the dialog, traps keyboard focus inside, and allows the Escape key to close it. This is far simpler and more robust than hand-rolling the same behavior.

Live Regions with aria-live

Screen readers announce page content as it loads. But when content changes dynamically — an error message appears after form submission, a chat message arrives, a notification pops up — the screen reader is not automatically notified. aria-live regions tell the browser to announce dynamic content changes.

LISTING 17.15 — aria-live Regions

<!-- Polite: announces when the user is not busy -->
<div aria-live="polite" aria-atomic="true">
  <p id="status"></p>
</div>

<!-- Assertive: interrupts immediately (use sparingly — only for critical errors) -->
<div role="alert" aria-live="assertive">
  <p id="error"></p>
</div>

<script>
// When you update content inside a live region,
// screen readers will announce the change
document.getElementById('status').textContent = 'Your changes have been saved.';
document.getElementById('error').textContent  = 'Error: network unavailable. Please retry.';
</script>
  • aria-live="polite" — waits until the user finishes their current interaction before announcing. Use for status messages, search results counts, chat messages.
  • aria-live="assertive" — interrupts immediately. Only use for truly critical errors that block the user. role="alert" is a shorthand that implies aria-live="assertive".
  • aria-atomic="true" — announces the entire region every time any part changes (rather than just the changed part).

Skip Navigation Links

Keyboard users navigate through every link and interactive element at the top of a page before reaching the main content. On a page with a 30-link navigation menu, pressing Tab 30 times before reaching the article body is exhausting. A skip navigation link solves this: a link at the very top of the page that jumps directly to the main content area when activated.

LISTING 17.16 — Skip Navigation Link

<!-- First element in <body> -->
<a class="skip-link" href="#main-content">Skip to main content</a>

<!-- ... header, navigation ... -->

<main id="main-content" tabindex="-1">
  <!-- Page content -->
</main>
/* Visually hidden by default; visible only on focus */
.skip-link {
  position: absolute;
  top: -100%;
  left: 0;
  background: #4f6df5;
  color: white;
  padding: 0.75rem 1.5rem;
  font-weight: 600;
  text-decoration: none;
  z-index: 9999;
  transition: top 0.1s;
}

.skip-link:focus {
  top: 0;
}

Common Accessibility Mistakes

The following errors account for the majority of accessibility failures found in web audits:

  • Missing or empty alt text. Every <img> must have an alt attribute. Decorative images that carry no information get alt="". Images that convey meaning need descriptive text.
  • Insufficient color contrast. WCAG 1.4.3 (Level AA) requires a contrast ratio of at least 4.5:1 for normal text and 3:1 for large text. Use the WebAIM Contrast Checker to verify your palette.
  • Removing focus outlines without replacement. outline: none without a replacement focus style makes keyboard navigation impossible to follow visually.
  • Interactive <div>s and <span>s. <div onclick> elements are not focusable by keyboard, have no ARIA role, and convey no semantic meaning. Use <button> for actions and <a> for navigation.
  • Form inputs without labels. Every input must have an associated <label> (or aria-label). Placeholder text disappears and is not a substitute for a label.
  • Inaccessible modals. Modals that don't trap focus, don't close on Escape, and don't return focus to the trigger when closed confuse and strand keyboard users.
  • Auto-playing media. Video or audio that plays automatically is disorienting and can be impossible to stop for some users. Always provide a pause control and do not autoplay audio.
  • Non-descriptive link text. Links labeled "click here" or "read more" provide no context for screen reader users who navigate by listing all links on a page.

Accessibility Testing Tools

No automated tool can catch all accessibility issues — some require human judgment — but automated tools can quickly identify a large percentage of common errors and are an essential first step.

Browser DevTools Lighthouse

Chrome and Edge include Google Lighthouse in DevTools. Open DevTools → Lighthouse tab → run an Accessibility audit. Lighthouse scores your page from 0–100 and lists specific failing criteria with links to explanations.

axe DevTools

The axe DevTools browser extension (by Deque) is the most widely used professional accessibility testing tool. It integrates into DevTools, finds issues that Lighthouse misses, and provides detailed remediation guidance for each violation.

WAVE

The WAVE Web Accessibility Evaluation Tool from WebAIM overlays accessibility information directly on your page — showing structure, labeling, contrast issues, and alerts visually in context. It is excellent for learning because you can see exactly which HTML elements have problems.

Screen Readers

Automated tools are a starting point, not a finish line. You must test with actual screen readers to understand the full user experience:

  • VoiceOver — built into macOS, iOS, and iPadOS. Activate with Cmd+F5 on Mac. Navigate with Tab, arrow keys, and VoiceOver-specific commands (Ctrl+Option+...).
  • NVDA — free screen reader for Windows. Download from nvaccess.org.
  • JAWS — professional screen reader for Windows, used by most blind professionals. Commercial license required.
  • TalkBack — screen reader built into Android.

Q&A

Q. Does CSS Nesting work the same as Sass/SCSS nesting?

A. Almost. Native CSS Nesting requires either the & selector or a nesting selector to identify descendent rules — some older Sass patterns that begin a nested rule with a plain element selector (like just p { } inside .card { }) are now valid in native CSS. The main difference is that native CSS computes selectors slightly differently than Sass in edge cases involving specificity and combinators, but for everyday usage the syntax and behavior are effectively the same.

Q. What's the difference between :is() and :where() in practice?

A. They work identically except for specificity. :is(h1, h2) has the specificity of h1 or h2 (whichever is higher in the list). :where(h1, h2) has zero specificity. Use :where() in base/reset stylesheets where you want any specific selector to easily override the rule. Use :is() in component or utility styles where normal specificity behavior is desired.

Q. If ARIA only adds to the accessibility tree and doesn't add keyboard behavior, what actually handles keyboard interaction for custom widgets?

A. JavaScript. When you build a custom widget — a date picker, a combobox, a tab interface — you are responsible for implementing all keyboard interactions. The WAI-ARIA Authoring Practices Guide at www.w3.org/WAI/ARIA/apg/ documents the expected keyboard behavior for every ARIA widget pattern. This is one of the strongest reasons to prefer native HTML elements: they come with keyboard behavior built in.

Q. Is using a CSS variable for theming the same as using light-dark()?

A. They achieve the same result but in different ways. The CSS custom properties approach — defining tokens on :root and overriding them in a media query — is more explicit and more controllable. You can also toggle dark mode via a class (useful for user-controlled theme switching). light-dark() is more concise and automatic, but it relies solely on the OS preference and is harder to override with a manual toggle. For most production sites, the custom properties approach is recommended; use light-dark() for simpler projects or when OS-only switching is sufficient.

Q. How do I implement a "dark mode toggle" button that lets users override the OS setting?

A. Store the user's preference in localStorage, then add a class (data-theme="dark") to the <html> element based on the saved preference. In CSS, define your color tokens for both @media (prefers-color-scheme: dark) (OS preference) and [data-theme="dark"] (manual toggle). The manual class should always win, so list it after the media query. Restore the preference from localStorage before the first paint to avoid a flash of the wrong theme — do this in a <script> tag in the <head>, before any stylesheets load.
