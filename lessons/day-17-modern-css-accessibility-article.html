Part 1: Modern CSS Features

CSS Nesting

For years, nested selectors were only possible with CSS preprocessors like Sass or Less. Native CSS Nesting — now supported in all modern browsers — brings this capability directly to the browser, no build step required. Related styles stay in one block instead of scattered across the file.

The core of nesting is the & selector — it always means "the current parent selector." Every nesting form is just combining & with selectors you already know.

LISTING 17.1 — CSS Nesting: Before and After

/* Without nesting — selectors scattered throughout the file */
.card { background: white; border-radius: 8px; }
.card:hover { box-shadow: 0 8px 24px rgba(0,0,0,0.12); }
.card .card-title { font-size: 1.25rem; }
.card .card-title:first-child { margin-top: 0; }
.card .card-body { padding: 1rem; color: #475569; }

/* With nesting — all card styles in one block */
.card {
  background: white;
  border-radius: 8px;

  &:hover {
    box-shadow: 0 8px 24px rgba(0,0,0,0.12);
  }

  .card-title {
    font-size: 1.25rem;
    &:first-child { margin-top: 0; }
  }

  .card-body {
    padding: 1rem;
    color: #475569;
  }
}

The table below shows every form the & selector can take, what it compiles to, and when to use it:

You writeCompiles toUse for
&:hover.card:hoverPseudo-classes
&:first-child.card:first-childStructural pseudo-classes
&::before.card::beforePseudo-elements
&.active.card.activeSame element + extra class
&[disabled].card[disabled]Attribute state
& .title.card .titleDescendant child
.title.card .titleSame — & is implied
& + p.card + pAdjacent sibling
& ~ p.card ~ pGeneral sibling

NOTE: If you nest a plain class like .title with no &, the browser automatically reads it as & .title (descendant). But if you need a pseudo-class or modifier on the same element, & is required: write &:hover, not :hover.

The following listing shows all nesting forms applied to a single .btn element so you can see them working together:

LISTING 17.2 — All Nesting Forms in One Example

.btn {
  background: blue;           /* own styles */

  &:hover { opacity: .85; }       /* pseudo-class */
  &:focus-visible {               /* keyboard focus only */
    outline: 3px solid blue;
  }
  &::after { content: " →"; }    /* pseudo-element */
  &.active { background: darkblue; }  /* same element + class */
  &[disabled] {                   /* attribute state */
    opacity: .4;
    cursor: not-allowed;
  }

  .btn-label { font-weight: 700; }    /* descendant (& implied) */
  & + .btn { margin-left: .5rem; }   /* adjacent sibling */

  .dark-theme & { background: #7c9bff; }  /* context parent */

  @media (max-width: 480px) {     /* nested media query */
    width: 100%;
  }
}

CAUTION: CSS Nesting is not available in very old browsers. Check caniuse.com/css-nesting for support. If you must support legacy browsers, use PostCSS to compile nesting syntax before deployment.

The :has() Selector — The Parent Selector

For decades, CSS had no way to style a parent based on what it contained. You could style a .card that had a certain class, but not a .card based on whether it contained an <img>. JavaScript was required. :has() changes that — supported in all modern browsers.

How to read it in plain English: .field:has(input:focus) — say it as "any .field that contains an input that is focused." The outer selector is what you're styling, and the argument is the condition inside it.

The table below shows every syntax form you will use:

Syntax formWhat it selectsExample
A:has(B)A that contains Bdiv:has(img)
A:has(B:state)A that contains B in a state.field:has(input:focus)
A:has(> B)A with a direct child Bsection:has(> h2)
A:has(B, C)A that contains B or Cform:has(input, select)
A:not(:has(B))A that does not contain B.sidebar:not(:has(li))
A:has(B) CStyle C when A contains B.field:has(input:focus) label

LISTING 17.3 — :has() in Practice

/* Card with image: remove top padding — image provides spacing */
.card:has(img) { padding-top: 0; }

/* Label highlights when its sibling input is focused — no JS */
.field:has(input:focus) label {
  color: #4f6df5;
  font-weight: 600;
}

/* Form shows error border when it contains an invalid field */
form:has(:invalid) {
  border: 2px solid #ef4444;
}

/* Hide sidebar entirely when it has no list items */
.sidebar:not(:has(li)) { display: none; }

/* Only direct-child headings trigger this rule */
section:has(> h2) { padding-top: 2rem; }

/* List items that contain a sub-list get a visual indicator */
li:has(ul) { list-style-type: "▸ "; }

:is() and :where()

Both selectors accept a list of selectors and match any element that fits at least one. They are identical except for specificity — which determines when to use which.

How to read them in plain English:

  • :is(header, main) :is(h1, h2, h3) — say it as "any h1, h2, or h3 that lives inside a header or a main." Read the outer :is() first (where), then the inner (what).
  • :where(p, ul, ol) — same as above, but this style has zero priority and any other rule beats it.
Syntax formUseSpecificity
:is(A, B)Match A or B — shorthand for a selector listHighest of A or B
:is(A, B) CC inside A or BHighest of A or B + C
:is(A):is(B)Element that is both A and BSum of both
:where(A, B)Same as :is() — but for base/reset stylesAlways 0
:not(:is(A, B))Elements that are neither A nor BHighest of A or B

LISTING 17.4 — :is() and :where() in Practice

/* Without :is() — 6 separate selectors */
header h1, header h2, header h3,
main   h1, main   h2, main   h3 {
  color: #0f172a;
  line-height: 1.2;
}

/* With :is() — 1 selector, same result */
:is(header, main) :is(h1, h2, h3) {
  color: #0f172a;
  line-height: 1.2;
}

/* :where() for zero-specificity base styles */
/* Before — reset has specificity (0,0,1) and blocks overrides */
p, ul, ol, dl { margin-block: 1em; }
.no-gap { margin-block: 0; }   /* ✗ loses — same specificity, source order wins */

/* After — :where() has specificity 0, anything beats it */
:where(p, ul, ol, dl) { margin-block: 1em; }
.no-gap { margin-block: 0; }   /* ✓ always wins */

Use :where() for base and reset styles that you always want easily overridden. Use :is() for component and utility styles where normal specificity behavior is expected.

CSS Cascade Layers with @layer

Specificity wars — where developers write increasingly specific selectors just to override library styles — are one of the most common sources of CSS frustration. Here is the problem in concrete terms: you import Bootstrap. Bootstrap's .btn loads first. Your own .btn rule has the same specificity, so Bootstrap wins because it loaded first. The only escape is writing longer selectors — body .wrap .btn — forever. Next week Bootstrap updates and you lose again.

@layer ends this. You declare a priority stack by name, and layer position beats specificity entirely.

LISTING 17.5 — @layer: The Specificity War and Its Fix

/* ✗ Without @layer — Bootstrap always wins */
@import url("bootstrap.css");
.btn { background: green; }          /* ✗ loses — Bootstrap loaded first */
body .wrap .btn { background: green; } /* ✓ barely wins — but fragile */

/* ✓ With @layer — you control the priority stack */

/* Step 1: declare order — lowest priority first */
@layer vendor, base, components, utilities;

/* Step 2: vendor — third-party libraries go here */
@import url("bootstrap.css") layer(vendor);

/* Step 3: base — your site-wide defaults */
@layer base {
  body { font-family: sans-serif; color: #1a1a1a; }
  * { box-sizing: border-box; }
}

/* Step 4: components — your reusable UI pieces */
@layer components {
  .btn { background: green; }  /* ✓ always beats Bootstrap */
  .card { border-radius: 8px; }
}

/* Step 5: utilities — single-purpose overrides */
@layer utilities {
  .hidden { display: none; }   /* ✓ beats components */
  .sr-only { position: absolute; clip: rect(0,0,0,0); }
}

/* Unlayered styles beat all layers above */
.override { color: red; }      /* ✓ beats everything */

Priority order (low → high): vendor → base → components → utilities → unlayered. Later in the @layer declaration always wins, regardless of specificity. Styles written outside any layer always beat everything inside a layer — so existing CSS will not break when you introduce @layer into a project.

Dark Mode with prefers-color-scheme and light-dark()

Operating systems expose a user preference for light or dark color schemes. CSS provides the prefers-color-scheme media query to respond to this preference, and two newer functions — color-mix() and light-dark() — to make theming cleaner than ever.

The Media Query Approach

LISTING 17.6 — Dark Mode with CSS Custom Properties

/* Define color tokens on :root for light mode */
:root {
  --color-bg:      #ffffff;
  --color-surface: #f8f9fa;
  --color-text:    #0f172a;
  --color-muted:   #64748b;
  --color-accent:  #4f6df5;
  --color-border:  #e2e8f0;
}

/* Override tokens for dark mode */
@media (prefers-color-scheme: dark) {
  :root {
    --color-bg:      #0f172a;
    --color-surface: #1e293b;
    --color-text:    #f1f5f9;
    --color-muted:   #94a3b8;
    --color-accent:  #7c9bff;
    --color-border:  #334155;
  }
}

/* Components use tokens — they switch automatically */
body  { background: var(--color-bg);      color: var(--color-text); }
.card { background: var(--color-surface); border: 1px solid var(--color-border); }

The light-dark() Function

The light-dark() function (supported in all modern browsers as of 2024) lets you specify both light and dark values in a single line — no media query block needed. You must set color-scheme on the root element to opt in.

LISTING 17.7 — light-dark() Function

:root {
  color-scheme: light dark; /* required — activates light-dark() */
}

body {
  /* light-dark(light-value, dark-value) */
  background: light-dark(#ffffff, #0f172a);
  color:       light-dark(#0f172a, #f1f5f9);
}

.card {
  background:   light-dark(#f8f9fa, #1e293b);
  border-color: light-dark(#e2e8f0, #334155);
}

The color-mix() Function

color-mix(in colorspace, color1 percentage, color2 percentage) blends two colors in a specified color space. The percentages control how much of each color contributes — they must add up to 100% (or the browser normalizes them). It is particularly useful for generating tints and shades of a brand color without hardcoding every variation.

LISTING 17.8 — color-mix() for Color Palettes

:root { --brand: #4f6df5; }

.btn-primary {
  background: var(--brand);
}

.btn-primary:hover {
  /* Mix brand with white: result is 80% brand + 20% white */
  background: color-mix(in srgb, var(--brand), white 20%);
}

.btn-ghost {
  /* Very light tint: 15% brand + 85% transparent */
  background: color-mix(in srgb, var(--brand), transparent 85%);
  color: var(--brand);
}

/* Color spaces affect the result — oklab gives perceptually even mixes */
.subtle { background: color-mix(in oklab, #ff0000, #0000ff 50%); }

Feature Queries with @supports

@supports tests whether the browser supports a given CSS property and value before applying a block of styles. It enables progressive enhancement: provide a working style for all browsers, then layer on advanced styles only for browsers that support them.

LISTING 17.9 — Progressive Enhancement with @supports

/* Basic layout — works everywhere */
.gallery { display: flex; flex-wrap: wrap; gap: 1rem; }
.gallery-item { flex: 1 1 200px; }

/* Enhance with Grid if supported */
@supports (display: grid) {
  .gallery {
    display: grid;
    grid-template-columns: repeat(auto-fill, minmax(200px, 1fr));
  }
  .gallery-item { flex: unset; }
}

/* Use container queries only if supported */
@supports (container-type: inline-size) {
  .card-wrapper { container-type: inline-size; }
  @container (min-width: 400px) {
    .card { flex-direction: row; }
  }
}

/* Test for a specific property-value pair */
@supports (backdrop-filter: blur(8px)) {
  .glass-card {
    backdrop-filter: blur(8px);
    background: rgba(255, 255, 255, 0.3);
  }
}

Part 2: Web Accessibility

Why Accessibility Matters

Accessibility means building websites that everyone can use — including people with disabilities. In the United States alone, 1 in 4 adults has some form of disability. If your app has 1,000 users and you skip accessibility, you are locking out 250 of them. That is not a minor bug — that is a broken product.

Here is what inaccessible code looks like, and what it takes to fix it:

LISTING 17.10 — Inaccessible vs. Accessible HTML

<!-- ✗ Inaccessible: Tab can't reach it. Screen reader: "Place Order" — no role -->
<div onclick="submitOrder()">Place Order</div>

<!-- ✗ Screen reader reads the filename: "pizza-2024-final.jpg" -->
<img src="pizza-2024-final.jpg">

<!-- ✗ Placeholder disappears on type. Screen reader says nothing useful -->
<input placeholder="Your address">


<!-- ✓ Tab-focusable. Screen reader: "Place Order, button" -->
<button onclick="submitOrder()">Place Order</button>

<!-- ✓ Screen reader: "Large pepperoni pizza, image" -->
<img src="pizza.jpg" alt="Large pepperoni pizza">

<!-- ✓ Label stays visible. Screen reader: "Your address, edit text" -->
<label for="addr">Your address</label>
<input id="addr" type="text">

Three small changes. Same visual result. One version works for everyone — the other locks out millions of users.

Here is why accessibility is non-negotiable:

  • Legal. In the United States, the ADA applies to websites the same way it applies to buildings that need ramps. Domino's Pizza was sued because a blind customer couldn't order online. Target, Netflix, and hundreds of other companies have faced lawsuits. The EU Web Accessibility Directive imposes similar requirements in Europe. Inaccessible websites cost real money in court.
  • Ethical. Imagine building a store with a front door that only some people can open. You'd never do that in real life. A website that only works for some people is the digital version of that locked door. Accessibility is not extra credit — it is basic respect.
  • Better for everyone. Closed captions were invented for deaf viewers — now everyone uses them in noisy coffee shops. Curb cuts were built for wheelchairs — now everyone uses them with strollers and bikes. Accessible features almost always improve the experience for everyone, and Google Search ranks accessible sites higher.

The Web Content Accessibility Guidelines (WCAG)

WCAG is the international standard for web accessibility, published by the W3C. Think of it like building codes for houses — specific rules that professionals are expected to follow. It organizes requirements into four principles: content must be Perceivable, Operable, Understandable, and Robust (POUR).

LISTING 17.11 — The POUR Principles with Code

<!-- P — Perceivable: can every user receive the content? -->
<img src="dog.jpg">                                   <!-- ✗ blind user hears nothing -->
<img src="dog.jpg" alt="Golden retriever running">    <!-- ✓ -->

<!-- O — Operable: can every user control the interface? -->
<div onclick="buy()">Buy</div>         <!-- ✗ Tab can't reach it -->
<button onclick="buy()">Buy</button>   <!-- ✓ keyboard works -->

<!-- U — Understandable: do errors explain what went wrong? -->
<input style="border: red">            <!-- ✗ red border says nothing -->
<input aria-describedby="err">
<p id="err">Enter a valid email address</p>  <!-- ✓ explains the problem -->

<!-- R — Robust: does it work with any assistive technology? -->
<div class="btn" role="button">Save</div>  <!-- ✗ fragile, no keyboard behavior -->
<button>Save</button>                       <!-- ✓ works in every AT, forever -->

Within POUR are success criteria at three conformance levels:

  • Level A — The minimum. Failing Level A means some users literally cannot use your site at all — not "it's hard," but "it's impossible."
  • Level AA — The legal standard in most countries and the target for all professional projects. This is what employers mean when they say "make it accessible."
  • Level AAA — The gold standard. Some AAA criteria are so strict they are impossible to meet for all content types. Aim for AA with AAA improvements where practical.

The current standard is WCAG 2.2, published in 2023. See the full specification at www.w3.org/TR/WCAG22/.

The Accessibility Tree

Your browser secretly builds two versions of every page. The first is the DOM — that is the one you see on screen. The second is the accessibility tree — a parallel, stripped-down version that screen readers and assistive technology read from. It contains no colors, no fonts, no layout. Just: what is this element, what is it called, and what state is it in?

Here is the same navigation written two ways, and what each means for the accessibility tree:

LISTING 17.12 — Bad HTML vs. Good HTML: What the Screen Reader Hears

<!-- ✗ Bad HTML -->
<div class="nav">
  <!-- ✗ <div> has no role — screen reader doesn't know this is navigation -->

  <div onclick="go('/')">Home</div>
  <!-- ✗ <div> is not focusable — Tab key skips it entirely
       ✗ onclick only works with a mouse
       ✗ no href — no real URL, no browser history
       ✗ no role — "Home" is just floating text with no context -->
</div>

<!-- Screen reader hears: "Home" — raw text with zero context. Tab reaches nothing. -->


<!-- ✓ Good HTML -->
<nav aria-label="Main">
  <!-- ✓ <nav> = landmark role "navigation"
       ✓ aria-label names it "Main" so users can tell it apart from other <nav>s -->

  <a href="/">Home</a>
  <!-- ✓ role "link" — screen reader says "link"
       ✓ Tab-focusable automatically
       ✓ Enter key activates it
       ✓ real URL — right-click, open in new tab all work -->
</nav>

<!-- Screen reader announces: "Main navigation" then "Home, link" -->

Every element in the accessibility tree gets four properties. These are what the screen reader reads aloud:

PropertyQuestion it answersWhere it comes from
RoleWhat type of thing is it?The HTML element (<button> → "button")
NameWhat is it called?Text content, alt, aria-label
StateIs it on, off, open, checked?aria-expanded, checked, disabled
ValueWhat does it currently show?The current value (sliders, inputs)

Here is a real element mapped to all four properties:

LISTING 17.13 — The 4 Accessibility Tree Properties on a Real Element

<button aria-expanded="false">Open FAQ</button>
<!-- Role:  "button"   ← from <button> element
     Name:  "Open FAQ" ← from the text inside
     State: "collapsed" ← from aria-expanded="false"
     Value: —          ← buttons don't hold a value

Screen reader announces: "Open FAQ, button, collapsed" -->

<!-- Value applies to things that hold a number or choice: -->
<input type="range" aria-label="Volume" value="75">
<!-- Role: "slider"   Name: "Volume"   Value: "75" -->

[IMG - Chrome DevTools Accessibility panel showing role, name, and state for a button]

NOTE: You can inspect the accessibility tree in Chrome DevTools: open the Elements panel, select any element, then click the Accessibility tab on the right. You'll see Role, Name, and State exactly as screen readers see them.

Semantic HTML is the single most impactful accessibility technique. Using <button> instead of <div onclick>, <nav> instead of <div class="nav">, and <h1> through <h6> in a meaningful hierarchy fills in all four tree properties automatically — for free.

ARIA: Accessible Rich Internet Applications

ARIA is a specification of HTML attributes that extend the accessibility tree with additional roles, states, and properties. ARIA fills the gaps where semantic HTML alone is not sufficient — for example, when building a custom dropdown, a tab panel, or a live chat interface.

NOTE: The first rule of ARIA: do not use ARIA if a native HTML element can do the same thing. A <button> is always preferable to <div role="button">. Native elements have built-in keyboard behavior, states, and accessibility semantics. ARIA only adds to the accessibility tree — it does not add keyboard behavior or change visual appearance.

ARIA Roles

The role attribute sets the element's type in the accessibility tree. The landmark roles are the most important for page structure — they let screen reader users jump directly to major sections of a page:

  • role="banner" — the page header (equivalent to <header> as a direct child of <body>)
  • role="navigation" — equivalent to <nav>
  • role="main" — equivalent to <main>
  • role="contentinfo" — the page footer (equivalent to <footer>)
  • role="complementary" — equivalent to <aside>
  • role="dialog" — a modal or non-modal dialog window
  • role="alert" — critical live information announced immediately (errors, warnings)
  • role="status" — polite live information (success messages, loading states)

ARIA Labels

ARIA labels provide accessible names for elements when visible text is insufficient or absent:

LISTING 17.14 — ARIA Labeling

<!-- aria-label: provides a name when there is no visible text -->
<button aria-label="Close dialog">✕</button>
<!-- Screen reader: "Close dialog, button" — not just "✕, button" ✓ -->

<!-- aria-labelledby: references another element's text as the label -->
<section aria-labelledby="pricing-heading">
  <h2 id="pricing-heading">Pricing Plans</h2>
</section>
<!-- Screen reader: "Pricing Plans, region" ✓ -->

<!-- aria-describedby: references a longer description -->
<input type="password" aria-describedby="pw-hint">
<p id="pw-hint">Must be at least 8 characters and include a number.</p>
<!-- Screen reader: "Password, edit text, Must be at least 8 characters…" ✓ -->

<!-- Multiple nav elements need labels to tell them apart -->
<nav aria-label="Main">...</nav>
<nav aria-label="Breadcrumb">...</nav>
<!-- Without labels, both announce: "navigation" — user can't distinguish them ✗ -->

ARIA States and Properties

States tell the screen reader about the current condition of an element. They must be updated with JavaScript whenever the UI changes — setting aria-expanded="false" in HTML and never toggling it is worse than not using it at all.

LISTING 17.15 — ARIA States

<!-- aria-expanded: does the user know whether a section is open? -->

<!-- ✗ Without aria-expanded -->
<button onclick="toggleMenu()">Menu ▼</button>
<!-- Screen reader: "Menu ▼, button"
     User clicks. Menu opens visually. Screen reader says NOTHING.
     Did it open? Did it close? No idea. -->

<!-- ✓ With aria-expanded -->
<button aria-expanded="false" aria-controls="mainMenu" onclick="toggleMenu(this)">Menu ▼</button>
<ul id="mainMenu" hidden>...</ul>
<!-- Screen reader: "Menu ▼, collapsed, button"
     User clicks. JS flips aria-expanded to "true" and removes hidden.
     Screen reader: "Menu ▼, expanded, button" ✓ -->

<!-- Other common states -->
aria-checked="true"   <!-- custom checkbox or toggle -->
aria-invalid="true"   <!-- form field has an error -->
aria-disabled="true"  <!-- item is not currently interactive -->
aria-current="page"   <!-- active nav link -->
aria-hidden="true"    <!-- hide decorative content from screen reader -->

Focus Management and Keyboard Navigation

Try this right now: go to any website and put your mouse on the other side of the desk. Use only Tab, Enter, and arrow keys. Can you navigate the site? Can you fill out forms? Can you click the buy button? Users with motor disabilities, tremors, or paralysis may not be able to use a mouse at all. The keyboard is their only way in. If your interactive element cannot be reached with Tab, it simply does not exist for them.

The rule is simple: if you can click it, you must be able to Tab to it and press Enter to activate it.

Key keyboard interactions:

  • Tab — move focus forward through interactive elements
  • Shift+Tab — move focus backward
  • Enter — activate a focused link or button
  • Space — activate a focused button or checkbox
  • Esc — close a modal or dismiss a popup
  • Arrow keys — navigate within components like menus, tabs, and sliders

The tabindex Attribute

LISTING 17.16 — tabindex Values

<!-- tabindex="0": add a non-interactive element to the natural tab order -->
<div role="button" tabindex="0" onclick="open()">Open</div>
<!-- Tab now reaches it. But use <button> instead whenever possible. -->

<!-- tabindex="-1": focusable by JavaScript only, never by Tab -->
<div id="modal" tabindex="-1">...</div>
<!-- Use when you need to move focus programmatically: -->
document.getElementById('modal').focus();

<!-- tabindex="1" or higher: NEVER use positive values -->
<button tabindex="3">...</button>  <!-- ✗ breaks the natural tab order -->

Focus Styles

Visible focus indicators are required by WCAG 2.4.7 (Level AA). Imagine playing a video game where the cursor turns invisible whenever you use a controller — mouse players are fine, but controller players have no idea where they are. That is exactly what happens when developers write outline: none without a replacement. The :focus-visible pseudo-class shows the focus ring only during keyboard navigation, hiding it for mouse users. Everyone wins.

LISTING 17.17 — Accessible Focus Styles

/* Remove browser outline for mouse users only */
:focus:not(:focus-visible) { outline: none; }

/* Show a clear focus ring for keyboard users */
:focus-visible {
  outline: 3px solid #4f6df5;
  outline-offset: 2px;
  border-radius: 2px;
}

/* High-contrast mode: use system colors */
@media (forced-colors: active) {
  :focus-visible { outline: 3px solid ButtonText; }
}

The inert Attribute

When a modal opens, keyboard users can still Tab to links and buttons hiding behind it in the background — elements they cannot even see. The inert attribute fixes this: add it to the background content and everything inside becomes untouchable, unclickable, and invisible to screen readers.

LISTING 17.18 — inert: Before and After

<!-- ✗ Without inert — focus leaks everywhere -->
<main id="bg">
  <a href="/about">About</a>        <!-- ✗ Tab still reaches it -->
  <button>Delete account</button>   <!-- ✗ Tab still reaches it -->
  <!-- User can trigger these without seeing them -->
</main>

<!-- ✓ With inert — background is completely locked -->
<main id="bg" inert>
  <a href="/about">About</a>        <!-- ✓ Tab skips it -->
  <button>Delete account</button>   <!-- ✓ Tab skips it -->
  <!-- Screen reader also skips everything inside -->
</main>

<script>
const modal   = document.getElementById('modal');
const content = document.getElementById('bg');
const openBtn = document.getElementById('open-btn');

openBtn.addEventListener('click', () => {
  modal.showModal();
  content.inert = true;    // lock the background
});

modal.addEventListener('close', () => {
  content.inert = false;   // restore the background
  openBtn.focus();         // return focus to the trigger
});
</script>

NOTE: inert replaces the need to manually set tabindex="-1" on every focusable child, toggle aria-hidden, and set pointer-events: none. One attribute does all of that. Supported in all modern browsers.

The Native <dialog> Element

Building accessible modal dialogs from scratch is notoriously difficult: you must trap focus inside, close on Escape, manage ARIA attributes, return focus to the trigger on close, and prevent background scrolling. The native <dialog> element handles all of this automatically with one method call: showModal().

Here is every piece of the <dialog> API you need to know:

PieceWhat it does
<dialog>The element itself — hidden in the DOM until opened
showModal()Opens as a true modal — traps focus, adds backdrop, Esc closes it
close(value)Closes the dialog; optional string records why it closed
dialog.returnValueThe string passed to close() — read it in the close event
"close" eventFires every time the dialog closes — the place to react
dialog::backdropCSS pseudo-element — style the dark overlay behind the dialog

LISTING 17.19 — The <dialog> Element: Complete Example

<!-- 1. The button that opens the dialog -->
<button id="openBtn">Delete photo</button>

<!-- 2. The dialog — hidden until showModal() is called -->
<dialog id="myDialog">
  <h2>Confirm Delete</h2>
  <p>This cannot be undone.</p>
  <button id="cancelBtn">Cancel</button>
  <button id="deleteBtn">Delete</button>
</dialog>

<script>
const dialog = document.getElementById('myDialog');

// 3. Open it — one line does everything
document.getElementById('openBtn').addEventListener('click', () => {
  dialog.showModal();
  // ✓ Screen reader: "Confirm Delete, dialog"
  // ✓ Focus moves inside automatically
  // ✓ Tab is trapped — can't reach the background page
  // ✓ Esc key closes it automatically
});

// 4. Close with a reason string
document.getElementById('cancelBtn').addEventListener('click', () => {
  dialog.close('cancelled');
  // ✓ Focus returns to #openBtn automatically
});
document.getElementById('deleteBtn').addEventListener('click', () => {
  dialog.close('deleted');
});

// 5. React to the close event
dialog.addEventListener('close', () => {
  console.log(dialog.returnValue); // "cancelled" or "deleted"
});
</script>

<!-- 6. Style the dark overlay in CSS -->
<style>
dialog::backdrop { background: rgba(0, 0, 0, 0.6); }
</style>

Live Regions with aria-live

Imagine ordering food delivery with your eyes closed. You submit the order, and 30 seconds later the page updates to say "Order confirmed!" But nobody told you. You are still sitting there wondering if it worked. That is the experience of a blind user on most websites — content updates happen visually, but screen readers only announce things when focus moves. aria-live fixes this by telling the browser: "watch this region — whenever the text inside it changes, read it aloud immediately, even if focus didn't move."

Here is every aria-live option you need to know:

SyntaxWhat it doesWhen to use it
aria-live="polite"Waits for screen reader to finish, then announcesSuccess messages, cart updates, search results
aria-live="assertive"Interrupts immediately — reads now, no waitingBlocking errors only — use sparingly
aria-live="off"Never announce changes (the default)Areas you don't want read aloud
role="status"Shorthand for aria-live="polite"Non-urgent status messages
role="alert"Shorthand for aria-live="assertive"Error messages — reads immediately
aria-atomic="true"Read the entire region, not just the changed partCounters, timers — where context matters

CAUTION: The live region must exist in the HTML before you update it. Add aria-live to an empty element at page load — the announcement fires when you inject text later. Setting aria-live at the same moment you add the text is too late and will be ignored by most screen readers.

LISTING 17.20 — aria-live Regions

<!-- ✗ Wrong — aria-live added at the same time as the update; too late -->
<p id="msg"></p>
<script>
  msg.setAttribute('aria-live', 'polite'); // ignored
  msg.textContent = 'Order confirmed!';    // silent
</script>

<!-- ✓ Correct — aria-live is in the HTML from the start -->
<p id="msg" aria-live="polite"></p>
<script>
  // Now any text change is announced automatically
  msg.textContent = 'Order confirmed! Arriving in 25 min.';
  // Screen reader: "Order confirmed! Arriving in 25 min." ✓
</script>

<!-- ✓ For errors — role="alert" interrupts immediately -->
<p id="err" role="alert"></p>
<script>
  err.textContent = 'Card number is invalid.';
  // Screen reader interrupts and reads this right now ✓
</script>

Skip Navigation Links

Imagine visiting ESPN every morning to check scores. ESPN's navigation has around 40 links before you reach the first headline. If you are using a keyboard, that is 40 Tab presses just to get to the content — every single page load, every single day. A skip link solves this: it is the very first thing on the page, invisible to mouse users until focused, but when a keyboard user presses Tab it appears and lets them jump straight to the main content with one keypress instead of forty. WCAG 2.4.1 (Level A) requires this — it is one of the highest-impact accessibility features you can add, and it takes about 10 lines of HTML and CSS.

LISTING 17.21 — Skip Navigation Link

<!-- First element in <body> -->
<a class="skip-link" href="#main-content">Skip to main content</a>

<header>...40 navigation links...</header>

<main id="main-content" tabindex="-1">
  <!-- tabindex="-1" allows programmatic focus from the skip link -->
</main>
/* Hidden by default — slides into view on focus */
.skip-link {
  position: absolute;
  top: -100%;
  left: 0;
  background: #4f6df5;
  color: white;
  padding: 0.75rem 1.5rem;
  font-weight: 600;
  text-decoration: none;
  z-index: 9999;
  transition: top 0.1s;
}

.skip-link:focus { top: 0; }

Summary

In the first half, you learned about modern CSS features now available natively in all modern browsers. CSS Nesting keeps related styles organized without a preprocessor — the & selector covers pseudo-classes, pseudo-elements, modifiers, siblings, context parents, and nested at-rules. :has() enables styling parent elements based on their contents, replacing significant amounts of JavaScript. :is() shortens repetitive selector lists while preserving specificity; :where() does the same with zero specificity for base styles. @layer solves specificity wars by introducing a priority stack where layer position beats specificity entirely. The prefers-color-scheme media query and light-dark() function make dark mode theming elegant, while color-mix() generates tints and shades without hardcoding values. @supports enables progressive enhancement — working experiences for all browsers, enhanced for those that support newer features.

In the second half, you learned that accessibility is not optional — it is a legal requirement in many jurisdictions, an ethical responsibility, and a business advantage. Every element in the accessibility tree has four properties: Role, Name, State, and Value — semantic HTML fills these in automatically, while <div>s fill in none of them. ARIA extends the tree with roles, labels, and states when native HTML is not sufficient — but the first rule of ARIA is to prefer native elements. You learned to manage focus with tabindex, show focus rings only during keyboard navigation with :focus-visible, and lock background content with inert. The native <dialog> element with showModal() handles focus trapping, Escape, backdrop, and accessibility attributes automatically. aria-live announces dynamic content changes to screen readers — the region must exist in HTML before you update it. Skip navigation links are a Level A WCAG requirement and take ten lines of code. Test with Lighthouse first, then axe, then a real screen reader.

Q&A

Q. Does CSS Nesting work the same as Sass/SCSS nesting?

A. Almost. Native CSS Nesting requires the & selector when writing pseudo-classes or modifiers on the parent element. If you nest a plain class like .title { } without &, the browser reads it as a descendant (& .title). In everyday usage the syntax and behavior are effectively the same as Sass, but in edge cases involving specificity and combinators, native CSS computes selectors slightly differently. For legacy browser support, PostCSS can compile native nesting syntax.

Q. When should I use :has() instead of JavaScript?

A. Any time you are adding a class to a parent element based on a child's state — for example, adding .is-focused to a .field wrapper when its input receives focus — that is a perfect candidate for :has(input:focus). No event listeners, no class toggling. Use JavaScript when you need to respond to the state change with behavior (side effects, API calls, rendering) rather than purely styling the page.

Q. What's the difference between :is() and :where() in practice?

A. They work identically except for specificity. :is(h1, h2) has the specificity of h1 or h2 (whichever is highest). :where(h1, h2) has zero specificity. Use :where() in base and reset stylesheets where you want any specific selector to easily override the rule. Use :is() in component or utility styles where normal specificity behavior is expected.

Q. If ARIA only adds to the accessibility tree and doesn't add keyboard behavior, what actually handles keyboard interaction for custom widgets?

A. JavaScript. When you build a custom widget — a date picker, a combobox, a tab interface — you are responsible for implementing all keyboard interactions. The WAI-ARIA Authoring Practices Guide at www.w3.org/WAI/ARIA/apg/ documents the expected keyboard behavior for every ARIA widget pattern. This is one of the strongest reasons to prefer native HTML elements: they come with keyboard behavior built in.

Q. Why does the aria-live region have to exist in the HTML before I update it?

A. Screen readers watch live regions continuously from the moment the page loads. If you add aria-live to an element at the same time you inject text into it, the screen reader has not had a chance to register the element as a live region yet and ignores the update. The correct pattern is to always have an empty live region in the HTML, then inject text into it later — the screen reader is already watching and announces the change automatically.

Q. Is using a CSS variable for theming the same as using light-dark()?

A. They achieve the same result but in different ways. The CSS custom properties approach — defining tokens on :root and overriding them in a media query — is more explicit and more controllable. You can also toggle dark mode via a class, which is useful for a user-controlled theme toggle button. light-dark() is more concise but relies solely on the OS preference and is harder to override with a manual toggle. For most production sites, the custom properties approach is recommended; use light-dark() for simpler projects or when OS-only switching is sufficient.

Q. How do I implement a "dark mode toggle" button that lets users override the OS setting?

A. Store the user's preference in localStorage, then add a data-theme="dark" attribute to the <html> element based on the saved preference. In CSS, define your color tokens for both @media (prefers-color-scheme: dark) (OS preference) and [data-theme="dark"] (manual toggle). The manual attribute should always win, so list it after the media query. Restore the preference from localStorage before the first paint — do this in a <script> tag in the <head>, before any stylesheets load, to avoid a flash of the wrong theme.
