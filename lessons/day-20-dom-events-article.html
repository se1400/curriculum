DOM Events & Event Handling

What You’ll Learn in This Lesson

  • What browser events are and how they make web pages interactive
  • How to attach and remove event listeners with addEventListener and removeEventListener
  • How to use the event object — event.target, event.type, event.preventDefault(), and more
  • The full range of mouse events and the modern Pointer Events API
  • Keyboard events: keydown, keyup, event.key, event.code, and modifier keys
  • Form events: submit, input, change, focus, blur, and their variants
  • Window and document events: DOMContentLoaded, load, resize, and scroll
  • How event bubbling works, and how to use it for event delegation
  • Event delegation — attaching one listener to a parent to handle many children
  • Modern addEventListener options: once and passive

Selecting elements from the DOM — with querySelector, getElementById, and their siblings — is only the first half of the equation. The second half is responding to what users do with those elements. That is what events are for.

Events are the heartbeat of every interactive web page. Every button click, every keystroke, every form submission, every page scroll — each of these is an event that your JavaScript code can listen for and respond to. By the end of this lesson you will be able to wire up rich interactivity with confidence, understand the mechanics that make it all work, and write clean, maintainable event-handling code using the patterns that professional developers use today.

What Events Are

Imagine you are sitting at a café and a bell above the door rings whenever a customer enters. The bell is not the customer — it is a signal that something happened. Your job as the barista is to respond to that signal: look up, say hello, take an order. In the browser, events work exactly this way. The browser is constantly generating signals — “someone clicked here,” “someone pressed a key,” “the page finished loading” — and your JavaScript code can listen for those signals and respond.

Events are not just user actions. They include browser lifecycle moments (the DOM finished parsing, the window was resized), network responses, timer completions, media playback changes, and more. But user interaction events are where you will spend the most time as a beginner, so that is where we focus first.

The Old Way: Inline Event Handlers

The earliest web pages wired up events directly in HTML attributes:

LISTING 20.1 — Inline Event Handlers (The Old Way)

<!-- Inline handler — considered bad practice today -->
<button onclick="alert('Hello!')">Click me</button>

<!-- Slightly better, but still problematic -->
<button onclick="handleClick()">Click me</button>

<script>
function handleClick() {
  alert('Hello!');
}
</script>

This approach worked, but it mixes behavior (JavaScript) directly into structure (HTML), making code harder to read, test, and maintain. It also has a practical limitation: you can only assign one handler per element per event type this way. The modern approach — addEventListener — solves all of these problems.

addEventListener — The Modern Approach

The addEventListener method is the standard way to register event handlers in modern JavaScript. You call it on any DOM element (or on window or document) and pass it two required arguments: the event type as a string, and a function to call when that event fires.

LISTING 20.2 — Basic addEventListener

// Get a reference to the element
const btn = document.querySelector('#myButton');

// Attach a listener: type is "click", handler is an arrow function
btn.addEventListener('click', function(event) {
  console.log('Button was clicked!');
});

// You can also pass an arrow function
btn.addEventListener('click', (event) => {
  console.log('Also fires on click!');
});

Notice that you can attach multiple listeners to the same element for the same event type. Both handlers above will run when the button is clicked. This is one of the key advantages over the old onclick property approach — assigning to element.onclick replaces any previous handler; addEventListener stacks them.

Why addEventListener Is Better

Here is a direct comparison to understand why addEventListener is the right choice:

LISTING 20.3 — onclick Property vs. addEventListener

const btn = document.querySelector('#btn');

// --- The onclick PROPERTY approach ---
btn.onclick = () => console.log('Handler A');
btn.onclick = () => console.log('Handler B'); // Replaces A! Only B runs.

// --- addEventListener approach ---
btn.addEventListener('click', () => console.log('Handler A')); // Runs
btn.addEventListener('click', () => console.log('Handler B')); // Also runs!

// addEventListener also separates your JavaScript from your HTML entirely.
// Your HTML stays clean; all behavior lives in .js files.

Removing Event Listeners

Sometimes you need to stop listening for an event — perhaps after a one-time action, or when a component is torn down. removeEventListener removes a specific listener. Here is the critical rule: you can only remove a listener if you have a reference to the exact same function that was passed to addEventListener. Anonymous functions are created fresh every time they are written in code, so two anonymous functions are never the same reference, even if they have identical bodies.

LISTING 20.4 — removeEventListener Requires a Named Function

const btn = document.querySelector('#btn');

// PROBLEM: anonymous function — cannot remove it
btn.addEventListener('click', () => console.log('I cannot be removed'));
btn.removeEventListener('click', () => console.log('I cannot be removed')); // Does nothing!
// These two arrow functions look identical but they are two different objects.

// SOLUTION: use a named function reference
function handleClick(event) {
  console.log('Click! I can be removed.');
}

btn.addEventListener('click', handleClick);

// Later, when you want to stop listening:
btn.removeEventListener('click', handleClick); // Works correctly

NOTE: The event type string passed to removeEventListener must match the one used in addEventListener exactly, and the capture flag (if used) must also match. If you added a listener in the capturing phase, you must remove it in the capturing phase.

The Event Object

When a browser fires an event and calls your handler function, it automatically passes one argument: the event object. This object is a rich description of exactly what happened — what was clicked, where the mouse was, which key was pressed, and much more. You access it by declaring a parameter in your handler function. It is commonly named event, evt, or just e.

LISTING 20.5 — The Event Object

document.querySelector('#btn').addEventListener('click', function(event) {
  // The event object is automatically passed as the first argument

  console.log(event.type);          // "click"
  console.log(event.target);        // The element that was actually clicked
  console.log(event.currentTarget); // The element this listener is attached to
  console.log(event.timeStamp);     // When the event fired, in milliseconds
});

event.target vs. event.currentTarget

These two properties are often confused, and understanding them is essential for event delegation. Think of it this way:

  • event.target — the element the user actually interacted with. If you click an <img> inside a <button>, event.target is the <img>.
  • event.currentTarget — the element the listener is attached to. In the same example, if the listener is on the <button>, event.currentTarget is always the <button>, regardless of what was clicked inside it.

LISTING 20.6 — target vs. currentTarget

const btn = document.querySelector('#btn');

btn.addEventListener('click', (event) => {
  // If the button contains an icon and the user clicks the icon:
  console.log(event.target);        // The <svg> icon element (what was clicked)
  console.log(event.currentTarget); // The <button> (where the listener lives)

  // In a regular function (not arrow), "this" equals currentTarget:
  // console.log(this); // <button>
});

event.preventDefault()

Many HTML elements have built-in default behaviors: links navigate to their href, form submit buttons send a form request and reload the page, right-clicking opens a context menu. Calling event.preventDefault() inside your handler cancels that default behavior, letting you take over with your own JavaScript logic.

LISTING 20.7 — Preventing Default Behavior

// Prevent a link from navigating
document.querySelector('a#no-follow').addEventListener('click', (event) => {
  event.preventDefault();
  console.log('Link click intercepted — no navigation occurred');
});

// Handle form submission with JavaScript instead of a page reload
document.querySelector('form').addEventListener('submit', (event) => {
  event.preventDefault(); // Stop the browser from sending an HTTP request

  const data = new FormData(event.target);
  console.log('Name:', data.get('name'));
  // Now you can validate, send via fetch(), update the UI, etc.
});

// Prevent the context menu (use sparingly — it's often unwelcome)
document.querySelector('.no-right-click').addEventListener('contextmenu', (event) => {
  event.preventDefault();
});

event.stopPropagation()

Events in the browser travel through the DOM in a process called propagation. event.stopPropagation() halts that travel. We will cover this fully in the bubbling and capturing section, but for now know that it exists and its purpose: to prevent an event from reaching parent elements after the current handler has run.

Mouse Events

Mouse events cover the full range of actions a pointing device can perform. Each event fires at a specific moment during an interaction, and choosing the right event matters for both correctness and performance.

Click Events

click fires when the user presses and releases a mouse button over an element. dblclick fires on a double click. Note that a double click also fires two click events before the dblclick — if you listen to both on the same element, you may need to debounce or distinguish between them.

LISTING 20.7a — Click Event: Counter and Toggle

// --- Counter ---
const countBtn = document.querySelector('#count-btn');
const countDisplay = document.querySelector('#count-display');
let count = 0;

countBtn.addEventListener('click', () => {
  count++;
  countDisplay.textContent = `Clicked ${count} time${count === 1 ? '' : 's'}`;
});

// --- Toggle visibility ---
const toggleBtn = document.querySelector('#toggle-btn');
const panel = document.querySelector('#panel');

toggleBtn.addEventListener('click', () => {
  const isHidden = panel.hidden;
  panel.hidden = !isHidden;
  toggleBtn.textContent = isHidden ? 'Hide panel' : 'Show panel';
});

Mouse Enter and Leave

mouseenter fires once when the pointer moves into an element. mouseleave fires once when the pointer moves out. Neither event bubbles — meaning they fire only on the target element, not on parent elements when the mouse enters a child. This makes them the preferred choice for hover effects.

mouseover and mouseout are the bubbling equivalents. They fire on the element AND bubble up through ancestors. Additionally, they fire again whenever the mouse moves between child elements inside the target — which can cause surprising repeated firing. In most cases, prefer mouseenter/mouseleave.

Mouse Press and Release

mousedown fires the moment a mouse button is pressed, before it is released. mouseup fires when the button is released. The click event fires after both — it is a composite of down and up. Use mousedown/mouseup when you need to react to the press and release separately, such as in drag-and-drop interfaces.

Mouse Move

mousemove fires continuously as the pointer moves across an element — potentially dozens or even hundreds of times per second. Never do expensive work directly inside a mousemove handler. If you need to respond to mouse movement, throttle or debounce your handler, or use requestAnimationFrame to limit work to once per frame.

Mouse Coordinates

The event object for mouse events includes coordinate properties:

  • event.clientX / event.clientY — position relative to the browser's visible viewport. These do not change when you scroll.
  • event.pageX / event.pageY — position relative to the entire document. These account for scroll position, so they continue to increase as you scroll down a long page.
  • event.offsetX / event.offsetY — position relative to the target element itself.

LISTING 20.8 — Mouse Events in Practice

const box = document.querySelector('.interactive-box');

// Fires once on entry, once on exit — no repeated firing for child moves
box.addEventListener('mouseenter', () => {
  box.classList.add('hovered');
});

box.addEventListener('mouseleave', () => {
  box.classList.remove('hovered');
});

// Track mouse position for a custom cursor or tooltip
box.addEventListener('mousemove', (event) => {
  const tooltip = document.querySelector('#tooltip');
  tooltip.style.left = event.clientX + 16 + 'px';
  tooltip.style.top  = event.clientY + 16 + 'px';
});

Pointer Events

Pointer events (pointerdown, pointerup, pointermove, pointerenter, pointerleave) are the modern way to handle both mouse AND touch interactions with a single set of handlers. Where you used to need separate mousedown and touchstart listeners, one pointerdown handler covers both.

LISTING 20.9 — Pointer Event Handles Both Mouse and Touch 
const card = document.querySelector('.card');

card.addEventListener('pointerdown', (event) => {
  card.classList.add('pressed');
  card.textContent = `Pressed with: ${event.pointerType}`; // "mouse", "touch", or "pen"
});

card.addEventListener('pointerup', () => {
  card.classList.remove('pressed');
});

For interactive UI components that need to work on both desktop and mobile, prefer pointer events over separate mouse/touch handlers.

Keyboard Events

Keyboard events let you respond to key presses. They are essential for building accessible custom widgets, keyboard shortcuts, game controls, and rich text editing experiences.

keydown and keyup

keydown fires the moment a key is pressed and repeats while the key is held down. Most keyboard shortcut implementations listen to keydown. keyup fires when the key is released — it fires only once per press, so it is more appropriate when you want to know the final state after a key interaction rather than every repeat tick.

CAUTION: The old keypress event is deprecated and should not be used in new code. It had inconsistent behavior across browsers and did not fire for keys like Escape, Backspace, or arrow keys. Use keydown instead.

event.key and event.code

The event object for keyboard events provides two properties for identifying what was pressed, and it is important to understand the difference:

  • event.key — the value of the key, accounting for the current keyboard layout and modifier keys. Pressing the “a” key produces "a"; pressing it with Shift produces "A". Special keys produce their name: "Enter", "Escape", "ArrowUp", "Backspace", " " (Space). Use event.key for most character input handling.
  • event.code — the physical key on the keyboard, regardless of layout. The key labeled “A” on a QWERTY keyboard is always "KeyA", whether you press it alone or with Shift, and even if the user has a DVORAK layout configured. Use event.code for game controls or any situation where physical key position matters more than character value.

Modifier Keys

The event object includes boolean properties for each modifier key:

  • event.shiftKey — true if Shift was held
  • event.ctrlKey — true if Ctrl was held
  • event.altKey — true if Alt (Option on Mac) was held
  • event.metaKey — true if the Meta key was held (the Windows key on PC; the Command key on Mac)

You combine these with event.key to detect keyboard shortcuts:

LISTING 20.10 — Keyboard Events: Showing the Pressed Key on Screen

// Display the last pressed key directly in the page — no console needed
const keyDisplay = document.querySelector('#key-display'); // a <span> in your HTML

document.addEventListener('keydown', (event) => {
  keyDisplay.textContent = event.key; // updates the span every time a key is pressed

  // Detect Ctrl+S (or Cmd+S on Mac) to trigger a save action
  if ((event.ctrlKey || event.metaKey) && event.key === 's') {
    event.preventDefault(); // Prevent the browser's "Save Page" dialog
    document.querySelector('#save-status').textContent = 'Saved!';
  }

  // Close a panel on Escape
  if (event.key === 'Escape') {
    document.querySelector('#panel').hidden = true;
  }
});

// Real-time character counter — updates a visible counter as the user types
const textarea = document.querySelector('#message');
const charCount = document.querySelector('#char-count');

textarea.addEventListener('keyup', () => {
  const remaining = 280 - textarea.value.length;
  charCount.textContent = remaining + ' characters remaining';
});

NOTE: Keyboard events only fire when an element has focus. For global keyboard shortcuts (like Ctrl+S to save), attach the listener to document rather than a specific element, as shown in the listing above.

Form Events

Forms are one of the most important areas of web interactivity. The form event API gives you fine-grained control over every moment of a user's interaction — from the first keystroke in a field to the final submission.

submit

The submit event fires on the <form> element when the user submits it, either by clicking a submit button or pressing Enter in a text field. Always call event.preventDefault() when handling form submission with JavaScript — without it, the browser will attempt to send an HTTP request and navigate away from (or reload) the page, wiping out your JavaScript state.

LISTING 20.11a — submit: Displaying Form Data on Screen

// HTML assumed:
// <form id="contact-form">
//   <input name="name" type="text" placeholder="Your name">
//   <input name="email" type="email" placeholder="Your email">
//   <button type="submit">Send</button>
// </form>
// <div id="results"></div>

const form    = document.querySelector('#contact-form');
const results = document.querySelector('#results');

form.addEventListener('submit', (event) => {
  event.preventDefault(); // Stop the page from reloading

  const data = new FormData(form);
  results.innerHTML = `
    <p><strong>Name:</strong> ${data.get('name')}</p>
    <p><strong>Email:</strong> ${data.get('email')}</p>
  `;
});

input

The input event fires on an <input>, <textarea>, or <select> element every time its value changes — on every keystroke, paste, or cut. It is your primary tool for real-time feedback: live validation, character counters, search-as-you-type, and dynamic previews.

LISTING 20.11b — input: Live Preview of Typed Text

// HTML assumed:
// <input id="username" type="text" placeholder="Choose a username">
// <p>Preview: <span id="preview">—</span></p>

const usernameInput = document.querySelector('#username');
const preview       = document.querySelector('#preview');

usernameInput.addEventListener('input', () => {
  preview.textContent = usernameInput.value || '—';
});

change

The change event fires when an element's value is committed rather than on every keystroke. For text inputs, it fires when the user tabs or clicks away from the field. For checkboxes, radio buttons, and <select> dropdowns, it fires immediately when the selection changes. Use change when you only care about the final value, not intermediate keystrokes.

LISTING 20.11c — change: Updating Visible Text for a Select

// HTML assumed:
// <select id="color-picker">
//   <option value="">Choose a color</option>
//   <option value="red">Red</option>
//   <option value="green">Green</option>
//   <option value="blue">Blue</option>
// </select>
// <p>Selected: <span id="color-label">none</span></p>

const picker     = document.querySelector('#color-picker');
const colorLabel = document.querySelector('#color-label');

picker.addEventListener('change', () => {
  colorLabel.textContent = picker.value || 'none';
});

focus and blur

focus fires when an element receives focus (either by clicking or tabbing). blur fires when it loses focus. These are useful for showing contextual help when a field is active, or triggering validation when the user leaves a field. One important note: neither focus nor blur bubbles, which can complicate event delegation. Their bubbling counterparts are focusin and focusout.

NOTE: Use CSS :focus and :focus-within pseudo-classes for purely visual focus styling. Reaching for JavaScript to add a class on focus is unnecessary in most cases and adds complexity. Save focus/blur event listeners for behavioral logic like showing or validating.

focusin and focusout

focusin and focusout are identical to focus and blur, except they do bubble. This makes them suitable for event delegation on a form or fieldset: you can attach one listener to the form and respond when any child input gains or loses focus.

reset

The reset event fires on a form when the user activates a reset button (<button type="reset">) or when you call form.reset() in JavaScript. Use it to clear any custom validation state or error messages you have added, since the browser only resets native field values, not your custom UI.

LISTING 20.11 — Form Events: Validation, Submission Result, and Reset

const form         = document.querySelector('#signup-form');
const email        = document.querySelector('#email');
const errorMsg     = document.querySelector('#error-msg');
const confirmation = document.querySelector('#confirmation');

// Real-time validation — show/hide error text as the user types
email.addEventListener('input', () => {
  const isValid = email.value.includes('@') && email.value.includes('.');
  email.classList.toggle('invalid', !isValid);
  email.classList.toggle('valid',    isValid);
  errorMsg.textContent = isValid ? '' : 'Enter a valid email address.';
});

// Validate when the user leaves the field
email.addEventListener('blur', () => {
  if (!email.value) {
    errorMsg.textContent = 'Email is required.';
  }
});

// Handle submission — show collected data on screen instead of reloading
form.addEventListener('submit', (event) => {
  event.preventDefault(); // Stop the page from reloading

  const data = new FormData(form);
  confirmation.innerHTML = `
    <p>Submitted! Email: <strong>${data.get('email')}</strong></p>
  `;
});

// Clean up custom UI when the form is reset
form.addEventListener('reset', () => {
  errorMsg.textContent     = '';
  confirmation.textContent = '';
  email.classList.remove('invalid', 'valid');
});

Window and Document Events

Not all events fire on interactive elements. Some of the most important events fire on window or document and relate to the overall lifecycle of the page.

DOMContentLoaded

DOMContentLoaded fires on document when the browser has finished parsing the HTML and building the DOM — but before images, stylesheets, and other external resources have finished loading. At this point, all elements exist in the DOM and can be safely queried and manipulated with JavaScript.

This is the event you almost always want when you need to run setup code as soon as possible after the page loads.

load

The load event fires on window when everything on the page has finished loading — HTML, CSS, images, fonts, scripts. It fires significantly later than DOMContentLoaded on pages with many or large resources. Use load when your code specifically depends on an image's dimensions, a font's metrics, or another resource that must be fully loaded to work correctly. For general DOM manipulation, prefer DOMContentLoaded.

LISTING 20.12 — DOMContentLoaded vs. load

// Fires as soon as the DOM is ready — images may still be loading
document.addEventListener('DOMContentLoaded', () => {
  console.log('DOM ready! Setting up event listeners...');
  initializeApp();
});

// Fires when EVERYTHING is loaded — use for image-dependent layout
window.addEventListener('load', () => {
  console.log('All resources loaded!');
  const img = document.querySelector('#hero-image');
  console.log('Image dimensions:', img.naturalWidth, 'x', img.naturalHeight);
});

// The common pattern: if your <script> tag is at the END of <body>,
// the DOM is already ready when your script runs — no listener needed.
// DOMContentLoaded is mainly needed if your script is in <head>.

resize

The resize event fires on window continuously as the browser window is resized. Like mousemove, it can fire at very high frequency. If you do expensive layout calculations inside a resize handler, you will make the resize feel sluggish. The solution is to debounce the handler — delay execution until the user has stopped resizing for a short period (typically 100–250 milliseconds).

scroll

The scroll event fires on window (or any scrollable element) as scrolling occurs. It also fires at high frequency and should be debounced or throttled for heavy operations. For reading scroll position to trigger animations, prefer the newer IntersectionObserver API, which is more performant than scroll event listeners for visibility-detection use cases.

beforeunload

The beforeunload event fires on window just before the user navigates away from the page — by closing the tab, typing a new URL, or following an external link. If you set a value on event.returnValue, most browsers show a generic confirmation dialog asking the user if they really want to leave. Use this sparingly — only when the user has unsaved changes that would be lost.

LISTING 20.13 — Window Events

// Debounce helper — delays execution until N ms after the last call
function debounce(fn, delay) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
}

// Debounced resize handler
window.addEventListener('resize', debounce(() => {
  console.log('Window is now', window.innerWidth, 'x', window.innerHeight);
  recalculateLayout();
}, 200));

// Warn before leaving with unsaved changes
let hasUnsavedChanges = false;
window.addEventListener('beforeunload', (event) => {
  if (hasUnsavedChanges) {
    event.returnValue = ''; // Triggers the browser's "Leave page?" dialog
  }
});

Event Bubbling and Capturing

Bubbling and Capturing

When you click an element, the event travels in two phases: first down through parent elements to the target (capturing phase), then back up through parents (bubbling phase). In practice you almost always use bubbling — capturing is an advanced use case you can look up when needed.

The third argument to addEventListener controls which phase to listen on — pass true for capture or false (the default) for bubble:

// Bubbling (default) — fires as event travels UP
element.addEventListener('click', handler);        // same as false
element.addEventListener('click', handler, false);

// Capturing — fires as event travels DOWN (advanced, rarely needed)
element.addEventListener('click', handler, true);

Which Events Do Not Bubble?

Most events bubble, but a handful do not. The important non-bubbling events are: focus, blur, load, mouseenter, and mouseleave. Their bubbling counterparts (focusin/focusout, mouseover/mouseout) do bubble, which is why they exist as alternatives for delegation scenarios.

Event Delegation

Now that you understand bubbling, you are ready for one of the most important patterns in JavaScript event handling: event delegation.

Imagine a to-do list with 200 items, each with a Delete button. The naive approach attaches a separate click listener to each of those 200 buttons — 200 listeners in memory, 200 function objects, and none of them cover items added after the page loads. Event delegation solves all three problems at once by exploiting bubbling: attach one listener to the parent and let clicks on children bubble up to it.

LISTING 20.17 — The Problem: Listeners on Individual Items

// Wasteful: one listener per item
document.querySelectorAll('.delete-btn').forEach(btn => {
  btn.addEventListener('click', () => deleteItem(btn));
});

// Problem 1: 200 listeners for 200 items
// Problem 2: Items added later get no listener
// Problem 3: Removing items must also remove their listeners

LISTING 20.18 — The Solution: Event Delegation

const list = document.querySelector('#task-list');

// ONE listener on the parent handles all current AND future items
list.addEventListener('click', (event) => {
  // Check if the clicked element (or an ancestor within the list) is a delete button
  const deleteBtn = event.target.closest('.delete-btn');
  if (deleteBtn) {
    const listItem = deleteBtn.closest('li');
    listItem.remove();
  }

  // Handle edit buttons in the same listener
  const editBtn = event.target.closest('.edit-btn');
  if (editBtn) {
    openEditModal(editBtn.closest('li'));
  }
});

event.target.matches() vs. event.target.closest()

There are two tools for filtering delegated events:

  • event.target.matches(selector) — returns true if the target element itself matches the selector. This works when the element that fires the click is exactly the one you are targeting.
  • event.target.closest(selector) — walks up the DOM from the target, returning the first ancestor that matches the selector (including the element itself), or null if none is found. This is more robust: if your button contains an icon (<button><svg>...</svg></button>), clicking the icon sets event.target to the <svg>, not the <button>. closest('.delete-btn') still finds the parent button correctly.

In practice, closest() is almost always the right choice for delegation. Use matches() only when you are certain the element you care about has no interactive children.

LISTING 20.19 — matches() vs. closest() in Delegation

list.addEventListener('click', (event) => {
  // FRAGILE: fails if the user clicks the icon inside the button
  if (event.target.matches('.delete-btn')) {
    // event.target might be <svg> inside .delete-btn — this check fails!
    deleteItem(event.target);
  }

  // ROBUST: works even if a child element was clicked
  const deleteBtn = event.target.closest('.delete-btn');
  if (deleteBtn) {
    // deleteBtn is always the .delete-btn element, not a descendant
    deleteItem(deleteBtn.closest('li'));
  }
});

Modern addEventListener Options

The third argument to addEventListener accepts either a boolean (for the capture flag, as shown earlier) or an options object with several useful properties. These options give you more control over how and how long a listener stays active.

once: true

When once: true is set, the listener fires exactly one time and then automatically removes itself. This is clean, readable, and eliminates the need to manually call removeEventListener inside the handler.

LISTING 20.20 — The once Option

const btn = document.querySelector('#submit-btn');

// Fires once, then the listener is automatically removed
btn.addEventListener('click', () => {
  console.log('This will only fire once!');
  submitForm();
}, { once: true });

// Equivalent without the option — more verbose:
btn.addEventListener('click', function handler() {
  console.log('This will only fire once!');
  submitForm();
  btn.removeEventListener('click', handler);
});

passive: true

When the browser sees a touchstart or wheel event listener, it must wait to see if your handler calls event.preventDefault() before it can scroll the page — because calling preventDefault() on those events would cancel scrolling. This waiting causes scroll jank on mobile devices. Setting passive: true is a promise to the browser that you will never call preventDefault() in this handler, freeing it to scroll immediately and smoothly.

LISTING 20.21 — The passive Option

// WITHOUT passive — browser waits before scrolling, causing potential jank
window.addEventListener('touchstart', handleTouchStart);

// WITH passive — browser scrolls immediately without waiting
window.addEventListener('touchstart', handleTouchStart, { passive: true });
window.addEventListener('wheel', handleWheelZoom, { passive: true });

// Modern browsers add passive: true by default for touchstart and wheel
// on the window and document, but it is good practice to be explicit.

function handleTouchStart(event) {
  // You CANNOT call event.preventDefault() here — it would throw a warning
  recordTouchStart(event.touches[0].clientX, event.touches[0].clientY);
}

AbortController

AbortController lets you cancel multiple event listeners at once — a clean-up pattern used in component-based frameworks. For most projects, removing individual listeners with removeEventListener (passing the same named function) covers your needs:

// Removing an individual listener
function handleClick(event) { /* ... */ }
button.addEventListener('click', handleClick);
button.removeEventListener('click', handleClick); // later

Summary

Events are the foundation of interactive web development. Every user action — every click, keystroke, form entry, and scroll — generates an event that your JavaScript can respond to. In this lesson you moved from the old inline onclick approach to the modern addEventListener method, understanding why it is superior: multiple listeners, clean separation of concerns, and the ability to remove listeners when they are no longer needed.

You explored the event object in depth — event.target, event.currentTarget, event.type, event.preventDefault(), and event.stopPropagation() — and you saw how these properties combine to give you precise control over what happens when events fire. You worked through the full range of mouse events and learned about the Pointer Events API, which unifies mouse and touch input into one set of handlers with pointerdown, pointerup, and friends. You learned keyboard events, the distinction between event.key (the character value) and event.code (the physical key), and how to combine modifier keys to detect shortcuts.

Form events — submit, input, change, focus, blur, and their bubbling variants — give you everything you need for real-time validation and custom form handling. Window and document events like DOMContentLoaded and load let you time your initialization code correctly. You understood how events bubble up through the DOM, and how that bubbling is the foundation of event delegation.

Event delegation — attaching one listener to a parent and using event.target.closest() to identify the relevant child — is one of the most important patterns in JavaScript. It handles dynamic elements automatically and keeps your code lean. Finally, you learned the modern options object for addEventListener: once for self-removing listeners, and passive for scroll performance. For removing individual listeners, pass a named function reference to removeEventListener.

Q&A

Q. What happens if I call addEventListener with the same function and event type twice on the same element?

A. The browser ignores the duplicate. If you call element.addEventListener('click', myFn) twice with the exact same function reference and the same capture flag setting, only one listener is registered. However, if you pass two different function objects — even if they have identical bodies, as with two separate arrow functions — both are registered as separate listeners and both will fire. This is a common source of accidental duplicate handlers when a setup function is called more than once without cleanup.

Q. When should I use event.stopPropagation() and when should I avoid it?

A. Use it carefully and deliberately. stopPropagation() can break other listeners higher in the DOM that legitimately need to know about the event — for example, a global click listener that closes an open dropdown whenever the user clicks anywhere. If you stop propagation inside the dropdown toggle button, that global listener never fires and the dropdown may not close correctly. A better pattern is often to check event.target in the global listener and decide whether to act, rather than stopping propagation at the source. Use stopPropagation() when you are deliberately building a contained widget with its own propagation rules, and you fully control all the listeners involved.

Q. Should I use Pointer Events instead of mouse events for all new projects?

A. For any interactive element that might be used on a touchscreen — which is most elements on a public-facing website — yes, Pointer Events are the right choice. They work identically on mouse, touch, and stylus input with one set of handlers. Mouse events still work everywhere and are fine for desktop-only tools, but Pointer Events are now universally supported and are the better default for interactive elements that need to work on both desktop and mobile.

Q. Why does my event delegation pattern sometimes fire when I click empty space in the list container, not on a list item?

A. Because clicks on empty space inside the container still bubble up to the container's listener. event.target will be the container element itself (or a non-list-item child). The fix is to always check the result of event.target.closest(selector) before acting. If closest() returns null, no matching ancestor was found and you should return early. The pattern is: const item = event.target.closest('li'); if (!item) return;. This guard makes your delegation handler robust against clicks in empty areas.

Q. What is the difference between DOMContentLoaded and putting my script at the bottom of the page?

A. Both achieve the same goal — ensuring the DOM exists before your script runs — but through different mechanisms. Placing your <script> tag at the end of <body>, just before </body>, means the browser has already parsed all the HTML above it by the time it executes your script. This is simple and reliable. DOMContentLoaded is useful when your script must live in <head> (for example, in a framework or CMS that controls page output), because it delays execution until the DOM is ready regardless of where the script tag is placed. Both approaches are valid; placing scripts at the bottom is simpler for beginners.

Q. When would I use AbortController instead of removeEventListener?

A. removeEventListener works well for removing a single listener — pass it the exact same named function you used in addEventListener and the listener is gone. AbortController becomes useful when you need to remove several listeners at once without tracking each function reference individually. It is a pattern you will encounter more in framework code than in everyday beginner projects, so removeEventListener with named functions is the right tool to reach for first.
