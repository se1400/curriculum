You have built card grids from arrays of strings: ['Tokyo', 'Lisbon', 'Chicago']. That works for simple lists, but real web applications deal with richer data. A product catalog needs names, prices, categories, and images. A recipe gallery needs titles, prep times, and difficulty levels. A team page needs names, roles, and bios.

JavaScript objects solve this. An object groups multiple related values under one name. Instead of three separate arrays for names, prices, and categories, you have one array where each item is an object containing all three. This is the data structure that powers every product listing, portfolio grid, and content gallery you have ever used.

JavaScript Objects

An object is a collection of key-value pairs wrapped in curly braces. Each key is a property name; each value is the data stored under that name:

const product = {
  name:     'Wireless Headphones',
  price:    79.99,
  category: 'Electronics',
  inStock:  true
};

Read this as a labeled container: a product with a name of 'Wireless Headphones', a price of 79.99, and so on. Each piece of related data lives together in one object rather than spread across separate variables.

Dot Notation

Access any property using a dot followed by the property name:

console.log(product.name);      // 'Wireless Headphones'
console.log(product.price);     // 79.99
console.log(product.inStock);   // true

You can also set or add properties the same way:

product.price  = 69.99;   // Update an existing property
product.rating = 4.5;     // Add a new property

Bracket Notation

An alternative syntax uses square brackets and a string for the key. This is equivalent to dot notation but necessary when the property name is dynamic (stored in a variable):

const key = 'name';
console.log(product[key]);   // 'Wireless Headphones' — same as product.name

NOTE
Use dot notation (product.name) in almost all cases — it is shorter and more readable. Use bracket notation (product[key]) only when the property name comes from a variable or is computed at runtime.

Arrays of Objects

Combine what you know about arrays with objects, and you get the core data structure of web development:

const products = [
  { id: 1, name: 'Headphones',  price: 79.99, category: 'Electronics' },
  { id: 2, name: 'Sneakers',    price: 49.99, category: 'Footwear'    },
  { id: 3, name: 'Coffee Mug',  price: 12.99, category: 'Kitchen'     },
  { id: 4, name: 'Yoga Mat',    price: 34.99, category: 'Fitness'     }
];

Each object in the array is one record — one product, one recipe, one team member. The array holds all the records. Every real-world content gallery, product listing, or data table follows this exact shape.

Note the id property. Every object should have a unique identifier. You will use it to connect rendered cards back to their source data when a user clicks one.

Accessing Nested Data

const first = products[0];        // The first product object
console.log(first.name);          // 'Headphones'
console.log(products[1].price);   // 49.99 — second product's price

Rendering Arrays of Objects as Cards

The renderCards() pattern works unchanged — you just access object properties in the template literal instead of the string directly:

const container = document.querySelector('.grid');

function renderProducts(items) {
  container.innerHTML = items.map(product => `
    <div class="card" data-id="${product.id}">
      <span class="badge">${product.category}</span>
      <h3 class="card-title">${product.name}</h3>
      <p class="card-price">$${product.price.toFixed(2)}</p>
    </div>
  `).join('');
}

renderProducts(products);

Two details to notice:

  • ►product.price.toFixed(2) formats the number to exactly two decimal places: 79.99, not 79.9 or 79.990000001.
  • ►data-id="${product.id}" stores the record’s ID in the HTML. When a user clicks the card, your JavaScript can read this attribute to identify which product was clicked.

Sorting Arrays

The .sort() method reorders an array in place. You provide a comparator function that tells JavaScript how to compare two items:

// Sort by price, low to high
const byPriceAsc = [...products].sort((a, b) => a.price - b.price);

// Sort by price, high to low
const byPriceDesc = [...products].sort((a, b) => b.price - a.price);

// Sort by name, alphabetically
const byName = [...products].sort((a, b) => a.name.localeCompare(b.name));

CAUTION
.sort() modifies the original array. Always sort a copy using the spread operator: [...products].sort(). If you sort the original, there is no way to restore the default order without reloading the page.

How the comparator works: if the function returns a negative number, a comes first. If positive, b comes first. If zero, their order is unchanged. For numbers, a.price - b.price produces the right sign automatically. For strings, localeCompare() handles alphabetical ordering correctly across all languages.

After sorting, pass the sorted copy to renderProducts() — the grid rebuilds with the new order:

document.querySelector('#sort-price').addEventListener('click', () => {
  const sorted = [...products].sort((a, b) => a.price - b.price);
  renderProducts(sorted);
});

Connecting Cards to Data: the data-id Pattern

When a user clicks a rendered card, you need to know which item they clicked so you can look up the full object. The data-id attribute makes this clean: the card carries the ID in the HTML, and your JavaScript reads it back with dataset.id.

container.addEventListener('click', function(event) {
  const card = event.target.closest('.card');
  if (!card) return;  // Click was not on a card

  const id      = Number(card.dataset.id);
  const product = products.find(p => p.id === id);

  if (product) {
    showDetail(product);
  }
});

Three things are happening here:

  • ►event.target.closest('.card') — finds the nearest ancestor with class card, even if the user clicked a child element like the title or badge
  • ►Number(card.dataset.id) — converts the ID string back to a number (dataset always returns strings)
  • ►products.find(p => p.id === id) — locates the matching object in the original array

NOTE
You are using a single event listener on the parent container rather than one on each card. This is called event delegation and it works because click events bubble up through the DOM. It is especially important for dynamic content: cards generated by JavaScript are not in the HTML when the page loads, so you cannot attach listeners to them directly at startup. A listener on the static parent container catches all clicks regardless of when the cards were added.

Empty States

When filtering or searching produces zero results, your render function should show a helpful message rather than a blank space:

function renderProducts(items) {
  if (items.length === 0) {
    container.innerHTML = '<p class="empty-state">No items match your search.</p>';
    return;
  }

  container.innerHTML = items.map(product => `
    <div class="card">...</div>
  `).join('');
}

Good empty states are part of strong UX — the “error prevention” and “visibility of system status” heuristics apply here. The user should never wonder whether the page is broken or simply has no results.

Building the Object

Data does not have to be hardcoded. You can build an object from form values and push it into the array:

form.addEventListener('submit', function(event) {
  event.preventDefault();

  const newProduct = {
    id:       products.length + 1,
    name:     document.querySelector('#name').value,
    price:    Number(document.querySelector('#price').value),
    category: document.querySelector('#category').value
  };

  products.push(newProduct);
  renderProducts(products);
});

Form fields become object properties. The new object gets pushed to the array. renderProducts() is called again — and the new card appears alongside all the others.

Best Practices

  • ►Give every object a unique id. Even if you do not need it immediately, you will need it the moment users start interacting with individual cards. Adding it from the start is cheap; retrofitting it later is not.
  • ►Sort copies, not originals. [...products].sort() every time. The original array is your single source of truth — protect it.
  • ►Convert numeric IDs immediately. dataset.id returns a string. Wrap it in Number() before comparing with === to numeric IDs.
  • ►Always handle the empty state. A render function that returns nothing when its input is empty is not finished. Add a check and a message.
  • ►Use data that feels real. Placeholder names like “Item 1, Item 2” make it hard to spot layout bugs. Use realistic names, prices, and categories from the start.

Summary

Objects group related data under one name using key-value pairs, and dot notation reads or writes any property. Arrays of objects are the core data structure of interactive web development: one array, one object per record, one render function that transforms them into HTML. You learned to sort arrays without mutating the original, use .toFixed(2) for formatted prices, store IDs in data-id attributes to connect rendered cards back to their source data, and handle empty states gracefully. The CSS card layouts you have designed are now powered entirely by data instead of handwritten HTML — and the data can come from anywhere.
