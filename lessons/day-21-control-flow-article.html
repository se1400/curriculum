Control Flow: Making Decisions in JavaScript

What You'll Learn in This Lesson

  • How if, else, and else if let your program choose different paths based on conditions
  • How comparison operators (===, !==, >, <, >=, <=) produce the boolean results your conditions depend on
  • How logical operators (&&, ||, !) combine conditions and how short-circuit evaluation works
  • How the ternary operator provides a compact, single-expression alternative to if/else
  • How the switch statement cleanly handles multiple discrete values of the same variable
  • How the nullish coalescing operator (??) provides defaults only for null and undefined — not for 0, "", or false
  • How optional chaining (?.) lets you safely navigate nested objects without throwing TypeError
  • How logical assignment operators (&&=, ||=, ??=) condense common assignment patterns into one line
  • How to combine all of these tools to build realistic, practical decision logic

What We're Building Toward

Control flow is what makes pages respond intelligently. Here is a small example of where this lesson is headed — a form that validates input and shows the right message to users:

LISTING 21.0 — Control Flow Powering a Real UI 
const ageInput = document.querySelector('#age');
const messageEl = document.querySelector('#message');

ageInput.addEventListener('input', () => {
  const age = Number(ageInput.value);

  if (!age) {
    messageEl.textContent = "Please enter your age.";
    messageEl.className = "neutral";
  } else if (age < 18) {
    messageEl.textContent = "You must be 18 or older to continue.";
    messageEl.className = "error";
  } else if (age > 120) {
    messageEl.textContent = "Please enter a valid age.";
    messageEl.className = "error";
  } else {
    messageEl.textContent = "Age accepted! Welcome.";
    messageEl.className = "success";
  }
});

Every concept in this lesson — conditionals, logical operators, comparisons — is a building block for exactly this kind of user-facing logic.

The programs you have written so far largely run from top to bottom: declare a variable, manipulate it, display a result. Real programs are rarely that linear. A shopping cart applies a discount code only if the cart total exceeds a threshold. A login form shows an error message only if the password is wrong. A weather app displays a rain warning only if the forecast includes precipitation. These programs make decisions, and the mechanism JavaScript uses to make decisions is called control flow.

This lesson is one of the most important in the curriculum. Every non-trivial program you will ever write uses control flow. By the end, you will be able to handle complex branching logic using the full modern JavaScript toolkit — not just the classic if statement, but the nullish coalescing operator, optional chaining, and logical assignment operators that make real-world code cleaner and more resilient.

What Control Flow Is

Think about a recipe. Most instructions are sequential: “Boil water. Add pasta. Stir for eight minutes.” But occasionally a recipe branches: “If the dough is too sticky, add more flour. If it is still sticky after that, refrigerate it for thirty minutes.” The cook does not perform both actions; they evaluate the situation and take the appropriate path. This is control flow.

Traffic lights work the same way. The signal system does not simply run through green-yellow-red in a fixed cycle without regard for conditions. A modern intersection evaluates sensor data: if a car is waiting at a cross-street and the main road has been green for more than ninety seconds, it changes the light. The outcome depends on input. That is control flow in the real world.

In JavaScript, control flow works through a handful of constructs:

  • Conditional statements (if, else if, else, switch) — run different blocks of code depending on a condition
  • Short-circuit operators (&&, ||, ??) — evaluate expressions lazily to short-circuit further evaluation
  • Optional chaining (?.) — conditionally access properties without crashing
  • Loops (for, while, for...of) — repeat blocks of code

This lesson covers everything above except loops. By the time you finish, you will have a complete mental model of how JavaScript evaluates conditions and routes execution through your program.

The if Statement

The if statement is the most fundamental decision-making tool in JavaScript. Its job is simple: evaluate a condition, and if that condition is truthy, execute a block of code. If the condition is falsy, skip it.

Basic Syntax

LISTING 21.1 — Basic if Statement

const cartTotal = 85;

if (cartTotal >= 50) {
  console.log("You qualify for free shipping!");
}

The condition goes inside parentheses. The code to run if the condition is truthy goes inside curly braces. When JavaScript reaches this statement, it evaluates cartTotal >= 50. Since 85 is indeed greater than or equal to 50, the condition produces true, and the block executes. If cartTotal were 30, the condition would produce false and the block would be skipped entirely.

The else Clause

An if statement handles the “yes” case. The else clause handles the “no” case — the code that should run when the condition is falsy.

LISTING 21.2 — if/else — Two Branches

const isLoggedIn = false;

if (isLoggedIn) {
  console.log("Welcome back! Here is your dashboard.");
} else {
  console.log("Please sign in to continue.");
}

Exactly one of these two blocks will execute — never both. The else clause has no condition of its own; it is simply the fallback when the original condition fails.

The else if Clause

Real programs often have more than two possibilities. A user might be an admin, a premium member, a regular member, or a guest. A grade might be an A, B, C, D, or F. The else if clause lets you test additional conditions in sequence.

LISTING 21.3 — else if — Multiple Conditions

const score = 74;

if (score >= 90) {
  console.log("Grade: A");
} else if (score >= 80) {
  console.log("Grade: B");
} else if (score >= 70) {
  console.log("Grade: C");
} else if (score >= 60) {
  console.log("Grade: D");
} else {
  console.log("Grade: F");
}

JavaScript evaluates these conditions in order, from top to bottom. As soon as it finds a truthy condition, it executes that block and skips all remaining branches. In the example above, score >= 90 is false (74 is not ≥ 90), score >= 80 is false, but score >= 70 is true (74 ≥ 70), so “Grade: C” is printed. The final else if and the trailing else are never evaluated at all.

This top-to-bottom, first-match-wins behavior is important to keep in mind. If you accidentally put the most general condition first, nothing more specific will ever match.

Comparison Operators

Conditions depend on comparison operators — expressions that compare two values and produce a boolean result. JavaScript provides six:

  • === — strict equality: same value and same type
  • !== — strict inequality: different value or different type
  • > — greater than
  • < — less than
  • >= — greater than or equal to
  • <= — less than or equal to

NOTE: Always use === (three equals signs) for equality checks, never == (two equals signs). The double-equals operator performs type coercion before comparing, which produces counterintuitive results: 0 == "" is true, 0 == false is true, and null == undefined is true. Strict equality refuses to coerce types, so 0 === "" is false as you would expect. There is essentially no good reason to use == in modern JavaScript.

Truthy and Falsy Revisited

The condition inside an if statement does not have to be a strict boolean — JavaScript will coerce any value to boolean automatically. The six falsy values are:

  • false
  • 0 (and -0 and 0n)
  • "" (empty string)
  • null
  • undefined
  • NaN

Everything else is truthy — including non-empty strings, non-zero numbers, objects, arrays (even empty ones), and functions. This means you can write conditions like:

LISTING 21.4 — Truthy/Falsy in Conditions

const username = "sarah_k";
const errorMessage = "";
const itemCount = 0;
const cartItems = [];

if (username) {
  console.log(`Hello, ${username}!`);  // runs — non-empty string is truthy
}

if (errorMessage) {
  console.log("Error:", errorMessage); // skipped — empty string is falsy
}

if (itemCount) {
  console.log("Items in stock");       // skipped — 0 is falsy
}

if (cartItems) {
  console.log("Cart exists");          // runs — [] is truthy, even when empty!
}

That last point catches beginners off guard: an empty array [] is truthy. If you want to check whether an array has items, test cartItems.length instead of cartItems.

A Realistic Example: Password Strength Check

LISTING 21.5 — Password Strength Evaluator

function evaluatePassword(password) {
  if (!password) {
    return "Please enter a password.";
  }

  if (password.length < 8) {
    return "Too short — minimum 8 characters.";
  }

  const hasUppercase = /[A-Z]/.test(password);
  const hasNumber    = /[0-9]/.test(password);
  const hasSymbol    = /[!@#$%^&*]/.test(password);

  if (hasUppercase && hasNumber && hasSymbol) {
    return "Strong password!";
  } else if (hasUppercase || hasNumber || hasSymbol) {
    return "Medium — add a mix of uppercase, numbers, and symbols.";
  } else {
    return "Weak — only lowercase letters detected.";
  }
}

console.log(evaluatePassword(""));           // "Please enter a password."
console.log(evaluatePassword("abc"));        // "Too short..."
console.log(evaluatePassword("helloworld")); // "Weak..."
console.log(evaluatePassword("Hello1world")); // "Medium..."
console.log(evaluatePassword("Hello1!World")); // "Strong password!"

Logical Operators in Conditions

Individual comparisons can only take you so far. Real decisions usually combine multiple conditions: a user must be logged in and have a verified email address before seeing sensitive data. A discount applies if a user has a coupon code or has spent more than $200 this month. You negate conditions too: “if the user is not an admin, hide this button.” JavaScript provides three logical operators for this: &&, ||, and !.

&& — AND

The && operator returns a truthy result only if both sides are truthy. If the left side is falsy, the right side is never evaluated. This is called short-circuit evaluation.

LISTING 21.6 — AND Operator

const isLoggedIn    = true;
const hasPermission = true;
const isPremium     = false;

if (isLoggedIn && hasPermission) {
  console.log("Access granted.");  // runs — both sides truthy
}

if (isLoggedIn && isPremium) {
  console.log("Premium content");  // skipped — isPremium is false
}

|| — OR

The || operator returns a truthy result if at least one side is truthy. If the left side is truthy, the right side is never evaluated (short-circuit).

LISTING 21.7 — OR Operator

const hasCoupon       = false;
const lifetimeSpend   = 350;

if (hasCoupon || lifetimeSpend >= 200) {
  console.log("Discount applied!");  // runs — lifetimeSpend >= 200 is true
}

! — NOT

The ! operator inverts a boolean value. !true becomes false, and !false becomes true. Applied to a non-boolean, it first coerces to boolean, then inverts.

LISTING 21.8 — NOT Operator

const isGuest = true;

if (!isGuest) {
  console.log("Show personalized content.");  // skipped — !true is false
} else {
  console.log("Show sign-up prompt.");         // runs
}

// Double NOT (!!) forces any value to a strict boolean
console.log(!!"hello");  // true
console.log(!!0);        // false
console.log(!!null);     // false

Short-Circuit Evaluation in Depth

Here is the nuance most beginners miss: && and || do not simply return true or false. They return one of their operand values — and this is intentional. Understanding short-circuit return values unlocks a set of powerful idioms used throughout professional JavaScript.

  • && returns the first falsy value it encounters, or the last value if all are truthy.
  • || returns the first truthy value it encounters, or the last value if all are falsy.

LISTING 21.9 — Short-Circuit Patterns in the DOM

// && — both conditions must be true before acting
const username = document.querySelector('#username').value;
const password = document.querySelector('#password').value;
if (username && password) {
  loginBtn.disabled = false;  // only enable when both fields are filled
}

// || — use the first truthy value as a fallback display name
const displayName = user.nickname || user.firstName || "Guest";
nameEl.textContent = displayName;

// ?? — only falls back on null/undefined (not 0 or "")
const score = user.score ?? 0;   // 0 is a valid score, it won't be overridden
scoreEl.textContent = score;

Practical Short-Circuit Patterns

These short-circuit return values enable two extremely common JavaScript patterns:

LISTING 21.10 — Guard Expression and Default Value Pattern

// GUARD: only access user.name if user exists
const user = null;
const name = user && user.name; // null — user is falsy, stops there safely
console.log(name); // null (no TypeError thrown)

// If user exists, we get the name:
const activeUser = { name: "Jordan", role: "editor" };
const activeName = activeUser && activeUser.name; // "Jordan"

// DEFAULT: provide fallback when value is falsy
const storedUsername = "";  // empty string from localStorage
const displayName = storedUsername || "Guest";
console.log(displayName); // "Guest"

const preferredTheme = localStorage.getItem("theme") || "light";
// If nothing is stored, default to "light"

NOTE: The || default-value pattern has a well-known weakness: it treats all falsy values as missing, including legitimate values like 0 and false. If a user's item count is 0, itemCount || 5 would incorrectly return 5. The nullish coalescing operator ?? was introduced precisely to fix this problem, and it is covered later in this lesson.

Operator Precedence with Logical Operators

When you mix &&, ||, and ! in a single expression, JavaScript evaluates them in a fixed order: ! first, then &&, then ||. This mirrors the rules of multiplication before addition in arithmetic.

LISTING 21.11 — Logical Operator Precedence

// Without parentheses — && binds tighter than ||
const result = false || true && false;
// Evaluated as: false || (true && false)
// = false || false
// = false

// With explicit parentheses for clarity:
const a = true;
const b = false;
const c = true;

if (a || (b && c)) {
  console.log("This runs");  // a is true, so the whole OR is true
}

// Guarding a complex condition
const age  = 17;
const hasParentalConsent = true;

if (age >= 18 || (age >= 13 && hasParentalConsent)) {
  console.log("User can proceed.");
}

The best practice is to use parentheses liberally whenever you combine operators. Even when the precedence is technically correct, parentheses make intent crystal clear — both for future readers and for yourself three months later.

The Ternary Operator

The if statement is a statement — it does something but does not produce a value. The ternary operator is an expression — it evaluates to a value. This distinction matters in contexts where you need to produce a result rather than execute side effects.

Syntax

condition ? valueIfTrue : valueIfFalse

The expression evaluates the condition. If it is truthy, the operator produces the value on the left of the colon. If falsy, it produces the value on the right. The result can be stored in a variable, returned from a function, or embedded directly in a template literal.

LISTING 21.12 — Ternary Operator Basics

const hour = new Date().getHours();

// With if/else:
let greeting;
if (hour < 12) {
  greeting = "Good morning";
} else {
  greeting = "Good afternoon";
}

// With ternary — same logic, one expression:
const greeting2 = hour < 12 ? "Good morning" : "Good afternoon";

// Put the result directly into the DOM:
document.querySelector('#greeting').textContent = greeting2;

// Ternary inside a template literal placed in the DOM:
const itemCount = 3;
document.querySelector('#cart-summary').textContent =
  `You have ${itemCount} ${itemCount === 1 ? "item" : "items"} in your cart.`;
// "You have 3 items in your cart."

More Realistic Ternary Examples

LISTING 21.13 — Ternary in Real Scenarios

const userRole = "admin";

// Set button label based on role
const manageBtn = document.querySelector('#action-btn');
manageBtn.textContent = userRole === "admin" ? "Manage Users" : "View Profile";

// Compute a price with conditional discount and display it
const memberPrice = 29.99;
const regularPrice = 49.99;
const isPremiumMember = true;
const price = isPremiumMember ? memberPrice : regularPrice;
document.querySelector('#price-display').textContent = `Your price: $${price}`;

// Ternary updating both text and CSS class — common UI pattern
const emailInput = document.querySelector('#email');
const statusLabel = document.querySelector('#email-status');
const isValid = emailInput.value.includes('@');
statusLabel.textContent = isValid ? "✓ Valid email" : "✗ Invalid email";
statusLabel.className = isValid ? "valid" : "invalid";

When NOT to Use the Ternary Operator

The ternary operator shines for simple, single-expression decisions. It is the wrong tool when the branches contain multiple statements or when ternaries are nested inside each other. Nested ternaries in particular are notoriously difficult to read and are considered a code smell by most style guides.

LISTING 21.14 — Nested Ternary — Avoid This

// AVOID — nested ternary is hard to parse
const accessLevel = role === "admin"
  ? "full"
  : role === "editor"
    ? "write"
    : role === "viewer"
      ? "read"
      : "none";

// PREFER — else if chain is far more readable
let accessLevel2;
if (role === "admin") {
  accessLevel2 = "full";
} else if (role === "editor") {
  accessLevel2 = "write";
} else if (role === "viewer") {
  accessLevel2 = "read";
} else {
  accessLevel2 = "none";
}

// Or use a switch (shown later in this lesson)

CAUTION: The ESLint rule no-nested-ternary exists specifically because nested ternaries cause bugs that are hard to see. Many teams ban them outright. When you find yourself writing a ternary inside a ternary, stop and use an if/else chain or a switch instead.

The switch Statement

When you need to test a single variable against many possible discrete values, a long chain of else if statements works but starts to feel repetitive. The switch statement is designed exactly for this scenario. It evaluates one expression once and then jumps to the matching case label.

Basic Syntax

LISTING 21.15 — switch Statement — User Roles

const userRole = "editor";

switch (userRole) {
  case "admin":
    console.log("Full access: manage users, content, and settings.");
    break;
  case "editor":
    console.log("Write and publish content.");
    break;
  case "viewer":
    console.log("Read-only access.");
    break;
  default:
    console.log("Unknown role — access denied.");
}

The switch expression (userRole) is evaluated once. JavaScript then checks each case value using strict equality (===). When it finds a match, it executes the code in that case block. The break statement then exits the switch entirely. The default block — which is optional but recommended — runs if no case matches.

The Importance of break

Without break, execution does not stop at the end of the matching case — it falls through into the next case and continues executing. This behavior surprises almost every beginner the first time they encounter it.

LISTING 21.16 — Fall-Through Without break

const day = "Monday";

switch (day) {
  case "Monday":
    console.log("Start of the work week.");
    // NO break here — falls through!
  case "Tuesday":
    console.log("Second day.");
    break;
  case "Wednesday":
    console.log("Mid-week.");
    break;
}

// Output:
// "Start of the work week."
// "Second day."
// Both lines print because Monday fell through to Tuesday!

Accidental fall-through is one of the most common switch bugs. Always add break unless you intentionally want fall-through behavior.

Intentional Fall-Through: Grouping Cases

Fall-through becomes useful when you want multiple cases to share the same block of code. Stack the case labels on top of each other to create a group.

LISTING 21.17 — Grouping Cases with Intentional Fall-Through

const day = "Saturday";

switch (day) {
  case "Monday":
  case "Tuesday":
  case "Wednesday":
  case "Thursday":
  case "Friday":
    console.log("Weekday — office is open.");
    break;
  case "Saturday":
  case "Sunday":
    console.log("Weekend — office is closed.");
    break;
  default:
    console.log("Invalid day.");
}

A Practical Example: Theme Switcher

LISTING 21.18 — switch for a UI Theme Switcher

const themeSelect = document.querySelector('#theme-select');
const body = document.body;

themeSelect.addEventListener('change', () => {
  switch (themeSelect.value) {
    case 'dark':
      body.className = 'theme-dark';
      break;
    case 'light':
      body.className = 'theme-light';
      break;
    case 'high-contrast':
      body.className = 'theme-high-contrast';
      break;
    default:
      body.className = '';
  }
});

Each case sets a CSS class on document.body. The actual colors are defined in your stylesheet under .theme-dark, .theme-light, and .theme-high-contrast. This separation of concerns — JavaScript controls state, CSS controls appearance — is the standard pattern in real UI work.

switch vs. else if — When to Use Each

Use switch when you are testing one variable against several discrete, known values. Use if/else if when:

  • You are testing different variables in each branch
  • Your conditions involve ranges (> 50, <= 100)
  • Your conditions involve complex boolean combinations

Nullish Coalescing: ??

You have seen how || can provide a default value: name || "Guest". This works great when an empty string or zero means “no value.” But what about cases where 0 or "" or false are perfectly valid, meaningful values that you want to keep? The || operator treats all falsy values as “missing,” which causes a class of subtle bugs. The nullish coalescing operator (??), introduced in ES2020, solves this precisely.

How ?? Works

leftSide ?? rightSide returns rightSide only if leftSide is null or undefined. For every other value — including 0, "", and false — it returns leftSide unchanged.

LISTING 21.19 — Nullish Coalescing vs. OR

// Scenario: a user's item count retrieved from an API
// The user genuinely has 0 items in their cart.
const itemCount = 0;

console.log(itemCount || 5);   // 5 — WRONG! 0 is falsy, so || replaces it
console.log(itemCount ?? 5);   // 0 — CORRECT! 0 is not null/undefined

// Another example: volume setting
const volume = 0;   // user muted the audio
const activeVolume = volume ?? 50;
console.log(activeVolume);  // 0 — the muted setting is preserved

// Only null/undefined trigger ??
const missingValue = null;
console.log(missingValue ?? "default"); // "default"

const alsoMissing = undefined;
console.log(alsoMissing ?? "default"); // "default"

// Everything else is passed through unchanged
console.log(false ?? true);  // false
console.log("" ?? "empty");  // ""
console.log(0 ?? 99);        // 0

A Real-World Use Case: API Configuration

LISTING 21.20 — Nullish Coalescing for Configuration Defaults

// API response — some fields may be missing (null/undefined)
const apiResponse = {
  username: "marco",
  postCount: 0,        // a real value — don't overwrite!
  followerCount: null, // missing from API
  bio: "",             // user chose to leave blank
  isVerified: false    // explicitly false
};

const username      = apiResponse.username      ?? "Anonymous";
const postCount     = apiResponse.postCount     ?? "No posts yet";
const followerCount = apiResponse.followerCount ?? 0;
const bio           = apiResponse.bio           ?? "No bio added.";
const isVerified    = apiResponse.isVerified    ?? false;

console.log(username);      // "marco"
console.log(postCount);     // 0 (preserved, not replaced with "No posts yet")
console.log(followerCount); // 0 (null was replaced)
console.log(bio);           // "" (empty string preserved — user's choice)
console.log(isVerified);    // false (preserved)

Compare this to using || for every line: apiResponse.postCount || "No posts yet" would incorrectly produce "No posts yet" when the count is genuinely 0, and apiResponse.bio || "No bio added." would incorrectly replace a deliberately blank bio with placeholder text.

Optional Chaining: ?.

One of the most common runtime errors in JavaScript is the TypeError: Cannot read properties of null (reading 'something'). It happens when you try to access a property on a value that is null or undefined. You have a user object, you try to access user.address.city, and at some point address is null — boom, crash. The optional chaining operator (?.), also introduced in ES2020, eliminates this entire category of error.

The Problem It Solves

LISTING 21.21 — The TypeError Problem

const user = {
  name: "Priya",
  // address property is missing entirely
};

// Old approach — verbose guard
let city;
if (user && user.address && user.address.city) {
  city = user.address.city;
}
console.log(city); // undefined (no crash, but a lot of typing)

// Without any guard — crashes
console.log(user.address.city);
// TypeError: Cannot read properties of undefined (reading 'city')

Optional Chaining Syntax

object?.property — If object is null or undefined, the expression short-circuits to undefined instead of throwing. If object has a value, property access proceeds normally.

LISTING 21.22 — Optional Chaining Basics

const user = {
  name: "Priya",
  // address is not present
};

// Optional chaining — safe access
console.log(user?.address?.city);       // undefined (no crash)
console.log(user?.name);                // "Priya"
console.log(user?.address?.zipCode);    // undefined

// With a fully populated user:
const fullUser = {
  name: "Jordan",
  address: {
    street: "123 Main St",
    city:   "Raleigh",
    zip:    "27601"
  }
};

console.log(fullUser?.address?.city);    // "Raleigh"
console.log(fullUser?.address?.country); // undefined (property absent, no crash)

Optional Chaining with Methods

You can also use ?. before a method call. object?.method?.() checks that method exists and is callable before invoking it.

LISTING 21.23 — Optional Chaining with Method Calls

const adminUser = {
  name: "Alex",
  getPermissions() {
    return ["read", "write", "delete"];
  }
};

const guestUser = {
  name: "Sam"
  // getPermissions is not defined on this object
};

console.log(adminUser.getPermissions?.());  // ["read", "write", "delete"]
console.log(guestUser.getPermissions?.());  // undefined (no crash)

// Practical: calling a callback only if it was provided
function fetchData(url, onSuccess) {
  // ... fetch logic ...
  const data = { items: [1, 2, 3] };
  onSuccess?.(data);  // calls onSuccess only if it's not null/undefined
}

fetchData("/api/items", (data) => console.log(data)); // works
fetchData("/api/items");  // onSuccess is undefined — ?. prevents the crash

Optional Chaining with Arrays

LISTING 21.24 — Optional Chaining with Array Indexing

const data = {
  users: [
    { name: "Alice", role: "admin" },
    { name: "Bob",   role: "viewer" }
  ]
};

const emptyData = { users: null };

console.log(data.users?.[0]?.name);        // "Alice"
console.log(data.users?.[5]?.name);        // undefined (index out of bounds, no crash)
console.log(emptyData.users?.[0]?.name);   // undefined (users is null, short-circuits)

Combining ?. and ??

Optional chaining and nullish coalescing are natural partners. Use ?. to safely navigate a potentially missing chain, and ?? to substitute a meaningful default when the chain produces undefined.

LISTING 21.25 — Combining Optional Chaining and Nullish Coalescing

const users = [
  { name: "Alice", profile: { city: "Austin", theme: "dark" } },
  { name: "Bob",   profile: null },
  { name: "Carol"  /* profile missing entirely */ }
];

for (const user of users) {
  const city  = user?.profile?.city  ?? "City not set";
  const theme = user?.profile?.theme ?? "light";
  console.log(`${user.name}: ${city} | Theme: ${theme}`);
}
// Alice: Austin | Theme: dark
// Bob: City not set | Theme: light
// Carol: City not set | Theme: light

Logical Assignment Operators

ES2021 introduced three shorthand operators that combine a logical check with assignment. These are not required beginner knowledge, but you will encounter them in real codebases. Each one is a condensed form of a pattern you already know.

||= — Assign if falsy

a ||= b is shorthand for a = a || b. It assigns b only if a is falsy.

LISTING 21.26 — Logical OR Assignment (||=)

// Fill in a placeholder only if the element has no text yet
const titleEl = document.querySelector('#page-title');
titleEl.textContent ||= "Untitled Page";
// If the element was empty (falsy), it now reads "Untitled Page"
// If it already had text, it is left alone

&&= — Assign only if truthy

a &&= b is shorthand for a = a && b. It assigns b only if a is already truthy — useful when you only want to update something that already has a value.

LISTING 21.27 — Logical AND Assignment (&&=)

// Update a status badge label only if it already exists
const badge = document.querySelector('#status-badge');
// badge.textContent is "Active" (truthy) — update it
badge.textContent &&= "Verified";
// If badge.textContent were "" (falsy), it would be left alone

??= — Assign only if null or undefined

a ??= b is shorthand for a = a ?? b. It assigns b only if a is null or undefined, leaving 0, "", and false untouched. This makes it the safest default-fill operator.

LISTING 21.28 — Nullish Assignment (??=)

// Fill in a score display only if no score has been set
let currentScore = null;
currentScore ??= 0;   // null — replaced with 0
scoreEl.textContent = currentScore;  // shows 0, not blank

// If the score were already 0 (a valid value), ??= would leave it alone
// Unlike ||=, which would mistakenly replace 0 with the default

NOTE: These three operators are shorthand — handy when you recognize them, but not something to reach for immediately. Focus first on using ||, &&, and ?? directly inside if statements and assignments. Logical assignment operators will feel natural once those fundamentals are solid.

LISTING 21.29 — Shipping Cost Calculator Using All Control Flow Tools

// Simulated user data from an API (some fields may be missing)
const customer = {
  name: "Maya Chen",
  membershipLevel: "gold",
  address: {
    country: "US",
    state: "CA"
    // zipCode missing from profile
  },
  savedPayment: null  // no payment method on file
};

// Order data
const order = {
  subtotal: 0,        // Will be set — 0 is valid, don't default-replace it
  itemCount: null,    // Missing from order object
  couponCode: "",     // Customer didn't enter one (empty string)
  isGift: false       // Explicitly not a gift
};

// Fill in defaults for missing values — use ??= to preserve 0
order.itemCount ??= 1;   // was null — default to 1
order.subtotal  ??= 0;   // already 0 — left alone (correctly)

// Resolve the subtotal (simulate items being added)
order.subtotal = 127.50;

// Safe address access
const country = customer?.address?.country ?? "US";
const state   = customer?.address?.state   ?? "unknown";
const zipCode = customer?.address?.zipCode ?? "not provided";

// Membership-based discount using switch
let memberDiscount = 0;
switch (customer?.membershipLevel) {
  case "platinum":
    memberDiscount = 0.20;
    break;
  case "gold":
    memberDiscount = 0.10;
    break;
  case "silver":
    memberDiscount = 0.05;
    break;
  default:
    memberDiscount = 0;
}

// Apply coupon code discount — only if a code was actually entered
const hasCoupon = order.couponCode !== "";
const couponDiscount = hasCoupon ? 0.15 : 0;

// Combine discounts (they stack, but total never exceeds 25%)
const totalDiscountRate = Math.min(memberDiscount + couponDiscount, 0.25);
const discountAmount    = order.subtotal * totalDiscountRate;
const discountedTotal   = order.subtotal - discountAmount;

// Calculate shipping
let shippingCost;
if (discountedTotal >= 100) {
  shippingCost = 0;  // free shipping over $100
} else if (country !== "US") {
  shippingCost = 24.99;  // international flat rate
} else if (state === "AK" || state === "HI") {
  shippingCost = 14.99;  // Alaska and Hawaii surcharge
} else {
  shippingCost = 7.99;   // standard domestic
}

// Gift surcharge
const giftSurcharge = order.isGift ? 5.00 : 0;

// Payment status
const paymentStatus = customer?.savedPayment
  ? `On file: ${customer.savedPayment}`
  : "No payment method saved — will prompt at checkout";

// Final total
const finalTotal = discountedTotal + shippingCost + giftSurcharge;

// Display results
console.log(`Customer: ${customer?.name ?? "Guest"}`);
console.log(`Membership: ${customer?.membershipLevel ?? "none"} (-${totalDiscountRate * 100}%)`);

console.log(`Gift wrap: ${giftSurcharge > 0 ? "$" + giftSurcharge.toFixed(2) : "N/A"}`);
console.log(`Total: $${finalTotal.toFixed(2)}`);
console.log(`Delivering to: ${state}, ${country} (ZIP: ${zipCode})`);
console.log(`Payment: ${paymentStatus}`);

Summary

Control flow is what transforms a script that runs the same way every time into a program that adapts to its inputs. You started this lesson with the fundamental if statement and built up to the full modern JavaScript control flow toolkit.

The if, else, and else if statements evaluate conditions in sequence and route execution to the appropriate branch. Conditions rely on comparison operators (===, !==, >, <, >=, <=) and on JavaScript's truthy/falsy system. The logical operators &&, ||, and ! combine conditions and leverage short-circuit evaluation — both to control truthiness and to implement guard expressions and defaults. The ternary operator provides a concise, expression-based alternative to if/else for simple single-value decisions, especially inside template literals and JSX-like patterns. The switch statement cleanly handles many discrete values of the same variable; remember to use break unless you want intentional fall-through.

The modern additions are equally important. Nullish coalescing (??) provides fallback values only for null and undefined, correctly preserving legitimate falsy values like 0 and "" where || would incorrectly replace them. Optional chaining (?.) eliminates the entire category of “cannot read property of null” TypeErrors by short-circuiting to undefined when a chain includes a null or undefined link. The two operators combine naturally: user?.address?.city ?? "Unknown" is a safe, readable, default-providing property access in a single expression. Finally, logical assignment operators (||=, &&=, ??=) condense common assignment-with-condition patterns into a single operator, making configuration and initialization code more concise and expressive.

The shipping calculator at the end of the lesson demonstrated that these tools are not alternatives to one another — they are complements, each suited to a specific job, and the best code picks the right tool for each decision.

Q&A

What is the difference between == and ===, and why should I always use ===?

The double-equals operator (==) performs type coercion before comparing: it converts one or both operands to a common type, then compares. This produces surprising results — 0 == false is true, "" == false is true, and null == undefined is true. The triple-equals operator (===) is strict: it requires both the value and the type to match, with no coercion. So 0 === false is false, as you would expect. Using === everywhere eliminates an entire class of hard-to-find bugs. The only reason to use == in modern JavaScript is if you want to deliberately check for both null and undefined at once (value == null), but even then many teams prefer the explicit value === null || value === undefined for clarity.

The ternary operator looks concise, but my teammates say to avoid it. Who is right?

Both perspectives have merit, and the answer is context-dependent. The ternary is excellent for simple, single-expression decisions — particularly when you need a value in the middle of a template literal or a JSX attribute. isLoggedIn ? "Dashboard" : "Login" is clear and saves several lines. The danger comes from nesting ternaries or using them for multi-line logic. A nested ternary like a ? b ? x : y : c ? m : n is nearly impossible to parse at a glance and should always be replaced with an if/else if chain. Most style guides and linters allow ternary expressions but ban or warn against nesting. A reasonable rule: if you cannot read the ternary out loud and understand it immediately, use an if statement instead.

When should I use switch instead of else if?

Use switch when you are comparing one variable against several known discrete values. If you find yourself writing if (role === "admin") ... else if (role === "editor") ... else if (role === "viewer"), the switch version is visually cleaner and communicates intent more clearly. However, switch is not a good fit for range-based conditions (> 50, <= 100), for conditions that compare different variables in each branch, or for complex boolean logic. In those cases, if/else if is the right choice. One practical concern: switch requires careful use of break statements. If fall-through is not your intention, missing a break introduces a hard-to-spot bug. Some developers prefer else if purely to avoid the fall-through risk.

Why doesn’t || work for providing default values when the value might legitimately be 0 or false?

The || operator is not really a “nullish check” — it is a truthiness check. It treats all six falsy values (false, 0, "", null, undefined, NaN) the same way: as signal to use the right-hand side. This is the wrong behavior when the left-hand value is 0 (a muted volume, an empty cart, a score of zero) or false (an explicit opt-out) or "" (a deliberately blank bio). The nullish coalescing operator ?? was introduced specifically to handle this. It only triggers on null and undefined — the two values that genuinely mean “this data does not exist.” The rule of thumb: use ?? when “missing” means null/undefined and falsy values are valid data. Use || when any falsy value genuinely means “no value was provided.”

What exactly does optional chaining (?.) prevent, and is there a performance cost?

Optional chaining prevents TypeError exceptions that occur when you access a property of null or undefined. Without it, user.address.city throws if address is null — even in a try/catch, that crash propagates. With user?.address?.city, JavaScript evaluates each step: if user is null or undefined, the entire expression short-circuits to undefined right there, without touching address or city. This is particularly valuable when dealing with data from APIs, where fields may be missing depending on the object's state. As for performance: the overhead is negligible. Optional chaining is compiled to simple null-checks by JavaScript engines, and the cost is far outweighed by the safety it provides. Use it freely whenever you are accessing nested properties on objects that might not be fully populated.

What is the difference between ||= and ??=, and how do I decide which to use?

The difference mirrors the distinction between || and ??. ||= assigns the right-hand value whenever the left-hand side is falsy — that includes 0, "", false, null, and undefined. ??= assigns the right-hand value only when the left-hand side is null or undefined; it leaves 0, "", and false untouched. The decision comes down to what “needs a default” means in your context. If you are initializing a configuration object where timeout: 0 is a valid setting that should not be overwritten, use ??=. If you are building a form where an empty string genuinely means the field was not filled in and you want to replace it, ||= may be appropriate. When in doubt, ??= is the safer choice for default initialization patterns, because it will never accidentally overwrite data the user or system intentionally provided as a falsy value.
