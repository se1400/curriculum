Local Storage & Practical Patterns

You can now write JavaScript that reads the DOM, responds to events, works with arrays of objects, and transforms data with iteration methods. There is one gap in the experience: every time a user refreshes the page, everything resets. The cart empties. The to-do list disappears. The theme snaps back to light. This lesson closes that gap.

The Web Storage API gives you a place to save data in the browser that survives page refreshes, closed tabs, and even closed browser windows. You will also learn to wrap risky operations in try/catch so your app fails gracefully instead of crashing, and you will see a set of code organization patterns that make JavaScript applications easier to read, debug, and build on.

What You’ll Learn

  • Save and retrieve data with localStorage and sessionStorage
  • Serialize objects with JSON.stringify() and deserialize with JSON.parse()
  • Protect code with try / catch / finally and throw custom errors
  • Detect when a user goes offline with navigator.onLine
  • Apply naming conventions and single-responsibility thinking
  • Understand the ES Module import/export pattern as a preview of modern JavaScript

The Web Storage API

Web Storage is a browser API that stores key-value pairs as strings. It lives on the window object, so localStorage and sessionStorage are always available in browser JavaScript without any import.

The difference between the two:

  • localStorage — data persists until you explicitly remove it. Survives refreshes, closed tabs, and closed windows. Scoped to the origin (protocol + domain + port).
  • sessionStorage — data is cleared when the tab closes. Survives refreshes within the same tab but not a new tab or window.

Both share the exact same API methods.

The Four Core Methods

LISTING 28.1 — Core localStorage methods 
// Save a value
localStorage.setItem('username', 'alex');

// Read a value (returns null if key doesn't exist)
const name = localStorage.getItem('username');
console.log(name); // "alex"

// Delete one key
localStorage.removeItem('username');

// Delete everything (use with caution)
localStorage.clear();

// How many keys are stored
console.log(localStorage.length);
NOTE: All values are stored as strings. If you pass a number like 42, it comes back as the string "42". This is why you must serialize objects with JSON.stringify() before storing them.

Storing Objects and Arrays

The most common mistake beginners make with localStorage is storing an object directly:

LISTING 28.2 — The object serialization problem and solution 
// ❌ Wrong — stores the string "[object Object]"
const user = { name: 'Alex', score: 42 };
localStorage.setItem('user', user);
localStorage.getItem('user'); // "[object Object]"

// ✅ Correct — serialize with JSON.stringify first
localStorage.setItem('user', JSON.stringify(user));

// Then deserialize with JSON.parse when reading back
const saved = JSON.parse(localStorage.getItem('user'));
console.log(saved.name); // "Alex"

The Null Guard Pattern

getItem() returns null when a key does not exist — not an empty string, not undefined, but null. Passing null to JSON.parse() throws an error. The null guard pattern prevents this:

LISTING 28.3 — Null guard with fallback 
// ❌ Crashes if 'cart' has never been set
const cart = JSON.parse(localStorage.getItem('cart'));

// ✅ Falls back to an empty array if key is missing
const cart = JSON.parse(localStorage.getItem('cart') || '[]');

// ✅ For objects, fall back to '{}'
const prefs = JSON.parse(localStorage.getItem('prefs') || '{}');

// ✅ Nullish coalescing is clearer than ||
const raw = localStorage.getItem('cart');
const cart = raw !== null ? JSON.parse(raw) : [];

A Persistent Theme Switcher

Here is a complete pattern for saving a user's theme preference:

LISTING 28.4 — Save and restore theme preference 
const toggleBtn = document.querySelector('#theme-toggle');

// Restore saved theme when the page loads
const savedTheme = localStorage.getItem('theme') || 'light';
document.body.dataset.theme = savedTheme;
toggleBtn.textContent = savedTheme === 'dark' ? '☀ Light' : '☾ Dark';

// Toggle and save on click
toggleBtn.addEventListener('click', () => {
  const current = document.body.dataset.theme;
  const next = current === 'dark' ? 'light' : 'dark';

  document.body.dataset.theme = next;
  localStorage.setItem('theme', next);
  toggleBtn.textContent = next === 'dark' ? '☀ Light' : '☾ Dark';
});

The storage Event

When localStorage changes in one tab, the browser fires a storage event in all other tabs from the same origin. This is how you can synchronize state across tabs — a shopping cart that updates in a background tab, a logout that clears other open windows:

LISTING 28.5 — Listening for cross-tab storage changes 
window.addEventListener('storage', (event) => {
  console.log('Key changed:', event.key);
  console.log('Old value:', event.oldValue);
  console.log('New value:', event.newValue);

  if (event.key === 'theme') {
    document.body.dataset.theme = event.newValue;
  }
});
NOTE: The storage event does NOT fire in the tab that made the change — only in other tabs. Use it for cross-tab synchronization, not as a general change listener.

Detecting Offline Status

navigator.onLine is a boolean that tells you whether the browser currently has network access. Combined with the online and offline events on window, you can show and hide an offline banner as connectivity changes:

LISTING 28.6 — Offline detection with navigator.onLine 
const banner = document.querySelector('#offline-banner');

function updateOnlineStatus() {
  if (navigator.onLine) {
    banner.hidden = true;
  } else {
    banner.hidden = false;
    banner.textContent = 'You are offline. Changes will be saved locally.';
  }
}

// Check immediately on load
updateOnlineStatus();

// Listen for changes
window.addEventListener('online', updateOnlineStatus);
window.addEventListener('offline', updateOnlineStatus);

Storage Limitations and Security

Before building on localStorage, know its limits:

  • Size limit: ~5 MB per origin in most browsers. Attempting to exceed it throws a QuotaExceededError.
  • Strings only: All data must be serialized to string. Binary data needs encoding.
  • Synchronous: Reading and writing blocks the main thread. For large amounts of data, IndexedDB is a better choice.
  • Private browsing: In some browsers, localStorage is disabled or throws in private/incognito mode. Always wrap it in try/catch.
  • Not secure: Any JavaScript running on the page can read localStorage. Never store passwords, authentication tokens, or sensitive personal data here.
[IMG - Chrome DevTools Application tab showing localStorage panel]

For comparison, here is when to use each storage option:

  • localStorage — user preferences, saved lists, last-visited state. Persists forever.
  • sessionStorage — form drafts, step-by-step wizard progress. Cleared on tab close.
  • Cookies — sent with every HTTP request; used for server-side sessions and authentication. Has expiry and domain options. Limited to ~4 KB.
  • IndexedDB — structured database in the browser. Handles large amounts of data, supports transactions and indexes. Complex API — most developers use a library wrapper.

Error Handling with try / catch / finally

JavaScript runs synchronously from top to bottom. When an error is thrown, execution stops and the rest of the code in that block does not run. The try/catch statement lets you contain that failure so the rest of your application keeps working.

LISTING 28.7 — Basic try / catch structure 
try {
  // Code that might fail
  const data = JSON.parse(someBadString);
  renderList(data);
} catch (error) {
  // Runs only if something in the try block throws
  console.error('Failed to parse data:', error.message);
  showErrorUI('Could not load your saved data.');
} finally {
  // Always runs — whether or not an error occurred
  hideLoadingSpinner();
}

The error object has three useful properties:

  • error.message — a human-readable description of what went wrong
  • error.name — the error type: TypeError, RangeError, SyntaxError, etc.
  • error.stack — a full stack trace (useful for debugging)

Wrapping localStorage in try / catch

Because localStorage can fail in private browsing mode or when storage is full, production code wraps it in try/catch:

LISTING 28.8 — Safe localStorage wrapper functions 
function saveData(key, value) {
  try {
    localStorage.setItem(key, JSON.stringify(value));
  } catch (error) {
    if (error.name === 'QuotaExceededError') {
      console.warn('Storage full — could not save:', key);
    } else {
      console.warn('localStorage unavailable:', error.message);
    }
  }
}

function loadData(key, fallback = null) {
  try {
    const raw = localStorage.getItem(key);
    return raw !== null ? JSON.parse(raw) : fallback;
  } catch {
    return fallback;
  }
}
NOTE: In ES2019 and later you can write catch { } without naming the error parameter when you do not need it — called "optional catch binding."

Throwing Custom Errors

You can throw errors yourself with throw. Use the built-in error constructors to give your errors a meaningful name and message:

LISTING 28.9 — Throwing typed errors 
function divide(a, b) {
  if (typeof a !== 'number' || typeof b !== 'number') {
    throw new TypeError('Both arguments must be numbers');
  }
  if (b === 0) {
    throw new RangeError('Cannot divide by zero');
  }
  return a / b;
}

try {
  divide(10, 0);
} catch (error) {
  if (error instanceof RangeError) {
    showErrorUI(error.message); // "Cannot divide by zero"
  } else {
    throw error; // Re-throw errors you didn't expect
  }
}

Re-throwing errors you did not handle intentionally (throw error inside catch) is an important pattern — it lets you handle specific cases without silently swallowing bugs you did not anticipate.

A Persistent To-Do List

The following example brings together localStorage, JSON serialization, the null guard, and event handling. Every change the user makes is saved immediately. When the page loads, the saved list is rendered from storage.

LISTING 28.10 — Persistent to-do list with localStorage 
const form = document.querySelector('#todo-form');
const input = document.querySelector('#todo-input');
const list = document.querySelector('#todo-list');

// ----- Storage helpers -----
function getTodos() {
  return JSON.parse(localStorage.getItem('todos') || '[]');
}

function saveTodos(todos) {
  localStorage.setItem('todos', JSON.stringify(todos));
}

// ----- Rendering -----
function renderTodos() {
  const todos = getTodos();
  list.innerHTML = todos
    .map((todo, i) => `
      <li>
        <label>
          <input type="checkbox" data-index="${i}" ${todo.done ? 'checked' : ''}>
          <span class="${todo.done ? 'done' : ''}">${todo.text}</span>
        </label>
        <button data-index="${i}" aria-label="Delete">×</button>
      </li>
    `)
    .join('');
}

// ----- Event: add -----
form.addEventListener('submit', (e) => {
  e.preventDefault();
  const text = input.value.trim();
  if (!text) return;

  const todos = getTodos();
  todos.push({ text, done: false });
  saveTodos(todos);
  renderTodos();
  input.value = '';
});

// ----- Event: check / delete (delegation) -----
list.addEventListener('click', (e) => {
  const todos = getTodos();
  const index = parseInt(e.target.dataset.index, 10);

  if (e.target.type === 'checkbox') {
    todos[index].done = e.target.checked;
    saveTodos(todos);
    renderTodos();
  }

  if (e.target.tagName === 'BUTTON') {
    todos.splice(index, 1);
    saveTodos(todos);
    renderTodos();
  }
});

// ----- Initialize -----
renderTodos();
NOTE: Notice that the data (the todos array) is always the source of truth. The DOM is only ever rendered from the data — never queried to determine the current state. This one-way data flow pattern is the foundation of how frameworks like React work.

Code Organization Patterns

As JavaScript applications grow past a single feature, unorganized code becomes hard to debug and hard to extend. A few lightweight patterns address this without requiring a framework.

Naming Conventions

Three conventions are standard across JavaScript codebases:

  • camelCase — variables, functions, object properties: getUserName, isLoggedIn, cartItems
  • PascalCase — classes, constructors, React components: UserProfile, ShoppingCart
  • UPPER_SNAKE_CASE — constants that will never change: MAX_RETRY_COUNT, API_BASE_URL

Booleans usually start with is, has, can, or should: isVisible, hasError, canSubmit. Functions that return a value start with a verb: getUser(), formatPrice(), buildCardHTML(). Functions that do something start with a verb: renderList(), saveToStorage(), showBanner().

Single Responsibility

A function should do one thing and do it well. If you find yourself writing a function named handleFormAndRenderAndSave(), that is a sign it should be three functions.

LISTING 28.11 — Separating concerns in a form handler 
// ❌ One function doing everything
function handleSubmit(e) {
  e.preventDefault();
  const text = document.querySelector('#input').value.trim();
  if (!text) return;
  const todos = JSON.parse(localStorage.getItem('todos') || '[]');
  todos.push({ text, done: false });
  localStorage.setItem('todos', JSON.stringify(todos));
  document.querySelector('#list').innerHTML = todos
    .map(t => `<li>${t.text}</li>`)
    .join('');
  document.querySelector('#input').value = '';
}

// ✅ Each function has one job
function getTodos() {
  return JSON.parse(localStorage.getItem('todos') || '[]');
}
function saveTodos(todos) {
  localStorage.setItem('todos', JSON.stringify(todos));
}
function buildTodoHTML(todo) {
  return `<li>${todo.text}</li>`;
}
function renderTodos() {
  document.querySelector('#list').innerHTML = getTodos()
    .map(buildTodoHTML).join('');
}
function addTodo(text) {
  const todos = getTodos();
  todos.push({ text, done: false });
  saveTodos(todos);
  renderTodos();
}
function handleSubmit(e) {
  e.preventDefault();
  const input = document.querySelector('#input');
  const text = input.value.trim();
  if (!text) return;
  addTodo(text);
  input.value = '';
}

Each function in the second version can be tested, debugged, or replaced independently. The handleSubmit function is now a thin coordinator — it does not know or care how addTodo works internally.

Separating Data, Logic, and Display

At a higher level, group your code into three layers:

  • Data layer: Functions that read from and write to storage. No DOM access. No business logic. Just get and set.
  • Logic layer: Functions that work with data — filtering, sorting, transforming. No DOM access. Pure functions where possible.
  • Display layer: Functions that read the current data and update the DOM. One function per component or view. Called only after data changes.

Even in a single-file script, grouping functions into these three sections makes your code dramatically easier to understand.

LISTING 28.12 — Three-layer separation in a single file 
// === DATA LAYER ===
function getCart() {
  return JSON.parse(localStorage.getItem('cart') || '[]');
}
function saveCart(items) {
  localStorage.setItem('cart', JSON.stringify(items));
}

// === LOGIC LAYER ===
function addItem(cart, product) {
  const existing = cart.find(item => item.id === product.id);
  if (existing) {
    return cart.map(item =>
      item.id === product.id
        ? { ...item, qty: item.qty + 1 }
        : item
    );
  }
  return [...cart, { ...product, qty: 1 }];
}
function getTotal(cart) {
  return cart.reduce((sum, item) => sum + item.price * item.qty, 0);
}

// === DISPLAY LAYER ===
function renderCart() {
  const cart = getCart();
  cartList.innerHTML = cart.map(item =>
    `<li>${item.name} × ${item.qty} — $${(item.price * item.qty).toFixed(2)}</li>`
  ).join('');
  totalEl.textContent = `$${getTotal(cart).toFixed(2)}`;
  badgeEl.textContent = cart.reduce((n, item) => n + item.qty, 0);
}

// === EVENT LAYER ===
addToCartBtns.forEach(btn => {
  btn.addEventListener('click', () => {
    const product = {
      id: btn.dataset.id,
      name: btn.dataset.name,
      price: Number(btn.dataset.price)
    };
    const cart = addItem(getCart(), product);
    saveCart(cart);
    renderCart();
  });
});

renderCart();

A Preview of ES Modules

Everything you have written so far lives in a single <script> tag. As your project grows, you will want to split code across multiple files. ES Modules are the standard way to do this in modern JavaScript.

LISTING 28.13 — ES Module import and export syntax 
// storage.js — export functions other files can use
export function getCart() {
  return JSON.parse(localStorage.getItem('cart') || '[]');
}
export function saveCart(items) {
  localStorage.setItem('cart', JSON.stringify(items));
}

// main.js — import the functions you need
import { getCart, saveCart } from './storage.js';

const cart = getCart();
saveCart([...cart, newItem]);

To use modules in a browser, add type="module" to your script tag:

<script type="module" src="main.js"></script>

Modules give you:

  • Each file has its own scope — no accidental global variables
  • Explicit dependencies via import — you always know where a function came from
  • Works with bundlers like Vite and esbuild, which is how all major frameworks are developed
NOTE: Modules require files to be served from a web server (even localhost). They do not work when opening an HTML file directly from the filesystem with file:// URLs.

Bringing It All Together

You now have a complete picture of client-side JavaScript. Every application you build will use the same five layers, in the same direction:

Data (objects & arrays) → Logic (functions & control flow) → Display (DOM) → Persistence (localStorage) → Interaction (events)

Data flows one way. The DOM is always rendered from data — never queried to determine state. Events trigger data changes, which trigger re-renders. Once this pattern clicks, you will recognize it in every framework you ever learn.

Common JavaScript Pitfalls Review

  • Mutation surprises: const arr2 = arr1 creates a reference, not a copy. Use spread ([...arr]) or structuredClone() to copy.
  • Storing objects in localStorage: Always JSON.stringify() before set and JSON.parse() after get.
  • Missing null guard: Always provide a fallback when reading from localStorage: || '[]' or || '{}'.
  • Synchronous DOM queries in loops: Read the DOM once, work with data, write to the DOM once. Never read and write the DOM inside the same loop.
  • Event handler accumulation: Adding a new addEventListener every time a list is re-rendered creates duplicate handlers. Use event delegation on a stable parent element.
  • Floating point math: 0.1 + 0.2 !== 0.3. Use toFixed() or Math.round() when displaying currency.
  • == vs ===: Always use strict equality === unless you have a specific reason for type coercion.

What to Learn Next

JavaScript running in a browser can only go so far without talking to a server. The natural next step is learning how to fetch data from an API — and that requires understanding asynchronous programming.

  • Promises & async/await — JavaScript's model for handling operations that take time (network requests, file reads). async function fetchData() { const data = await fetch(url); } is the pattern you will use constantly.
  • The Fetch API — the browser's built-in tool for making HTTP requests. Replaces older techniques like XMLHttpRequest.
  • REST APIs — nearly every website you use has a JSON API behind it. Learning to read, call, and parse API responses opens up a huge range of project ideas.
  • JavaScript frameworks — React, Vue, and Svelte automate the render cycle you have been writing by hand. The data-to-DOM one-way flow you practiced here is exactly how React's state model works.
  • Node.js — JavaScript on the server. The same language, different APIs. Enables full-stack development with one language on both sides.
  • TypeScript — adds static types to JavaScript. Catches an entire category of bug at write time rather than run time. Widely used in production codebases.

Every one of these topics builds directly on what you practiced here. The patterns — functions that do one thing, data flowing one way, events triggering re-renders — are universal.

Summary

  • localStorage stores string key-value pairs that persist across page refreshes. sessionStorage stores key-value pairs that clear when the tab closes.
  • Objects and arrays must be serialized with JSON.stringify() before storing and deserialized with JSON.parse() after reading.
  • Use a null guard when reading from storage: JSON.parse(localStorage.getItem(key) || '[]').
  • try / catch / finally contains errors so one failure does not crash the entire application. The finally block always runs regardless of whether an error occurred.
  • Wrap localStorage calls in try/catch — it can throw in private browsing and when storage is full.
  • navigator.onLine detects connectivity. The online and offline events fire when connectivity changes.
  • Never store passwords, tokens, or sensitive data in localStorage — it is readable by any JavaScript on the page.
  • Organize code into three layers: data (storage read/write), logic (pure transformations), and display (DOM rendering).
  • ES Modules (import / export) are the standard way to split code across files in modern JavaScript.

Q&A

What is the difference between localStorage and sessionStorage?
Both store key-value string pairs and share the same API. localStorage data persists until explicitly deleted — it survives page refreshes, closed tabs, and browser restarts. sessionStorage data is cleared when the tab or window that created it is closed, but survives refreshes within the same tab.
Why do I have to use JSON.stringify and JSON.parse with localStorage?
The Web Storage API only stores strings. If you pass an object directly, JavaScript calls .toString() on it, producing "[object Object]" — not the data you intended. JSON.stringify() converts your object into a JSON string, and JSON.parse() converts it back to an object when you read it.
What happens if localStorage is unavailable?
In private/incognito browsing in some browsers, and when storage is full, calls to localStorage.setItem() throw a QuotaExceededError. Wrapping storage calls in try/catch lets you detect this and fall back gracefully — for example, storing data in memory for the duration of the session instead.
When should I use try/catch?
Use try/catch at the boundaries where failure is likely and where you want to show a user-facing error rather than crash. Good candidates: parsing JSON from external sources or storage, network requests, reading files, and calls to third-party APIs. For internal code where errors indicate programming bugs (not user or environmental failures), let errors propagate so you catch them during development.
What does it mean to re-throw an error?
When you catch an error but only know how to handle a specific type, you should re-throw errors of all other types: if (error instanceof RangeError) { handleIt(); } else { throw error; }. This prevents catch from silently swallowing bugs you did not anticipate — those errors still bubble up to be caught by an outer handler or appear in the console.
What is the storage event and when does it fire?
The storage event fires on window when localStorage changes — but only in other open tabs from the same origin, not in the tab that made the change. It provides an event.key, event.oldValue, and event.newValue so you can synchronize state across tabs.
Why should I never store tokens or passwords in localStorage?
Any JavaScript running on the page — including code from third-party scripts, browser extensions, and XSS vulnerabilities — can read localStorage with localStorage.getItem(). Sensitive data like authentication tokens and passwords should be stored in httpOnly cookies, which cannot be read by JavaScript at all.
What are ES Modules and why do they matter?
ES Modules are the standard system for sharing code across JavaScript files using import and export. They solve two problems with older patterns: they prevent global variable pollution (each module has its own scope) and they make dependencies explicit (you always see where a function or value came from). All major frameworks (React, Vue, Svelte) and bundlers (Vite, webpack) are built around ES Modules.
