classList.toggle() and the UI Patterns Every Website Uses

What You’ll Learn in This Lesson:

  • ►The central principle of interactive UI: CSS defines states, JavaScript switches between them
  • ►How to build a dark mode toggle with one line of JavaScript
  • ►How to build an accordion using max-height transitions
  • ►How to replace the CSS-only hamburger menu hack with clean JavaScript
  • ►How to build a tab system and a modal dialog
  • ►How to schedule delayed actions with setTimeout
  • ►How to run repeated actions with setInterval

You now have two powerful tools working together. CSS can define multiple visual states — a card can be small or large, a panel can be hidden or visible, a theme can be light or dark. JavaScript can respond to user events with addEventListener. This lesson brings both tools to their natural conclusion by combining them into the interactive patterns that appear on nearly every modern website.

The insight that makes this day click: JavaScript does not create visual effects. CSS does. JavaScript’s job is simply to switch a class on or off. Your CSS — the transitions and animations you have already built, the custom properties, the dark mode patterns — handles everything that moves and changes. One line of JavaScript. Everything you already wrote does the rest.

NOTE
This principle has a name in the industry: separation of concerns. CSS concerns itself with appearance. JavaScript concerns itself with state. When you keep these separate, your code is cleaner, easier to debug, and easier to hand off. It also means your CSS transitions work automatically — you never have to write animation logic in JavaScript.

The Central Principle

Every pattern in this lesson follows the same structure. Your CSS defines two states for an element:

/* State 1: Default */
.panel {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.4s ease;
}

/* State 2: Open */
.panel.open {
  max-height: 500px;
}

Your JavaScript toggles between them with a single line:

panel.classList.toggle('open');

That’s it. The CSS transition fires automatically the moment the class changes. You wrote the animation in CSS — JavaScript just flips the switch.

[IMG - Diagram: two boxes labeled “default state” and “.open state” with a classList.toggle() arrow between them, and a CSS transition arrow showing the animated change]

This exact pattern — CSS defines two states, JS toggles the class — powers dark mode, accordions, modals, navigation menus, tabs, tooltips, dropdowns, and virtually every other interactive UI component you have ever seen. Learn this once. Build everything.

Dark Mode Toggle

Dark mode with prefers-color-scheme automatically reads the user’s operating system preference. But what if the user wants to override that preference on your specific page? That requires JavaScript.

The CSS pattern uses custom properties on two selectors: the default and the .dark-mode class applied to the body:

:root {
  --bg:   #ffffff;
  --text: #111111;
  --card: #f5f5f5;
}

body.dark-mode {
  --bg:   #1a1a2e;
  --text: #eeeeee;
  --card: #16213e;
}

body {
  background: var(--bg);
  color: var(--text);
  transition: background 0.3s, color 0.3s;
}

The JavaScript is one line:

document.body.classList.toggle('dark-mode');

Because --bg, --text, and --card cascade to every element using them, that single class toggle updates the entire page simultaneously. The transition on body means the shift is smooth. You wrote all of this CSS already — the JavaScript is the part you were missing.

You can also update the button label and icon with the ternary operator:

const isDark = document.body.classList.contains('dark-mode');
btn.textContent = isDark ? '☀ Light Mode' : '☾ Dark Mode';

Accordion

An accordion is a list of sections where clicking a heading reveals or hides its content. The max-height transition trick makes this smooth without JavaScript needing to know the exact height of the content.

/* CSS */
.accordion-body {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.35s ease;
}

.accordion-item.open .accordion-body {
  max-height: 400px;  /* Must be larger than any content height */
}
// JavaScript
const headers = document.querySelectorAll('.accordion-header');

headers.forEach(function(header) {
  header.addEventListener('click', function() {
    const item = header.parentElement;

    // Close all other items (single-open mode)
    document.querySelectorAll('.accordion-item').forEach(function(el) {
      if (el !== item) el.classList.remove('open');
    });

    // Toggle this one
    item.classList.toggle('open');

    // Update aria-expanded for accessibility
    const isOpen = item.classList.contains('open');
    header.setAttribute('aria-expanded', isOpen);
  });
});

NOTE
Toggling aria-expanded keeps the accordion accessible to screen readers, which announce “expanded” or “collapsed” as the user navigates. It takes one line and makes your component usable by everyone.

Real Hamburger Menu

There is a CSS-only hamburger menu technique that uses a hidden checkbox and the sibling selector — a clever hack that requires no JavaScript. JavaScript lets you do it properly. The CSS stays almost the same, but instead of relying on a checkbox’s :checked state, you toggle a class on the navigation element directly.

/* CSS: nav hidden by default at mobile */
.nav-links {
  position: fixed;
  top: 0; right: -100%;
  width: 75%;
  height: 100vh;
  background: var(--card);
  transition: right 0.3s ease;
  padding-top: 5rem;
}

/* Class toggle reveals it */
.nav-links.nav-open {
  right: 0;
}
// JavaScript: three lines
const menuBtn   = document.querySelector('.menu-btn');
const navLinks  = document.querySelector('.nav-links');

menuBtn.addEventListener('click', function() {
  navLinks.classList.toggle('nav-open');
  const isOpen = navLinks.classList.contains('nav-open');
  menuBtn.setAttribute('aria-expanded', isOpen);
  menuBtn.textContent = isOpen ? '✕' : '☰';
});

The checkbox hack was a creative workaround for the absence of JavaScript. Now that you have JavaScript, the clean approach is always to toggle a class. The result is identical visually — but the code is simpler, more readable, and the aria-expanded attribute makes it accessible.

Tabs

A tab system shows one panel at a time. Clicking a tab activates it and shows its corresponding content panel. The pattern requires managing an “active” class across a set of elements: remove it from all, add it to the one that was clicked.

const tabs   = document.querySelectorAll('.tab');
const panels = document.querySelectorAll('.tab-panel');

tabs.forEach(function(tab) {
  tab.addEventListener('click', function() {
    // Deactivate everything
    tabs.forEach(function(t)   { t.classList.remove('active'); });
    panels.forEach(function(p) { p.classList.remove('active'); });

    // Activate the clicked tab and its matching panel
    tab.classList.add('active');
    const targetId = tab.dataset.tab;
    document.querySelector('#' + targetId).classList.add('active');
  });
});

The data-tab attribute on each button stores the ID of the panel it controls — the same data-* and dataset pattern you have seen before, now used to wire up a real UI component.

Modal Dialog

A modal is an overlay that appears on top of the page, requiring the user’s attention before returning to the main content. The pattern involves a backdrop element and a dialog box, both hidden by default:

/* CSS */
.modal-overlay {
  position: fixed;
  inset: 0;
  background: rgba(0,0,0,0.6);
  display: flex;
  align-items: center;
  justify-content: center;
  opacity: 0;
  pointer-events: none;
  transition: opacity 0.2s;
}

.modal-overlay.open {
  opacity: 1;
  pointer-events: auto;
}
// JavaScript
const overlay = document.querySelector('.modal-overlay');

function openModal()  { overlay.classList.add('open'); }
function closeModal() { overlay.classList.remove('open'); }

document.querySelector('.open-modal-btn').addEventListener('click', openModal);
document.querySelector('.close-btn').addEventListener('click', closeModal);

// Close when clicking the backdrop (but not the dialog itself)
overlay.addEventListener('click', function(event) {
  if (event.target === overlay) closeModal();
});

[IMG - Modal pattern: a dimmed backdrop with a centered white dialog box, showing the open and closed CSS states side by side]

NOTE
Using opacity: 0; pointer-events: none; rather than display: none as the hidden state allows CSS transitions to work. display: none removes the element from the layout instantly — there is nothing to transition from. opacity and pointer-events keep the element present but invisible and non-interactive, so the transition has something to animate.

setTimeout — Delayed Actions

setTimeout schedules a function to run once after a specified delay in milliseconds:

setTimeout(function() {
  toast.classList.remove('visible');
}, 3000);  // Runs after 3 seconds (3000 ms)

The most common use is a toast notification — a brief message that appears after an action and disappears automatically:

function showToast(message) {
  const toast = document.querySelector('.toast');
  toast.textContent = message;
  toast.classList.add('visible');

  setTimeout(function() {
    toast.classList.remove('visible');
  }, 3000);
}

setTimeout returns an ID you can use to cancel the scheduled action with clearTimeout(id) if you need to — for example, if the user manually dismisses the toast before the timer fires.

setInterval — Repeated Actions

setInterval schedules a function to run repeatedly at a fixed interval:

const clockId = setInterval(function() {
  updateClock();
}, 1000);  // Runs every 1 second

Always cancel an interval when it is no longer needed using clearInterval(id). A running interval that is never cancelled will keep executing even after the user has navigated away, which wastes resources and can cause bugs.

// Start and stop a clock
let clockId = null;

function startClock() {
  clockId = setInterval(updateClock, 1000);
}

function stopClock() {
  clearInterval(clockId);
}

The classic use is a live clock, which connects to the Date object you may have seen in the Intro to JavaScript slide:

function updateClock() {
  const now  = new Date();
  const h    = String(now.getHours()).padStart(2, '0');
  const m    = String(now.getMinutes()).padStart(2, '0');
  const s    = String(now.getSeconds()).padStart(2, '0');
  document.querySelector('#clock').textContent = h + ':' + m + ':' + s;
}

Best Practices

  • ►Let CSS do the animating. Define both states in CSS with a transition. JavaScript only adds or removes the class. Never write animation timings or easing in JavaScript.
  • ►Toggle aria-expanded on interactive components. Buttons that show/hide content should have aria-expanded="true" when open and aria-expanded="false" when closed. One line of JavaScript keeps your components accessible.
  • ►Always clear intervals you no longer need. Every setInterval call should have a corresponding clearInterval that runs when the interval is done. Forgetting this is a common source of subtle bugs.
  • ►Use opacity + pointer-events for animated show/hide. display: none makes transitions impossible. Use opacity: 0; pointer-events: none; as the hidden state and transition opacity to reveal the element.
  • ►Keep modal close logic in one reusable function. Multiple triggers close the same modal (X button, backdrop click, Escape key). A closeModal() function called by all three is cleaner than duplicating the logic.

Summary

This is the payoff of everything you have built in CSS. The transitions, the @keyframes animations, the CSS custom properties, the dark mode patterns, the hamburger menu — all of it now has a JavaScript remote control. The central insight is simple and applies universally: CSS defines two states, JavaScript switches between them with classList.toggle(). You built a dark mode toggle with one line, an accordion with three, a hamburger menu with four, and a modal with five. setTimeout lets you schedule actions for the future, and setInterval lets you repeat them on a cadence. None of these required you to write animation logic in JavaScript — the CSS you already know handles all of that.
