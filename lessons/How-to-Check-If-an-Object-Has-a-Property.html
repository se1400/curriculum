Checking If an Object Has a Property

Why Check for Properties?

Understanding how to check if an object has a specific property is important for:

  • Working with data from external sources
  • Ensuring certain properties exist before using them
  • Avoiding errors when accessing properties
  • Writing defensive code that handles missing data

Three Common Methods

1. hasOwnProperty()

Checks if property exists as the object's own property

obj.hasOwnProperty("prop")

2. in Operator

Checks if property exists in the object or its prototype chain

"prop" in obj

3. Check !== undefined

Checks if property value is not undefined

obj.prop !== undefined
Key Point:

All three methods return boolean values (true or false), but they work slightly differently. Understanding these differences will help you choose the right method for your needs.

hasOwnProperty() Method

What Is hasOwnProperty()?

This method returns a boolean indicating whether the object has the specified property as its own property (not inherited). 

const person = {
    name: "Alice",
    age: 30
};

console.log(person.hasOwnProperty("name")); // true
console.log(person.hasOwnProperty("job"));  // false
true
false

How It Works

person.hasOwnProperty("name") → "name" exists → true
person.hasOwnProperty("job") → "job" doesn't exist → false
Use Case:

Use hasOwnProperty() when you want to check if a property belongs directly to the object, excluding inherited properties from the prototype chain.

The in Operator

How the in Operator Works

The in operator returns true if the property exists in the object or its prototype chain. 

const person = {
    name: "Bob",
    age: 25
};

console.log("name" in person); // true
console.log("job" in person);  // false
true
false

Syntax Difference

Notice the Property Is a String!

hasOwnProperty: person.hasOwnProperty("name")
in operator: "name" in person

With the in operator, the property name comes first and must be a string.

Both Return the Same Result:

For properties that exist directly on the object, hasOwnProperty() and in return the same result. The difference becomes important when dealing with inherited properties.

Checking Against undefined

The undefined Method

You can check if a property exists by verifying it's not undefined. 

const car = {
    brand: "Toyota",
    model: "Corolla",
    year: 2020
};

console.log(car.brand !== undefined); // true
console.log(car.color !== undefined); // false
true
false

Why This Works

Accessing a non-existent property on an object returns undefined.

car.brand exists → returns "Toyota" → "Toyota" !== undefined → true
car.color doesn't exist → returns undefined → undefined !== undefined → false
⚠️ Limitation:

This method can give false negatives if a property explicitly has the value undefined. We'll explore this issue on the next slide.

⚠️ Limitation: Explicit undefined Values

The Problem

What happens when a property exists but its value is explicitly set to undefined? 

const obj = {
    a: undefined
};

console.log(obj.a !== undefined);      // false (property exists!)
console.log(obj.hasOwnProperty("a"));  // true
console.log("a" in obj);               // true
false
true
true
❌ False Negative!

The property does exist (it's been explicitly defined), but checking against undefined returns false because the value is undefined.

obj.a !== undefined → undefined !== undefined → false ❌

Comparison: All Three Methods

obj.a !== undefined → false (incorrect!)
obj.hasOwnProperty("a") → true (correct!)
"a" in obj → true (correct!)
Best Practice:

Use hasOwnProperty() or in when you need to reliably check if a property exists, regardless of its value.

Comparing All Three Methods

Side-by-Side Examples

const person = {
    name: "Alice",
    age: 30,
    job: undefined
};

// Checking for "name" (exists with value)
console.log(person.hasOwnProperty("name")); // true
console.log("name" in person);              // true
console.log(person.name !== undefined);    // true

// Checking for "job" (exists but value is undefined)
console.log(person.hasOwnProperty("job"));  // true
console.log("job" in person);               // true
console.log(person.job !== undefined);     // false ⚠️

// Checking for "city" (doesn't exist)
console.log(person.hasOwnProperty("city")); // false
console.log("city" in person);              // false
console.log(person.city !== undefined);    // false

Results Summary

Property hasOwnProperty() in !== undefined
"name" (exists) true true true
"job" (= undefined) true true false ⚠️
"city" (missing) false false false

Summary & Best Practices

Quick Reference Guide

hasOwnProperty()

Pros:
  • Reliable
  • Checks own properties only
  • Works with undefined values
Best for:

When you want to check if a property belongs directly to the object

in Operator

Pros:
  • Concise syntax
  • Checks prototype chain
  • Works with undefined values
Best for:

When you want to check existence including inherited properties

!== undefined

Pros:
  • Simple and readable
  • Quick to type
Cons:
  • ⚠️ Fails with undefined values
Best for:

Quick checks when you know the property won't be undefined

Decision Guide

Use hasOwnProperty() when:

You need to verify a property exists directly on the object

Use in operator when:

You want to check for properties including inherited ones

Use !== undefined when:

You're doing a quick check and know the property won't be explicitly undefined

Final Takeaway:

The choice between these methods often depends on the specific requirements of your code. Understanding the differences between them will help you make the right choice in different scenarios. For most cases, hasOwnProperty() or in are the safest options.
