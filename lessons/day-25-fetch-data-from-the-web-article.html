Fetch — Data from the Web

APIs, JSON, and Loading Real Data with fetch()

What You’ll Learn in This Lesson:

  • ►What an API is and how the browser requests data from one
  • ►The fetch() function — the browser’s built-in tool for making HTTP requests
  • ►The .then() chain — how to work with data after it arrives
  • ►How to show a loading state while waiting for a response
  • ►How to handle errors with .catch()
  • ►Why your renderCards() function works exactly the same whether data comes from an array or an API

Every gallery, card grid, and list you have built so far used a hardcoded array — data written directly in the JavaScript file. Real websites do not work that way. Product listings come from a database. Weather data comes from a weather service. Movie details come from a film catalog API. The data lives on a server, and the browser requests it over the network.

fetch() is the browser’s built-in function for making those requests. It sends an HTTP request to a URL and returns the response — usually as JSON, the same format you read and write with JSON.parse() and JSON.stringify(). The key insight: your renderCards() function does not need to change at all. The only thing that changes is where the data comes from.

What Is an API?

An API (Application Programming Interface) is a server that sends back data in response to a request. Instead of returning HTML for a browser to display, an API returns raw data — typically formatted as JSON — for JavaScript to use.

You send a request to a URL like https://api.example.com/products. The server responds with JSON like this:

[
  { "id": 1, "name": "Espresso", "category": "Coffee", "price": 3.50 },
  { "id": 2, "name": "Cold Brew", "category": "Coffee", "price": 4.75 }
]

That is exactly the same shape as the arrays of objects you have been building by hand. The only difference is the data comes from a server instead of being written in your code.

The fetch() Function

The fetch recipe has four lines:

fetch('https://api.example.com/products')
  .then(function(response) { return response.json(); })
  .then(function(data) {
    renderCards(data);
  })
  .catch(function(error) {
    container.innerHTML = '<p class="error">Failed to load data. Please try again.</p>';
  });

Here is what each line does:

  • ►Line 1: fetch(url) — sends an HTTP GET request to the URL. This does not happen instantly. The response will arrive sometime in the future.
  • ►Line 2: .then(response => response.json()) — when the response arrives, convert its body from a JSON string into a JavaScript array or object.
  • ►Line 3: .then(data => renderCards(data)) — when the conversion is done, use the data. Pass it to your render function.
  • ►Line 4: .catch(error => { ... }) — if anything goes wrong (network down, bad URL, server error), show a message. Never leave the user looking at a blank page.

NOTE
fetch() is asynchronous — it does not wait for the response before moving on. The code after your fetch() call runs immediately while the request is still in flight. The .then() callbacks run later, when the response arrives. This is why the data is not available in a regular variable after the fetch() line — you must use it inside the .then().

Loading States

Network requests take time — anywhere from a few milliseconds to several seconds depending on the connection. While the user is waiting, show a loading indicator rather than leaving the container empty:

// Show loading state before the request starts
container.innerHTML = '<p class="loading">Loading…</p>';

fetch('https://api.example.com/products')
  .then(function(response) { return response.json(); })
  .then(function(data) {
    renderCards(data);  // Replaces the loading message
  })
  .catch(function(error) {
    container.innerHTML = '<p class="error">Failed to load. Please try again.</p>';
  });

Set the loading message before the request. When the data arrives, renderCards() overwrites it. If the request fails, .catch() overwrites it with an error message. The container never shows blank space.

Replacing Hardcoded Data with fetch()

Here is how little changes when you swap a hardcoded array for an API. Before:

// Hardcoded array
const products = [
  { id: 1, name: 'Espresso',  category: 'Coffee', price: 3.50 },
  { id: 2, name: 'Cold Brew', category: 'Coffee', price: 4.75 }
];

renderCards(products);

After, with fetch():

// Same data, from the network
container.innerHTML = '<p class="loading">Loading…</p>';

fetch('https://www.themealdb.com/api/json/v1/1/search.php?s=pasta')
  .then(function(response) { return response.json(); })
  .then(function(data) {
    renderCards(data.meals);  // The API wraps its array in a property
  })
  .catch(function() {
    container.innerHTML = '<p class="error">Failed to load.</p>';
  });

The renderCards() function is completely unchanged. The filter, search, modal, and localStorage patterns you have already built are completely unchanged. The data just arrives from a different place.

NOTE
Real APIs often wrap their data in a property. TheMealDB returns { meals: [ ... ] }, not a bare array. Always check the structure of the response in the browser DevTools Network tab or the API’s documentation so you know exactly which property to pass to your render function.

Reading the Response in DevTools

Before writing any code, look at what an API actually returns:

  1. Open DevTools and click the Network tab
  2. Filter by Fetch/XHR
  3. Run your fetch or visit the URL directly in a new tab
  4. Click the request in the list, then open the Preview or Response tab
  5. Inspect the structure: is the array at the top level, or inside a property like data.results?

This single step saves most of the confusion when working with a new API. You are just reading a JSON object the same way you have been reading objects all week.

async/await — A Cleaner Syntax for the Same Thing

The .then() chain works well, but there is a more readable syntax for the same operation: async/await. Both do exactly the same thing under the hood.

// .then() version
fetch(url)
  .then(response => response.json())
  .then(data => renderCards(data))
  .catch(error => { container.innerHTML = '<p>Error.</p>'; });

// async/await version — same result, reads like regular code
async function loadData() {
  try {
    const response = await fetch(url);
    const data     = await response.json();
    renderCards(data);
  } catch (error) {
    container.innerHTML = '<p>Error.</p>';
  }
}

loadData();

await pauses execution inside an async function until the promise resolves — meaning you can write asynchronous code that looks synchronous. The try/catch pattern you already know handles errors here too.

NOTE
Both syntaxes are common. Many real codebases use async/await because it is easier to read and debug. Start with .then() to understand what is happening, then try rewriting the same fetch with async/await to see the difference. They are interchangeable.

Best Practices

  • ►Always show a loading state before fetching. Set the container content before the request. If the response is fast, users barely see it. If it is slow, they know the page is working.
  • ►Always handle errors with .catch() or a try/catch block. Networks fail. Servers go down. A blank page with a console error is a broken experience; a friendly error message is not.
  • ►Inspect the API response before writing any render code. Check the Network tab in DevTools. Know the exact shape of the data so you can access the right properties.
  • ►Keep your render function generic. A renderCards(items) that takes an array works whether the array came from a hardcoded constant or an API. Do not mix fetch logic into your render function.
  • ►Handle the empty state. Some API responses return an empty array (no results for a search query, for example). Your render function already handles this — make sure it does.

Summary

An API is a URL that returns data — usually JSON — instead of an HTML page. fetch() sends an HTTP request to that URL and returns the response asynchronously. The .then() chain handles the response in two steps: first response.json() converts the raw response into a JavaScript value, then the second .then() uses that value. .catch() handles anything that goes wrong. A loading state set before the request and replaced by real data (or an error) is a simple, professional UX pattern. async/await is an alternative syntax for the same operation that reads more like regular sequential code. Most importantly: the renderCards() function, the filter logic, the search, the modal pattern, and the localStorage persistence you have already built all work unchanged when the data source switches from a hardcoded array to a network request. Learning fetch() is the last piece. Everything else you already built.
