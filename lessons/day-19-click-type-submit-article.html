In Lesson 18, every JavaScript change happened once — when the page loaded. The variables ran, the elements updated, and then JavaScript went silent. Your page was still fundamentally static: it changed once, then waited. This lesson changes that entirely.

Events are how JavaScript listens to the user. Every click, every keystroke, every slider drag, every form submission — the browser tracks all of it. Your job is to tell JavaScript which events to listen for and what to do when they occur. Once you understand events, your pages stop being documents and start being applications.

The visual payoff arrives immediately. Range sliders that shift your entire page color scheme as you drag them. Password fields that show a live strength bar. Forms that validate as you type rather than scolding you after submission. All of it runs on the same two ideas: listen for an event, then decide what to do.

addEventListener — The Right Way

In the Intro to JavaScript slide, you saw inline event handlers: onclick="doSomething()" written directly in the HTML. That approach works, but it has a significant limitation — it mixes your JavaScript logic into your HTML markup, making both harder to read and maintain.

The modern approach separates concerns cleanly. You select an element with querySelector, then attach a listener to it from your JavaScript file:

const btn = document.querySelector('#submit-btn');

btn.addEventListener('click', function() {
  console.log('Button was clicked!');
});

Read this as: “Find the element with id submit-btn. When a click event occurs on it, run this function.”

The function you pass as the second argument is called a handler. It runs every time the event fires — not just once on page load. The same button can be clicked fifty times and the handler runs fifty times.

NOTE
You can attach multiple addEventListener calls to the same element — each one stacks. Inline onclick can only hold one handler; the second one overwrites the first. This is one of several reasons addEventListener is preferred in professional code.

The Four Events You’ll Use Most

click

Fires when the user clicks an element — buttons, cards, links, labels, anything. This is the most common event you’ll attach:

const btn = document.querySelector('.toggle-btn');
btn.addEventListener('click', function() {
  document.body.classList.toggle('dark-mode');
});

input

Fires on every single keystroke or value change in a form field — text inputs, range sliders, color pickers. It fires continuously as the user interacts, which makes it perfect for live updates:

const slider = document.querySelector('#hue-slider');
slider.addEventListener('input', function() {
  document.documentElement.style.setProperty('--hue', slider.value);
});

This is the event that powers live color theme editors, search-as-you-type, and character counters.

change

Similar to input, but fires only after the user has finished interacting — when they leave the field or make a selection. Better for dropdowns and checkboxes where you want to respond once, not on every intermediate state:

const select = document.querySelector('#font-select');
select.addEventListener('change', function() {
  document.body.style.fontFamily = select.value;
});

submit

Fires when a form is submitted — either by clicking a submit button or pressing Enter inside a text field. You will use this with event.preventDefault() every single time:

const form = document.querySelector('#signup-form');
form.addEventListener('submit', function(event) {
  event.preventDefault();  // Stop the page from refreshing
  // Now read the form values and do something with them
});

NOTE
Without event.preventDefault(), a form submission reloads the page. This has been the default browser behavior since 1995 and was a source of constant frustration in Lesson 14. Now you can stop it with one line.

The Event Object

When an event fires, the browser automatically passes an event object to your handler. This object contains information about exactly what happened: which element was interacted with, what the current value is, whether any modifier keys were held, and more.

To access it, add a parameter to your handler function — conventionally named event or just e:

btn.addEventListener('click', function(event) {
  console.log(event);         // The full event object
  console.log(event.type);    // 'click'
  console.log(event.target);  // The element that was clicked
});

event.target

event.target refers to the specific element the user interacted with. This is useful when you want to respond to the thing that was actually clicked or typed in, without querying for it again:

const btn = document.querySelector('#theme-btn');
btn.addEventListener('click', function(event) {
  event.target.classList.toggle('active');  // Toggle on the clicked button itself
});

event.target.value

For any form input, event.target.value gives you the current value at the moment the event fired:

const input = document.querySelector('#search');
input.addEventListener('input', function(event) {
  const query = event.target.value;
  console.log('Currently typed:', query);
});

Reading Form Values

Form elements expose their current value through properties. The one you need depends on the element type.

Text, Email, Password, Search

const nameInput = document.querySelector('#name');
const value = nameInput.value;  // Returns a string

Number and Range

const slider = document.querySelector('#volume');
const num = Number(slider.value);  // .value is always a string — convert it

CAUTION
.value on any form input always returns a string, even if the input is type="number" or type="range". If you need to do math with the value, wrap it in Number(): const score = Number(input.value); Without this, "5" + 3 gives you "53" instead of 8.

Select (Dropdown)

const select = document.querySelector('#color-scheme');
const selected = select.value;  // The value attribute of the chosen <option>

Checkbox

const checkbox = document.querySelector('#agree');
const isChecked = checkbox.checked;  // Returns true or false

Color Picker

const colorPicker = document.querySelector('#accent-color');
const hex = colorPicker.value;  // Returns a hex string: '#6c5ce7'

Making Decisions: if, else if, else

Events tell JavaScript when something happened. Conditional statements tell JavaScript what to do about it. Together they power every interaction pattern you’ll build.

An if statement runs a block of code only when a condition is true:

const score = Number(input.value);

if (score >= 90) {
  label.textContent = 'Excellent';
  label.className = 'badge badge--green';
}

Add else if for additional conditions, and else as a catch-all for anything that didn’t match:

if (score >= 90) {
  label.textContent = 'Excellent';
} else if (score >= 70) {
  label.textContent = 'Good';
} else if (score >= 50) {
  label.textContent = 'Pass';
} else {
  label.textContent = 'Not yet';
}

JavaScript evaluates conditions top to bottom and stops at the first one that is true. Order matters: put the most specific conditions first.

Comparison Operators

These operators compare two values and return true or false:

score === 100   // Strictly equal (value AND type must match)
score !== 0     // Not equal
score > 50      // Greater than
score < 100     // Less than
score >= 60     // Greater than or equal to
score <= 99     // Less than or equal to

NOTE
Always use === (triple equals), never == (double equals). Double equals performs type coercion, which can produce surprising results: "5" == 5 is true. Triple equals compares both value and type: "5" === 5 is false, which is the correct behavior. Treat == as a legacy feature to avoid.

Logical Operators

Combine multiple conditions into a single expression:

// AND — both must be true
if (password.length >= 8 && password.includes('@')) { ... }

// OR — at least one must be true
if (role === 'admin' || role === 'editor') { ... }

// NOT — flips true to false and false to true
if (!isLoggedIn) { redirectToLogin(); }

Truthy and Falsy

In JavaScript, every value is either truthy or falsy when used in a condition. You don’t always need an explicit comparison — JavaScript converts the value automatically.

These six values are falsy — they behave like false in an if statement:

  • ►false
  • ►0
  • ►"" (empty string)
  • ►null
  • ►undefined
  • ►NaN

Everything else is truthy — any non-empty string, any non-zero number, any object or array.

This matters most when validating form fields. An empty text input has a value of "" — which is falsy — so you can check for it concisely:

if (!nameInput.value) {
  showError('Name is required');
}

// This is the same as:
if (nameInput.value === '') {
  showError('Name is required');
}

The first version is shorter and reads naturally. Both are correct.

The Ternary Operator

The ternary operator is a compact one-line version of if/else. It’s useful when you’re choosing between two values — not running two blocks of code:

condition ? valueIfTrue : valueIfFalse

Examples:

// Set text based on a condition
btn.textContent = isDark ? 'Switch to Light' : 'Switch to Dark';

// Choose a CSS class
badge.className = score >= 60 ? 'badge badge--pass' : 'badge badge--fail';

// Set an attribute
icon.setAttribute('aria-label', isOpen ? 'Close menu' : 'Open menu');

Use the ternary for simple value choices. If you need to run multiple lines of code in each branch, use a regular if/else — ternary becomes hard to read when nested or complex.

Live CSS Updates with Events

This is where Lesson 19 pays off your CSS knowledge most directly. Range sliders connected to CSS custom properties via addEventListener('input') give the user a live control panel for your design. Every element using var(--hue) updates instantly as the slider moves — no button click required.

const hueSlider = document.querySelector('#hue');
const satSlider = document.querySelector('#saturation');

hueSlider.addEventListener('input', function() {
  document.documentElement.style.setProperty('--hue', hueSlider.value);
});

satSlider.addEventListener('input', function() {
  document.documentElement.style.setProperty('--saturation', satSlider.value + '%');
});

Combine this with a <input type="color"> picker and you can let users choose their own accent color:

const colorPicker = document.querySelector('#accent');
colorPicker.addEventListener('input', function() {
  document.documentElement.style.setProperty('--accent', colorPicker.value);
});

The same technique powers dark mode toggles, animation speed controls, font size sliders, and spacing adjusters — any design parameter that students have expressed in a CSS custom property can now be given to the user as a live control.

Best Practices

  • ►Store element references before adding listeners. Call querySelector() once and keep the result in a variable. Do not call it inside the event handler on every fire.
  • ►Always use event.preventDefault() on form submissions. Without it, the page reloads and all your JavaScript state is lost.
  • ►Convert numeric form values immediately. Wrap .value in Number() the moment you read it from a number or range input, so the rest of your code works with actual numbers.
  • ►Use input for live updates, change for final selections. input fires continuously as the user types or drags; change fires once when they finish. Pick whichever fits the interaction.
  • ►Keep handlers short. If the handler function grows long, extract the logic into a named function and call it from the handler. This makes the code easier to read and test.

Summary

Events are the bridge between a user’s action and JavaScript’s response. You learned to attach handlers with addEventListener, work with the event object to read event.target and call event.preventDefault(), and read values from every form element type. Conditional statements with if/else if/else let JavaScript make decisions based on those values, and the ternary operator gives you a concise way to choose between two outcomes. The most immediately visual technique — connecting range sliders and color pickers to CSS custom properties — shows how events and the CSS knowledge you already have combine into something that feels like a real product feature.

In the next lesson, you will bring together everything from Lessons 18 and 19 to build the UI patterns that appear on nearly every modern website: dark mode toggles, accordions, hamburger navigation menus, and modals. All of them are built on the same single idea: CSS defines two states, JavaScript toggles between them.
