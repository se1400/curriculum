What Are Comments?

Comments in programming are used to provide additional context for the code or leave notes for yourself and others.

Key Characteristic

Comments are lines or blocks of text that are ignored by the JavaScript engine when your code is executed.

Purpose of Comments

They are there solely for the benefit of people reading the code, whether that's you or someone else.

  • Explain complex logic
  • Document important decisions
  • Leave reminders for yourself
  • Help other developers understand your code

Two Types of Comments

JavaScript provides two ways to add comments to your code:

Single-Line Comments

//

Created using two forward slashes

Multi-Line Comments

/* */

Created using slash-asterisk pairs

Let's explore each type in detail!

Single-Line Comments

Single-line comments are created using // (two forward slashes).

Example: Basic Single-Line Comment

// I am a single line comment in JavaScript

When to Use Single-Line Comments

This type of comment is great for brief explanations or clarifications.

Example: Commenting Code

let price = 10;  // Store the item price
let quantity = 5; // Number of items purchased

Common Uses

  • Quick notes next to a line of code
  • Brief explanations of variables or values
  • Short clarifications about logic

Real-World Example: Single-Line Comments

Here's a real-world example from the freeCodeCamp curriculum project files:

// This is to allow English to build without having to download the i18n files.
// It fails when trying to resolve the i18n-curriculum path if they don't exist.
const curriculumLocale = process.env.CURRICULUM_LOCALE ?? 'english';
const I18N_CURRICULUM_DIR = path.resolve(
  __dirname,
  curriculumLocale === 'english' ? '.' : 'i18n-curriculum/curriculum'
);

Focus on the Comment, Not the Code

Don't worry about trying to understand what the code is actually doing because it's more advanced than what you've learned so far.

Instead, focus on the comment left by the developer. This comment provides important context for why this code exists.

Multi-Line Comments

When you need to write longer descriptions or notes, use multi-line comments.

Basic Syntax

/*
 I am a multiline comment.
 This is helpful for longer explanations.
*/

When to Use Multi-Line Comments

Multi-line comments are useful when you need to write longer descriptions, explanations, or notes in your code.

Example: Documenting Logic

/*
 This function calculates the total cost including tax.
 It takes the base price and quantity as parameters.
 Tax rate is set at 8.5% for this region.
*/
function calculateTotal(price, quantity) {
  // Function code here
}

Understanding the Context

Just like before, ignore all of the JavaScript code because it uses concepts that haven't been taught yet. Instead, focus on the comment left by the developer.

A developer on the team, or even a new contributor working on the project, can understand why this piece of code is here and have the full context before working on this area of the project.

Avoiding Over-Commenting

‚ö†Ô∏è Important: Don't Over-Comment

While comments are useful in programming, it's important to avoid over-commenting.

You don't need to comment on every single line of code, especially if the code is straightforward and self-explanatory.

Example: Unnecessary Comments

// This code uses the const keyword to create a new variable called price.
// We are assigning the number 10 to the price variable.
const price = 10;

‚ùå Bad: There's no need to add any comments here because the code is self-explanatory.

The goal is to enhance readability, not clutter the code with unnecessary explanations.

Personal Projects vs Team Projects

üìù Personal Learning Projects

If you want to add comments to your personal projects as you are learning to code, that's fine.

Comments can help you remember what you were thinking or learning.

üë• Team Projects

Once you start working on real-world projects with other developers, it's important not to use comments for code that is self-explanatory.

Only comment when providing valuable context.

Don't Use Comments to Hide Bad Code

It's also important not to use comments to help explain away confusing, overly complicated, or poorly written code.

In those situations, it's best to refactor, or change, your code so other developers will better understand what is going on.

Comment Best Practices

When to Use Comments

‚úì DO Use Comments For:

  • Explaining why code exists, not what it does
  • Documenting complex logic or algorithms
  • Providing context about business decisions
  • Warning about potential issues or edge cases
  • Documenting assumptions or requirements

‚úó DON'T Use Comments For:

  • Explaining obvious code
  • Repeating what the code clearly shows
  • Commenting every single line
  • Covering up poorly written code (refactor instead!)
  • Leaving commented-out code in production

Comments Summary

Key Takeaways

Powerful tools for documenting your code and making it easier to understand.

Two Types

When to Use Them

You should use comments to provide context or leave notes for yourself and others, but avoid over-commenting obvious code.

  • They're for people, not computers
  • Explain the "why", not the "what"
