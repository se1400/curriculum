CSS Transitions and Transforms

What You'll Learn in This Lesson:

  • What CSS transitions are and how to control their four sub-properties
  • How timing functions control the pace and feel of a transition
  • Which CSS properties can be animated, and why transform and opacity are special
  • How to move, rotate, scale, and skew elements with 2D transforms
  • How to create depth effects and interactive card flips with 3D transforms
  • How combining transitions with transforms produces polished UI interactions
  • How to respect user motion preferences with prefers-reduced-motion

A static web page that simply displays text and images will only take you so far. Modern web design relies on motion — subtle hover effects, smooth reveals, interactive controls that respond to touch — to communicate state changes, guide attention, and make interfaces feel alive. In this lesson, you'll learn the two CSS features that power almost all of that motion: transitions and transforms.

Transitions let you tell the browser: "when this property changes, animate it smoothly instead of snapping." Transforms let you move, rotate, scale, and flip elements without disrupting the layout around them. Together, they give you the tools to build everything from a simple hover glow on a button to a full 3D card flip — entirely in CSS, with no JavaScript required.

The Evolution of Web Animation

To appreciate what CSS transitions and transforms give you, it helps to understand what life looked like before them. In the early 2000s, complex animations on the web belonged to Adobe Flash — a plugin that let designers build frame-by-frame motion graphics. When Flash fell out of favor (partly due to security issues, partly because it didn't work on mobile), developers turned to JavaScript. Libraries like jQuery offered a convenient .animate() method that could gradually change an element's position, size, or color by running a timer and nudging property values dozens of times per second.

The problem with JavaScript animation is that it runs on the browser's main thread — the same thread handling user input, page rendering, and everything else. Under load, that single thread gets busy, and animations become choppy or freeze entirely. The technical term for this is jank. CSS transitions and transforms solve the problem by handing animation work off to the browser's compositor and, on modern hardware, to the GPU. The browser knows how to interpolate a CSS transition smoothly at 60 frames per second without touching the main thread at all.

One piece of history you'll encounter in older code: early implementations of transforms and transitions required vendor prefixes — special versions of the property for each browser engine. You may see code like this in legacy stylesheets:

LISTING 14.1 — Vendor-prefixed transforms from legacy code (for reference only)

/* You may see this in old code — do not write it for new projects */
-webkit-transform: rotate(45deg);   /* Chrome, Safari, older iOS */
   -moz-transform: rotate(45deg);   /* Firefox 3.5–15 */
    -ms-transform: rotate(45deg);   /* Internet Explorer 9 */
        transform: rotate(45deg);   /* Modern standard */

Vendor prefixes are no longer needed for transitions or transforms in any actively maintained browser. Write the unprefixed standard property and move on.

Understanding CSS Transitions

A CSS transition watches a property for changes. When the property changes — because the user hovered, a class was toggled, or focus moved — the transition smoothly animates from the old value to the new one over a specified duration. Without a transition, the change is instantaneous. With one, it's animated.

Four sub-properties control exactly how that animation behaves.

The Four Transition Properties

transition-property specifies which CSS property to watch. You can name a specific property (background-color, transform, opacity), use all to watch every property, or list multiple properties separated by commas.

transition-duration sets how long the animation takes, in seconds (s) or milliseconds (ms). 0.3s and 300ms are the same. A duration of 0s means no animation — the change happens instantly.

transition-timing-function controls the pace of the animation — whether it starts slow and ends fast, runs at a constant speed, or overshoots and bounces back. You'll learn all the options in the next section.

transition-delay sets how long the browser waits before starting the animation after the property changes. Useful for staggering multiple transitions.

LISTING 14.2 — The four transition sub-properties written individually

button {
  background-color: #4f6df5;
  color: white;
  border-radius: 4px;
  padding: 0.75rem 1.5rem;

  transition-property: background-color, transform;
  transition-duration: 0.2s;
  transition-timing-function: ease-out;
  transition-delay: 0s;
}

button:hover {
  background-color: #3a56d4;
  transform: translateY(-2px);  /* We'll cover transforms shortly */
}

The Transition Shorthand

In practice, most developers use the transition shorthand, which packs all four values into one declaration. The order is: property, duration, timing-function, delay.

LISTING 14.3 — Transition shorthand syntax

/* Single property */
button {
  transition: background-color 0.2s ease-out;
}

/* Multiple properties — comma-separated */
button {
  transition: background-color 0.2s ease-out,
              transform 0.15s ease-out;
}

/* All properties at once (use with caution — can be expensive) */
button {
  transition: all 0.2s ease-out;
}

NOTE: The transition property goes on the default state of the element, not on the :hover rule. This ensures the transition runs both when the property changes to the hovered value and back when the hover ends. If you put the transition only on :hover, the animation plays going in but snaps back instantly when the cursor leaves.

Timing Functions: Controlling the Pace

The timing function is what makes an animation feel natural or mechanical, snappy or sluggish. The browser offers several built-in keyword values, a mathematical function for custom curves, a step-based function for discrete motion, and a newer function for spring-like effects.

Built-In Keyword Values

  • ease — Starts slow, accelerates through the middle, decelerates at the end. This is the default. It feels natural and is the right choice for most UI interactions.
  • ease-in — Starts slow, ends fast. Good for elements leaving the screen — they "launch."
  • ease-out — Starts fast, ends slow. Good for elements arriving on screen — they "land." This is the most common choice for hover effects.
  • ease-in-out — Slow start, fast middle, slow end. Symmetrical. Use for longer animations or elements that travel across the screen.
  • linear — Constant speed throughout. Feels mechanical. Good for spinning indicators or anything that loops continuously.

Custom Curves with cubic-bezier()

All the keyword values above are shortcuts for specific cubic-bezier() curves. A cubic bezier curve is defined by four numbers — two control point coordinates — that determine the shape of the speed curve. You can craft any easing curve, including ones that overshoot (creating a springy "bounce past the target" feel).

LISTING 14.4 — cubic-bezier() examples

/* ease is equivalent to: */
transition-timing-function: cubic-bezier(0.25, 0.1, 0.25, 1.0);

/* A springy, slightly overshooting curve */
transition-timing-function: cubic-bezier(0.34, 1.56, 0.64, 1);

/* A sharp deceleration (snappy landing) */
transition-timing-function: cubic-bezier(0.22, 1, 0.36, 1);

Chrome DevTools has a built-in cubic-bezier editor — click the small curve icon that appears next to any timing function value in the Styles panel to open a visual editor where you can drag the control points and see a live preview.

[IMG - Chrome DevTools cubic-bezier curve editor open]

Stepped Animation with steps()

The steps(n) function divides the transition into n discrete jumps instead of a smooth interpolation. Each jump happens instantly, with no animation between steps. This is useful for sprite sheet animations or anything where you want a deliberately "retro" or mechanical look.

LISTING 14.5 — Using steps() for a sprite-sheet animation

.sprite {
  width: 64px;
  height: 64px;
  background-position: 0 0;
  /* Jump through 8 frames in 0.8 seconds */
  transition: background-position 0.8s steps(8);
}

.sprite:hover {
  background-position: -512px 0; /* 8 frames × 64px wide */
}

The Modern linear() Function

A newer addition to CSS (supported in all modern browsers as of 2023) is the linear() easing function. Unlike the older linear keyword (which is just constant speed), linear() accepts a list of value stops and creates a custom piecewise linear curve — meaning you can approximate spring physics, bounce effects, and other complex motions without writing @keyframes

LISTING 14.6 — linear() for a spring-like bounce

/* A bouncy spring effect using linear() */
.spring {
  transition: transform 0.6s linear(
    0, 0.5, 1.05, 0.95, 1.02, 0.98, 1
  );
}

.spring:hover {
  transform: translateY(-20px);
}

For generating complex linear() curves, use the generator at linear-easing-generator.netlify.app.

What Can (and Cannot) Be Transitioned

CSS transitions only work on properties that have a numeric or color value the browser can interpolate between. You can transition from opacity: 0 to opacity: 1 because the browser can generate every decimal value in between. You can transition background-color from red to blue because colors have numeric RGB components. Properties that accept only discrete, non-numeric values — like display, visibility (sort of), or font-family — cannot be smoothly transitioned in the traditional sense.

Commonly transitioned properties include: opacity, transform, background-color, color, border-color, box-shadow, width, height, padding, margin, font-size, border-radius.

NOTE: Transitioning layout properties like width, height, padding, or margin causes the browser to recalculate the entire page layout on every frame of the animation. This is expensive and can cause performance problems on complex pages. Whenever possible, reach for transform and opacity instead — they bypass layout entirely.

The display: none Problem — and the Modern Solution

One of the longest-standing frustrations in CSS was the inability to transition an element into or out of display: none. Because display is a discrete property, the browser would either show the element or not — with no animated fade-in or slide-in. Developers worked around this with JavaScript class toggling, or by combining visibility with opacity.

As of 2023–2024, modern browsers support two new features that solve this cleanly:

@starting-style lets you define the property values an element should start from when it first appears in the DOM or becomes visible. The browser transitions from those starting values to the element's normal values.

transition-behavior: allow-discrete tells the browser to include discrete properties (like display) in the transition sequence, holding the "off" value until the transition finishes when hiding, and applying it immediately when showing.

LISTING 14.7 — Animating an element from display: none using @starting-style

.dialog {
  display: none;
  opacity: 0;
  transform: translateY(12px);
  transition: opacity 0.3s ease, transform 0.3s ease,
              display 0.3s allow-discrete;
  /* OR: transition-behavior: allow-discrete; */
}

.dialog.is-open {
  display: block;
  opacity: 1;
  transform: translateY(0);
}

/* Define where the animation starts FROM when the element appears */
@starting-style {
  .dialog.is-open {
    opacity: 0;
    transform: translateY(12px);
  }
}

NOTE: @starting-style and transition-behavior: allow-discrete are supported in Chrome 117+, Firefox 129+, and Safari 17.5+. All are current as of early 2025. For projects requiring older browser support, JavaScript-based class toggling remains the safe fallback.

Transitions and Performance

Not all CSS animations are equal in terms of how much work they ask of the browser. Understanding the performance implications will help you write transitions that feel smooth even on lower-powered devices.

GPU-Accelerated Properties

Modern browsers can hand certain animations off to the GPU (Graphics Processing Unit), which is purpose-built for the kind of math involved in compositing visual layers. Two properties receive this treatment:

  • transform — Moving, rotating, scaling, and skewing with transforms never triggers layout or paint. The browser promotes the element to its own compositor layer and lets the GPU handle the interpolation.
  • opacity — Fading an element in or out is also handled entirely by the compositor layer.

This is why you should almost always prefer transform: translateX(100px) over animating left: 100px, and prefer fading with opacity over animating visibility or toggling colors. The visual result may look identical, but the GPU path is dramatically smoother.

The will-change Property

The will-change property is a hint to the browser that an element is about to be animated. The browser can then prepare a compositor layer for it in advance, reducing the overhead at the moment the animation starts.

LISTING 14.8 — Using will-change

/* Tell the browser this element will be transformed */
.card {
  will-change: transform;
}

.card:hover {
  transform: scale(1.03);
  transition: transform 0.2s ease-out;
}

CAUTION: will-change costs memory. The browser allocates a separate GPU layer for each element that declares it. Do not apply will-change to large numbers of elements or to elements that aren't actually animated. Apply it only to elements with complex transitions, and consider adding it dynamically with JavaScript just before an animation begins and removing it after. When overused, will-change can hurt performance rather than help it.

Respecting User Motion Preferences

Some users configure their operating system to reduce or eliminate motion effects — because of vestibular disorders, migraines, epilepsy, or personal preference. The CSS media feature prefers-reduced-motion lets you detect this setting and tone down or eliminate your animations for those users.

LISTING 14.9 — Respecting prefers-reduced-motion

/* Default: animated */
.button {
  transition: transform 0.2s ease-out, box-shadow 0.2s ease-out;
}

.button:hover {
  transform: translateY(-3px);
  box-shadow: 0 8px 20px rgba(0, 0, 0, 0.15);
}

/* Reduced motion: instant state change, no animation */
@media (prefers-reduced-motion: reduce) {
  .button {
    transition: none;
  }
}

You can test this in Chrome DevTools: open the Rendering panel (from the three-dot menu → More tools → Rendering) and find Emulate CSS media feature prefers-reduced-motion.

NOTE: prefers-reduced-motion does not mean "no motion at all." It means "less motion." A fade from 0 to 1 opacity is generally fine; a parallax effect that slides many elements at different speeds while scrolling is not. Use your judgment to distinguish between functional transitions (like a button state change) and purely decorative motion (like a page-entry fly-in).

2D Transforms

The transform property lets you visually reposition and reshape an element without affecting the document flow. The element still occupies its original space in the layout — other elements do not move to accommodate the transform. Think of it like picking up a piece of paper and rotating it while the rest of the table stays still.

All 2D transform functions are values passed to the transform property. You can apply one or combine several.

translate() — Moving Elements

translate(x, y) moves an element horizontally and vertically from its current position. The values can be lengths (px, rem, em) or percentages. A positive X value moves right; a positive Y value moves down.

LISTING 14.10 — translate() examples

/* Move 50px right and 20px down */
transform: translate(50px, 20px);

/* Shorthand for one axis only */
transform: translateX(50px);
transform: translateY(-20px);  /* Move 20px UP */

/* Percentage is relative to the element's own size */
transform: translateX(100%);  /* Move right by the element's own width */

rotate() — Spinning Elements

rotate(angle) spins an element clockwise around its center point by default. Angles are expressed in degrees (deg), turns (turn), or radians (rad).

LISTING 14.11 — rotate() examples

transform: rotate(45deg);      /* 45 degrees clockwise */
transform: rotate(-90deg);     /* 90 degrees counter-clockwise */
transform: rotate(0.25turn);   /* Quarter turn clockwise (same as 90deg) */
transform: rotate(1turn);      /* Full rotation (good for spin animations) */

scale() — Resizing Elements

scale(x, y) resizes an element from its center. A value of 1 is normal size, 1.5 is 50% larger, 0.5 is half size. If you provide only one value, it scales proportionally in both axes.

LISTING 14.12 — scale() examples

transform: scale(1.2);         /* 20% larger in both dimensions */
transform: scale(1.5, 0.8);    /* Wider and shorter */
transform: scaleX(2);          /* Double width only */
transform: scale(0);           /* Shrink to nothing (good for hide animations) */

skew() — Shearing Elements

skew(x-angle, y-angle) tilts an element along one or both axes, creating a slanted or parallelogram effect. This is used less frequently than the other transforms but can create interesting decorative effects.

LISTING 14.13 — skew() examples

transform: skewX(15deg);        /* Slant left-to-right */
transform: skewY(5deg);         /* Slant top-to-bottom */
transform: skew(15deg, 5deg);   /* Both axes */

Combining Multiple Transforms

You can apply multiple transform functions in a single transform declaration by separating them with spaces. Order matters. Transforms are applied right to left — the last function in the list executes first. This means rotate(45deg) translateX(100px) (rotate first, then translate along the rotated axis) produces a different result than translateX(100px) rotate(45deg) (translate first, then rotate).

LISTING 14.14 — Combining transforms — order affects the result

/* Translate first, then rotate around the new position */
transform: translateX(100px) rotate(45deg);

/* Rotate first, then translate along the rotated axis */
transform: rotate(45deg) translateX(100px);

/* Scale, then translate */
transform: scale(1.5) translateY(-30px);

/* A full hover effect: lift, scale, and add a slight rotation */
.card:hover {
  transform: translateY(-8px) scale(1.02) rotate(-1deg);
}

Changing the Pivot Point: transform-origin

By default, transforms happen around the center of the element. The transform-origin property changes that pivot point. It accepts two values — a horizontal and a vertical position — using keywords (top, center, bottom, left, right) or lengths and percentages.

LISTING 14.15 — transform-origin examples

/* Rotate from the top-left corner (like turning a page) */
transform-origin: top left;
transform: rotate(45deg);

/* Scale from the bottom center (like growing upward) */
transform-origin: bottom center;
transform: scaleY(1.5);

/* Exact pixel positions */
transform-origin: 0 50%;   /* Left edge, vertical center */

The Centering Trick

One of the most practical applications of translate is centering an element of unknown size. When using position: absolute, the standard approach is to set top: 50%; left: 50%, which moves the element's top-left corner to the center. The trick is then to translate it back by half its own dimensions:

LISTING 14.16 — Centering with position absolute and translate

.container {
  position: relative;
}

.centered-element {
  position: absolute;
  top: 50%;
  left: 50%;
  /* Move back by 50% of the element's own width and height */
  transform: translate(-50%, -50%);
}

/* Modern equivalent using newer CSS (when you control the parent): */
.container {
  display: grid;
  place-items: center;
}

Modern Individual Transform Properties

A newer approach (supported in all current browsers since 2022) allows you to set translate, rotate, and scale as their own CSS properties, separate from the transform property. This is significant for animation because each can be transitioned or animated independently without interfering with the others.

LISTING 14.17 — Individual transform properties (modern syntax)

/* Old way: all in one transform */
.card:hover {
  transform: translateY(-8px) scale(1.02);
}

/* New way: each is its own property */
.card {
  translate: 0 0;
  scale: 1;
  /* Can transition each independently */
  transition: translate 0.2s ease-out, scale 0.15s ease-out;
}

.card:hover {
  translate: 0 -8px;
  scale: 1.02;
}

The individual properties are applied in a fixed order — translate, then rotate, then scale — regardless of the order in your CSS. This makes them more predictable than the combined transform property, where order matters.

3D Transforms

CSS 3D transforms add a third axis (Z) that points toward and away from the viewer, creating the illusion of depth. Elements can be rotated in three-dimensional space, moved toward or away from the viewer, and stacked to create objects with a front and a back.

Establishing Depth with perspective

Before 3D transforms become visible, you need to establish a perspective — a distance from the viewer to the "screen" that determines how strong the 3D effect appears. This is set on the parent element of the transformed elements, not on the element itself.

LISTING 14.18 — Setting perspective on the parent element

.scene {
  perspective: 800px;   /* Distance from viewer to the z=0 plane */
}

.card {
  transform: rotateY(30deg);   /* Now has visible 3D depth */
}

A smaller perspective value (like 200px) creates a dramatic, fish-eye 3D effect. A larger value (like 1200px) creates a subtle, gentle perspective. Values around 600px–1000px are typical for UI effects.

[IMG - Three perspective values compared: 200px, 600px, 1200px]

3D Rotation Functions

rotateX(angle) tilts an element forward and backward (like nodding your head). rotateY(angle) spins it left and right (like shaking your head). rotateZ(angle) is the same as the 2D rotate() function.

LISTING 14.19 — 3D rotation examples

/* Tilt the top toward the viewer */
transform: rotateX(45deg);

/* Spin halfway around the vertical axis (shows the back) */
transform: rotateY(180deg);

/* Combine: a corner curl effect */
transform: rotateX(10deg) rotateY(-15deg);

/* Moving toward/away from viewer on the Z axis */
transform: translateZ(100px);    /* Move 100px closer to viewer */
transform: translateZ(-100px);   /* Move 100px further away */

Preserving 3D Space: transform-style: preserve-3d

When you apply 3D transforms to a parent element that contains children, the children are "flattened" into the parent's 2D plane by default. Setting transform-style: preserve-3d on the parent tells the browser to maintain the 3D context so that children can have their own 3D positions relative to the parent.

This is essential for building objects with a front face and a back face.

Hiding the Back with backface-visibility

When you rotate an element 180 degrees, you're looking at its back face — a mirror image of the front. In most cases, you want to hide this. Setting backface-visibility: hidden makes an element invisible when its back is facing the viewer.

The combination of preserve-3d on a container, two child elements (front and back), backface-visibility: hidden on both, and a rotateY(180deg) on the back creates the classic CSS card flip.

LISTING 14.20 — A 3D card flip with preserve-3d and backface-visibility

/* The scene provides perspective */
.scene {
  perspective: 800px;
  width: 300px;
  height: 200px;
}

/* The card is the 3D container — it flips */
.card {
  width: 100%;
  height: 100%;
  position: relative;
  transform-style: preserve-3d;         /* Keep children in 3D space */
  transition: transform 0.6s ease-in-out;
}

.scene:hover .card {
  transform: rotateY(180deg);           /* Flip on hover */
}

/* Both faces absolutely fill the card */
.card__face {
  position: absolute;
  width: 100%;
  height: 100%;
  backface-visibility: hidden;          /* Hide when facing away */
}

/* Back face starts rotated 180 degrees */
.card__face--back {
  transform: rotateY(180deg);
}

Putting It All Together: Transitions and Transforms in Practice

Transitions and transforms are most powerful when combined. A transform defines what happens visually; a transition defines how smoothly it happens. Here are the patterns you'll reach for most often in real projects.

Button and Interactive Element States

A subtle lift effect — combining a small upward translate with a box-shadow change — communicates that a button is interactive and responsive without being distracting:

LISTING 14.21 — Button lift effect using transform and transition

.btn {
  transform: translateY(0);
  box-shadow: 0 2px 6px rgba(0, 0, 0, 0.12);
  transition: transform 0.15s ease-out, box-shadow 0.15s ease-out;
}

.btn:hover {
  transform: translateY(-3px);
  box-shadow: 0 8px 20px rgba(0, 0, 0, 0.18);
}

.btn:active {
  transform: translateY(-1px);
  box-shadow: 0 4px 10px rgba(0, 0, 0, 0.15);
}

Image Zoom on Hover

Scaling an image inside a clipped container creates a smooth zoom effect without resizing the container itself:

LISTING 14.22 — Image zoom within a fixed container

.image-wrapper {
  overflow: hidden;      /* Clip the image when it scales beyond its container */
  border-radius: 8px;
}

.image-wrapper img {
  width: 100%;
  display: block;
  transform: scale(1);
  transition: transform 0.4s ease-out;
}

.image-wrapper:hover img {
  transform: scale(1.08);
}

Fade-In on Appear

A common pattern for elements that are revealed (via a class toggled by JavaScript) is to transition opacity and a small translate together:

LISTING 14.23 — Fade and slide in when a class is added

.panel {
  opacity: 0;
  transform: translateY(12px);
  transition: opacity 0.3s ease, transform 0.3s ease;
}

.panel.is-visible {
  opacity: 1;
  transform: translateY(0);
}

/* Respect reduced motion preference */
@media (prefers-reduced-motion: reduce) {
  .panel {
    transition: none;
  }
}

Summary

In this lesson, you learned two of the most-used CSS features in modern web design: transitions and transforms. Here are the key takeaways:

  • CSS transitions smoothly animate property changes. Four sub-properties control the transition: transition-property (what to animate), transition-duration (how long), transition-timing-function (the pace), and transition-delay (when to start). The transition shorthand combines all four.
  • Timing functions determine the character of an animation. ease-out is the most natural for hover effects. cubic-bezier() lets you create custom curves including springy overshoots. The newer linear() function enables spring physics without keyframes.
  • Only properties with numeric or color values can be smoothly transitioned. The modern @starting-style and transition-behavior: allow-discrete solve the longstanding problem of animating elements in and out of display: none.
  • transform and opacity are the highest-performance animatable properties because they are handled by the GPU compositor, bypassing layout and paint entirely. Use them whenever possible instead of animating top, left, width, or height.
  • Always include a prefers-reduced-motion media query to disable or reduce animations for users who have requested that preference in their operating system.
  • 2D transforms — translate, rotate, scale, skew — reposition and reshape elements visually without affecting document flow. When combining multiple transforms, order matters: transforms are applied right to left. Modern individual transform properties (translate, rotate, scale) allow independent animation of each.
  • 3D transforms require a perspective value on the parent element to establish depth. transform-style: preserve-3d maintains a 3D context for children. backface-visibility: hidden hides an element when its back face is visible — essential for card flip effects.
  • The most common practical patterns are: button lift (translateY + box-shadow transition), image zoom (scale inside overflow:hidden), and fade/slide in (opacity + translateY).

Q & A

Q: What is the difference between animating left: 100px and transform: translateX(100px)? They look the same on screen.

A: They look identical, but the browser's work is very different. Animating left requires the browser to recalculate layout on every frame — it has to figure out whether any other elements need to move as a result. transform: translateX() is handled entirely by the GPU compositor, which just moves a pre-composited layer. On a simple page the difference may not be perceptible. On a complex page with many elements, the difference between smooth 60fps motion and noticeable jank can come down to this choice.

Q: Should I always use transition: all to keep my CSS simple?

A: Avoid transition: all in production code. It means the browser monitors every property on that element for changes. If a property changes that you didn't intend to animate — like a color calculated from a CSS variable during a theme switch — it will animate unnecessarily. It can also cause unexpected transitions when browser state changes affect computed values. Name your properties explicitly: transition: transform 0.2s ease-out, opacity 0.2s ease-out.

Q: The card flip looks odd in some browsers — the front face shows through the back. What's wrong?

A: This is almost always a missing transform-style: preserve-3d on the card container, or a missing backface-visibility: hidden on the faces. Make sure both faces have backface-visibility: hidden, the parent card element has transform-style: preserve-3d, and the back face starts with transform: rotateY(180deg). Also check that no parent element has a CSS property that forces 2D flattening — properties like overflow: hidden, clip-path, filter, or mix-blend-mode on an ancestor can break preserve-3d.

Q: I want an animation to play when the page loads, not just on hover. Can CSS transitions do that?

A: CSS transitions only react to property changes — they need a before-and-after state. For page-load animations, you have two options. The classic approach is JavaScript: set the initial state in CSS, then use a short setTimeout to add a class after the page loads, triggering the transition. The modern approach is @starting-style: define the element's starting values in a @starting-style block, and the browser will automatically transition from those values to the element's normal values the first time it renders. @starting-style is supported in all current browsers as of 2024.

Q: How do I make a transition only run in one direction — animate on hover, but snap back instantly when hover ends?

A: Put the transition on the :hover rule instead of the base element rule. When the element is hovered, the :hover styles apply, including the transition. When the hover ends, the :hover styles are removed and the transition is gone too, so the return to the base state happens instantly. Use this intentionally and sparingly — most of the time the two-way transition feels more polished.
