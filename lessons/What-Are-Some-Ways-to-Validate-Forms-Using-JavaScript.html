The Constraint Validation API

Certain HTML elements, such as the textarea and input elements, expose a Constraint Validation API.

This API allows you to assert that the user's provided value for that element passes any HTML-level validation you have written, such as minimum length or pattern matching.

The Problem: Company Email Validation

Real-World Scenario

Let's say you wanted employees at a company to send feedback messages through a form.

Basic Email Input

We use the email input which comes with built-in validation to check for basic validation like if the input includes the at (@) sign.

‚ö†Ô∏è The Problem

But what if the user provides an email address like example@email.com?

This would pass the basic validation, but we want to be more specific about accepting emails from those with a company email address.

Basic Form (Without Pattern)

<form> <label>Enter your email: </label> <input required type="email" /> <label>Enter your feedback: </label> <textarea required placeholder="Your feedback here..."></textarea> <button type="submit">Submit Feedback</button> </form>

This validates that it's an email format, but accepts any email address including example@email.com.

Solution: The pattern Attribute

More Specific Validation

This is where we can use the pattern attribute to specify that the email address must end in a company email address.

Updated Form (With Pattern)

<form> <label>Enter your email: </label> <input required placeholder="username@sampleCompany.com" type="email" pattern="[a-z0-9._%+-]+@sampleCompany\.com$" /> <label>Enter your feedback: </label> <textarea required placeholder="Your feedback here..."></textarea> <button type="submit">Submit Feedback</button> </form>

What the Pattern Does

The regex pattern [a-z0-9._%+-]+@sampleCompany\.com$ ensures:

‚Ä¢ Email must end with @sampleCompany.com

‚Ä¢ Only accepts company email addresses

‚Ä¢ Rejects emails like user@gmail.com

‚ö†Ô∏è Still a Problem

Now, if you try to submit the feedback, you will see a message saying "Please match the requested format."

Even though the input does have placeholder text showing them the desired format, it would be better to also include a customized error message using JavaScript.

The checkValidity() Method

Part of the Constraint Validation API

Let's first take a look at the checkValidity() method.

element.checkValidity()

What It Does

The checkValidity() method returns true if the element matches all HTML validation (based on its attributes), and false if it fails.

Example Code

const input = document.querySelector("input[type='email']"); input.addEventListener("input", (e) => { console.log(e.target.checkValidity()); });

Understanding the Code

‚Ä¢ We've queried our input from the DOM

‚Ä¢ Added an input event listener

‚Ä¢ e.target refers to the element that triggered the event (our input)

‚Ä¢ checkValidity() checks if input meets all validation rules

Result

When we try with an invalid input, we see false gets logged in the console.

Now that we know the input is invalid, let's report the invalidity!

The reportValidity() Method

Showing the Error to Users

Now let's report the invalidity to the user.

element.reportValidity()

Updated Code

const input = document.querySelector("input[type='email']"); input.addEventListener("input", (e) => { const isValid = e.target.checkValidity(); if (!isValid) { e.target.reportValidity(); } });

What reportValidity() Does

‚Ä¢ Shows the browser's built-in error message

‚Ä¢ Reports the invalid state immediately

‚Ä¢ No need to wait for form submission

‚ö†Ô∏è Still Using Default Message

As a result, you will see the browser's error message: "Please match the requested format."

It reports the invalid state immediately, instead of waiting for us to submit the form. But it's still using the default message.

This is because the reportValidity method only tells the browser that the input is invalid. The browser still chooses how to display why it's invalid.

That's where the setCustomValidity method comes in.

The setCustomValidity() Method

Custom Error Messages

This method accepts a custom error message, which is displayed to the user.

element.setCustomValidity(message)

Complete Solution

const input = document.querySelector("input[type='email']"); input.addEventListener("input", (e) => { const isValid = e.target.checkValidity(); if (!isValid) { e.target.setCustomValidity( "You must use a company email address that ends in @sampleCompany.com" ); e.target.reportValidity(); } else { // Clear the custom message when valid e.target.setCustomValidity(""); } });

‚úì Result

As a result, you will see the custom error message:

"You must use a company email address that ends in @sampleCompany.com"

‚ö†Ô∏è Important!

Always clear the custom validity message when the input becomes valid by calling:

element.setCustomValidity("")

Otherwise, the form will remain invalid even with valid input!

The validity Property

Understanding Why Validation Failed

If you are interested in exploring more about the different types of validity states and why a particular validation has failed, you can log out the validity property.

Accessing the validity Property

const input = document.querySelector("input[type='email']"); input.addEventListener("input", (e) => { console.log(e.target.validity); });

ValidityState Object

The validity property is an instance of the ValidityState object.

Here is an example of what the object might look like in the browser:

ValidityState {
badInput: false,
customError: false,
patternMismatch: true,
rangeOverflow: false,
rangeUnderflow: false,
stepMismatch: false,
tooLong: false,
tooShort: false,
typeMismatch: false,
valueMissing: false,
valid: false
}

ValidityState Properties

All Boolean Properties

There are several helpful properties which all hold the value of a boolean of true or false.

Common Properties

valueMissing: true when a required input field is left empty

patternMismatch: true if the value doesn't match the specified regex pattern

typeMismatch: true if value doesn't match input type (e.g., not a valid email)

tooShort: true if value is shorter than minlength

tooLong: true if value is longer than maxlength

More Properties

rangeUnderflow: true if value < min

rangeOverflow: true if value > max

stepMismatch: true if value doesn't fit step attribute

badInput: true if browser can't convert input

customError: true if custom error set via setCustomValidity()

valid: true if all validations pass

Complete Working Example

HTML

<form id="feedbackForm"> <label>Enter your email: </label> <input id="email" type="email" required placeholder="username@sampleCompany.com" pattern="[a-z0-9._%+-]+@sampleCompany\.com$" /> <label>Enter your feedback: </label> <textarea id="feedback" required></textarea> <button type="submit">Submit</button> </form>

JavaScript

const emailInput = document.getElementById("email"); const feedbackInput = document.getElementById("feedback"); // Email validation emailInput.addEventListener("input", (e) => { if (!e.target.checkValidity()) { e.target.setCustomValidity( "You must use a company email that ends in @sampleCompany.com" ); } else { e.target.setCustomValidity(""); } }); // Form submission document.getElementById("feedbackForm").addEventListener("submit", (e) => { e.preventDefault(); console.log("Form submitted successfully!"); });

Key Takeaways

The Constraint Validation API

Provides powerful JavaScript methods to control form validation with custom error messages and real-time feedback.

‚úì Three Key Methods

checkValidity(): Returns true/false if element is valid

reportValidity(): Shows browser's error message

setCustomValidity(): Sets custom error message

ValidityState Properties

valueMissing: Required field empty

patternMismatch: Doesn't match regex

typeMismatch: Wrong type

tooShort/tooLong: Length issues

valid: All validations pass

‚ö†Ô∏è Remember

Always clear custom validity with setCustomValidity("") when input becomes valid!

üí° Best Practices

‚úì Use pattern attribute for regex validation

‚úì Provide clear, helpful error messages

‚úì Give real-time feedback on input events

‚úì Check validity state to understand why validation failed

‚úì Combine HTML and JavaScript validation for best results
