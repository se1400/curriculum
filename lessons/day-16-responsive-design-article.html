Responsive Web Design

What You'll Learn in This Lesson

  • Why mobile-first design is the correct starting approach
  • How the viewport meta tag controls layout on mobile devices
  • How to write CSS @media queries using min-width
  • How to create fluid typography and spacing with clamp()
  • How to make images responsive with srcset, sizes, and <picture>
  • How to build intrinsically responsive grids without media queries
  • How to use container queries to respond to a component's parent size

When Tim Berners-Lee published the first website in 1991, there was exactly one screen size: the desktop monitor. In 2010, Ethan Marcotte coined the term responsive web design in an article that would change the industry. Today, the web is accessed on phones, tablets, laptops, desktops, televisions, watches, and e-readers. According to Statcounter, mobile devices account for more than half of all global web traffic. Designing only for one screen size is no longer an option — it is a failure.

Responsive web design is the practice of building a single HTML document that adapts its layout and appearance to fit any screen size. It is achieved through three foundational techniques: a fluid grid, flexible images, and CSS media queries. Modern CSS has added powerful new tools to this foundation, including clamp(), intrinsic sizing, and container queries.

The Problem with Fixed-Width Layouts

In the early 2000s, most websites were built with a fixed width — typically 800px or 960px — because almost all desktop monitors were that wide. When smartphones arrived, these fixed-width sites simply overflowed the screen. Users had to pinch-zoom and pan horizontally to read content — a terrible experience.

The first "solution" was to build two entirely separate websites: one for desktop (www.example.com) and one for mobile (m.example.com). Developers had to maintain two codebases, two sets of content, and detect user agents to redirect users. This approach was fragile and expensive.

Responsive design solved this by building one flexible site that works everywhere.

Mobile-First Design

Mobile-first is not just a technical pattern — it is a design philosophy. It means you design and code for the smallest screen first, then progressively add layout complexity for larger screens. This approach produces better results for three reasons:

  • Content priority. When screen space is limited, you are forced to identify what matters most and present it clearly. The most important content comes first.
  • Performance. Mobile devices are often on slower networks. By starting with a minimal, fast baseline and adding enhancements for faster connections and larger screens, you serve the most constrained users best.
  • CSS cascade efficiency. Adding styles for larger screens with min-width media queries is simpler than stripping styles back with max-width. Smaller baseline means less CSS to override.

NOTE: Mobile-first does not mean you ignore desktop. It means you build the foundation for mobile, then enhance for larger screens. The final result must work beautifully on all sizes.

The Viewport Meta Tag

Before writing a single line of responsive CSS, you must include one essential HTML tag in your <head>:

LISTING 16.1 — The Viewport Meta Tag

<meta name="viewport" content="width=device-width, initial-scale=1">

Without this tag, mobile browsers apply a default virtual viewport — typically 980px wide — and then zoom out to fit that wide viewport onto the narrow physical screen. Your perfectly crafted CSS media queries will appear to fire too late or not at all, because the browser thinks the viewport is 980px when it is actually 375px.

This tag instructs the browser to:

  • width=device-width — set the viewport width to the actual screen width of the device
  • initial-scale=1 — start at 1:1 zoom, with no scaling applied

[IMG - Side-by-side page without viewport meta tag versus with it on mobile]

CAUTION: You may see older examples that include maximum-scale=1, user-scalable=no. Never use these. Preventing users from zooming is an accessibility barrier — it makes content impossible to read for users with low vision. WCAG 1.4.4 requires that text can be resized up to 200% without loss of functionality.

CSS Media Queries

A media query is a conditional block of CSS: if a specified condition is true, the styles inside apply. The condition most used in responsive design is the viewport width.

Basic Syntax

LISTING 16.2 — Basic Media Query Syntax

/* Styles in here apply to ALL screen sizes (the default) */
.container {
  padding: 1rem;
}

/* Styles in here apply ONLY when viewport is 640px or wider */
@media (min-width: 640px) {
  .container {
    padding: 2rem;
  }
}

/* Styles in here apply ONLY when viewport is 1024px or wider */
@media (min-width: 1024px) {
  .container {
    padding: 3rem;
    max-width: 1200px;
    margin-inline: auto;
  }
}

min-width vs. max-width

There are two ways to write media queries: min-width (mobile-first) and max-width (desktop-first). They are opposites of each other.

  • min-width — "apply these styles when the screen is at least this wide." Start with mobile styles, add desktop styles upward.
  • max-width — "apply these styles when the screen is at most this wide." Start with desktop styles, add mobile styles downward.

Mobile-first using min-width is the correct modern approach. It produces cleaner CSS because the simpler default styles cascade naturally to small screens, and complexity is added progressively for larger ones.

LISTING 16.3 — Mobile-First Layout (min-width)

/* Mobile: single column by default */
.card-grid {
  display: grid;
  grid-template-columns: 1fr;
  gap: 1rem;
}

/* Tablet: 2 columns */
@media (min-width: 640px) {
  .card-grid {
    grid-template-columns: repeat(2, 1fr);
  }
}

/* Desktop: 3 columns */
@media (min-width: 1024px) {
  .card-grid {
    grid-template-columns: repeat(3, 1fr);
  }
}

Combining Conditions

You can combine multiple conditions with and, or (written as a comma), and not. The modern Media Queries Level 4 specification also supports the and, or, and not keywords without requiring the @media type prefix:

LISTING 16.4 — Combining Media Query Conditions

/* Applies between 640px and 1023px */
@media (min-width: 640px) and (max-width: 1023px) {
  .sidebar { display: none; }
}

/* Applies to screen or print */
@media screen, print {
  .page { max-width: 1200px; }
}

/* Modern Level 4 range syntax */
@media (640px <= width < 1024px) {
  .sidebar { display: none; }
}

Choosing Breakpoints

A breakpoint is the specific width at which your layout changes. Beginners often choose breakpoints based on specific device sizes — "375px for iPhone, 768px for iPad." This is a mistake. Device sizes change with every product cycle, and no set of device-specific breakpoints will ever cover all devices.

Instead, choose breakpoints based on your content: add a breakpoint when your design starts to look awkward or uncomfortable at a given width. Resize your browser slowly from narrow to wide, watching for the point where a layout change improves readability or usability. That is your breakpoint.

That said, a set of commonly used breakpoints provides a useful starting framework:

  • Small (default) — below 640px — single column, stacked navigation
  • Medium — 640px and up — two-column layouts, horizontal navigation
  • Large — 1024px and up — three or more columns, sidebar layouts
  • Extra large — 1280px and up — wide content, larger type, more generous whitespace

Responsive Typography with clamp()

Traditional responsive typography requires multiple media queries to set different font sizes at each breakpoint. The result is fonts that jump in size at breakpoints rather than scaling fluidly. The clamp() function solves this elegantly.

clamp(minimum, preferred, maximum) returns the preferred value as long as it falls between the minimum and maximum. By using a viewport-relative unit (vw) as the preferred value, the font size scales proportionally with the viewport width — with hard floors and ceilings to prevent extremes.

LISTING 16.5 — Fluid Typography with clamp()

/* Font size scales from 1rem (16px) to 2rem (32px)
   proportionally as viewport grows from 320px to 1280px */
h1 {
  font-size: clamp(1.5rem, 4vw + 1rem, 3rem);
}

h2 {
  font-size: clamp(1.25rem, 3vw + 0.5rem, 2rem);
}

p {
  font-size: clamp(1rem, 1.5vw + 0.5rem, 1.25rem);
  line-height: clamp(1.5, 1.5 + 0.5vw, 1.8);
}

The formula 4vw + 1rem means "4% of viewport width plus a fixed 1rem offset." At small sizes the clamp minimum prevents the text from becoming too small to read; at large sizes the maximum prevents it from becoming grotesquely large.

LISTING 16.6 — Fluid Spacing with clamp()

/* Padding scales from 1rem on mobile to 3rem on wide screens */
.section {
  padding-block: clamp(1rem, 5vw, 3rem);
  padding-inline: clamp(1rem, 8vw, 6rem);
}

/* A consistent fluid gap for grids and stacks */
:root {
  --space-fluid: clamp(1rem, 3vw, 2rem);
}

.card-grid { gap: var(--space-fluid); }

Responsive Images

Images are the heaviest assets on most web pages. A poorly configured image setup causes mobile users to download a 4000px wide image just to display it at 375px — wasting bandwidth and battery life.

The Baseline: max-width: 100%

At a minimum, every <img> should have this CSS rule applied:

img {
  max-width: 100%;
  height: auto;
}

This prevents images from overflowing their containers and maintains the original aspect ratio when width is constrained. Add it to every project as a global reset rule.

object-fit for Image Containers

When you need an image to fill a fixed-size container — a card thumbnail, an avatar, a hero slot — the object-fit property controls how the image is sized and positioned within it:

LISTING 16.7 — object-fit for Consistent Thumbnails

.card-thumb {
  width: 100%;
  height: 200px;
  object-fit: cover;    /* fills the area, crops if needed */
  object-position: top; /* focus on the top of the image */
}

/* Other values: */
/* object-fit: contain  — fits entirely inside, may letterbox */
/* object-fit: fill     — stretches to fill exactly (distorts) */
/* object-fit: none     — displays at natural size, no scaling */

srcset and sizes

The HTML srcset attribute lists multiple versions of the same image at different resolutions. The browser chooses which file to download based on the device's screen density and the current display width of the image. You save bandwidth — a 1× screen does not download the 3× retina file.

LISTING 16.8 — srcset for Density Switching

<img
  src="hero-800.jpg"
  srcset="hero-400.jpg 400w,
          hero-800.jpg 800w,
          hero-1600.jpg 1600w"
  sizes="(max-width: 640px) 100vw,
         (max-width: 1024px) 50vw,
         800px"
  alt="A panoramic mountain landscape"
  width="800"
  height="450"
  loading="lazy"
>

The sizes attribute tells the browser how wide the image will be displayed at each breakpoint. The browser uses this hint — combined with the device's pixel density — to pick the most appropriate file from srcset.

The <picture> Element

The <picture> element goes further: it lets you swap out a completely different image file (not just a different resolution of the same image) based on media conditions. This technique — called art direction — is used when you want a tightly cropped portrait version on mobile and a wide landscape version on desktop.

LISTING 16.9 — Art Direction with <picture>

<picture>
  <!-- Use WebP on browsers that support it -->
  <source
    type="image/webp"
    srcset="hero-wide.webp 1200w, hero-narrow.webp 600w"
    sizes="(max-width: 640px) 100vw, 1200px"
  >
  <!-- Tall, cropped version on small screens -->
  <source
    media="(max-width: 640px)"
    srcset="hero-portrait.jpg"
  >
  <!-- Wide landscape version on larger screens -->
  <img
    src="hero-wide.jpg"
    alt="A mountain panorama at sunrise"
    width="1200"
    height="630"
    loading="lazy"
  >
</picture>

The <img> inside <picture> is required and serves as both the fallback for browsers that do not support <picture> and the element that carries the alt text, width, and height attributes.

Responsive Layout Patterns

Stack-to-Row with Flexbox

The most common responsive layout pattern: items stack vertically on small screens and line up horizontally on wider screens. Flexbox makes this trivial.

LISTING 16.10 — Stack-to-Row Navigation

.nav {
  display: flex;
  flex-direction: column;  /* stacked on mobile */
  gap: 0.5rem;
}

@media (min-width: 640px) {
  .nav {
    flex-direction: row;   /* horizontal on tablet+ */
    gap: 1.5rem;
  }
}

Intrinsically Responsive Grid (No Media Queries)

CSS Grid's auto-fill with minmax() creates a truly responsive layout that adds and removes columns based on available space — with no media queries at all. This technique, sometimes called the "RAM pattern" (Repeat, Auto, Minmax), is one of the most useful tools in modern CSS.

LISTING 16.11 — Intrinsic Grid with auto-fill

.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(280px, 1fr));
  gap: 1.5rem;
}
/*
  This grid:
  - Creates as many 280px columns as will fit in the container
  - Stretches all columns to equal widths with 1fr
  - Automatically adds/removes columns as the container resizes
  - No media queries needed!
*/

The difference between auto-fill and auto-fit:

  • auto-fill — creates columns to fill the row width, even if some are empty (useful for consistent column widths)
  • auto-fit — collapses empty columns, allowing filled columns to expand further (useful when you have fewer items than columns)

The aspect-ratio Property

Responsive media — especially video embeds — has always required a CSS hack: a wrapper element with a calculated padding-bottom percentage. The aspect-ratio property replaces that workaround cleanly.

LISTING 16.12 — Responsive Video Embed with aspect-ratio

/* Old padding-bottom hack (no longer needed) */
.video-wrapper-old {
  position: relative;
  padding-bottom: 56.25%; /* 16:9 */
  height: 0;
}
.video-wrapper-old iframe {
  position: absolute;
  inset: 0;
  width: 100%;
  height: 100%;
}

/* Modern: aspect-ratio */
.video-wrapper {
  width: 100%;
  aspect-ratio: 16 / 9;
}

.video-wrapper iframe {
  width: 100%;
  height: 100%;
}

/* Card with consistent image thumbnail */
.card-image {
  width: 100%;
  aspect-ratio: 4 / 3;
  object-fit: cover;
}

New Dynamic Viewport Units

You already know vw (viewport width) and vh (viewport height). But vh has a long-standing problem on mobile browsers: the address bar and tab bar take up part of the screen, and they appear and disappear as the user scrolls. A 100vh element may be taller than the visual viewport, causing content to be cut off behind the browser chrome.

CSS now provides three families of dynamic viewport units to address this:

  • svh / svw — Small Viewport — the smallest possible viewport (browser bars fully visible). Safe to use when you do not want content hidden behind browser chrome.
  • lvh / lvw — Large Viewport — the largest possible viewport (browser bars fully retracted). Used when you want to fill the full screen after the browser chrome hides.
  • dvh / dvw — Dynamic Viewport — updates in real time as browser chrome appears or disappears. Best for full-screen layouts that should always fill the visible area.

LISTING 16.13 — Dynamic Viewport Units

/* Old approach — can be hidden behind mobile address bar */
.hero { min-height: 100vh; }

/* Better: use dvh for full-screen sections */
.hero { min-height: 100dvh; }

/* Safe: svh guarantees nothing is cut off */
.modal-overlay { height: 100svh; }

/* With a fallback for older browsers */
.hero {
  min-height: 100vh;       /* fallback */
  min-height: 100dvh;      /* overrides if supported */
}

Container Queries

Media queries respond to the viewport size. But components — cards, sidebars, widgets — are often placed in different contexts with different available widths. A card component in a three-column grid has less space than the same card used as a single wide block. With media queries alone, you cannot write CSS that says "make this card compact when its container is narrow."

Container queries solve this. You declare a container on a parent element, then write queries that respond to the container's size rather than the viewport.

LISTING 16.14 — Container Queries Syntax

/* Step 1: Declare a containment context on the parent */
.card-wrapper {
  container-type: inline-size;
  /* or: container-type: size (tracks both inline and block dimensions) */
}

/* Optional: give the container a name for nested queries */
.sidebar {
  container-type: inline-size;
  container-name: sidebar;
}

/* Step 2: Query the container width (not the viewport) */
.card {
  display: flex;
  flex-direction: column;
}

@container (min-width: 400px) {
  .card {
    flex-direction: row;    /* side-by-side when the container is wide */
  }
}

/* Query a named container */
@container sidebar (min-width: 300px) {
  .sidebar-widget { font-size: 1.1rem; }
}

Container queries make truly portable, context-aware components possible. A single card component can automatically adapt whether it is in a narrow sidebar, a two-column grid, or a full-width hero area — all without any changes to the HTML or JavaScript.

Responsive Navigation Patterns

Navigation presents a unique responsive challenge: a horizontal navbar on desktop must collapse into something compact on mobile. There are several patterns, each with tradeoffs.

The Hamburger Menu

The most common pattern: a ☰ icon button on small screens that toggles navigation visibility. It requires a small amount of JavaScript to toggle a class that shows and hides the menu.

LISTING 16.15 — Hamburger Menu Pattern

<header>
  <a href="/" class="logo">Brand</a>
  <button class="menu-toggle" aria-expanded="false" aria-controls="nav-menu"
          aria-label="Toggle navigation">
    <span></span><span></span><span></span>
  </button>
  <nav id="nav-menu" class="nav-menu" hidden>
    <a href="/">Home</a>
    <a href="/about">About</a>
    <a href="/work">Work</a>
    <a href="/contact">Contact</a>
  </nav>
</header>

/* Mobile: hide menu by default */
.nav-menu[hidden] { display: none; }

/* Show when toggled */
.nav-menu.is-open {
  display: flex;
  flex-direction: column;
}

/* Desktop: always show */
@media (min-width: 768px) {
  .menu-toggle { display: none; }
  .nav-menu[hidden] { display: flex !important; }
  .nav-menu { flex-direction: row; gap: 1.5rem; }
}
// JavaScript to toggle the menu
const toggle = document.querySelector('.menu-toggle');
const menu   = document.querySelector('.nav-menu');

toggle.addEventListener('click', () => {
  const expanded = toggle.getAttribute('aria-expanded') === 'true';
  toggle.setAttribute('aria-expanded', String(!expanded));
  menu.hidden = expanded;
  menu.classList.toggle('is-open', !expanded);
});

Testing Responsive Designs

Browser developer tools include a built-in device emulation mode that lets you test responsive designs without a physical device. In Chrome and Edge, open DevTools (F12), then click the device toolbar icon (or press Ctrl+Shift+M / Cmd+Shift+M). You can:

  • Select preset device sizes (iPhone, Pixel, iPad, Galaxy) or enter custom dimensions
  • Simulate different device pixel ratios (1×, 2×, 3×)
  • Throttle the network to simulate a slow mobile connection
  • Drag the viewport edge to resize interactively and watch breakpoints fire

NOTE: Emulation in DevTools is a useful starting point, but it is not a substitute for testing on real devices. Touch interactions, font rendering, scroll behavior, and browser chrome all behave differently on physical hardware. Test on at least one real iOS and one real Android device before launching.

Summary

In this lesson you learned the philosophy and techniques of responsive web design. You began with the mobile-first approach — starting with minimal styles for small screens and progressively enhancing for larger ones. You learned that the viewport meta tag is non-negotiable for mobile layouts to work correctly, and that @media (min-width) queries are the standard way to apply layout changes at different screen sizes.

You explored clamp() for fluid typography and spacing that scales smoothly between a minimum and maximum without media query jumps. You learned how to serve appropriately sized images using srcset, sizes, and the <picture> element for art direction. You saw how CSS Grid's auto-fill with minmax() creates intrinsically responsive layouts without any media queries at all.

Finally, you learned about two modern CSS features: the new dynamic viewport units (dvh, svh, lvh) that solve the mobile browser chrome problem, and container queries (@container), which make components responsive to their parent's width rather than the viewport — enabling truly portable, context-aware components for the first time.

Q&A

Q. What is the difference between responsive design and adaptive design?

A. Responsive design uses a fluid, flexible layout that adjusts continuously to any screen size, primarily through CSS. Adaptive design uses multiple fixed layouts, each designed for a specific breakpoint; the server or client detects the device and serves the appropriate layout. Responsive design is the industry standard today because it handles the full spectrum of device sizes with one codebase.

Q. Should I use px or rem for my breakpoints?

A. There is ongoing debate, but many accessibility experts recommend rem for media queries. If a user has set their browser base font size larger than 16px (a common accessibility setting), px-based media queries fire at the same viewport width regardless — potentially breaking the layout for that user. rem-based breakpoints scale with the user's font size preference. However, px breakpoints are simpler and are used in most popular frameworks. The practical impact is small for most sites.

Q. What is the difference between auto-fill and auto-fit in CSS Grid?

A. Both create as many columns as will fit in the available space. The difference appears when there are fewer items than possible columns. auto-fill maintains the column tracks even if they are empty, keeping item widths consistent. auto-fit collapses empty tracks and stretches the filled items to take up all available space. For most card layouts, auto-fill is preferable because it produces more predictable column widths.

Q. When should I use container queries instead of media queries?

A. Use container queries for components that need to adapt to their context, not the viewport. For example, a card component used in both a narrow sidebar and a wide main column should use container queries so it can adapt to the available container width. Use media queries for page-level layout changes — things like shifting from a single column to a two-column layout, or showing/hiding the mobile navigation.

Q. Is it okay to use display: none to hide content on mobile?

A. From a CSS perspective, yes — but be careful. display: none removes the element from the rendering tree, so it does not consume space and is not visible. However, the HTML is still downloaded by the browser, and screen readers may or may not ignore it depending on context. If you need to hide content from all users (including screen readers) at a breakpoint, display: none combined with aria-hidden="true" or the hidden attribute is appropriate. Never hide content that is the only way to access important information — provide an alternative.
