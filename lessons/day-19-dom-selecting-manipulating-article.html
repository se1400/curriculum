So far in this curriculum, you have written HTML to describe content and CSS to style it. JavaScript is the language that brings a web page to life — and the way JavaScript interacts with an HTML document is through a system called the Document Object Model, or DOM. Understanding the DOM is the single most important foundation you can build before writing any interactive JavaScript. Everything else — event handling, form validation, dynamic content, animations — depends on it.

This lesson is hands-on and concrete. By the end, you will know how to find any element on a page, read and change its content, modify its appearance, build new elements from scratch, and navigate the tree of elements that makes up a document. These are the fundamental operations every front-end developer performs every single day.

What the DOM Is

When a browser loads an HTML file, it does not simply display the text. It parses the markup and builds an in-memory representation of the entire document — a tree of objects, where each object corresponds to a piece of the HTML. This representation is the Document Object Model.

Think of the DOM as a living model of your web page. Every element, every text node, every attribute is an object in this model, and JavaScript can read and modify any of it. When you change something in the DOM, the browser immediately updates what the user sees. The DOM is the bridge between the inert text of an HTML file and the dynamic, interactive experience in the browser.

The DOM Tree

The DOM is structured as a tree — a hierarchy of nodes. At the very top sits the document object, which represents the entire HTML document. Below it is the root <html> element, which branches into <head> and <body>. Each of those branches further into child elements, which have their own children, all the way down to the individual text nodes that hold the actual words on the page.

[IMG - DOM tree diagram showing document to html to head body branches]

LISTING 19.1 — A Simple HTML Structure and Its DOM Tree

<!-- This HTML... -->
<body>
  <main>
    <h1>Hello, DOM</h1>
    <p class="intro">Welcome to the <strong>Document Object Model</strong>.</p>
  </main>
</body>

/* ...becomes this tree of objects:
   document
   └── html
       ├── head
       └── body
           └── main
               ├── h1
               │   └── [text: "Hello, DOM"]
               └── p.intro
                   ├── [text: "Welcome to the "]
                   ├── strong
                   │   └── [text: "Document Object Model"]
                   └── [text: "."]
*/

Notice that text inside an element is also a node — a text node. And attributes live on element nodes as properties. Everything in the HTML document is represented as some type of node in this tree.

The DOM Is “Live”

One of the most important things to understand about the DOM is that it is live. When JavaScript changes a DOM object — say, setting an element's text content or adding a CSS class — that change is immediately reflected in the browser. You do not reload the page. The rendering engine notices the change and repaints the affected part of the screen. This is what makes dynamic web applications possible.

NOTE: The DOM is distinct from the HTML source file. If JavaScript changes the DOM, those changes do not write back to the .html file on disk. The source file is unchanged; only the in-memory representation has been modified. Right-clicking and choosing “View Page Source” always shows the original HTML. The Elements panel in DevTools, on the other hand, shows the current live DOM.

Selecting Elements

Before you can do anything with a DOM element, you need a JavaScript reference to it. Selecting elements is the act of finding a specific element (or group of elements) in the DOM tree so that you have an object to work with. Modern JavaScript provides two methods that work with CSS selector syntax — the same selectors you already know from writing CSS.

document.querySelector()

document.querySelector(selector) searches the entire document and returns the first element that matches the given CSS selector. If nothing matches, it returns null. The argument is a string containing any valid CSS selector — a tag name, a class, an ID, a complex combinator, a pseudo-class, anything.

LISTING 19.2 — querySelector Examples

// Select by tag name (first <h1> on the page)
const heading = document.querySelector('h1');

// Select by class (first element with class "intro")
const intro = document.querySelector('.intro');

// Select by ID
const nav = document.querySelector('#main-nav');

// Select using a descendant combinator
const firstNavLink = document.querySelector('nav a');

// Select with an attribute selector
const emailInput = document.querySelector('input[type="email"]');

// Select with a pseudo-class
const secondItem = document.querySelector('li:nth-child(2)');

// Always check for null before using the result
const modal = document.querySelector('.modal');
if (modal) {
  // safe to work with modal
}

// querySelector returns the element object — use it directly on screen
const resultBox = document.querySelector('#qs-result');
if (resultBox) {
  resultBox.textContent =
    `heading found: ${heading ? heading.tagName : 'null'} | ` +
    `intro found: ${intro ? intro.tagName + '.' + intro.className : 'null'} | ` +
    `nav found: ${nav ? '#' + nav.id : 'null'}`;
}

The key detail is the return value: a single element object or null. If your selector could match many elements but you only need the first, querySelector is the right tool. When you need all matches, reach for its sibling method.

document.querySelectorAll()

document.querySelectorAll(selector) returns all matching elements as a NodeList — a list-like object similar to an array. Unlike querySelector, it never returns null; if nothing matches, you get an empty NodeList.

LISTING 19.3 — querySelectorAll Examples

// All paragraphs on the page
const allParagraphs = document.querySelectorAll('p');

// All elements with class "card"
const cards = document.querySelectorAll('.card');

// All inputs that are required
const requiredInputs = document.querySelectorAll('input[required]');

// Multiple selectors at once (comma-separated, just like CSS)
const headings = document.querySelectorAll('h1, h2, h3');

// Iterate with forEach — add a class to every paragraph
allParagraphs.forEach(p => {
  p.classList.add('highlighted');
});

// Or with a for...of loop
for (const card of cards) {
  card.classList.add('visible');
}

// cards.length tells you how many matched
const countDisplay = document.querySelector('#card-count');
if (countDisplay) {
  countDisplay.textContent = `Found ${cards.length} card(s)`;
}

NodeList vs. Array

A NodeList looks like an array — it has a length property and you can access items by index (nodeList[0]) — but it is not a true JavaScript array. It supports forEach(), which covers the vast majority of everyday tasks. For beginners, querySelectorAll() combined with forEach() covers 95% of use cases. You only need Array.from() or spread [...nodeList] when you want array methods like filter() or map() — and you will learn those soon in a dedicated lesson.

LISTING 19.4 — querySelectorAll with forEach

// querySelectorAll + forEach is the everyday pattern
const cards = document.querySelectorAll('.card');

cards.forEach(card => {
  card.classList.add('visible');
});

NOTE: The NodeList returned by querySelectorAll is static — it is a snapshot taken at the moment of the call. If elements are added or removed from the DOM afterward, the NodeList does not update. This is actually useful: you get a stable list to work with.

Legacy Selectors

You will encounter older JavaScript code that uses different selection methods. You do not need to use these, but you should recognize them:

  • document.getElementById('my-id') — selects a single element by its ID (no # prefix needed). Very fast. Returns the element or null.
  • document.getElementsByClassName('my-class') — returns a live HTMLCollection of elements with that class.
  • document.getElementsByTagName('p') — returns a live HTMLCollection of all elements with that tag name.

The critical difference between these older methods and querySelectorAll is that getElementsByClassName and getElementsByTagName return live HTMLCollections — they update automatically if the DOM changes. This can cause surprising bugs if you are iterating over the collection while modifying the DOM. For new code, querySelector and querySelectorAll are always preferred.

Reading and Setting Content

Once you have a reference to an element, you can read what is inside it or replace its contents entirely. JavaScript provides three properties for this, each with different behavior that matters in different situations.

element.textContent

textContent gives you the plain text inside an element — all the text, stripped of any HTML tags. It also includes the text of all descendant elements. You can both read from it and write to it.

LISTING 19.5 — textContent for Reading and Writing

<p id="greeting">Hello, <strong>world</strong>!</p>
<p id="read-result"></p>

<script>
const p = document.querySelector('#greeting');

// Reading — strips HTML tags, returns raw text
const readText = p.textContent;  // "Hello, world!"
document.querySelector('#read-result').textContent = 'Read: ' + readText;

// Writing — replaces ALL content with plain text
// Any HTML in the string is treated as literal characters, not markup
p.textContent = 'Goodbye, <em>everyone</em>!';
// The p now displays:  Goodbye, <em>everyone</em>!
// (The angle brackets appear as visible text, not as HTML)
</script>

This behavior — treating the written string as plain text — is what makes textContent safe. Even if the string contains HTML characters or script tags, they will appear as literal text, not be interpreted as code. Use textContent whenever you are working with plain text.

element.innerHTML

innerHTML reads or writes the HTML markup inside an element. When you read it, you get the full HTML string including all child tags. When you write to it, the browser parses the string as HTML and builds new DOM nodes from it.

LISTING 19.6 — innerHTML for Reading and Writing Markup

<div id="card">
  <h2>My Card</h2>
  <p>Some content.</p>
</div>
<pre id="html-preview"></pre>

<script>
const card = document.querySelector('#card');

// Reading — returns the raw HTML string inside
const rawHTML = card.innerHTML;
document.querySelector('#html-preview').textContent = 'innerHTML: ' + rawHTML;
// Displays: innerHTML: <h2>My Card</h2>\n  <p>Some content.</p>

// Writing — replaces content with parsed HTML
card.innerHTML = '<h2>Updated Card</h2><p>New content with a <a href="#">link</a>.</p>';

// Building HTML from your own data is fine
const products = ['Apple', 'Banana', 'Cherry'];
const list = document.querySelector('#fruit-list');
list.innerHTML = products.map(name => `<li>${name}</li>`).join('');
</script>

CAUTION: Never set innerHTML using content that comes from user input. If a user types <img src=x onerror="stealYourData()"> into a form and you insert that directly into the DOM via innerHTML, the browser executes the embedded script. This is called a Cross-Site Scripting (XSS) attack and is one of the most common web security vulnerabilities. If you need to display user-supplied text, always use textContent. Only use innerHTML with HTML strings you have written yourself or that come from a trusted source.

element.innerText

innerText is similar to textContent but with a key difference: it is layout-aware. It returns only the text that is currently rendered on screen, respecting CSS visibility. Elements with display: none or visibility: hidden are excluded from innerText but included in textContent. It also normalizes whitespace to match what the user sees.

Because innerText has to look at layout information, reading it can trigger a browser reflow — a potentially expensive recalculation. For most use cases where you just want the text content of an element, textContent is faster and more predictable.

Which Property to Use

  • Use textContent as your default for reading or writing plain text. It is fast, safe, and predictable.
  • Use innerHTML when you need to insert actual HTML markup — but only with HTML you control, never with user input.
  • Use innerText when you specifically need to know what text is visible on screen as rendered (relatively rare in practice).

Working with Attributes

Every HTML attribute — src, href, class, id, disabled, data-* and so on — is accessible from JavaScript. There are two main approaches: the generic attribute methods, and direct property access.

The Attribute Methods

Four methods cover every attribute manipulation scenario:

LISTING 19.7 — getAttribute, setAttribute, removeAttribute, hasAttribute

<a id="link" href="https://example.com" target="_blank">Example</a>
<p id="attr-read-result"></p>

<script>
const link = document.querySelector('#link');
const result = document.querySelector('#attr-read-result');

// Read an attribute's current value (always returns a string, or null)
const href    = link.getAttribute('href');    // "https://example.com"
const target  = link.getAttribute('target');  // "_blank"
const missing = link.getAttribute('class');   // null (attribute doesn't exist)

// Set or change an attribute
link.setAttribute('href', 'https://developer.mozilla.org');
link.setAttribute('aria-label', 'Visit MDN Web Docs');

// Check if an attribute exists (boolean)
const hasTarget = link.hasAttribute('target');  // true
const hasClass  = link.hasAttribute('class');   // false

// Remove an attribute entirely
link.removeAttribute('target');
// The link no longer has target="_blank"

// Display what we read from the original attributes
result.textContent =
  `href: "${href}" | target: "${target}" | class: ${missing} | ` +
  `hasAttribute('target'): ${hasTarget} | hasAttribute('class'): ${hasClass}`;
</script>

The data-* Attributes and the dataset API

Custom data-* attributes are the standard, clean way to embed application-specific data into HTML elements — without making up non-standard attributes or cramming data into class names. You have probably seen patterns like data-id="42" or data-user-role="admin" in the wild. These are perfectly valid HTML and are specifically designed for JavaScript to read.

The dataset property gives you a convenient object-based API to read and write all data-* attributes on an element. There is one naming rule: the data- prefix is dropped, and any hyphens in the name are converted to camelCase.

LISTING 19.8 — data-* Attributes and dataset

<div id="product"
     data-product-id="SKU-1042"
     data-category="electronics"
     data-price="299.99"
     data-in-stock="true">
  Wireless Headphones
</div>
<p id="product-info"></p>

<script>
const product = document.querySelector('#product');

// Reading via dataset (camelCase conversion from data-* attribute names)
const productId = product.dataset.productId;  // "SKU-1042"
const category  = product.dataset.category;   // "electronics"
const priceStr  = product.dataset.price;      // "299.99" (always a string!)
const inStockStr = product.dataset.inStock;   // "true"

// Note: values are always strings — convert if needed
const price = parseFloat(priceStr);           // 299.99 (number)
const inStock = inStockStr === 'true';        // true (boolean)

// Display what was read from the dataset
document.querySelector('#product-info').textContent =
  `ID: ${productId} | Category: ${category} | Price: $${price} | In Stock: ${inStock}`;

// Writing via dataset creates or updates the attribute
product.dataset.discount = '10';  // adds data-discount="10"
product.dataset.category = 'audio'; // updates data-category="audio"

// Deleting a dataset property removes the attribute
delete product.dataset.inStock;
// The element no longer has data-in-stock
</script>

Direct Property Access

For the most common HTML attributes, JavaScript elements expose them as direct properties. These are generally more convenient than calling getAttribute:

LISTING 19.9 — Direct Property Access for Common Attributes

const img = document.querySelector('img');
const input = document.querySelector('input[type="text"]');
const checkbox = document.querySelector('input[type="checkbox"]');
const btn = document.querySelector('button');
const attrLog = document.querySelector('#attr-log');

// Image attributes
img.src = 'new-photo.jpg';       // changes the image source
img.alt = 'A new description';   // changes the alt text

// Input value (reflects what the user has typed or what you set)
const currentValue = input.value;  // read the current value
input.value = 'Pre-filled text';   // programmatically fill the input

// Boolean attributes (represented as true/false booleans in JS)
const isChecked = checkbox.checked;  // read: true if checked
checkbox.checked = true;             // check it programmatically

btn.disabled = true;             // disable the button
btn.disabled = false;            // re-enable it

// Display the read values on screen
attrLog.textContent =
  `input.value was: "${currentValue}" | checkbox.checked was: ${isChecked}`;

The hidden Attribute

The hidden HTML attribute is a boolean attribute that hides an element — the browser applies display: none automatically. It is semantically meaningful (it tells the browser the element is not currently relevant) and works well with JavaScript:

LISTING 19.10 — The hidden Attribute

<div id="error-message" hidden>Something went wrong.</div>

<script>
const errorBox = document.querySelector('#error-message');

// Show the element
errorBox.hidden = false;  // removes the hidden attribute

// Hide it again
errorBox.hidden = true;   // adds the hidden attribute back

// Or use removeAttribute / setAttribute
errorBox.removeAttribute('hidden');  // shows
errorBox.setAttribute('hidden', ''); // hides again
</script>

The difference between hidden and display: none via CSS is mostly semantic. The hidden attribute communicates intent through HTML, which can be useful. Either approach is valid; choose whichever fits your code organization better.

Modifying Styles and Classes

JavaScript can change the visual appearance of elements in two ways: by setting inline styles directly, or by adding and removing CSS classes. Understanding which approach to use — and why — is an important part of writing maintainable code.

Inline Styles with element.style

The style property on a DOM element gives you access to that element's inline style. You set individual CSS properties as JavaScript property names, using camelCase instead of kebab-case:

LISTING 19.11 — Setting Inline Styles

const box = document.querySelector('.box');

// Kebab-case CSS properties become camelCase in JS
box.style.backgroundColor = '#4f6df5';  // background-color
box.style.fontSize = '1.25rem';          // font-size
box.style.marginTop = '2rem';            // margin-top
box.style.borderRadius = '8px';          // border-radius

// To remove an inline style, set it to an empty string
box.style.backgroundColor = '';          // reverts to stylesheet value

// Reading inline styles (only reads styles set via JS or inline HTML)
const inlineFontSize = box.style.fontSize;  // "1.25rem" (if set via JS)

// To read the actual computed style (including stylesheets), use:
const computed = getComputedStyle(box);
const computedFontSize = computed.fontSize;  // "20px" (the rendered value)

const styleLog = document.querySelector('#style-log');
if (styleLog) {
  styleLog.textContent =
    `inline fontSize: "${inlineFontSize}" | computed fontSize: "${computedFontSize}"`;
}

Inline styles are the nuclear option of CSS: they have the highest specificity (short of !important), making them nearly impossible to override from a stylesheet. They also scatter style decisions across your JavaScript, making the codebase harder to maintain. As a result, setting inline styles directly is rarely the right approach for anything beyond quick, one-off computed values like animation positions.

The classList API

The right way to change the visual state of an element is to toggle CSS classes. You define what each class means visually in your stylesheet, then use JavaScript to apply or remove those classes in response to user actions. This keeps your styles in CSS (where they belong) and your logic in JavaScript (where it belongs).

The classList property on any element provides a clean API for exactly this:

LISTING 19.12 — The classList API

const card = document.querySelector('.card');
const btn  = document.querySelector('.toggle-btn');

// classList.add() — adds one or more classes
card.classList.add('is-active');
card.classList.add('is-featured', 'has-shadow'); // multiple at once

// classList.remove() — removes one or more classes (no error if absent)
card.classList.remove('is-active');
card.classList.remove('is-featured', 'has-shadow');

// classList.toggle() — adds if absent, removes if present
// Returns true if the class was added, false if it was removed
btn.addEventListener('click', () => {
  const isOpen = card.classList.toggle('is-expanded');
  btn.textContent = isOpen ? 'Collapse' : 'Expand';
});

// toggle() accepts an optional second argument: force it on or off
card.classList.toggle('is-dark', window.matchMedia('(prefers-color-scheme: dark)').matches);

// classList.contains() — check if a class is present (returns boolean)
if (card.classList.contains('is-featured')) {
  card.querySelector('h3')?.textContent = 'Featured Card';
}

// classList.replace() — swap one class for another
card.classList.replace('theme-light', 'theme-dark');

Compare this to the older approach of manipulating element.className directly. className is just a string, so adding a class requires string concatenation (element.className += ' active'), and removing one requires splitting, filtering, and joining. The classList API is far more readable, less error-prone, and handles edge cases (like duplicate classes) automatically.

Creating and Inserting Elements

Selecting and modifying existing elements covers a lot of ground, but many real interactions require creating new elements on the fly — adding a comment to a list, rendering a new product card, inserting an error message next to a form field. JavaScript provides everything you need to create, configure, and insert elements into the live DOM.

Creating Elements

document.createElement(tagName) creates a new element in memory. It is not yet part of the DOM — it exists only in JavaScript. You typically set its content, classes, and attributes before inserting it.

LISTING 19.13 — Creating and Configuring an Element

// Create a new list item
const li = document.createElement('li');

// Configure it before inserting
li.textContent = 'New task item';
li.classList.add('task-item');
li.dataset.taskId = '99';

// You can build nested structures
const article = document.createElement('article');
article.classList.add('card');

const h3 = document.createElement('h3');
h3.textContent = 'New Article Title';

const p = document.createElement('p');
p.textContent = 'Article preview text goes here.';

// Append children to the article before inserting
article.appendChild(h3);
article.appendChild(p);

Inserting Elements

Once an element is ready, you insert it into the DOM tree. Several methods give you control over exactly where it lands:

LISTING 19.14 — Inserting Elements Into the DOM

const list = document.querySelector('#task-list');
const firstItem = list.firstElementChild;

// appendChild() — adds as the LAST child of the parent
list.appendChild(li);

// prepend() — adds as the FIRST child (or multiple nodes/strings)
list.prepend(li);

// before() — inserts BEFORE the element (as a sibling)
firstItem.before(li);

// after() — inserts AFTER the element (as a sibling)
firstItem.after(li);

// You can insert multiple nodes or strings at once
list.prepend(li, 'Some text', anotherElement);

Inserting HTML Strings: insertAdjacentHTML()

When you want to insert a chunk of HTML markup — rather than building it element by element — insertAdjacentHTML(position, htmlString) is often the most convenient tool. It accepts a position keyword and an HTML string, parses the string, and inserts the resulting elements at the specified position.

The four position values are relative to the element you call the method on:

LISTING 19.15 — insertAdjacentHTML Positions

<!-- Visual guide to the four positions -->
<!--  "beforebegin" — before the element's opening tag -->
<ul id="list">                    <!-- "afterbegin" — inside, before first child -->
  <li>Existing item</li>
</ul>                             <!-- "beforeend" — inside, after last child -->
<!-- "afterend" — after the element's closing tag -->

<script>
const list = document.querySelector('#list');

// afterbegin: new item will be the first child
list.insertAdjacentHTML('afterbegin', '<li class="new">First item</li>');

// beforeend: new item will be the last child (most commonly used)
list.insertAdjacentHTML('beforeend', '<li class="new">Last item</li>');

// before the list element itself
list.insertAdjacentHTML('beforebegin', '<h3>Task List</h3>');

// after the list element itself
list.insertAdjacentHTML('afterend', '<p>End of list.</p>');
</script>

The same XSS caution applies here: never pass user-supplied text to insertAdjacentHTML directly. For user content, build elements with createElement and set textContent instead.

Removing Elements

The modern way to remove an element from the DOM is simply to call .remove() on it:

LISTING 19.16 — Removing Elements

const notification = document.querySelector('.notification');

// Remove the element from the DOM entirely
notification.remove();

// Common pattern: find and remove
document.querySelector('#old-banner')?.remove();
// The ?. (optional chaining) prevents an error if the element doesn't exist

Modern Alternatives: replaceWith() and replaceChildren()

Two modern methods make common replacement patterns much cleaner:

LISTING 19.17 — replaceWith and replaceChildren

// replaceWith() — replaces an element with one or more new nodes
const oldCard = document.querySelector('.card-v1');
const newCard = document.createElement('div');
newCard.classList.add('card-v2');
newCard.textContent = 'Updated card content';

oldCard.replaceWith(newCard);
// oldCard is gone; newCard takes its place in the DOM

// replaceChildren() — clears ALL children and replaces them
// Called on the parent element
const taskList = document.querySelector('#task-list');

// This removes all existing children and adds new ones in one operation
const firstItem = document.createElement('li');
firstItem.textContent = 'Reset item 1';
const secondItem = document.createElement('li');
secondItem.textContent = 'Reset item 2';

taskList.replaceChildren(firstItem, secondItem);
// All previous <li> elements are gone

// replaceChildren() with no arguments clears all children (like innerHTML = '')
taskList.replaceChildren(); // empties the list

Traversing the DOM

Selection methods let you find elements from the top down, starting at document and searching by selector. But sometimes you already have an element and you need to navigate relative to it — to its parent, its children, its siblings. This is called traversal, and it is especially useful in event handlers where you receive a reference to one element and need to act on related elements nearby.

Moving Up: Parent

element.parentElement returns the direct parent element of the current element. If the element has no parent (it is the root <html> element) or if the parent is a non-element node, it returns null.

LISTING 19.18 — Navigating to Parent Elements

<ul id="list">
  <li id="item-1">Item One</li>
  <li id="item-2">Item Two</li>
</ul>
<p id="parent-info"></p>

<script>
const item = document.querySelector('#item-1');
const info = document.querySelector('#parent-info');

// parentElement gives you the containing element
const parent = item.parentElement;           // the <ul id="list"> element
info.textContent = `Parent tag: ${parent.tagName}, id: "${parent.id}"`;
// Displays: Parent tag: UL, id: "list"
</script>

Moving Down: Children

LISTING 19.19 — Navigating to Child Elements

const list = document.querySelector('#list');
const summary = document.querySelector('#list-summary');

// .children returns a live HTMLCollection of child ELEMENTS (no text nodes)
const childCount = list.children.length;  // 2
const firstName  = list.firstElementChild.textContent;  // "Item One"
const lastName   = list.lastElementChild.textContent;   // "Item Two"

summary.textContent =
  `${childCount} items — first: "${firstName}", last: "${lastName}"`;
// Displays: 2 items — first: "Item One", last: "Item Two"

Moving Sideways: Siblings

LISTING 19.20 — Navigating to Sibling Elements

const item1 = document.querySelector('#item-1');
const siblingLog = document.querySelector('#sibling-log');

// Next and previous ELEMENT siblings (skip text nodes)
const next = item1.nextElementSibling;       // the <li id="item-2"> element
const prev = item1.previousElementSibling;   // null — item-1 is the first

// Walk all following siblings and collect their text
const siblingTexts = [];
let sibling = item1.nextElementSibling;
while (sibling) {
  siblingTexts.push(sibling.textContent);
  sibling = sibling.nextElementSibling;
}

siblingLog.textContent =
  `Next sibling: "${next?.textContent}", previous: ${prev}, following: [${siblingTexts.join(', ')}]`;
// Displays: Next sibling: "Item Two", previous: null, following: [Item Two]

element.closest() — Walking Up the Tree

closest(selector) is one of the most useful DOM methods in modern JavaScript. Starting with the element itself, it walks up the DOM tree and returns the first ancestor (or the element itself) that matches the given CSS selector. If no match is found, it returns null.

This is extremely useful in event handlers, where you often know what was clicked but need to find the containing component it belongs to:

LISTING 19.21 — element.closest() for Finding Ancestors

<ul class="task-list">
  <li class="task-item">
    <span class="task-label">Buy groceries</span>
    <button class="delete-btn">Delete</button>
  </li>
  <li class="task-item">
    <span class="task-label">Walk the dog</span>
    <button class="delete-btn">Delete</button>
  </li>
</ul>

<script>
document.querySelector('.task-list').addEventListener('click', event => {
  const btn = event.target;

  // Walk up from the clicked button to find the containing list item
  const taskItem = btn.closest('.task-item');

  if (taskItem) {
    taskItem.remove();  // remove the entire task row
  }
});

// closest() also matches the element itself
const btn = document.querySelector('.delete-btn');
const closestBtn       = btn.closest('button');      // the button itself
const closestTaskItem  = btn.closest('.task-item');  // the parent <li>
const closestTable     = btn.closest('table');       // null — no <table> ancestor

const closestInfo = document.querySelector('#closest-info');
if (closestInfo) {
  closestInfo.textContent =
    `closest('button'): ${closestBtn?.tagName ?? 'null'} | ` +
    `closest('.task-item'): ${closestTaskItem?.className ?? 'null'} | ` +
    `closest('table'): ${closestTable}`;
  // Displays: closest('button'): BUTTON | closest('.task-item'): task-item | closest('table'): null
}
</script>

Without closest(), you would have to manually chain parentElement calls and check each one — fragile code that breaks the moment you adjust your HTML structure. closest() makes the intent clear and the code robust.

element.matches() — Testing an Element Against a Selector

matches(selector) returns a boolean: does this element match the given CSS selector? It is especially useful inside event handlers for filtering which elements you want to respond to.

LISTING 19.22 — element.matches() for Selector Testing

const item = document.querySelector('.task-item');
const matchResult = document.querySelector('#match-result');

// matches() returns a boolean — display the results on screen
matchResult.textContent = [
  `matches('.task-item'): ${item.matches('.task-item')}`,          // true
  `matches('[data-completed]'): ${item.matches('[data-completed]')}`, // false
  `matches('li'): ${item.matches('li')}`,                          // true
  `matches('li.task-item'): ${item.matches('li.task-item')}`,      // true
].join(' | ');

// Common use: filtering in an event handler (previews event delegation)
document.addEventListener('click', event => {
  if (event.target.matches('.delete-btn')) {
    // Handle delete button clicks
    const item = event.target.closest('.task-item');
    item?.remove();
  }

  if (event.target.matches('.edit-btn')) {
    // Handle edit button clicks differently
  }
});

The pattern of attaching a single event listener to a parent container and using matches() to filter click targets is one of the most important techniques in front-end JavaScript.

Summary

The Document Object Model is the browser's live, object-based representation of your HTML document — a tree of nodes that JavaScript can read and modify in real time. Every change you make to the DOM is instantly reflected on screen, which is the foundation of all interactive web behavior.

You learned to select elements using querySelector and querySelectorAll with the same CSS selector syntax you already know from stylesheets. You saw that the result of querySelectorAll is a NodeList — not a true Array — but that pairing it with forEach() covers the vast majority of everyday tasks. When you need array methods like filter() or map(), Array.from() or the spread operator will convert it.

Reading and writing element content comes down to three properties: textContent for safe, fast plain-text access; innerHTML for reading and inserting HTML markup you control; and innerText for layout-aware rendered text. The cardinal rule is never to set innerHTML from user-supplied input.

Attributes are accessible through the generic getAttribute, setAttribute, removeAttribute, and hasAttribute methods, or directly as element properties for common attributes. The data-* attribute pattern combined with the dataset API gives you a clean, standards-compliant way to embed custom data in your HTML elements.

For styling, the classList API — with its add, remove, toggle, contains, and replace methods — is almost always the right tool. It keeps your visual definitions in CSS where they belong, while JavaScript handles the logic of when to apply them.

Creating new elements follows a simple pattern: createElement to make the element, configure it with content and attributes, then insert it with appendChild, prepend, before, after, or insertAdjacentHTML. The modern replaceWith() and replaceChildren() methods make common replacement operations clean and concise.

Finally, DOM traversal — parentElement, children, firstElementChild, nextElementSibling, and their counterparts — lets you navigate the tree relative to any element you already have. closest() is especially powerful, walking up the tree to find an ancestor by selector, while matches() tests an element against a selector and is indispensable for event filtering. Both become even more important when tackling event delegation.

Q&A

Q. What is the difference between the DOM and the HTML source file?

A. The HTML source file is static text on disk. The DOM is the browser's live in-memory interpretation of that text — a tree of JavaScript objects. When your JavaScript modifies the DOM, the source file is untouched; only the in-memory model changes. This is why “View Page Source” shows the original HTML, while the DevTools Elements panel shows the current state of the DOM after any JavaScript modifications.

Q. When should I use querySelector versus the legacy getElementById or getElementsByClassName?

A. For new code, prefer querySelector and querySelectorAll. They support the full CSS selector syntax, which is both more expressive and more familiar. getElementById is marginally faster (it does a direct hash lookup), so you may still see it in performance-sensitive code or legacy codebases, but the speed difference is negligible for typical page interactions. The main reason to know the legacy methods is to recognize them when reading existing code, not necessarily to write them.

Q. Why does my querySelectorAll result not have a map() method when arrays do?

A. querySelectorAll returns a NodeList, which is a DOM type, not a JavaScript Array. NodeList supports forEach() and a few other iteration patterns, but it does not inherit from Array.prototype, so methods like map(), filter(), and find() are not available. The fix is to convert it first: Array.from(nodeList) or [...nodeList]. After that, you have a true array with all methods available.

Q. Is it ever acceptable to use innerHTML with user-generated content?

A. Only if you sanitize the content first. Raw user input must never be passed directly to innerHTML. If you genuinely need to support user-authored HTML (for example, a rich text editor), you must sanitize it with a trusted library like DOMPurify, which strips potentially dangerous markup and attributes before insertion. For displaying user-typed text, always use textContent, which treats the string as literal characters and never executes any code within it.

Q. What is the difference between classList.toggle('active') and manually checking classList.contains before adding or removing?

A. They produce the same result, but toggle() is concise and atomic. Instead of writing if (el.classList.contains('active')) { el.classList.remove('active'); } else { el.classList.add('active'); }, you write el.classList.toggle('active') and it handles both cases automatically. Additionally, toggle() returns a boolean indicating whether the class was added (true) or removed (false), which is useful for updating related state like button labels. The optional second argument — toggle('active', condition) — lets you force the class on or off based on a boolean expression, which is cleaner than an if/else with add and remove.

Q. Why does element.closest() check the element itself in addition to its ancestors?

A. Because closest() is designed to answer the question “does this element or any of its ancestors match this selector?” Including the element itself makes the method more useful: if you have a click handler on a button and the button itself has the class you are searching for, btn.closest('.btn') correctly returns the button rather than skipping it and returning an ancestor. This self-inclusive behavior means you can safely call closest() without having to special-case whether the element itself is the match.
