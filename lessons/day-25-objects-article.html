Introduction

Imagine building a user profile page. Without objects, you might declare four separate variables: firstName, lastName, email, and age. That works for one user, but the moment you have ten users, you have forty separate variables with no clear connection between them. Objects solve this by bundling all the related data for one entity under a single name — user.firstName, user.lastName, user.email, user.age — so that one variable holds the whole picture. Objects are the fundamental unit of structured data in JavaScript, and they appear in virtually every program you will ever read or write.

The visual connection between objects and the screen is direct. One object becomes one rendered card. An array of objects becomes an entire product grid or search results page. By the end of this lesson you will be able to take a plain JavaScript object, access its properties, transform its data, and render it as styled HTML. You will also learn JSON — the text format that APIs use to send structured data across the network — so that when your code talks to a server, you know exactly what is being exchanged and how to work with it.

What Is an Object?

An object is a collection of key-value pairs. Each key is a string (or a Symbol) that names a piece of data, and each value can be anything: a number, a string, a boolean, an array, another object, or even a function. The key-value pair is called a property. When the value is a function, the property is called a method.

Here is a product object and the card it can render:

LISTING 25.1 — A Product Object and Its Rendered Card 
const product = {
  name: "Wireless Headphones",
  price: 79.99,
  category: "Audio",
  inStock: true,
};

// Render the object as an HTML card
const card = `
  <div class="card">
    <h3>${product.name}</h3>
    <p>${product.category}</p>
    <strong>$${product.price}</strong>
    <span>${product.inStock ? "In Stock" : "Out of Stock"}</span>
  </div>
`;

document.querySelector('#product-container').innerHTML = card;

One object. One card. The keys (name, price, category, inStock) become the labeled data inside the card. This is the core pattern of front-end development: data lives in objects, HTML is generated from that data.

Creating and Accessing Objects

The most common way to create an object is with an object literal: curly braces containing a comma-separated list of key-value pairs. Each pair uses a colon to separate the key from the value.

LISTING 25.2 — Object Literal Syntax 
const user = {
  name: "Alex",
  age: 28,
  isAdmin: false,
};

Dot Notation

Dot notation is the most common way to read and write properties. Write the variable name, a dot, and then the property key:

LISTING 25.3 — Reading Properties with Dot Notation 
console.log(user.name);    // "Alex"
console.log(user.age);     // 28
console.log(user.isAdmin); // false

Bracket Notation

Bracket notation uses a string inside square brackets. It is equivalent to dot notation but opens up two additional capabilities: accessing keys that contain special characters, and using a variable as the key name.

LISTING 25.4 — Bracket Notation and Dynamic Keys 
// Equivalent to dot notation
console.log(user["name"]); // "Alex"

// Dynamic key: the key name comes from a variable
const fieldName = "age";
console.log(user[fieldName]); // 28

// Keys with special characters require bracket notation
const config = {
  "max-retries": 3,
  "Content-Type": "application/json",
};
console.log(config["max-retries"]); // 3

NOTE: Use dot notation as your default. Switch to bracket notation when the key is stored in a variable, or when the key contains hyphens, spaces, or other characters that are not valid identifier characters.

Modifying, Adding, and Deleting Properties

You can change an existing property simply by assigning to it. You can add a new property the same way — if the key does not exist, JavaScript creates it. The delete operator removes a property entirely.

LISTING 25.5 — Modifying, Adding, and Deleting Properties 
const user = { name: "Alex", age: 28 };

// Modify an existing property
user.name = "Jordan";
console.log(user.name); // "Jordan"

// Add a new property
user.phone = "555-0100";
console.log(user.phone); // "555-0100"

// Delete a property
delete user.phone;
console.log(user.phone); // undefined

Checking If a Property Exists

Accessing a property that does not exist returns undefined rather than throwing an error. That silent undefined can cause subtle bugs — especially when a missing property and a property explicitly set to undefined look identical. Two tools give you a reliable existence check.

LISTING 25.6 — Checking Property Existence 
const user = { name: "Alex", role: undefined };

// `in` operator: checks own AND inherited properties
console.log("name" in user);   // true
console.log("role" in user);   // true  (property exists, value is undefined)
console.log("email" in user);  // false

// Object.hasOwn(): modern, checks ONLY own properties (not inherited)
console.log(Object.hasOwn(user, "name"));  // true
console.log(Object.hasOwn(user, "email")); // false

Object.hasOwn() was introduced in ES2022 and replaces the older obj.hasOwnProperty("key"). Prefer it in modern code. The in operator is still useful when you want to check inherited properties as well, but for most practical use cases Object.hasOwn() is the clearer choice.

Shorthand and Computed Properties

Property Shorthand

When a variable name matches the property key you want to use, you can write the variable name once instead of twice. This shorthand appears constantly in modern JavaScript — in function return values, in object construction from form fields, and in module exports.

LISTING 25.7 — Property Shorthand 
const name = "Alex";
const age = 28;
const email = "alex@example.com";

// Long form
const user1 = { name: name, age: age, email: email };

// Shorthand — identical result
const user2 = { name, age, email };

console.log(user2); // { name: "Alex", age: 28, email: "alex@example.com" }

Computed Property Names

Square brackets inside an object literal let you use an expression as the key name. This makes it possible to build objects dynamically — for example, constructing an object from a form where the field name is only known at runtime.

LISTING 25.8 — Computed Property Names 
const key = "email";
const value = "alex@example.com";

const obj = { [key]: value };
console.log(obj); // { email: "alex@example.com" }

// Real use: building an object from form data
function buildUpdate(fieldName, newValue) {
  return { [fieldName]: newValue };
}

const patch = buildUpdate("role", "admin");
console.log(patch); // { role: "admin" }

Methods and this

A function stored as a property of an object is called a method. Methods let an object carry its own behavior alongside its data. Modern JavaScript provides a shorthand syntax for methods that removes the function keyword entirely.

LISTING 25.9 — Method Shorthand and this 
const user = {
  name: "Alex",
  role: "editor",

  // Method shorthand (preferred)
  greet() {
    return `Hi, I'm ${this.name} and I'm an ${this.role}.`;
  },

  // Equivalent long form
  // greet: function() {
  //   return `Hi, I'm ${this.name}.`;
  // },
};

console.log(user.greet()); // "Hi, I'm Alex and I'm an editor."

Inside a method, this refers to the object that the method was called on — in this case, user. So this.name is user.name, and this.role is user.role. The value of this is determined at the moment the method is called, not when it is defined.

Arrow Functions and this

Arrow functions do not have their own this. Instead, they inherit this from the surrounding scope where they were defined. This makes arrow functions unsuitable as object methods when you need to reference the object via this.

LISTING 25.10 — Why Arrow Functions Break Object Methods 
const user = {
  name: "Alex",

  // Arrow function — this is NOT the user object
  greetArrow: () => {
    return `Hi, I'm ${this.name}.`; // this.name is undefined
  },

  // Regular method shorthand — this IS the user object
  greetMethod() {
    return `Hi, I'm ${this.name}.`; // "Alex"
  },
};

console.log(user.greetArrow());  // "Hi, I'm undefined."
console.log(user.greetMethod()); // "Hi, I'm Alex."

NOTE: Use method shorthand (methodName() {}) when you need this to refer to the object. Use arrow functions when you need this to refer to an outer context — for example, inside a callback nested within a method.

Object Utilities

JavaScript provides a set of static methods on the Object constructor that let you inspect, iterate, and transform objects. These are among the most useful tools for working with data.

Object.keys(), Object.values(), Object.entries()

LISTING 25.11 — Object.keys(), Object.values(), Object.entries() 
const settings = {
  theme: "dark",
  fontSize: 16,
  notifications: true,
  language: "en",
};

Object.keys(settings);
// ["theme", "fontSize", "notifications", "language"]

Object.values(settings);
// ["dark", 16, true, "en"]

Object.entries(settings);
// [["theme", "dark"], ["fontSize", 16], ["notifications", true], ["language", "en"]]

Object.entries() is particularly powerful for rendering objects as tables or settings panels, because it gives you both the key and the value together as you iterate:

LISTING 25.12 — Rendering a Settings Panel with Object.entries() 
const settings = { theme: "dark", fontSize: 16, language: "en" };

const rows = Object.entries(settings).map(([key, value]) => `
  <tr>
    <td><strong>${key}</strong></td>
    <td>${value}</td>
  </tr>
`).join('');

document.querySelector('#settings-table tbody').innerHTML = rows;

Object.fromEntries()

Object.fromEntries() is the inverse of Object.entries(). It takes an array of [key, value] pairs and constructs an object. This is useful for transforming all values in an object via a map call and then converting the result back into an object.

LISTING 25.13 — Transforming an Object with fromEntries 
const prices = { shirt: 29.99, hat: 14.99, bag: 49.99 };

// Apply a 10% discount to every price
const discounted = Object.fromEntries(
  Object.entries(prices).map(([item, price]) => [item, price * 0.9])
);

console.log(discounted);
// { shirt: 26.991, hat: 13.491, bag: 44.991 }

Object.assign()

Object.assign(target, source1, source2, ...) copies all enumerable own properties from the source objects into the target object and returns the target. Sources are applied left to right, so later sources overwrite earlier ones if keys conflict.

LISTING 25.14 — Merging Objects with Object.assign() 
const defaults = { theme: "light", fontSize: 14, notifications: true };
const userPrefs = { theme: "dark", fontSize: 16 };

// Merge userPrefs into defaults (userPrefs wins on conflicts)
const config = Object.assign({}, defaults, userPrefs);
console.log(config);
// { theme: "dark", fontSize: 16, notifications: true }

NOTE: Pass an empty object {} as the first argument to avoid mutating either source. Object.assign(defaults, userPrefs) would modify the defaults object in place, which is usually not what you want.

Object.freeze()

Object.freeze(obj) makes an object immutable: existing properties cannot be changed, new properties cannot be added, and properties cannot be deleted. Attempting any of these operations silently fails (or throws in strict mode). Use it for configuration constants that should never be modified at runtime.

LISTING 25.15 — Object.freeze() for Constants 
const API_CONFIG = Object.freeze({
  baseUrl: "https://api.example.com",
  version: "v2",
  timeout: 5000,
});

API_CONFIG.version = "v3"; // silently ignored
console.log(API_CONFIG.version); // "v2"

Copying Objects

Assignment in JavaScript copies a reference, not the value. When you write const obj2 = obj1, both variables point to the exact same object in memory. Modifying obj2 modifies obj1 as well. This is one of the most common sources of bugs for developers new to JavaScript.

LISTING 25.16 — Reference Trap: Assignment Is Not a Copy 
const original = { name: "Alex", score: 100 };
const copy = original; // NOT a copy — same reference

copy.score = 0;
console.log(original.score); // 0  ← original was mutated!

Shallow Copy

Both the spread operator and Object.assign() create a shallow copy — a new object whose top-level properties are independent from the source, but whose nested objects are still shared references.

LISTING 25.17 — Shallow Copy with Spread and Object.assign() 
const original = { name: "Alex", score: 100 };

// Spread — shallow copy
const copy1 = { ...original };
copy1.score = 0;
console.log(original.score); // 100 — independent at top level ✓

// Object.assign() — also a shallow copy
const copy2 = Object.assign({}, original);
copy2.name = "Jordan";
console.log(original.name); // "Alex" — independent ✓

The Shallow Copy Problem with Nested Objects

When an object contains a nested object (or array), a shallow copy copies the reference to that nested object. Both the original and the copy point to the same nested object, so mutating it through either variable affects both.

LISTING 25.18 — Shallow Copy Fails for Nested Objects 
const original = {
  name: "Alex",
  address: { city: "Portland", state: "OR" },
};

const copy = { ...original };

copy.name = "Jordan";          // independent ✓
copy.address.city = "Seattle"; // mutates the SHARED nested object!

console.log(original.name);         // "Alex"    — independent ✓
console.log(original.address.city); // "Seattle" — was mutated! ✗

Deep Copy with structuredClone()

structuredClone() creates a completely independent deep copy of an object, including all nested objects and arrays. It was added to JavaScript natively in 2022 and is now the recommended way to make deep copies. It handles nested objects, arrays, dates, maps, and sets — but it does not copy functions.

LISTING 25.19 — Deep Copy with structuredClone() 
const original = {
  name: "Alex",
  address: { city: "Portland", state: "OR" },
  scores: [95, 87, 100],
};

const copy = structuredClone(original);

copy.name = "Jordan";
copy.address.city = "Seattle";
copy.scores.push(72);

console.log(original.name);         // "Alex"    — unchanged ✓
console.log(original.address.city); // "Portland" — unchanged ✓
console.log(original.scores);       // [95, 87, 100] — unchanged ✓

When to use which: Use spread or Object.assign() for flat objects (no nested objects or arrays). Use structuredClone() whenever the object contains nested structures that you need to be truly independent.

Object Destructuring

Destructuring lets you extract properties from an object directly into named variables in a single statement. Instead of writing const name = user.name; const age = user.age; on separate lines, you declare them together on one line.

LISTING 25.20 — Basic Destructuring 
const user = { name: "Alex", age: 28, role: "editor", email: "alex@example.com" };

// Extract name and age into variables
const { name, age } = user;
console.log(name); // "Alex"
console.log(age);  // 28

Renaming While Destructuring

Add a colon after the property name to rename the variable. The key on the left matches the object property; the identifier on the right is the new variable name.

LISTING 25.21 — Destructuring with Rename 
const { name: userName, role: userRole } = user;
console.log(userName); // "Alex"
console.log(userRole); // "editor"
// `name` and `role` are NOT declared as variables here

Default Values

Use = to provide a default value that is used when the property is missing (undefined) from the object.

LISTING 25.22 — Destructuring with Default Values 
const { name, role = "viewer", theme = "light" } = user;
console.log(role);  // "editor" — property exists, default not used
console.log(theme); // "light"  — property missing, default used

Destructuring in Function Parameters

One of the most valuable uses of destructuring is in function parameters. Instead of accepting an entire object and accessing obj.name inside the function, you destructure the properties you need right in the parameter list. This makes the function signature self-documenting — anyone reading the function call immediately knows which properties are used.

LISTING 25.23 — Destructuring in Function Parameters 
// Without destructuring
function displayUser(user) {
  return `${user.name} <${user.email}>`;
}

// With destructuring in the parameter
function displayUser({ name, email }) {
  return `${name} <${email}>`;
}

displayUser({ name: "Alex", age: 28, email: "alex@example.com" });
// "Alex "

Nested Destructuring

You can destructure into nested objects by mirroring the shape of the object in the destructuring pattern.

LISTING 25.24 — Nested Destructuring 
const user = {
  name: "Alex",
  address: {
    city: "Portland",
    state: "OR",
  },
};

const { name, address: { city, state } } = user;
console.log(city);  // "Portland"
console.log(state); // "OR"
// `address` is NOT declared as a variable here

Optional Chaining and Nullish Coalescing

When working with data from APIs or user input, some properties may be absent entirely. Accessing a property on null or undefined throws a TypeError and stops your code. Optional chaining (?.) prevents that by short-circuiting to undefined the moment any link in the chain is null or undefined.

LISTING 25.25 — Optional Chaining 
const user = {
  name: "Alex",
  address: {
    city: "Portland",
  },
};

// Without optional chaining — throws if address is missing
// console.log(user.contact.phone); // TypeError!

// With optional chaining — returns undefined safely
console.log(user?.address?.city);     // "Portland"
console.log(user?.contact?.phone);    // undefined (no error)

// Also works for method calls
console.log(user?.getAddress?.());    // undefined (no error if method missing)

Nullish Coalescing (??)

The nullish coalescing operator (??) provides a fallback value when the left side is null or undefined. Unlike the logical OR operator (||), it does not treat 0, false, or empty string as falsy — only null and undefined trigger the fallback.

LISTING 25.26 — Optional Chaining with Nullish Coalescing 
const user = { name: "Alex", score: 0 };

// ?? only falls back on null/undefined
console.log(user?.address?.city ?? "Unknown");  // "Unknown"
console.log(user?.score ?? "No score");          // 0 (not "No score")

// || would incorrectly trigger on 0
console.log(user?.score || "No score");          // "No score" — wrong!

NOTE: Combine optional chaining and nullish coalescing as a pair: obj?.deep?.property ?? fallback. This pattern handles missing data gracefully and is a hallmark of production-quality JavaScript.

Arrays of Objects

The most common data pattern in front-end development is an array of objects. Product listings, user directories, blog posts, search results, shopping cart items — virtually every list of real-world data is an array where each element is an object with the same shape.

LISTING 25.27 — An Array of Product Objects 
const products = [
  { name: "Wireless Headphones", price: 79.99, category: "Audio",  inStock: true  },
  { name: "USB-C Hub",           price: 34.99, category: "Tech",   inStock: true  },
  { name: "Notebook",            price: 12.99, category: "Office", inStock: false },
  { name: "Desk Lamp",           price: 44.99, category: "Office", inStock: true  },
  { name: "Webcam",              price: 89.99, category: "Tech",   inStock: false },
];

// Access by index, then by property
console.log(products[0].name);   // "Wireless Headphones"
console.log(products[2].price);  // 12.99

To render all of these products as cards, loop over the array and build an HTML string from each object’s properties:

LISTING 25.28 — Rendering an Array of Objects as a Card Grid 
const grid = document.querySelector('#product-grid');

let html = '';

for (const product of products) {
  html += `
    <div class="card ${product.inStock ? '' : 'out-of-stock'}">
      <span class="category">${product.category}</span>
      <h3>${product.name}</h3>
      <strong>$${product.price.toFixed(2)}</strong>
      <p>${product.inStock ? '✓ In Stock' : '✗ Out of Stock'}</p>
    </div>
  `;
}

grid.innerHTML = html;

Five products in the array, five cards on screen. Change the array, the display changes. This is the direct relationship between data and UI that defines modern web development.

JSON

JSON (JavaScript Object Notation) is a text format for representing structured data. It was derived from JavaScript object literal syntax, but it is language-agnostic — Python, Ruby, Java, Go, and virtually every other language can read and write JSON. When web applications talk to servers, when you save data to local storage, when you configure tools and services, JSON is almost always the format being used.

Why JSON Exists

A JavaScript object lives in your program’s memory. You cannot send memory directly over a network or write it to a file. You need to serialize the object — convert it into a flat text representation that can be transmitted and then reconstructed on the other side. JSON is that text representation.

Think of it like this: an object is furniture arranged in a room. JSON is the packing list that describes every piece precisely enough that someone else can unpack and reassemble the room identically.

JSON vs JavaScript Object Literal

JSON looks nearly identical to a JavaScript object literal, but the rules are stricter:

LISTING 25.29 — JSON vs JavaScript Object Literal 
// JavaScript object literal — flexible
const user = {
  name: 'Alex',       // single or double quotes for strings
  age: 28,
  isAdmin: false,
  greet() { return "Hi"; }, // functions allowed
  lastLogin: undefined,     // undefined allowed
};

// JSON — strict rules
// {
//   "name": "Alex",         ← keys MUST be double-quoted
//   "age": 28,              ← numbers are fine
//   "isAdmin": false,       ← booleans: true/false (lowercase)
//   "scores": [95, 87],     ← arrays allowed
//   "address": { "city": "Portland" }  ← nested objects allowed
// }
//
// NOT allowed in JSON:
//   functions, undefined, Date objects, comments, trailing commas

JSON supports only these value types: strings (double-quoted only), numbers, booleans (true / false), null, arrays, and objects. Everything else — functions, undefined, Dates, Symbols, class instances — cannot be represented in JSON.

JSON.stringify() — Object to JSON String

JSON.stringify(value, replacer, space) converts a JavaScript value into a JSON string. The second argument is a replacer (pass null to include all properties). The third argument is the number of spaces to use for indentation in the output; pass 2 for readable pretty-printed JSON.

LISTING 25.30 — JSON.stringify() 
const user = { name: "Alex", age: 28, isAdmin: false };

// Compact string (for sending over a network)
const compact = JSON.stringify(user);
console.log(compact);
// '{"name":"Alex","age":28,"isAdmin":false}'

// Pretty-printed (for display or debugging)
const pretty = JSON.stringify(user, null, 2);
console.log(pretty);
// {
//   "name": "Alex",
//   "age": 28,
//   "isAdmin": false
// }

// Properties with undefined or function values are dropped
const partial = JSON.stringify({ name: "Alex", fn: () => {}, x: undefined });
console.log(partial); // '{"name":"Alex"}'

JSON.parse() — JSON String to Object

JSON.parse(jsonString) converts a JSON string back into a JavaScript value. This is what you call after receiving data from a server.

LISTING 25.31 — JSON.parse() 
const jsonString = '{"name":"Alex","age":28,"scores":[95,87,100]}';

const user = JSON.parse(jsonString);
console.log(user.name);    // "Alex"
console.log(user.scores);  // [95, 87, 100]
console.log(typeof user);  // "object"

The Null Guard Pattern

When reading from storage or a source that may not have data yet, the retrieved value might be null. Passing null to JSON.parse() throws an error. Use the || operator to provide a fallback JSON string.

LISTING 25.32 — Null Guard with JSON.parse() 
// localStorage.getItem() returns null if the key doesn't exist
const raw = localStorage.getItem('cart'); // null on first visit

// Without guard: JSON.parse(null) → throws SyntaxError
// With guard: if raw is null/falsy, parse the empty array string instead
const cart = JSON.parse(raw || '[]');
console.log(cart); // [] on first visit, saved data on return visits

// Same pattern for objects
const prefs = JSON.parse(localStorage.getItem('prefs') || '{}');

A Complete JSON Round Trip

Here is the full cycle: create an object, serialize it to JSON, simulate sending/storing it, parse it back, and use the result.

LISTING 25.33 — JSON Round Trip 
// 1. Start with a JavaScript object
const order = {
  id: "ORD-001",
  customer: "Alex",
  items: ["Headphones", "USB-C Hub"],
  total: 114.98,
};

// 2. Stringify to send over a network or save to storage
const jsonString = JSON.stringify(order);
console.log(typeof jsonString); // "string"
console.log(jsonString);
// '{"id":"ORD-001","customer":"Alex","items":["Headphones","USB-C Hub"],"total":114.98}'

// 3. --- string travels over the network or sits in storage ---

// 4. Parse on the receiving end to get a JavaScript object back
const receivedOrder = JSON.parse(jsonString);
console.log(typeof receivedOrder); // "object"
console.log(receivedOrder.customer); // "Alex"
console.log(receivedOrder.items[0]); // "Headphones"
console.log(receivedOrder.total);    // 114.98

Real-World Context: Fetching API Data

When your JavaScript code calls a web API — a weather service, a search endpoint, a product catalog — the server responds with a JSON string. The fetch() API (which you will learn about when covering network requests) provides a .json() method that calls JSON.parse() for you. But under the hood, the same process is always happening: a JSON string arrives, gets parsed, and you work with the resulting JavaScript object.

LISTING 25.34 — Simulating an API Response 
// Imagine this string arrived from a server
const apiResponse = `{
  "status": "ok",
  "results": [
    { "id": 1, "title": "Post One",   "published": true  },
    { "id": 2, "title": "Post Two",   "published": false },
    { "id": 3, "title": "Post Three", "published": true  }
  ]
}`;

const data = JSON.parse(apiResponse);
console.log(data.status); // "ok"

// Filter and render only published posts
const published = data.results.filter(post => post.published);

published.forEach(post => {
  console.log(post.title); // "Post One", "Post Three"
});

NOTE: JSON.parse() throws a SyntaxError if the string is not valid JSON. In production code that parses data from external sources, wrap JSON.parse() in a try...catch block to handle malformed input gracefully.

Summary

  • Objects are collections of key-value pairs. Keys are strings (or Symbols); values can be any type, including other objects, arrays, and functions.
  • Dot notation (obj.key) is the default. Bracket notation (obj["key"] or obj[variable]) is required for dynamic keys and keys with special characters.
  • Property shorthand, computed property names, and method shorthand are modern syntactic conveniences that reduce repetition and appear throughout real codebases.
  • this inside a method refers to the object the method is called on. Arrow functions do not have their own this, making them unsuitable as object methods when this is needed.
  • Object.keys(), Object.values(), and Object.entries() convert an object’s keys and values into arrays, enabling iteration and transformation. Object.fromEntries() reconstructs an object from an array of pairs.
  • Assignment (const b = a) copies a reference, not the object. Spread and Object.assign() make shallow copies (nested objects remain shared). structuredClone() makes a fully independent deep copy of any object tree.
  • Destructuring extracts properties into variables cleanly. You can rename, set defaults, destructure in function parameters, and destructure nested objects — all patterns that appear constantly in modern JavaScript.
  • Optional chaining (?.) short-circuits to undefined when any link in a property chain is null or undefined, preventing TypeErrors. Pair it with nullish coalescing (??) to supply a fallback value.
  • Arrays of objects are the dominant data pattern in front-end development. A for...of loop over an array of objects, combined with template literals and innerHTML, generates complete UI sections from data.
  • JSON is a strict text format for structured data exchange. JSON.stringify() converts a JavaScript object to a JSON string; JSON.parse() converts a JSON string back to an object. JSON keys must be double-quoted, and functions, undefined, and Dates are not valid JSON values.

Q&A

What is the difference between dot notation and bracket notation?
Both read and write object properties, and both produce identical results when the key is a known, valid identifier. Dot notation (user.name) is shorter and more readable when the key is a fixed string with no special characters. Bracket notation (user["name"] or user[fieldName]) is required in two situations: when the key contains characters that are not valid in an identifier (hyphens, spaces, starting with a number), and when the key is stored in a variable and only known at runtime. In practice, dot notation is used most of the time, with bracket notation reserved for those specific cases.
When should I use bracket notation instead of dot notation?
Use bracket notation when you need to access a property whose key is computed at runtime — for example, user[fieldName] where fieldName is a variable that might hold "name", "email", or any other string. Also use it when the key itself contains characters that make dot notation syntactically impossible, such as config["max-retries"] or headers["Content-Type"]. A common pattern is building a form-field-to-object mapper where you receive the field name as a string and use it directly as the key: data[field.name] = field.value.
What does this refer to inside an object method?
this refers to the object on which the method was called. When you write user.greet(), JavaScript sets this to user for the duration of that call. So this.name inside greet resolves to user.name. This is determined at call time, not at definition time. Arrow functions are the notable exception: they capture this from the enclosing lexical scope when they are defined, so an arrow function used as an object method will not have the object as its this. Use regular method shorthand (methodName() {}) any time you need this to refer to the object itself.
What is the difference between Object.assign() and spread?
Both create a shallow copy and both merge source properties into a target. The practical differences are: Object.assign(target, ...sources) mutates the target object in place and returns it, while spread ({ ...source }) always creates a new object literal. For making copies without mutation, spread is cleaner and more idiomatic: const copy = { ...original } is shorter than const copy = Object.assign({}, original). For merging multiple sources in one step with fine-grained control over the target, Object.assign() is explicit. In modern codebases, spread is used for copies and merges, while Object.assign() appears mainly in legacy code or when the intent is to explicitly modify an existing target.
What is shallow copy vs deep copy, and when does it matter?
A shallow copy creates a new object at the top level, but any nested objects or arrays inside it are still shared references pointing to the same memory as the original. A deep copy creates entirely independent copies of all nested structures as well. It matters whenever you need to modify a copy without affecting the original, and the object contains nested objects or arrays. If your object is flat — all values are primitives like strings and numbers — a shallow copy is sufficient because primitives are copied by value. If your object contains a nested address object, scores array, or any other reference type, a shallow copy is a mutation trap. Use structuredClone() for deep copies.
Why can't I just send a JavaScript object over the network?
A JavaScript object exists only in your program’s memory — it is a data structure in RAM, specific to the JavaScript engine running your code. A network request carries a stream of bytes. There is no standard way for a server written in Python or Java to interpret raw JavaScript memory. Serialization bridges this gap: you convert the object into a universally understood text format (JSON), send that text, and the receiving application parses it back into its own equivalent data structure. This is why JSON.stringify() exists for outgoing data and JSON.parse() exists for incoming data.
What makes JSON different from a regular JavaScript object literal?
JSON is a strict subset of JavaScript with several important restrictions. First, all keys must be double-quoted strings — {"name": "Alex"} is valid JSON, but {name: "Alex"} is not. Second, string values must use double quotes, not single quotes. Third, only six value types are permitted: strings, numbers, booleans (true/false), null, arrays, and objects. Functions, undefined, Date objects, Symbols, and class instances cannot be represented. Fourth, trailing commas are not allowed. A JavaScript object literal has none of these restrictions — it can have unquoted keys, single-quoted strings, function values, and trailing commas. JSON’s strictness is a feature: it ensures that any language can parse it reliably.
What does optional chaining (?.) do, and when should I use it?
Optional chaining is a short-circuit operator: when evaluating a chain of property accesses like a?.b?.c, if any value in the chain is null or undefined, the entire expression immediately evaluates to undefined instead of throwing a TypeError. Without it, user.address.city throws if user.address is null or undefined. With it, user?.address?.city safely returns undefined. Use it whenever you are working with data that may be partially populated — API responses where some fields are optional, user data that may not have filled in every profile field, or DOM queries that may return null. It also works for method calls: user?.getAddress?.() will not throw if getAddress is not defined.
