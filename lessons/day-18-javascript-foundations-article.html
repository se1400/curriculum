You have spent the past seventeen lessons building beautiful, well-structured, thoughtfully designed web pages. You know HTML inside and out. Your CSS creates smooth transitions, elegant layouts, and rich visual effects. You’ve written pages that respond to :hover and :focus states, and you understand CSS custom properties, animations, and responsive design. But everything you’ve built so far is static — once the page loads, it doesn’t respond to what the user does.

This lesson changes that. JavaScript is the programming language of the web browser. It makes your page respond — to clicks, to typing, to scrolling, to any interaction a user has. Here is the most important idea to hold onto from the very beginning: JavaScript does not replace the HTML and CSS you know. It controls it. The transitions and animations you have built in CSS, the custom properties, the class names you have been styling — JavaScript is the remote control that operates all of it.

What Is JavaScript?

JavaScript was created in 1995 by Brendan Eich at Netscape Communications. The original goal was modest: give web browsers a lightweight scripting language so designers could add basic interactivity to pages without a full programming language. Today, JavaScript has grown into one of the most widely used programming languages in the world, running in every browser, on servers, and in mobile and desktop applications.

To understand how JavaScript fits into what you already know, think of a web page as a building:

  • ►HTML is the structure — the walls, floors, and rooms
  • ►CSS is the design — the paint, lighting, and furniture
  • ►JavaScript is the electricity — the switches and sensors that make everything respond

[IMG - Three connected pillars labeled HTML structure, CSS style, JavaScript behavior]

When a smart home’s lights turn on, nobody tears down walls or repaints the rooms. The switches just control what’s already there. JavaScript works the same way: you write your HTML and CSS, and JavaScript operates it.

How JavaScript Fits Into Your Page

JavaScript lives in .js files and connects to your HTML using the <script> tag. The modern recommended approach is to place the <script> tag in the <head> section with a defer attribute:

<head>
  <meta charset="UTF-8">
  <title>My Page</title>
  <link rel="stylesheet" href="styles.css">
  <script defer src="script.js"></script>
</head>

The defer attribute tells the browser: “Download this script in the background, but do not run it until the entire HTML has finished loading.” This means your JavaScript will always have access to every element on the page — you never have to worry about the script running before the page is ready.

NOTE
You will also see JavaScript placed in a <script> tag right before the closing </body> tag. This was the standard approach before defer became widely supported and works just as well. Both are valid; placing the script in <head> with defer is the cleaner modern convention.

You can also write JavaScript directly inside a <script> tag without a src attribute. This is called an inline script, and it is the style you will use in CodePen examples throughout these lessons:

<script>
  // JavaScript code goes here
  console.log('Hello!');
</script>

CAUTION
You will encounter older tutorials that use document.write() to add content, or alert() boxes for output. Both are outdated and avoided in modern development. document.write() can corrupt your page if called at the wrong time, and alert() is intrusive to users. In this course, JavaScript will always update elements on the page — not write directly to the document or pop up dialog boxes.

The Browser Console — Your Testing Playground

Before you write a single line of code that changes your page, you need to know about the browser console. You were introduced to DevTools in the Validating and Debugging lesson. The console is not just for finding errors — it is a live environment where you can test JavaScript immediately, without touching any files.

Open the console right now: press F12 on Windows (or Option + Command + J on Mac), then click the Console tab. Type the following line and press Enter:

document.querySelector('h1').textContent = 'It works!'

If the page has an <h1>, it just changed. That is JavaScript modifying the DOM — live, in real time, with no setup required. The console is your scratchpad. Use it constantly as you learn.

In your scripts, console.log() is your most important debugging tool. Think of it as leaving a sticky note for yourself while you build — messages appear in the console, invisible to your users, helpful to you:

console.log('Page loaded!');             // A basic message
console.log('Current color:', color);    // A message plus a value
console.warn('Double-check this');       // A yellow warning
console.error('Something went wrong');   // A red error message
console.table([{ name: 'Ada' }, { name: 'Grace' }]); // A formatted table

[IMG - Browser console panel showing log, warn, error, and table output side by side]

TRY IT YOURSELF
Open the console on any webpage. Type console.log('Hello from the console!') and press Enter. Then try document.body.style.backgroundColor = 'coral'. The page background just changed — and you can refresh to undo it. The console is completely safe to experiment in.

JavaScript Syntax Basics

Before exploring what JavaScript can do, a few quick rules will save you from common frustration. These are not complicated — but skipping them causes mysterious errors.

Statements and Semicolons

A statement is a single instruction — one thing JavaScript should do. Each statement typically lives on its own line and ends with a semicolon:

const name = 'Ada Lovelace';
document.querySelector('h1').textContent = name;
console.log(name);

Semicolons are not strictly required if each statement is on its own line, but they prevent subtle bugs and make your intent unambiguous. Make it a habit.

Comments

Comments are notes inside your code that the browser ignores entirely. They are essential for explaining what your code does — for others reading it and for your future self:

// This is a single-line comment

/* This is a multi-line comment.
   Use it when you need more space to explain something. */

const price = 9.99; // You can also add a comment at the end of a line

Case Sensitivity and Naming

JavaScript is case-sensitive. myVariable, MyVariable, and MYVARIABLE are three completely separate things. By convention, JavaScript uses camelCase for variable and function names: the first word is lowercase, and each additional word starts with a capital letter.

  • ►Variable names can contain letters, numbers, underscores, and dollar signs
  • ►Names must begin with a letter, underscore, or dollar sign — not a number
  • ►Names cannot be reserved words: const, let, if, for, function, etc.
  • ►Make names meaningful: accentColor is far better than c

Variables: const and let

A variable is a named container that holds a value. Imagine a row of labeled jars in a kitchen: you can see what each one contains without opening it, replace the contents without getting a new jar, and use the label in a recipe instead of describing the contents every time.

const accentColor = '#6c5ce7';  // A jar labeled "accentColor" holding a color value
const userName = 'Maria';        // A jar labeled "userName" holding a name

Modern JavaScript has two keywords for creating variables: const and let.

const — For Values That Won’t Change

const declares a variable whose value stays fixed. This is your default choice for almost everything:

const siteName = 'Portfolio';
const primaryColor = '#2d3436';
const maxScore = 100;

If you try to reassign a const, JavaScript throws an error. This is actually a feature, not a limitation — it protects you from accidentally overwriting something important.

let — For Values That Will Change

let declares a variable you expect to update as the script runs — a score counter, a current selection, an accumulating total:

let score = 0;
score = score + 10;      // Works fine with let

let activeSection = 'home';
activeSection = 'about'; // Can be updated at any time

NOTE
You will encounter a third keyword, var, in older JavaScript code and tutorials. It works, but it has subtle and confusing behaviors that const and let fix. In this course, always use const and let. If you see var in legacy code, just know it is the older approach and treat it mentally like let.

A practical rule: start every variable with const. If you later discover you need to reassign it, change it to let. This keeps your code safer and easier to read.

The Three Data Types You’ll Use Most

Every value in JavaScript has a type — a category describing what kind of data it is. Three types cover the vast majority of everyday JavaScript work:

Strings

A string is any text. Wrap it in single quotes, double quotes, or backtick characters:

const headline = 'Introduction to JavaScript';
const author = "Grace Hopper";
const message = `Hello, world!`;

All three produce a string. Most developers pick one style and stay consistent. In this course, single quotes for standard strings and backticks for template literals (explained next).

Numbers

Numbers do not need quotes — just write them directly. They can be whole numbers or decimals:

const age = 28;
const price = 49.99;
const temperature = -5;
const taxRate = 0.08;

Numbers can be used in math expressions directly: price * (1 + taxRate) gives the price with tax applied.

Booleans

A boolean is a light switch: it is either true or false, with no other options:

const isLoggedIn = true;
const hasSubmittedForm = false;
const isDarkMode = true;

You will use booleans constantly to make decisions: Is the menu open? Did the form validate? Is the user authorized?

NOTE
You will also encounter two special values: null means “intentionally empty” (like a cleared field), and undefined means “this variable was declared but never given a value.” You do not need to create these yourself right now — just recognize them when they appear in the console.

Template Literals

Template literals are strings that embed variable values directly inside the text. They use backtick characters instead of quotes, and any expression wrapped in ${} gets evaluated and inserted:

const name = 'Sofia';
const role = 'UX Designer';

const message = `Hello, ${name}! Your role is ${role}.`;
// Result: "Hello, Sofia! Your role is UX Designer."

This is far cleaner than the older approach of concatenating strings with + signs. Template literals also preserve line breaks, making them ideal for building multi-line HTML strings:

const card = `
  <div class="card">
    <h3>${name}</h3>
    <p class="role">${role}</p>
  </div>
`;

If you have worked with CSS custom properties, the ${} syntax will feel intuitive — it is the same idea of “insert a value here.” You will use template literals constantly when you generate card grids from data.

Math: Random Numbers and More

JavaScript has a built-in Math object that provides mathematical functions. Two are especially useful for creating visual effects:

Math.random()

Math.random() returns a random decimal number from 0 (inclusive) up to, but never reaching, 1:

Math.random()  // Might return: 0.4823, 0.0042, 0.9997, 0.3141...

Math.floor()

Math.floor() rounds a number down to the nearest whole number, discarding everything after the decimal:

Math.floor(4.9)  // Returns 4
Math.floor(4.1)  // Returns 4
Math.floor(0.9)  // Returns 0

Getting a Random Integer in a Range

By combining the two, you can get a random whole number within any range. To get a random number between 0 and 359 (perfect for HSL color hue values):

const randomHue = Math.floor(Math.random() * 360);
// Returns a random integer: 0, 1, 2 ... 357, 358, or 359

To get a number between 1 and 6 (a dice roll), multiply by 6 and add 1:

const roll = Math.floor(Math.random() * 6) + 1;
// Returns 1, 2, 3, 4, 5, or 6

TRY IT YOURSELF
Open the browser console and type Math.floor(Math.random() * 360), then press Enter several times. Notice you get a different random integer between 0 and 359 each time. This is the technique we will use shortly to generate random color themes.

The DOM: Your Page as a Data Structure

When a browser loads an HTML page, it does not just display it — it builds a model of the entire document in memory. This model is called the Document Object Model, or DOM. Think of it as the browser’s internal blueprint of your HTML, organized as a tree of connected objects, where every element, attribute, and piece of text is a node that JavaScript can find and modify.

[IMG - DOM tree diagram showing a webpage HTML as a tree of connected nodes]

Given this HTML:

<div class="card">
  <h3 id="title">My First Card</h3>
  <p>A short description goes here.</p>
</div>

The browser creates a tree in memory: a div node with two children — an h3 node and a p node. JavaScript can reach into this tree, grab any node, and change it. The browser immediately re-renders what the user sees.

This is the fundamental pattern behind all of JavaScript in this course:

  1. Find an element in the DOM
  2. Change it (content, style, classes, or attributes)
  3. The page updates instantly

Selecting Elements with querySelector

To change an element, you first need to find it. The primary tool is document.querySelector(). The argument it takes is a CSS selector — the exact same syntax you have been writing in your stylesheets for seventeen lessons:

document.querySelector('.card')        // First element with class "card"
document.querySelector('#title')       // Element with id "title"
document.querySelector('h2')           // First <h2> element on the page
document.querySelector('.nav li')      // First <li> inside an element with class "nav"
document.querySelector('[data-id]')    // First element that has a data-id attribute

This is not a coincidence — querySelector() was designed to accept CSS selectors so that your stylesheet knowledge transfers directly to JavaScript. If you can write it in CSS, you can use it here.

querySelector() returns the first matching element. To get all matching elements, use querySelectorAll():

document.querySelectorAll('.card')  // Returns every element with class "card"

Always store the result in a variable so you do not search the DOM repeatedly for the same element:

const heading = document.querySelector('h1');
const allCards = document.querySelectorAll('.card');

Changing Content: textContent and innerHTML

Once you have an element, you can change what is inside it. Two properties handle this, and choosing the right one matters.

textContent

textContent sets or reads the plain text inside an element. Any HTML tags in the string are treated as literal characters — they will appear on screen, not be parsed:

const heading = document.querySelector('h1');
heading.textContent = 'Welcome to JavaScript';
// The h1 now displays: Welcome to JavaScript

Use textContent whenever you are displaying text that comes from user input or external data. It is the safe option because it cannot accidentally execute HTML.

innerHTML

innerHTML sets or reads the HTML inside an element. Tags in the string are parsed and rendered as actual HTML elements:

const container = document.querySelector('.card-grid');
container.innerHTML = '<div class="card"><h3>New Card</h3></div>';
// A real styled card now exists in the DOM

Use innerHTML when you are building HTML yourself — for example, rendering a grid of cards from a data array. You will use this pattern heavily when building data-driven card grids.

CAUTION
Never use innerHTML to display content that came directly from a user — such as a form field value or a URL parameter. If a user types HTML or JavaScript into a field and you inject it with innerHTML, that code will execute in the browser. This is a serious security vulnerability called Cross-Site Scripting (XSS). Always use textContent for user-provided content.

Changing Styles and Classes

This is where your CSS knowledge becomes your greatest asset. JavaScript can change an element’s visual appearance in two ways: directly through the style property, or — far more powerfully — by toggling CSS classes.

The style Property

Every element has a style property that lets you set inline styles from JavaScript. CSS property names with hyphens become camelCase: background-color becomes backgroundColor, font-size becomes fontSize:

const card = document.querySelector('.card');
card.style.backgroundColor = '#6c5ce7';
card.style.borderRadius = '12px';
card.style.padding = '1.5rem';

This works, but it writes inline styles that override your stylesheet and are difficult to undo. For most situations, toggling CSS classes is the better approach.

Setting CSS Custom Properties from JavaScript

There is a more powerful version: style.setProperty() can update CSS custom properties (CSS variables) directly. Because a single custom property can influence dozens of elements, changing it from JavaScript updates your entire theme at once:

// Update a CSS custom property on the root element
document.documentElement.style.setProperty('--accent', '#6c5ce7');
document.documentElement.style.setProperty('--bg', '#f5f5f5');

If your stylesheet uses var(--accent) on buttons, links, borders, and headings, that one line changes all of them simultaneously. Combine this with Math.random() and you can generate entirely new color themes on demand:

const hue = Math.floor(Math.random() * 360);
document.documentElement.style.setProperty('--hue', hue);
// The entire page shifts to a new color scheme

classList — The Bridge Between JavaScript and CSS

The most common and cleanest way to change visual appearance with JavaScript is by adding, removing, or toggling CSS classes. You have been writing those classes for seventeen lessons. The classList property is the switch that connects your CSS to user actions.

const card = document.querySelector('.card');

card.classList.add('featured');       // Adds the class "featured"
card.classList.remove('featured');    // Removes the class "featured"
card.classList.toggle('featured');    // Adds it if absent, removes it if present
card.classList.contains('featured');  // Returns true or false

Here is the insight that powers every dark mode toggle, accordion, modal, and navigation menu you will build: CSS defines what both states look like. JavaScript simply switches between them.

For example, your CSS might define two states for a card:

.card {
  transform: scale(1);
  box-shadow: 0 2px 8px rgba(0,0,0,0.1);
  transition: transform 0.3s, box-shadow 0.3s;
}

.card.featured {
  transform: scale(1.05);
  box-shadow: 0 8px 24px rgba(108, 92, 231, 0.3);
  border-color: #6c5ce7;
}

And your JavaScript is a single line:

card.classList.toggle('featured');

One line of JavaScript. The CSS transition handles all the animation automatically. JavaScript toggles the class — CSS animates the change. This principle is the foundation of every interactive pattern in this course.

Creating and Removing Elements

You have been selecting elements that already exist in your HTML. JavaScript can also create entirely new elements and add them to the page, or remove existing ones, with no page reload required.

Creating an Element

The process has four steps: create, configure, optionally set attributes, then append to the page:

// Step 1: Create a new empty element
const newCard = document.createElement('div');

// Step 2: Configure it
newCard.classList.add('card');
newCard.textContent = 'A New Card';

// Step 3: Set attributes (optional)
newCard.setAttribute('data-id', '42');

// Step 4: Append it to a container that already exists in the HTML
const grid = document.querySelector('.card-grid');
grid.appendChild(newCard);

The new element appears at the end of the container, exactly as if you had typed it in the HTML file.

Removing an Element

Removing is even simpler — call .remove() directly on the element:

const card = document.querySelector('.card');
card.remove();  // Element is gone from the DOM immediately

data-* Attributes and dataset

You have used data-* attributes in your HTML to store extra information on elements. JavaScript reads these through the dataset property:

<!-- HTML -->
<div class="swatch" data-color="#6c5ce7" data-name="Purple"></div>
<div class="swatch" data-color="#00b894" data-name="Teal"></div>
<div class="swatch" data-color="#e17055" data-name="Coral"></div>
// JavaScript
const swatch = document.querySelector('.swatch');
console.log(swatch.dataset.color);  // "#6c5ce7"
console.log(swatch.dataset.name);   // "Purple"

The rule: data-color becomes dataset.color and data-name becomes dataset.name. Hyphens in the attribute name convert to camelCase in JavaScript: data-background-color becomes dataset.backgroundColor.

You can also set data-* attributes from JavaScript:

swatch.dataset.selected = 'true';
// Adds data-selected="true" to the element in the DOM

This is how JavaScript tracks state without cluttering class names or creating hidden inputs. Class names are for styling. data-* attributes carry data — that is their purpose, and JavaScript reads them naturally.

Best Practices for Modern JavaScript

As you begin writing JavaScript, a few habits will save you considerable frustration and produce code you are proud of:

  • ►Use const by default. Switch to let only when you need to reassign. Never use var.
  • ►Always use === for comparisons, never ==. The double-equals performs type coercion and produces unexpected results. Triple-equals compares both value and type, which is always what you want.
  • ►Store element selections in variables. Call document.querySelector() once and store the result: const btn = document.querySelector('.btn'). Calling it multiple times for the same element wastes effort.
  • ►Use console.log() constantly. Log values after you set them, log what a variable contains, log what an event returns. It is your best window into what JavaScript is actually doing.
  • ►Name things clearly. submitButton is infinitely better than btn2. accentColor is better than c. Code is communication — write it so a human can read it.
  • ►Add semicolons at the end of statements. They are not strictly required but prevent subtle bugs and signal where each statement ends.
  • ►Use the defer attribute when linking external JavaScript: <script defer src="script.js"></script>.
  • ►Write comments as you go. A comment on a non-obvious line now takes five seconds. Figuring out what that line does two weeks from now takes five minutes.
