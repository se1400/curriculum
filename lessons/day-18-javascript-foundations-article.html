JavaScript connects HTML and CSS to real interactivity: linking a script file to an HTML page, writing console.log() calls, responding to button clicks, and manipulating the DOM. This lesson goes deeper into the foundations that underpin all of that.

Every program you will ever write — in JavaScript or any other language — revolves around data. You receive data from users, fetch it from servers, transform it, combine it, validate it, and display it back. Variables in the Browser

Before diving into theory, let's see what variables are actually for. Every variable you learn to declare is a tool for controlling what users see on the page.

LISTING 18.0 — Variables Controlling What the User Sees

const greeting = "Hello";
const userName = "Alex";
const fullMessage = greeting + ", " + userName + "!";

document.querySelector('h1').textContent = fullMessage;
// The h1 on the page now reads: "Hello, Alex!"

const buttonColor = "royalblue";
document.querySelector('button').style.backgroundColor = buttonColor;
// The button is now royal blue

Everything in this lesson — data types, variables, operators — is building toward this: controlling what appears on screen. Keep that goal in mind as you read.

Variables: let, const, and var

A variable is a named container for a value. You create a variable by declaring it with a keyword. JavaScript offers three declaration keywords: let, const, and var. Two of them are modern and useful; one is legacy and should be avoided.

let — Block-Scoped and Reassignable

let declares a variable that can be reassigned after its initial value is set. It is block-scoped, meaning the variable only exists within the nearest enclosing pair of curly braces { } — an if block, a for loop, a function body. Outside that block, the variable does not exist.

LISTING 18.1 — let — Reassignable and Block-Scoped

let score = 0;
score = 10;       // reassignment is fine
score = score + 5; // now 15

if (true) {
  let blockVar = "I only exist inside this if block";
  console.log(blockVar); // "I only exist inside this if block"
}

console.log(blockVar); // ReferenceError: blockVar is not defined

Use let when you know the value will need to change — a counter that increments, a running total that accumulates, a flag that toggles.

const — Block-Scoped and Non-Reassignable

const declares a variable whose binding cannot be reassigned. Once you assign a value to a const, you cannot point that name at a different value.

LISTING 18.2 — const — Cannot Be Reassigned

const TAX_RATE = 0.08;
TAX_RATE = 0.09; // TypeError: Assignment to constant variable.

const siteName = "LearnCode";
siteName = "OtherSite"; // TypeError — not allowed

There is an important nuance that trips up beginners: const prevents reassignment of the variable binding, but it does not make the value itself immutable. If the value is an object or an array, the contents of that object or array can still be changed.

LISTING 18.3 — const with Objects and Arrays (Contents Can Change)

const user = { name: "Alex", age: 25 };
user.age = 26;        // fine — modifying a property
user.email = "a@b.c"; // fine — adding a property
console.log(user);    // { name: "Alex", age: 26, email: "a@b.c" }

user = {};            // TypeError — cannot reassign the binding

const colors = ["red", "green"];
colors.push("blue");  // fine — modifying the array
console.log(colors);  // ["red", "green", "blue"]

colors = ["yellow"];  // TypeError — cannot reassign the binding

NOTE: Prefer const by default. Declare every variable with const first. If you find yourself needing to reassign it, change it to let. This habit makes code easier to read — readers know a const binding will never point to a different value, reducing what they need to track mentally.

var — The Old Way (Avoid It)

var was the only way to declare variables in JavaScript before ES6 (2015). It has two behaviors that make it error-prone and are not found in let or const:

  • Function-scoped, not block-scoped. A var declared inside an if block or for loop leaks out into the enclosing function scope — or the global scope if there is no function.
  • Hoisted and initialized to undefined. The declaration is moved to the top of its scope, meaning you can reference a var variable before its declaration line without getting an error — you simply get undefined. This silently masks bugs.

LISTING 18.4 — var Scope Leak — A Classic Bug

for (var i = 0; i < 3; i++) {
  // i is being declared with var
}
console.log(i); // 3 — i leaked out of the for loop!

// With let, this is a ReferenceError:
for (let j = 0; j < 3; j++) { }
console.log(j); // ReferenceError: j is not defined

LISTING 18.5 — var Leaks Out of Blocks

// var leaks out of an if block
if (true) {
  var leaked = "I escaped the block!";
}
console.log(leaked); // "I escaped the block!" — leaked into the outer scope

// let stays contained inside the block
if (true) {
  let contained = "I stay inside";
}
console.log(contained); // ReferenceError: contained is not defined

// Same issue in for loops — var i leaks out after the loop ends
for (var i = 0; i < 3; i++) { }
console.log(i); // 3 — leaked out

for (let j = 0; j < 3; j++) { }
console.log(j); // ReferenceError: j is not defined

Hoisting and the Temporal Dead Zone (TDZ)

JavaScript processes declarations before executing code. Both let and const are hoisted (the engine is aware of them), but they are not initialized until their declaration line is reached. The gap between the start of the block and the declaration line is called the Temporal Dead Zone — any access during this window throws a ReferenceError.

LISTING 18.6 — The Temporal Dead Zone

// ReferenceError: Cannot access 'color' before initialization
console.log(color);
let color = "blue"; // TDZ ends here
console.log(color); // "blue"

The practical rule: always declare variables at the top of their block. This makes the TDZ a non-issue and keeps your code predictable.

The Five Primitive Data Types

JavaScript has seven primitive data types. Five of them are the ones you will use constantly: string, number, boolean, null, and undefined. Two newer ones — BigInt and Symbol — cover specialized needs. All primitives are immutable: the values themselves cannot be changed, though variables can be reassigned to hold different values.

string

A sequence of characters enclosed in quotes. Strings represent text. JavaScript has three ways to delimit strings: single quotes ('), double quotes ("), and backticks (`). The first two are interchangeable; backtick strings (template literals) have special capabilities covered in depth below.

number

JavaScript has a single numeric type that represents both integers and floating-point numbers. There is no separate int or float. The details — including floating-point precision, NaN, Infinity, and safe integer limits — are covered in depth below.

boolean

A boolean has exactly two possible values: true and false. Booleans are the result of comparisons and are used as the condition in if statements, while loops, and ternary expressions.

LISTING 18.7 — Boolean Values

const isLoggedIn = true;
const hasError   = false;

const age  = 20;
const isAdult = age >= 18; // true
const hasTicket = false;

if (isAdult && hasTicket) {
  console.log("Welcome!");
} else {
  console.log("Sorry, you cannot enter.");
}

null

null represents the intentional absence of a value. It is a deliberate, programmer-assigned empty slot. You use null when you want to explicitly say "this variable has no value right now, and that is intentional."

LISTING 18.8 — null — Intentional Absence

let currentUser = null;  // No user is logged in — intentionally empty

// Later, after login:
currentUser = { id: 42, name: "Taylor" };

// An API result that returned no match:
const searchResult = null;

undefined

undefined means a value has not yet been assigned. JavaScript assigns undefined automatically in several situations: a declared variable with no assignment, a function parameter that was not provided, and accessing a non-existent object property. Unlike null, you rarely assign undefined yourself — JavaScript uses it as a default.

LISTING 18.9 — undefined — Not Yet Assigned

let score;
console.log(score); // undefined — declared but not assigned

function greet(name) {
  console.log(name); // undefined if called as greet()
}
greet();

const obj = { a: 1 };
console.log(obj.b); // undefined — property does not exist

// Convention: use null for intentional absence, not undefined
// Let JavaScript use undefined on its own; assign null yourself
NOTE: BigInt (42n) represents integers larger than JavaScript can handle with normal numbers — needed only in cryptography or large ID processing, rarely in everyday web development. Symbol is a primitive for creating unique identifiers primarily used inside JavaScript frameworks — you will not need to write Symbols yourself as a beginner.

The typeof Operator

The typeof operator returns a string describing the type of its operand. It is an invaluable debugging tool and is used frequently for type-checking in code that handles unknown input.

LISTING 18.10 — typeof on Every Type

console.log(typeof "hello");        // "string"
console.log(typeof 42);             // "number"
console.log(typeof 3.14);           // "number"
console.log(typeof true);           // "boolean"
console.log(typeof undefined);      // "undefined"
console.log(typeof null);           // "object"  ← famous bug!
console.log(typeof Symbol("x"));    // "symbol"
console.log(typeof 42n);            // "bigint"
console.log(typeof {});             // "object"
console.log(typeof []);             // "object"  ← arrays are objects
console.log(typeof function(){});   // "function"

[IMG - JavaScript typeof results in browser console]

Two results deserve special attention:

  • typeof null === "object" — This is a well-known bug in JavaScript that has existed since the language's first version in 1995. Fixing it would break too much existing code, so it will never be corrected. To check specifically for null, use value === null (strict equality), never typeof.
  • typeof [] === "object" — Arrays are a kind of object in JavaScript. To check whether something is an array, use Array.isArray(value).

LISTING 18.11 — Correct Type Checks for null and Arrays

const value = null;

// Wrong: typeof cannot distinguish null from objects
if (typeof value === "object") {
  console.log("object or null?"); // ambiguous!
}

// Correct: use strict equality for null checks
if (value === null) {
  console.log("definitely null");
}

// Checking for arrays:
const arr = [1, 2, 3];
console.log(typeof arr);           // "object" — not helpful
console.log(Array.isArray(arr));   // true — correct

// Combining checks: is it a non-null object?
function isObject(val) {
  return typeof val === "object" && val !== null;
}
console.log(isObject({}));    // true
console.log(isObject(null));  // false
console.log(isObject("hi"));  // false

Strings in Depth

Strings are arguably the most frequently used data type in web development. They represent user names, URLs, HTML content, error messages, labels — nearly every piece of data that touches a user interface passes through a string at some point.

Single Quotes and Double Quotes

Single and double quotes are completely interchangeable in JavaScript. The choice is purely stylistic — pick one and be consistent within a project. The main practical reason to choose one over the other is to avoid escaping: if your string contains single quotes, wrap it in double quotes (and vice versa).

LISTING 18.12 — Single and Double Quotes

const a = 'Hello, World!';
const b = "Hello, World!";

// Use double quotes when the string contains a single quote:
const message = "It's a great day!";

// Use single quotes when the string contains a double quote:
const html = '<a href="/">Home</a>';

// Escaping the alternative — works, but less readable:
const escaped = 'It\'s a great day!';
const escaped2 = "She said \"hello\"";

Template Literals (Backticks)

Template literals, introduced in ES6, are strings delimited by backtick characters (`). They provide two capabilities that single and double quote strings lack:

  • String interpolation — embed any JavaScript expression directly in the string using ${ expression }. The expression is evaluated and its result is converted to a string and inserted.
  • Multi-line strings — a template literal can span multiple lines without escape characters or concatenation.

LISTING 18.13 — Template Literals — Interpolation and Multi-line

const name    = "Jordan";
const points  = 1450;
const rank    = 3;

// Old concatenation style — awkward and error-prone:
const oldMsg = "Hello, " + name + "! You have " + points + " points and are ranked #" + rank + ".";

// Template literal — clean and readable:
const newMsg = `Hello, ${name}! You have ${points} points and are ranked #${rank}.`;

// Expressions inside ${ } are evaluated:
const price = 29.99;
const qty   = 3;
const receipt = `Subtotal: $${(price * qty).toFixed(2)}`;
console.log(receipt); // "Subtotal: $89.97"

// Multi-line string (preserves newlines and indentation):
const card = `
  <div class="card">
    <h2>${name}</h2>
    <p>Points: ${points}</p>
  </div>
`;

Escape Characters

Some characters cannot appear literally inside a string — the newline character, for example, would just break the line. Escape sequences solve this: a backslash followed by a special character code produces the intended character.

LISTING 18.14 — Common Escape Characters

console.log("Line one\nLine two");     // newline
console.log("Name:\tJordan");          // tab
console.log("C:\\Users\\Jordan");      // literal backslash
console.log("She said \"hello\"");     // double quote inside double-quoted string
console.log('It\'s working');          // single quote inside single-quoted string
console.log("\u2764");                 // Unicode character: ❤

// In template literals, \n is still valid but you can also just press Enter:
const multiline = `First line
Second line`;  // same result as "First line\nSecond line"

String Immutability

Strings are immutable in JavaScript. You cannot change individual characters in a string. String methods — toUpperCase(), slice(), replace(), and so on — do not modify the original string; they always return a new string.

LISTING 18.15 — String Immutability

const greeting = "hello";
const upper    = greeting.toUpperCase(); // returns a NEW string
console.log(greeting); // "hello" — unchanged
console.log(upper);    // "HELLO"

// You cannot change individual characters:
let word = "cat";
word[0] = "b"; // silently fails in non-strict mode
console.log(word); // "cat" — unchanged

// To "change" a string, reassign to a new value:
let city = "london";
city = city[0].toUpperCase() + city.slice(1);
console.log(city); // "London"

Numbers in Depth

JavaScript has exactly one numeric type. Whether you write 42 or 3.14, the value is stored as a 64-bit floating-point number following the IEEE 754 standard — the same format used by C's double. This design keeps the language simple (one type instead of several), but it introduces a few behaviors you must understand to avoid bugs.

Integers and Decimals

LISTING 18.16 — Integer and Decimal Numbers

const integer = 42;
const decimal = 3.14159;
const negative = -17;
const big = 1_000_000; // numeric separator (ES2021) — same as 1000000, more readable

console.log(typeof 42);    // "number"
console.log(typeof 3.14);  // "number" — same type!

The Floating-Point Precision Problem

Because all numbers are stored as 64-bit floats, some decimal fractions cannot be represented exactly in binary. This leads to a famous gotcha:

[IMG - Floating point 0.1 plus 0.2 result in console]

LISTING 18.17 — Floating-Point Precision and Number.EPSILON

console.log(0.1 + 0.2);          // 0.30000000000000004 — not 0.3!
console.log(0.1 + 0.2 === 0.3);  // false — do NOT compare floats with ===

// Safe float comparison using Number.EPSILON:
// Number.EPSILON is the smallest difference between two representable numbers
// ~2.22e-16

function floatsEqual(a, b) {
  return Math.abs(a - b) < Number.EPSILON;
}

console.log(floatsEqual(0.1 + 0.2, 0.3)); // true

// For display purposes: round to desired precision
const total = 0.1 + 0.2;
console.log(total.toFixed(2)); // "0.30" — string, rounded for display

Special Numeric Values

JavaScript's number type includes three special values that represent edge cases in arithmetic:

LISTING 18.18 — Infinity, -Infinity, and NaN

// Infinity and -Infinity
console.log(1 / 0);           // Infinity
console.log(-1 / 0);          // -Infinity
console.log(Infinity + 1);    // Infinity
console.log(typeof Infinity);  // "number"

// NaN — Not a Number
console.log(0 / 0);           // NaN
console.log("hello" * 2);     // NaN
console.log(Math.sqrt(-1));   // NaN
console.log(typeof NaN);       // "number" (another quirk!)

// NaN is the only value in JS not equal to itself:
console.log(NaN === NaN); // false — bizarre but true

// Checking for NaN:
console.log(isNaN("hello"));        // true — but MISLEADING (coerces first)
console.log(isNaN(undefined));      // true — undefined coerces to NaN
console.log(Number.isNaN("hello")); // false — not actually NaN
console.log(Number.isNaN(NaN));     // true — correct!
console.log(Number.isNaN(undefined)); // false — correct

Always use Number.isNaN() instead of the global isNaN(). The global function converts its argument to a number first, so isNaN("hello") returns true because Number("hello") is NaN — even though the string "hello" is not NaN. Number.isNaN() does no conversion and only returns true for the actual NaN value.

The Math Object

JavaScript's built-in Math object provides mathematical constants and functions. Unlike most objects, you do not create instances of Math — you call its methods directly as Math.method().

LISTING 18.20 — Math Rounding Methods

console.log(Math.round(4.4));  // 4 — rounds to nearest integer
console.log(Math.round(4.5));  // 5 — .5 rounds up
console.log(Math.floor(4.9));  // 4 — always rounds DOWN (toward -Infinity)
console.log(Math.ceil(4.1));   // 5 — always rounds UP (toward +Infinity)
console.log(Math.trunc(4.9));  // 4 — removes decimal part (toward 0)
console.log(Math.trunc(-4.9)); // -4 — note: different from floor for negatives!
console.log(Math.floor(-4.1)); // -5 — floor goes toward -Infinity

LISTING 18.21 — Math.random() and Random Integer Patterns

// Math.random() returns a float in [0, 1) — 0 is possible, 1 is never returned
console.log(Math.random()); // e.g., 0.7341829...

// Random integer from 0 to max-1:
function randomInt(max) {
  return Math.floor(Math.random() * max);
}
console.log(randomInt(6)); // 0, 1, 2, 3, 4, or 5 (like a die minus one)

// Random integer from min to max (inclusive):
function randomRange(min, max) {
  return Math.floor(Math.random() * (max - min + 1)) + min;
}
console.log(randomRange(1, 6));   // 1, 2, 3, 4, 5, or 6 (like a die)
console.log(randomRange(10, 20)); // integer between 10 and 20 inclusive

LISTING 18.22 — Other Essential Math Methods

// Min and max
console.log(Math.min(3, 1, 4, 1, 5));  // 1
console.log(Math.max(3, 1, 4, 1, 5));  // 5
console.log(Math.min(...[10, 5, 20])); // 5 — spread array into arguments

// Absolute value, power, square root
console.log(Math.abs(-42));      // 42
console.log(Math.pow(2, 10));    // 1024 (2 to the 10th power)
console.log(2 ** 10);            // 1024 — ** operator does the same thing
console.log(Math.sqrt(144));     // 12
console.log(Math.cbrt(27));      // 3 — cube root

// Constants
console.log(Math.PI);   // 3.141592653589793
console.log(Math.E);    // 2.718281828459045 (Euler's number)
console.log(Math.LN2);  // 0.6931471805599453 (natural log of 2)

Truthy and Falsy Values

JavaScript conditionals do not require a strict boolean. Any value placed in a boolean context — the condition of an if statement, a while loop, a ternary, or the operand of ! — is implicitly converted to either true or false. Values that convert to true are called truthy; values that convert to false are called falsy.

There are exactly six falsy values in JavaScript:

  • false — the boolean false
  • 0 — the number zero (also -0 and 0n BigInt zero)
  • "" — an empty string (single quotes, double quotes, or backticks — all empty)
  • null — intentional absence of value
  • undefined — unassigned value
  • NaN — Not a Number

Every other value is truthy — including 0's neighbors, non-empty strings, all objects, and all arrays (even empty ones).

[IMG - Truthy and falsy values comparison table]

LISTING 18.23 — Falsy Values in if Statements

// All six falsy values:
if (false)     { } // skipped
if (0)         { } // skipped
if ("")        { } // skipped
if (null)      { } // skipped
if (undefined) { } // skipped
if (NaN)       { } // skipped

// Truthy — these blocks DO execute:
if (1)         { console.log("truthy"); }
if ("hello")   { console.log("truthy"); }
if ([])        { console.log("truthy — empty array is truthy!"); }
if ({})        { console.log("truthy — empty object is truthy!"); }
if (-1)        { console.log("truthy — negative numbers too"); }
if ("false")   { console.log("truthy — the STRING 'false' is not the boolean false!"); }

LISTING 18.24 — Practical Truthy/Falsy Patterns

// Guard pattern: only proceed if value exists and is non-empty
function processName(name) {
  if (!name) {
    return "Name is required";
  }
  return `Hello, ${name}!`;
}
console.log(processName(""));        // "Name is required"
console.log(processName(null));      // "Name is required"
console.log(processName(undefined)); // "Name is required"
console.log(processName("Sam"));     // "Hello, Sam!"

// Default value with || (OR):
const username = userInput || "Guest";
// If userInput is falsy (empty string, null, undefined), use "Guest"

// Nullish coalescing ?? — only falls back for null/undefined (not 0 or ""):
const timeout = userTimeout ?? 5000;
// ?? is safer than || when 0 or "" are valid values

NOTE: The empty array [] and the empty object {} are both truthy. This is a common source of confusion. To check if an array is empty, use array.length === 0. To check if an object has no properties, use Object.keys(obj).length === 0.

Loose vs. Strict Equality

JavaScript provides two equality operators: == (loose equality, also called abstract equality) and === (strict equality, also called identity). They look similar but behave very differently.

Loose Equality: ==

The == operator performs type coercion before comparing. If the two operands are of different types, JavaScript converts one or both to a common type first, then compares. This produces results that are impossible to memorize reliably — and causes real bugs when you work with DOM values.

LISTING 18.25 — == Bugs with DOM Input Values

// Bug 1: user input is always a string — == silently coerces it
const input = document.querySelector('#age').value; // "25" not 25

if (input == 25) {
  document.querySelector('#result').textContent = "Old enough"; // passes! (coercion)
}
if (input === 25) {
  document.querySelector('#result').textContent = "Old enough"; // fails correctly
}
// Rule: always use === when comparing form values to numbers.
// Or convert first: if (Number(input) === 25)

// Bug 2: empty string is falsy, so == false passes when input is ""
if (input == false) {
  document.querySelector('#error').textContent = "Please enter your age";
  // Fires when input is "" OR "0" OR "false" — all coerce the same way
}
if (!input) {
  document.querySelector('#error').textContent = "Please enter your age";
  // Clearer intent: falsy check, catches empty string
}

// Bug 3: a checkbox value is the string "on" or undefined, not true/false
const checked = document.querySelector('#terms').value; // "on" not true
if (checked == true) { ... }  // never passes — "on" does not coerce to true
if (checked === "on") { ... } // correct

Strict Equality: ===

The === operator never coerces types. If the two operands have different types, the result is immediately false. If they have the same type, it compares their values directly.

LISTING 18.26 — Strict Equality — Predictable Results

console.log(0 === false);       // false — different types
console.log("" === false);      // false — different types
console.log("" === 0);          // false — different types
console.log(null === undefined); // false — different types
console.log("1" === 1);         // false — different types
console.log(1 === 1);           // true
console.log("hello" === "hello"); // true
console.log(null === null);     // true
console.log(undefined === undefined); // true

// Use !== for not-equal:
console.log(1 !== 2);   // true
console.log(1 !== "1"); // true — strict, different types

NOTE: Always use === and !==. Never use == or != in modern JavaScript. The only exception some developers make is value == null to check for both null and undefined simultaneously — but even then, the nullish coalescing operator ?? and optional chaining ?. are clearer alternatives.

Type Conversion

Type conversion is the process of changing a value from one data type to another. JavaScript does this in two ways: implicitly (coercion, done automatically by the engine) and explicitly (conversion, done intentionally by the programmer).

Implicit Coercion

JavaScript coerces types automatically in many operations. The rules are not always intuitive, which is why coercion is a frequent source of bugs for developers who have not studied it carefully.

LISTING 18.27 — Implicit Type Coercion

// The + operator with a string: coerces to string (concatenation wins)
console.log("5" + 3);    // "53" — 3 is coerced to "3", then concatenated
console.log("5" + true); // "5true"
console.log("" + null);  // "null"
console.log("" + undefined); // "undefined"

// Arithmetic operators other than +: coerce to number
console.log("5" - 3);    // 2 — "5" is coerced to 5
console.log("5" * "2");  // 10 — both coerced to numbers
console.log("5" / "2");  // 2.5
console.log(true + true); // 2 — true is 1, false is 0
console.log(false + 1);   // 1

// Unary + coerces to number:
console.log(+"5");         // 5 — string to number
console.log(+true);        // 1
console.log(+false);       // 0
console.log(+null);        // 0
console.log(+undefined);   // NaN
console.log(+"hello");     // NaN

Explicit Conversion

Explicit conversion is always preferable to relying on coercion, because the intent is clear and the behavior is predictable.

LISTING 18.28 — Explicit Type Conversion

// To Number:
console.log(Number("42"));       // 42
console.log(Number("3.14"));     // 3.14
console.log(Number("42px"));     // NaN — can't convert "42px" to a number
console.log(Number(""));         // 0 — empty string to 0
console.log(Number(true));       // 1
console.log(Number(false));      // 0
console.log(Number(null));       // 0
console.log(Number(undefined));  // NaN

// parseInt and parseFloat — more lenient than Number():
console.log(parseInt("42px"));     // 42 — reads integer up to non-numeric char
console.log(parseInt("3.9"));      // 3 — truncates decimal (does not round)
console.log(parseFloat("3.14em")); // 3.14 — reads decimal up to non-numeric char
console.log(parseInt("px42"));     // NaN — must start with a digit

// To String:
console.log(String(42));        // "42"
console.log(String(true));      // "true"
console.log(String(false));     // "false"
console.log(String(null));      // "null"
console.log(String(undefined)); // "undefined"
console.log(String(NaN));       // "NaN"

// To Boolean:
console.log(Boolean(1));         // true
console.log(Boolean(0));         // false
console.log(Boolean("hello"));   // true
console.log(Boolean(""));        // false
console.log(Boolean(null));      // false
console.log(Boolean(undefined)); // false
console.log(Boolean([]));        // true — empty array is truthy!
console.log(Boolean({}));        // true — empty object is truthy!

Summary

This lesson covered the foundational layer of JavaScript: how to store data and what kinds of data the language understands.

You learned to declare variables with const (the default choice), let (when reassignment is needed), and why var should be avoided — because it is function-scoped rather than block-scoped and is hoisted in a way that silently masks bugs. Both let and const experience the Temporal Dead Zone, which produces a clear ReferenceError if you access them before their declaration line, preventing the silent-undefined behavior of var.

JavaScript has seven primitive types. The five you will use daily are string, number, boolean, null, and undefined. Two modern additions are BigInt (for integers beyond Number.MAX_SAFE_INTEGER) and Symbol (for guaranteed-unique property keys). The typeof operator identifies types at runtime, with two famous quirks: typeof null === "object" (a historical bug) and typeof [] === "object" (because arrays are objects).

Strings are immutable sequences of characters that can be written with single quotes, double quotes, or backtick template literals. Template literals are the modern choice, supporting ${expression} interpolation and multi-line strings. JavaScript numbers are all 64-bit floating-point values, which means integer arithmetic is exact only up to Number.MAX_SAFE_INTEGER and decimal arithmetic has precision limits — compare floats using Number.EPSILON, never with ===. Use Number.isNaN() rather than global isNaN(). The Math object provides rounding, randomness, and mathematical functions.

Six values are falsy: false, 0, "", null, undefined, and NaN. Everything else is truthy — including empty arrays and empty objects. Always use strict equality === and !==, never the loose operators == and !=, which perform type coercion and produce deeply counterintuitive results. When you need to convert types explicitly, prefer Number(), String(), Boolean(), parseInt(), and parseFloat().

Q&A

Q. What is the difference between null and undefined? When should I use each?

A. Both represent the absence of a meaningful value, but they communicate different things. undefined is JavaScript's way of saying "this has not been given a value yet" — it is the default state of a declared-but-unassigned variable, a missing function argument, or a non-existent object property. You rarely assign undefined yourself. null is the programmer's way of saying "I am intentionally setting this to empty" — for example, let currentUser = null clearly signals that no user is logged in and this is deliberate. Think of undefined as "not yet" and null as "intentionally nothing."

Q. Why does typeof null return "object"? Is null an object?

A. No, null is not an object — it is a primitive. The "object" result is a bug from JavaScript's very first implementation in 1995. In the original engine, values were stored with a type tag in the low bits of a 32-bit word. The type tag for objects was 000. The value null was represented as all zeros — a null pointer — which happened to have the 000 tag, so typeof misidentified it as an object. The TC39 committee proposed fixing this in ES6, but the change was rejected because it would break existing code that relies on the behavior. It remains a permanent, unfixable quirk of the language. Always check for null using value === null.

Q. If const prevents reassignment, why can I still push items into a const array?

A. Because const locks the variable binding, not the value itself. When you write const arr = [1, 2, 3], the variable arr holds a reference — a memory address pointing to the array. const prevents you from changing that memory address (you cannot do arr = []). But the array at that address is still a mutable object. Calling arr.push(4) modifies the array in place at the same memory address — the binding does not change, so const has no objection. If you need a truly immutable array or object, use Object.freeze() (shallow only) or reach for a library like Immer.

Q. When should I use parseInt() vs Number() to convert a string to a number?

A. Use parseInt() when the string contains a number followed by non-numeric characters and you want to extract the leading integer — for example, parsing a CSS value like "42px" yields 42. Always pass 10 as the second argument to ensure decimal interpretation: parseInt("42px", 10). Use Number() when you want strict conversion: if the string contains anything that is not a valid number, Number() returns NaN immediately, which is often what you want for form validation. Use parseFloat() the same way as parseInt() but for decimals: parseFloat("3.14em") returns 3.14.

Q. What is the Temporal Dead Zone, and does it matter in practice?

A. The Temporal Dead Zone (TDZ) is the period between the start of a block scope and the line where a let or const variable is declared. During this window, the variable exists (the engine is aware of it because it was hoisted) but is not yet initialized. Any attempt to access it throws a ReferenceError. In practice, the TDZ matters most in two situations: when you accidentally reference a variable before its declaration (the error helps you find the bug quickly) and in class field initializers and function bodies where declaration order can be subtle. The TDZ is a feature, not a flaw — it catches a class of bugs that var's silent undefined hoisting would hide.
