HTML Forms and Form Accessibility

What You'll Learn in This Lesson:

  • How the <form> element works and when to use GET vs. POST
  • Why pairing every input with a label is critical for usability and accessibility
  • All major HTML input types — text, email, password, date, checkbox, radio, and more
  • How to group related inputs with <fieldset> and <legend>
  • How to add autocomplete suggestions with <datalist>
  • How to use HTML5 built-in validation to catch errors before submission
  • How CSS pseudo-classes let you style fields based on their state
  • Core accessibility techniques: ARIA attributes and keyboard navigation

Every website you've used that collected information from you — a sign-up page, a search box, a checkout form — did so with an HTML form. Forms are how users send data back to a server. Without them, the web would be a one-way street: you could read content, but you couldn't interact.

In this lesson, you'll build a comprehensive understanding of HTML forms, from the basic <form> element all the way to HTML5 validation and accessibility best practices. By the end, you'll be able to create forms that work correctly, look professional, and are usable by everyone — including people who rely on keyboard navigation or screen readers.

The <form> Element

The <form> element is the container for all form controls. On its own, a <form> tag doesn't display anything visible, but it defines two critical pieces of information: where the data goes and how it gets there.

LISTING 13.1 — The basic form element with action and method

<form action="/submit" method="post">
  <!-- form controls go here -->
</form>

The action attribute specifies the URL where the form data will be sent when the user submits. If you omit action, the form submits to the current page's URL.

The method attribute tells the browser how to send the data. There are two choices:

  • method="get" — The form data is appended to the URL as a query string: example.com/search?q=hello. Use GET for searches and anything where the URL should be bookmarkable or shareable.
  • method="post" — The form data is sent in the request body, not the URL. Use POST for sensitive data (like passwords), large data payloads, or anything that creates or changes data on the server (like creating an account).

[IMG - Browser address bar with GET query string visible]

NOTE: The HTML5 specification adds other method values, but get and post are the only values you'll use in everyday web development.

Pairing Labels with Inputs

The single most important rule for accessible forms: every input must have a label.

A <label> element associates a text description with a form control. This matters for three reasons:

  1. Screen readers announce the label when a user focuses the input — without it, a blind user hears "text field" with no context about what to type.
  2. Larger click targets — clicking or tapping a label focuses its associated input, making the interactive area bigger. This is especially helpful on mobile.
  3. Better tooling support — autofill software, password managers, and browser autocomplete all use labels to understand what a field is for.

There are two ways to associate a label with an input.

Using for and id:

LISTING 13.2 — Associating a label using for and id

<label for="email">Email Address</label>
<input type="email" id="email" name="email">

The for attribute on the <label> must exactly match the id attribute on the <input>. They don't have to be adjacent in the HTML, but keeping them together is a good habit.

Wrapping the input inside the label:

LISTING 13.3 — Wrapping an input inside its label

<label>
  Email Address
  <input type="email" name="email">
</label>

When you wrap the input inside the label, the association is implicit — no for/id needed. Both approaches are valid. The wrapping technique is more concise; the for/id technique gives you more layout flexibility.

CAUTION: Never use the placeholder attribute as a substitute for a <label>. Placeholder text disappears when the user starts typing, leaving them no reminder of what the field is for. Use placeholder for example values like jane@example.com, not as the field's label.

Text-Type Inputs

All text-type inputs use <input> with a type attribute. The browser renders them as single-line text fields, but each type gives the browser information about what kind of data to expect — which unlocks virtual keyboard layouts on mobile, built-in validation, and browser autofill.

LISTING 13.4 — Common text-type inputs

<input type="text"     name="username"  placeholder="Your username">
<input type="email"    name="email"     placeholder="jane@example.com">
<input type="password" name="password"  placeholder="At least 8 characters">
<input type="url"      name="website"   placeholder="https://example.com">
<input type="tel"      name="phone"     placeholder="(555) 555-5555">
<input type="search"   name="q"         placeholder="Search...">
<input type="number"   name="quantity"  min="1" max="99">

Here's what each type does differently:

  • type="text" — The generic single-line text input. Use it when none of the more specific types apply.
  • type="email" — The browser validates that the value looks like an email address. On mobile, it shows a keyboard with @ and . prominently placed.
  • type="password" — Characters are masked (shown as dots or asterisks) for privacy. Password managers automatically recognize and fill this type.
  • type="url" — The browser validates the format matches a URL. Mobile keyboards show /, ., and http:// shortcuts.
  • type="tel" — Triggers a numeric phone keypad on mobile. The browser doesn't validate the phone format (there are too many international formats), so use pattern if you need validation.
  • type="search" — Functionally identical to type="text", but browsers may add a clear (×) button. Semantically signals this is a search field.
  • type="number" — Shows spinner arrows for incrementing and decrementing. Use min and max to set a valid range and step to control the increment size.

NOTE: The name attribute is required for form data to be submitted. The name becomes the key in the submitted data (for example, email=jane@example.com). An <input> without a name attribute is never included in the form submission.

Date and Time Inputs

HTML provides a set of inputs for capturing dates and times. The browser renders a native date/time picker — which looks different in each browser — saving you from having to build a calendar widget from scratch with JavaScript.

LISTING 13.5 — Date and time input types

<input type="date"           name="birthday">
<input type="time"           name="meeting-time">
<input type="datetime-local" name="appointment">
<input type="month"          name="billing-month">
<input type="week"           name="sprint-week">
  • type="date" — Picks a calendar date (year, month, day). The submitted value format is always YYYY-MM-DD.
  • type="time" — Picks a time in hours and minutes (optionally seconds). Value format: HH:MM.
  • type="datetime-local" — Combines a date and time picker in one control. Value format: YYYY-MM-DDTHH:MM.
  • type="month" — Picks just a year and month. Useful for credit card expiration dates.
  • type="week" — Picks a week number within a year (ISO week format).

You can set default values, minimum dates, and maximum dates using the value, min, and max attributes. The values must match the input type's expected format:

LISTING 13.6 — Date input with min, max, and a default value

<input type="date" name="travel-date"
       value="2025-06-01"
       min="2025-01-01"
       max="2025-12-31">

[IMG - Native date picker in Chrome, Firefox, and Safari side by side]

CAUTION: Date input appearance varies significantly across browsers and operating systems. For production projects that need consistent styling or more advanced date-range behavior, consider a JavaScript date picker library. For learning purposes and basic projects, native date inputs work well and require zero JavaScript.

Selection Inputs: Checkboxes, Radio Buttons, Range, and Color

Not all inputs collect text. These four types capture different kinds of choices or values.

Checkboxes let users select zero or more options from a set. Each checkbox is independent of the others.

LISTING 13.7 — Checkboxes for multiple selections

<label><input type="checkbox" name="toppings" value="cheese"> Cheese</label>
<label><input type="checkbox" name="toppings" value="peppers"> Peppers</label>
<label><input type="checkbox" name="toppings" value="mushrooms"> Mushrooms</label>

Add the checked attribute to pre-select a checkbox: <input type="checkbox" checked>.

Radio buttons let users select exactly one option from a group. All radio buttons in the same group share the same name.

LISTING 13.8 — Radio buttons for single selection

<label><input type="radio" name="size" value="small"> Small</label>
<label><input type="radio" name="size" value="medium" checked> Medium</label>
<label><input type="radio" name="size" value="large"> Large</label>

Selecting any radio button in the group automatically deselects the others. Unlike checkboxes, once a user selects a radio button they can't deselect it — they can only choose a different option in the group.

The range input creates a slider for selecting a number within a range.

LISTING 13.9 — Range slider with a live value display

<label for="volume">Volume: <output id="volume-display">50</output></label>
<input type="range" id="volume" name="volume"
       min="0" max="100" value="50" step="1">

The slider itself doesn't display the current number — you need to add that with the <output> element or JavaScript to wire them together.

The color input opens a native color picker.

LISTING 13.10 — Color picker input

<label for="theme-color">Favorite Color</label>
<input type="color" id="theme-color" name="theme-color" value="#3498db">

The value must be a six-digit hex color. The browser ignores named colors like "red".

The <select> Element

When you have a predefined list of options — more than four or five — a drop-down list is often more practical than a group of radio buttons. Use the <select> element with <option> children.

LISTING 13.11 — A basic select drop-down

<label for="country">Country</label>
<select id="country" name="country">
  <option value="">-- Select a country --</option>
  <option value="us">United States</option>
  <option value="ca">Canada</option>
  <option value="uk">United Kingdom</option>
  <option value="au">Australia</option>
</select>

The first <option> has an empty value and serves as a prompt. When the form is submitted, the value of the selected <option> is sent using the name of the <select>.

To organize a long list, use <optgroup> to create labeled section headers:

LISTING 13.12 — Grouping options with optgroup

<select name="country">
  <optgroup label="North America">
    <option value="us">United States</option>
    <option value="ca">Canada</option>
    <option value="mx">Mexico</option>
  </optgroup>
  <optgroup label="Europe">
    <option value="uk">United Kingdom</option>
    <option value="de">Germany</option>
    <option value="fr">France</option>
  </optgroup>
</select>

The label attribute on <optgroup> appears as a non-selectable header in the list.

The <textarea> Element

For longer, multi-line text input, use <textarea> instead of a text <input>.

LISTING 13.13 — A textarea for multi-line input

<label for="message">Your Message</label>
<textarea id="message" name="message" rows="5"
          placeholder="Write your message here..."></textarea>

Unlike <input>, <textarea> is a container element with a closing tag. Any text placed between the opening and closing tags becomes the default value.

The rows attribute sets the initial visible height. Users can resize the textarea by dragging the handle in the bottom-right corner. You can control resizing behavior with CSS:

  • resize: none — prevents resizing completely
  • resize: vertical — allows only vertical resizing (usually the best option)
  • resize: horizontal — allows only horizontal resizing
  • resize: both — the default behavior

NOTE: The cols attribute is largely irrelevant today because CSS is used to control width. Setting a CSS width on a textarea overrides cols. The rows attribute is still useful as a way to set an initial height.

The <datalist> Element

A <datalist> provides autocomplete suggestions for a text input while still allowing the user to type anything they want. It's a hybrid between a free-text input and a <select>.

LISTING 13.14 — A text input with datalist suggestions

<label for="browser">Favorite Browser</label>
<input type="text" id="browser" name="browser" list="browser-options"
       placeholder="Start typing...">

<datalist id="browser-options">
  <option value="Chrome">
  <option value="Firefox">
  <option value="Safari">
  <option value="Edge">
  <option value="Opera">
</datalist>

The list attribute on the <input> must match the id of the <datalist>. As the user types, the browser shows matching suggestions. Unlike <select>, the user can ignore the suggestions and type anything they want.

Grouping Fields with <fieldset> and <legend>

When a form has several logically related groups of inputs — like separate sections for billing and shipping addresses, or a group of radio buttons — wrap each group in a <fieldset> and label it with a <legend>.

LISTING 13.15 — Fieldset and legend for a radio button group

<fieldset>
  <legend>Contact Preferences</legend>
  <label><input type="radio" name="contact" value="email"> Email</label>
  <label><input type="radio" name="contact" value="phone"> Phone</label>
  <label><input type="radio" name="contact" value="text"> Text Message</label>
</fieldset>

The <legend> appears at the top of the <fieldset> border by default. Screen readers announce the legend text when a user moves focus into the group — so a user hears "Contact Preferences, Email, radio button" instead of just "Email, radio button." This context is critical for understanding what the group is about.

<fieldset> is especially important for radio button groups because it explicitly communicates to assistive technology that these controls are related choices from a single group.

Special Inputs: Hidden Fields, Output, and Buttons

Hidden inputs store data that needs to be submitted with the form but shouldn't be visible or editable by the user — like a session token, a user ID, or a pre-populated reference value.

LISTING 13.16 — Hidden inputs

<input type="hidden" name="user_id" value="1042">
<input type="hidden" name="source" value="homepage-cta">

CAUTION: Hidden inputs are not secure. Anyone can view the page source and see their values. Never use hidden inputs for sensitive security decisions. They're appropriate for convenience reference values — not secrets.

<button> vs. <input type="submit"> — Both are valid ways to add a submit button, but <button> is preferred for modern HTML because it's more flexible:

LISTING 13.17 — Button element vs. input type submit

<!-- Old approach -->
<input type="submit" value="Send Message">

<!-- Modern approach: button can contain HTML -->
<button type="submit">Send Message</button>

<!-- Button with an icon (not possible with input submit) -->
<button type="submit">
  <img src="send-icon.svg" alt=""> Send Message
</button>

The <button> element can contain HTML markup inside it — icons, styled text, images — while <input type="submit"> can only show plain text. Always include type="submit" explicitly on a <button> inside a form. Without it, the button defaults to type="submit" anyway, but being explicit makes your intent clear.

The <output> element displays a calculated result, often connected to a range slider or number input via JavaScript:

LISTING 13.18 — Output element for displaying calculated values

<label for="price">Price: $<output id="price-display">10</output></label>
<input type="range" id="price" min="10" max="100" step="5" value="10">

HTML5 Validation Attributes

Before HTML5, all form validation required JavaScript. Now browsers handle common validation automatically. Add these attributes to your inputs and the browser will check them before allowing the form to submit — no code required.

LISTING 13.19 — Common validation attributes

<!-- Required: field must not be empty -->
<input type="text" name="username" required>

<!-- Text length limits -->
<input type="text" name="username" minlength="3" maxlength="20">

<!-- Numeric range -->
<input type="number" name="age" min="18" max="120">

<!-- Custom pattern using a regular expression -->
<input type="tel" name="phone"
       pattern="[0-9]{3}-[0-9]{3}-[0-9]{4}"
       placeholder="555-555-5555">

<!-- Step: controls valid increments -->
<input type="number" name="quantity" min="0" step="5">

Here's a complete reference of the validation attributes:

  • required — The field must have a value before the form can submit.
  • minlength / maxlength — Minimum and maximum number of characters (for text inputs).
  • min / max — Minimum and maximum values (for number, date, time, and range inputs).
  • pattern — A regular expression the value must match. The browser won't explain the pattern in its error tooltip by default — add a title attribute with a plain-English description.
  • step — Controls valid increments for number and range inputs. step="5" means only multiples of 5 are valid.

LISTING 13.20 — Pattern attribute with a title tooltip for context

<label for="zip">ZIP Code</label>
<input type="text" id="zip" name="zip"
       pattern="[0-9]{5}(-[0-9]{4})?"
       title="Enter a 5-digit ZIP code, optionally followed by a dash and 4 digits"
       placeholder="12345 or 12345-6789"
       required>

The autocomplete attribute helps browsers and password managers fill in fields automatically. For common fields, use standardized token values that browsers and autofill tools recognize:

LISTING 13.21 — Autocomplete tokens for common fields

<input type="text"  name="fname" autocomplete="given-name">
<input type="text"  name="lname" autocomplete="family-name">
<input type="email" name="email" autocomplete="email">
<input type="tel"   name="phone" autocomplete="tel">

Common autocomplete values: "name", "given-name", "family-name", "email", "tel", "street-address", "postal-code", "country", "current-password", "new-password". See the full list in MDN's autocomplete documentation.

Form Pseudo-Classes

CSS provides a set of pseudo-classes to style form controls based on their current state. You can use these to give users visual feedback about whether a field's value is valid, whether it's required, or whether they've started typing.

LISTING 13.22 — Styling form fields based on their state

/* Red border on invalid fields */
input:invalid {
  border-color: red;
}

/* Green border on valid fields */
input:valid {
  border-color: green;
}

/* Visual indicator for required fields */
input:required {
  border-left: 3px solid orange;
}

/* Style when placeholder text is visible (field is empty) */
input:placeholder-shown {
  background-color: #fffde7;
}

Here's a reference of the form-related pseudo-classes:

  • :valid — Matches when the input's value satisfies all validation constraints (or the input has no constraints).
  • :invalid — Matches when the value fails any validation constraint.
  • :required — Matches inputs that have the required attribute.
  • :optional — Matches inputs that do not have the required attribute.
  • :placeholder-shown — Matches when placeholder text is currently visible, meaning the field is empty.
  • :focus — Matches when an element has keyboard focus.

CAUTION: The :invalid pseudo-class matches immediately on page load for required or pattern-constrained fields — before the user has even touched the form. This causes jarring red error styles to appear the moment the page loads. To avoid this, use the newer :user-invalid pseudo-class (supported in all modern browsers), which only applies after the user has interacted with the field. As a fallback, combine :not(:placeholder-shown) with :invalid to suppress errors on untouched fields.

LISTING 13.23 — Using :user-invalid for better UX

/* Only show invalid style after user has interacted with the field */
input:user-invalid {
  border-color: red;
  background-color: #fff0f0;
}

/* Fallback for older browsers */
input:not(:placeholder-shown):invalid {
  border-color: red;
}

Form Accessibility

Built-in HTML validation catches formatting errors, but good form accessibility goes further. Here are the essential techniques.

Providing Help Text with aria-describedby

When a field has additional instructions beyond the label, link them to the input using aria-describedby:

LISTING 13.24 — Linking help text with aria-describedby

<label for="username">Username</label>
<input type="text" id="username" name="username"
       minlength="3" maxlength="20"
       aria-describedby="username-hint">
<p id="username-hint">3–20 characters. Letters, numbers, and underscores only.</p>

The value of aria-describedby is the id of the help text element. When the user focuses the input, a screen reader announces both the label ("Username") and the description ("3–20 characters, letters, numbers, and underscores only").

Announcing Errors with aria-invalid

When a form is submitted with errors, set aria-invalid="true" on each failing field. Screen readers announce this as part of the field's description, letting users know which fields need attention:

LISTING 13.25 — Using aria-invalid and role="alert" for validation errors

<label for="email">Email Address</label>
<input type="email" id="email" name="email"
       aria-invalid="true"
       aria-describedby="email-error">
<p id="email-error" role="alert">Please enter a valid email address.</p>

Using role="alert" on the error message causes screen readers to announce it immediately when it appears in the DOM, even if the user's focus hasn't moved there yet.

Custom Validation with novalidate

If you want to handle all validation yourself with JavaScript — for full control over error messages, styling, and logic — add the novalidate attribute to the <form> element:

LISTING 13.26 — Disabling browser validation for custom JavaScript validation

<form action="/submit" method="post" novalidate>
  <!-- Form controls -->
</form>

With novalidate, the browser stops showing its built-in validation popups. However, :valid and :invalid pseudo-classes still work, and you can access input.validity properties in JavaScript to check each constraint programmatically.

Tab Order and Keyboard Navigation

Every form control must be reachable and usable with the keyboard alone. By default, interactive elements — inputs, selects, textareas, and buttons — are included in the tab order in the order they appear in the HTML.

Keep your visual order and DOM order in sync. If CSS moves a form field to a different visual position than it appears in the HTML, keyboard users experience the fields in HTML order — which may feel backward or disorienting.

Avoid setting positive tabindex values (like tabindex="3") on form controls. Overriding the natural tab order almost always makes the experience worse. If you need to make a custom element focusable, use tabindex="0" to add it to the natural tab order, or tabindex="-1" to make it programmatically focusable without adding it to the tab sequence.

[IMG - DevTools accessibility panel showing form focus order highlighted]

NOTE: To test your form's keyboard experience, set your mouse aside and tab through the form yourself. Every field should be reachable, every label should be clearly readable, and pressing Enter should submit the form. If anything is confusing without a mouse, fix it before shipping.

Summary

In this lesson, you learned how to build complete, accessible HTML forms. Here are the key takeaways:

  • The <form> element uses action to specify where data is sent and method to specify how. Use post for sensitive data and form submissions that change server data; use get for searches and bookmarkable requests.
  • Every input must have an associated <label>, either paired with for/id or by wrapping. Labels are required for accessibility and improve usability for everyone.
  • HTML provides specific input types for text, email, password, URL, phone, search, numbers, dates, times, checkboxes, radio buttons, ranges, and colors — and the browser handles keyboard optimization and basic validation for each type automatically.
  • The <select> element creates drop-down lists. Use <optgroup> to organize long option lists into labeled groups.
  • <textarea> is for multi-line text input. <datalist> adds autocomplete suggestions to a text input while still allowing free-text entry.
  • Group related controls with <fieldset> and <legend>, especially radio button groups, so screen readers can announce the group context.
  • HTML5 validation attributes — required, minlength, maxlength, min, max, pattern, step — let you enforce rules before the form is submitted, without writing JavaScript.
  • CSS pseudo-classes — :valid, :invalid, :required, :placeholder-shown, :user-invalid — let you style fields based on their current state.
  • Accessibility requires: labels paired with every input, help text linked via aria-describedby, errors marked with aria-invalid, dynamic error messages announced with role="alert", and a logical keyboard tab order.

Q & A

Q: Do I always need a <form> element? What if I'm just showing a search box that filters a list on the page with JavaScript?

A: No — if you're handling the interaction entirely client-side with JavaScript (like filtering a visible list as the user types), you don't need a <form> element. A standalone <input> with a <label> is fine in that case. The <form> element is primarily meaningful when data needs to travel to a server.

Q: My required fields show red error borders the moment the page loads, before the user has done anything. How do I fix that?

A: This happens because :invalid fires immediately on page load for any required field that's empty — and an empty required field is technically invalid. The fix is to use :user-invalid instead of :invalid. This newer pseudo-class only applies after the user has actually interacted with the field. For older browser support, combine :not(:placeholder-shown) with :invalid to prevent error styles from appearing on untouched fields.

Q: What's the difference between the name and id attributes on an input? Do I need both?

A: They serve different purposes. id is a unique identifier used to link a <label> (via for) and for targeting with CSS or JavaScript. name is what the form uses when submitting data — it becomes the key in the key-value pair sent to the server (e.g., email=jane@example.com). You need name on any field whose data you want submitted. You need id to connect a <label> using the for/id approach. In most cases you'll have both.

Q: When should I use radio buttons instead of a <select> drop-down?

A: A useful rule of thumb: use radio buttons when you have five or fewer options and want them all visible at once. Use <select> when you have six or more options, or when screen space is limited. Radio buttons make all choices visible simultaneously, which is better for clarity. Drop-down lists save space but require an extra click or tap to see the full list of options.

Q: Is HTML5 validation enough, or do I still need JavaScript validation too?

A: HTML5 validation handles the common cases well — required fields, email format, number ranges, and custom patterns. But you'll still need JavaScript for cross-field logic: checking that two password fields match, verifying that a date is in the future, requiring at least one of several optional fields to be filled out, or when you need full control over the styling and content of error messages. In production, always validate on the server as well — browser-side validation, whether HTML5 or JavaScript, can always be bypassed by a determined user.
