Forms That Remember

Validation, localStorage, and Data That Survives a Refresh

What You’ll Learn in This Lesson:

  • ►How to read every form input type from JavaScript
  • ►How to validate fields and show inline error messages
  • ►localStorage — the browser’s built-in storage that persists across refreshes
  • ►Why localStorage only stores strings — and how JSON.stringify() and JSON.parse() solve that
  • ►How to load saved data when the page starts
  • ►How to use try/catch to handle corrupted data gracefully

You have built forms that look great and validate in CSS. You have written event handlers that read input values and conditionals that catch empty fields. But every time the page refreshes, everything disappears. Users have to start over. Data that should persist — saved entries, theme preferences, a half-filled form — is gone.

This lesson fixes that. You will learn how to read every type of form input cleanly, validate it with helpful error messages, and save the result to localStorage — the browser’s built-in key-value store that survives page refreshes, tab closes, and even browser restarts. The defining moment is when you submit a form, refresh the page, and see your data still there.

Reading Form Values

Different input types store their values in different ways. You need to know how to read each one:

// Text, email, password, textarea — all return a string
const name     = document.querySelector('#name').value;
const email    = document.querySelector('#email').value;

// Number inputs — always convert with Number()
const age      = Number(document.querySelector('#age').value);

// Select (dropdown) — returns the selected option's value attribute
const country  = document.querySelector('#country').value;

// Checkbox — returns true or false, not a string
const agreed   = document.querySelector('#terms').checked;

// Radio buttons — find the one that is checked
const size     = document.querySelector('input[name="size"]:checked').value;

// Range slider — returns a string number; convert it
const volume   = Number(document.querySelector('#volume').value);

NOTE
.value always returns a string — even for <input type="number">. Wrap any input you need as a number in Number() immediately. The same applies to values read back from localStorage.

Validation with if Checks

HTML5 validation attributes (required, minlength, pattern) are a great first line of defense, but JavaScript validation gives you control over the exact messages users see and when they appear. The pattern is:

  1. Set an isValid flag to true
  2. Check each field — flip the flag to false and show an error if it fails
  3. Only save/submit if the flag is still true when all checks are done
form.addEventListener('submit', function(event) {
  event.preventDefault();
  let isValid = true;

  // Check name
  if (nameInput.value.trim() === '') {
    showError(nameError, 'Name is required');
    isValid = false;
  } else {
    clearError(nameError);
  }

  // Check email
  if (!emailInput.value.includes('@')) {
    showError(emailError, 'Enter a valid email address');
    isValid = false;
  } else {
    clearError(emailError);
  }

  if (isValid) {
    saveEntry();
  }
});

value.trim() removes leading and trailing whitespace before checking. A field containing only spaces should fail the “not empty” test.

Showing and Hiding Error Messages

Each field gets a companion element for its error message. By default it is hidden. When a check fails, you set the text and make it visible using classList:

// HTML: <span class="error-msg hidden" id="name-error"></span>

function showError(element, message) {
  element.textContent = message;
  element.classList.remove('hidden');
}

function clearError(element) {
  element.textContent = '';
  element.classList.add('hidden');
}

With CSS transitions, the message can animate in smoothly — the same technique used to toggle visibility with classList:

.error-msg {
  color: #e74c3c;
  font-size: 0.85rem;
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.2s ease;
}

.error-msg:not(.hidden) {
  max-height: 40px;
}

localStorage — Data That Survives a Refresh

localStorage is a key-value store built into every browser. Unlike regular JavaScript variables, data saved to localStorage persists after the page refreshes, after the tab closes, and after the browser restarts. It belongs to the current domain and is never sent to a server.

Three methods cover almost everything you need:

// Save a value
localStorage.setItem('username', 'Maria');

// Read a value back
const name = localStorage.getItem('username'); // 'Maria'

// Delete a value
localStorage.removeItem('username');

// Delete everything in this domain's storage
localStorage.clear();

You can inspect localStorage live in DevTools: open the Application tab, then expand Local Storage in the sidebar. Every key-value pair you set will appear there instantly.

[IMG - Screenshot: DevTools Application tab showing Local Storage panel with key-value pairs visible]

NOTE
localStorage.getItem() returns null — not undefined, not an empty string — when the key does not exist. Always check for this or use a fallback value when reading.

Storing Objects: JSON.stringify and JSON.parse

localStorage can only store strings. If you try to save an array or object directly, it gets silently converted to "[object Object]" — useless. The solution is JSON:

const entries = [
  { name: 'Maria', score: 92 },
  { name: 'Luca',  score: 88 }
];

// Convert to string before saving
localStorage.setItem('entries', JSON.stringify(entries));

// Convert back from string when loading
const saved = JSON.parse(localStorage.getItem('entries'));

JSON.stringify() converts a JavaScript array or object into a JSON string — a text representation that looks like '[{"name":"Maria","score":92},{"name":"Luca","score":88}]'. This string can be stored in localStorage.

JSON.parse() does the reverse: converts a JSON string back into a usable JavaScript array or object.

[IMG - Diagram: JavaScript array → JSON.stringify() → JSON string → localStorage → JSON.parse() → JavaScript array again]

The Null Guard

The first time a user visits your page, the localStorage key you are trying to read does not exist yet. getItem() returns null, and passing null to JSON.parse() returns null, not an empty array. Calling .length or .forEach() on null crashes your script.

The null guard pattern uses || to provide a fallback:

const entries = JSON.parse(localStorage.getItem('entries')) || [];

Read this as: “Parse the stored value. If the result is null (or any other falsy value), use an empty array instead.” One line handles both the first visit and all subsequent visits.

Loading Saved Data on Page Start

To restore data when the page loads, run your load-and-render code at the top of your script — outside any event listener, so it runs immediately when the page is ready:

// At the top of your script — runs on every page load
const entries = JSON.parse(localStorage.getItem('entries')) || [];
renderEntries(entries);

// Save a new entry
function saveEntry() {
  const entry = {
    name:  nameInput.value.trim(),
    email: emailInput.value.trim(),
    date:  new Date().toLocaleDateString()
  };

  entries.push(entry);
  localStorage.setItem('entries', JSON.stringify(entries));
  renderEntries(entries);
}

The variable entries starts with whatever was in storage (or an empty array). New submissions are pushed in, the array is saved back to storage, and the render function is called. Refresh the page — the top-of-script load restores everything.

try/catch for Corrupted Data

JSON.parse() throws an error if it receives invalid JSON. This can happen if someone manually edits the browser’s storage, if an older version of your app saved data in a different format, or if the storage was partially written due to a browser crash. A try/catch block handles this gracefully so the page never shows an error to the user:

let entries = [];

try {
  entries = JSON.parse(localStorage.getItem('entries')) || [];
} catch {
  // Storage was corrupted — start fresh
  entries = [];
  localStorage.removeItem('entries');
}

The code inside try runs normally. If JSON.parse() throws, execution jumps immediately to the catch block. The page starts with a clean slate rather than crashing.

NOTE
This is the same try/catch pattern from the JavaScript Foundations lesson. Any code that can fail for reasons outside your control — parsing data from external sources, reading storage written by older code — is a good candidate for wrapping in try/catch.

Putting It All Together

The complete pattern combines all of the above: read a form, validate it, build an object, push it to an array, save the array to localStorage, and re-render the list. On every page load, restore the array from storage and render immediately:

// 1. Load saved data on startup
let entries = [];
try {
  entries = JSON.parse(localStorage.getItem('entries')) || [];
} catch {
  entries = [];
}
renderEntries(entries);

// 2. Handle new submission
form.addEventListener('submit', function(event) {
  event.preventDefault();

  // Validate
  if (nameInput.value.trim() === '') {
    showError(nameError, 'Name is required');
    return;
  }
  clearError(nameError);

  // Build the entry object
  const entry = {
    name: nameInput.value.trim(),
    date: new Date().toLocaleDateString()
  };

  // Save and re-render
  entries.push(entry);
  localStorage.setItem('entries', JSON.stringify(entries));
  renderEntries(entries);
  form.reset();
});

// 3. Render function
function renderEntries(list) {
  if (list.length === 0) {
    container.innerHTML = '<p class="empty">No entries yet.</p>';
    return;
  }
  container.innerHTML = list.map(e => `
    <div class="card">
      <strong>${e.name}</strong>
      <span>${e.date}</span>
    </div>
  `).join('');
}

Best Practices

  • ►Always convert .value to the type you need. Every input returns a string. Wrap number inputs in Number(). Trim text inputs with .trim() before checking or saving.
  • ►Always use the null guard. JSON.parse(localStorage.getItem('key')) || [] handles both first-time visitors and returning users with one line. Never assume the key exists.
  • ►Wrap JSON.parse in try/catch for data you didn’t write. If there is any chance the stored data could be corrupted or in a format you don’t expect, protect against it.
  • ►Clear errors at the start of each submission. Run clearError() on all fields before checking them. Otherwise, an error from a previous failed submission stays visible even after the user fixes it.
  • ►Reset the form after a successful save. form.reset() clears all fields and returns them to their default values. Without this, old values linger in the inputs after submission.
  • ►Inspect localStorage in DevTools. Application → Local Storage. You can see every key, edit values, and delete entries to test your null guard and error handling during development.

Summary

Every form input type has its own way of reporting its value: .value for text-based inputs (always a string), .checked for checkboxes, and a :checked query for radio buttons. Validation with an isValid flag lets you check all fields at once and show specific inline error messages. localStorage stores key-value pairs that survive page refreshes and browser restarts. Because it only holds strings, arrays and objects must be converted with JSON.stringify() before saving and restored with JSON.parse() when loading. The null guard pattern (|| []) handles first-time visitors cleanly, and try/catch handles corrupted data without crashing the page. Loading saved data at the top of the script — before any event listeners — is what makes the “survives a refresh” moment work.

The next step is replacing hardcoded data arrays with data fetched from a real server. fetch() makes an HTTP request and returns the response as JSON — the same format you just learned to parse. Your renderCards() function will not change at all. Only the source of the data changes.
